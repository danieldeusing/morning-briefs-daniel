Claude Code · 2.1.152 bringt Auto-Fix und Skill-Kontrolle [1]

Nach der Vorwoche, in der Claude Code bei 2.1.150 (23.05.) festhing, ist heute 2.1.152 da — und es ist der substanziellste Build der Reihe seit Tagen. Drei Dinge lohnen den Blick.[1]

/code-review --fix schreibt zurück

Bisher war /code-review ein Read-only-Gutachten. Jetzt schreibt --fix die Befunde direkt in den Working-Tree — Reuse-, Vereinfachungs- und Effizienz-Vorschläge werden angewandt statt nur aufgelistet. /simplify ruft intern dasselbe --fix auf. Das ist der Schritt vom „der Reviewer sagt dir was" zum „der Reviewer macht's". Try Für Daniels Workflow der relevanteste Punkt: einen Review-Pass abnicken und die mechanischen Änderungen gleich im Diff sehen, statt sie von Hand nachzutragen.

code.claude.com/docs/changelog

Skills dürfen jetzt Werkzeug wegnehmen

Skills und Slash-Commands können in ihrem Frontmatter disallowed-tools setzen — Tools, die dem Modell entzogen werden, solange der Skill aktiv ist. Dazu kommt /reload-skills (Skill-Verzeichnisse neu scannen ohne Session-Neustart) und ein SessionStart-Hook, der reloadSkills: true zurückgeben kann, damit ein vom Hook installierter Skill in derselben Session sofort verfügbar ist.

disallowed-tools im Skill: Stell dir einen Werkzeugkasten vor, aus dem du für eine bestimmte Aufgabe absichtlich die scharfe Säge rausnimmst, damit niemand sie versehentlich greift. Genauso kann ein Skill jetzt sagen „solange ich laufe, kein Bash" — enger Rahmen = weniger Wege, etwas kaputtzumachen.
code.claude.com/docs/changelog

Kleinkram mit Folgen

Drei Zeilen, die im Alltag zählen: ein neuer MessageDisplay-Hook kann Assistant-Text beim Anzeigen umschreiben oder verstecken; bei nicht gefundenem Primär-Modell schaltet Claude Code auf --fallback-model für den Rest der Session, statt jede Anfrage scheitern zu lassen; und „Auto mode no longer requires opt-in consent" — der autonome Modus läuft ohne vorgeschaltete Zustimmungsfrage. Letzteres ist eine Bequemlichkeit mit Beigeschmack: ein Klick weniger, aber auch eine Bestätigung weniger vor unbeaufsichtigten Läufen.

Codex · 0.134.0 von Alpha auf stable [2]

Die Alpha ist durch

Was seit dem 22.05. als 0.134.0-alpha.1…3 reifte, ist gestern (26.05., 19:13 UTC) als stable 0.134.0 veröffentlicht — hoch vom Stable 0.133.0 (21.05.). Die nennenswerten Neuerungen: durchsuchbare lokale Gesprächs-Historie (case-insensitive), --profile ist jetzt der primäre Profil-Selektor (Legacy-Profil-Configs werden abgelehnt), per-Server-MCP-Env-Targeting plus OAuth für streamable HTTP, und read-only MCP-Tools laufen jetzt parallel statt seriell.[2]

Die letzten beiden Punkte sind das eigentliche Signal: Codex schraubt diese Woche sichtbar am MCP-Unterbau — und passt zum MCP-Spec-Schub auf der anderen Seite (siehe rechts). Read-only-Tools parallel zu fahren ist genau die Art Latenz-Gewinn, die agentische Läufe spürbar schneller macht, ohne das Risiko paralleler Schreibzugriffe.

MCP · Governance-Schub im Spec-Repo [7][8]

Während die CLIs neue Builds shippten, lief im modelcontextprotocol-Repo der aktivste Tag seit Wochen — und der Schwerpunkt ist auffällig: Sicherheit, Auditierbarkeit, Governance.

Gemergt: Codex/ChatGPT lesen Server-Instructions

PR #2790 (gemergt 26.05.) hält fest, dass Codex und ChatGPT jetzt MCP-Server-Instructions unterstützen — der vom Server mitgelieferte Bedienungs-Hinweistext wird vom Client tatsächlich gelesen und ins Modell gespeist. Jüngster Merge im Repo ist inzwischen #2800 (27.05., Release-Workflow-Kram); die veröffentlichte Spec bleibt 2025-11-25, das RC-Ziel der 28.07. (PR #2750 gemergt).[7]

MCP-Server-Instructions: Wie der Beipackzettel an einem Gerät — der Server sagt dem KI-Client von sich aus „so benutzt du mich richtig" (welche Tools wofür, welche Reihenfolge). Bisher ignorierten manche Clients den Zettel; Codex und ChatGPT lesen ihn jetzt.
MCP PR #2790

Drei neue SEPs an einem Tag

Am 26.05. landeten gleich drei Proposals: SEP „Tool Risk Metadata" (#2793) erweitert die ToolAnnotations um maschinenlesbare Risiko-Angaben, damit ein Client automatisch entscheiden kann, was ohne Rückfrage laufen darf und was nicht; #2792 bringt Per-Request-Sprachverhandlung (i18n); #2778 „Adding constraints". Die SEP-2787-Attestation (signierte Tool-Call-Belege, EU-AI-Act Art. 12) bleibt offen, bekam aber am 26.05. mit #2789 erste Test-Vektoren. Alle offen, alle Extensions-Track — aber die Richtung ist klar.[8]

Tool Risk Metadata: Ein Etikett an jedem Werkzeug, das sagt, wie gefährlich es ist — „nur lesen" (grün) vs. „löscht Dateien" (rot). Der Agent kann dann die grünen ohne Nachfragen nutzen und nur bei den roten anhalten. Heute trifft diese Entscheidung oft eine handgepflegte Liste; das Etikett macht sie maschinenlesbar.

Community · die Kehrseite der Agenten

Simon Willison dokumentiert diese Woche eine Prompt-Injection-Exfiltration über Microsoft Copilot „Cowork" — ein Angreifer schmuggelt Anweisungen in Daten, die der Agent verarbeitet, und schleust darüber Informationen hinaus. Dazu Daniel Stenbergs „The Pressure" über die Flut KI-generierter, stimmloser Security-Reports, die curl-Maintainer überrollt. Zwei Geschichten, eine Pointe: je autonomer die Agenten, desto wichtiger werden Audit-Spuren und Risiko-Etiketten — genau das, woran MCP gerade schraubt.

Prompt-Injection: Wie ein gefälschter Notizzettel im Posteingang eines allzu gehorsamen Assistenten — im Text, den der Agent verarbeiten soll, stecken versteckte Befehle („schick alles an diese Adresse"). Der Agent kann harmlose Daten nicht von eingeschmuggelten Anweisungen unterscheiden und führt sie aus.
