Opus 4.8 · die Anbieter-Zahlen und das Verified-Loch [1][2][3]

Gestern um 18:00 UTC ist Claude Opus 4.8 live gegangen — überall gleichzeitig (claude.ai, API, Claude Code, Bedrock, Vertex, Foundry). Anthropic positioniert es als „more effective collaborator": schärferes Urteil, mehr Ehrlichkeit über den eigenen Fortschritt, weniger Schummeln im Code. Der Marketing-Satz, der hängenbleibt: Opus 4.8 ist „viermal seltener als sein Vorgänger" bereit, eigene Code-Fehler ohne Kommentar durchzuwinken.[1]

„Code-Fehler durchwinken": Wenn ein Agent eine Funktion schreibt, die einen Test bricht, und das Modell trotzdem sagt „done", obwohl es den Test-Fail gesehen hat — das ist „durchwinken". Bei Opus 4.7 passierte das laut Anthropic in X% der Fälle, bei 4.8 in X/4%. Die Zahlenbasis veröffentlicht Anthropic nicht; das ist ein Verhalten, das man nur in eigenen Logs verifizieren kann.

Die Benchmark-Tabelle, die Anthropic im Begleit-Blog mitschickt, sieht respektabel aus: Agentic Coding 64,3 → 69,2 %, Multidisciplinary Reasoning with Tools 54,7 → 57,9 %, Agentic Computer Use 82,8 → 83,4 %, Wissen-Arbeit (Elo) 1.753 → 1.890, Agentic Financial Analysis 51,5 → 53,9 %.[2] Was nicht auf dieser Liste steht und auch nicht auf dem prüfbaren SWE-Bench-Verified: ein Score für Opus 4.8. Wir haben die Verified-JSON direkt aus dem Repo gezogen — kein Eintrag für Opus 4.8, kein Eintrag für Mythos, kein Eintrag für Sonnet 4.6 oder 4.7. Die jüngste sichtbare Partition (bash-only) datiert vom 17.02.2026.[3]

Dynamic Workflows · Claude schreibt sein eigenes Skript [5][6]

Die unscheinbar wirkende, aber strukturell wichtigere Hälfte der gestrigen Anthropic-Auslieferung: Dynamic Workflows, Research Preview, freigeschaltet ab Claude Code v2.1.154. Statt dass Claude turn-by-turn Subagenten spawnt und jedes Zwischenergebnis sein Kontextfenster auffrisst, bekommt jeder „Workflow"-Auftrag jetzt ein eigenständiges JavaScript-Skript, das Claude für deinen Task selbst schreibt — und das anschließend ein dedizierter Runtime im Hintergrund ausführt, während die Konversation responsive bleibt.[5]

Dynamic Workflow: Stell dir vor, du bittest einen Koch um ein Drei-Gänge-Menü. Bisher hat er für jedes Gericht parallel überlegt und den ganzen Plan im Kopf gehalten — irgendwann fiel ihm Gang eins wieder raus, weil sein Arbeitsspeicher voll war. Jetzt schreibt er dir vorher einen Zettel: „Vorspeise zwei Köche kalt, Hauptgang vier Köche heiß, Dessert nacheinander" — der Zettel ist das Skript, die Küche ist der Runtime. Der Koch (Claude) kann jederzeit in die Konversation zurückkehren, ohne dass die Arbeit unterbrochen wird; das Skript läuft separat. Und der Zettel ist code — du kannst ihn lesen, abspeichern, beim nächsten Mal mit einem Befehl wiederausführen.

Die harten Limits stehen in der offiziellen Doku: Bis zu 16 gleichzeitig laufende Agenten pro Lauf (weniger auf schwacher Hardware), 1.000 Agenten total pro Lauf (das verhindert Runaway-Loops), kein User-Input mitten im Lauf (nur Permission-Prompts können pausieren). Verfügbar auf Pro, Max, Team, Enterprise, plus Anthropic API, Bedrock, Vertex und Microsoft Foundry. Auf Pro muss man in /config die Dynamic-Workflows-Zeile umlegen.[6]

Drei Wege, einen Workflow zu starten

  • Im Prompt das Wort „workflow" stehen lassen — Claude Code hebt es hervor und schreibt das Skript anstatt turn-by-turn zu arbeiten. (alt+w deaktiviert die Auslösung pro Prompt.)
  • Bundled Workflow: /deep-research <frage> ist der mitgelieferte — fanout Web-Searches auf eine Frage, Quellen werden gegenseitig gecheckt, Ergebnis ist ein zitierter Report, in dem Claims, die das Cross-Check nicht überlebt haben, bereits rausgefiltert sind.
  • /effort ultracode: koppelt xhigh-Reasoning mit automatischer Workflow-Orchestrierung — Claude entscheidet selbst, wann ein Task einen Workflow braucht. Höherer Token-Verbrauch, längere Latenz pro Request; Default-Modus bleibt manuelle Auslösung.

Workflow als versionierbarer Befehl + Bun-Show-Case

Wenn ein Lauf das Gewünschte tut, drückt man im /workflows-View s — das Skript wandert als slash-Command ins Repo (.claude/workflows/ oder ~/.claude/workflows/) und ist beim nächsten Branch als /<name> reproduzierbar. Doku-Vergleich ist sauber: Subagent = Worker, Skill = Instructions, Workflow = das Skript, das beides orchestriert. Anthropics eigener Show-Case: ein Workflow-getriebener Zig→Rust-Port des Bun-Runtimes mit „rund 750.000 Zeilen Rust", 99,8 % Tests grün, in 11 Tagen.[5] Anbieter-Use-Case, kein unabhängiger Benchmark — Bun ist auch keine repräsentative Mittelstands-Code-Base. Zeigt das Maximum, nicht den Montag-Erwartungswert. Try Niedriger Einstieg: /deep-research auf einer Recherche-Frage; eigene Workflows: das Skript anschauen, bevor man es per s committet.

Zweite Geschichte, die strukturell wichtigere Hälfte der gestrigen Auslieferung: Dynamic Workflows in Claude Code. Statt dass Claude einen Subagenten nach dem anderen spawnt und jedes Zwischenergebnis sein Kontextfenster auffrisst, schreibt Claude jetzt selbst ein JavaScript-Skript für den Auftrag, und ein separater Runtime führt das Skript im Hintergrund aus. Bis zu sechzehn Agenten parallel, bis zu tausend total pro Lauf. Verfügbar als Research Preview ab Claude Code Version zwei-Punkt-eins-Punkt-hundertvierundfünfzig, auf Pro, Max, Team und Enterprise. Drei Auslöser: das Wort „workflow" in den Prompt schreiben, oder den eingebauten Befehl Slash-Deep-Research benutzen, oder Slash-Effort-Ultracode setzen — dann entscheidet Claude selbst, wann ein Task einen Workflow braucht. Wenn der Lauf gut ist, drückt man im Workflow-View die Taste S, und das Skript wird als versionierbarer Slash-Befehl ins Repo gespeichert. Für Daniels Solo-Stack ist das der direkteste Hebel, den die gestrige Auslieferung anbietet — niedrige Einstiegshürde, hohe Substanz.
