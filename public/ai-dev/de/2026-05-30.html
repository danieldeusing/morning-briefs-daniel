Gemini 3.5 Flash · der seltene Fall einer Anbieter-Zahl, die hält [1][2][3]

Auf der Google I/O am 19.05. — also vor elf Tagen, in der Anthropic-Opus-4.8-Woche untergegangen — ist Gemini 3.5 Flash in GA gegangen und ist seitdem Googles Default-Modell. Google positioniert es als agentisches Coding-Modell: ein Flash-Modell (das günstige, schnelle Tier), das laut Google das vorherige 3.1 Pro auf den Coding-Benchmarks schlägt — und das, gemessen an Frontier-Modellen allgemein, „viermal schneller" in Output-Tokens pro Sekunde und „oft zu weniger als der halben Kosten" läuft (Googles Vergleich gilt anderen Frontier-Modellen generell, nicht speziell 3.1 Pro).[1]

Die Zahlen, die Google zeigt: Terminal-Bench 2.1 76,2 %, MCP Atlas 83,6 %, GDPval-AA 1.656 Elo, CharXiv-Reasoning 84,2 %. Gegen 3.1 Pro sind das spürbare Sprünge (Terminal-Bench 70,3 → 76,2; MCP Atlas 78,2 → 83,6; Finance Agent v2 43,0 → 57,9).[2]

Terminal-Bench / MCP Atlas / GDPval: Drei Tests, die agentisches Können messen, nicht nur Code-Schnipsel. Terminal-Bench setzt das Modell in eine echte Shell und lässt es eine Aufgabe über mehrere Befehle lösen — wie ein Praktikant, der sich allein durch dein Terminal arbeitet. MCP Atlas prüft, ob das Modell externe Werkzeuge über das Model-Context-Protocol korrekt anspricht (MCP = ein USB-Standard, über den ein Modell Tools wie GitHub oder eine Datenbank ansteuert). GDPval-AA ist ein Elo-Score über realistische Wissensarbeit — wie eine Schach-Wertung, nur für Berufsaufgaben.

Normalerweise ist hier der Punkt, an dem dieser Brief „Anbieter-Zahl, nicht gespiegelt" schreibt. Diesmal ist es anders: Artificial Analysis hat genau diese Werte unabhängig nachgemessen — Terminal-Bench 2.1, MCP Atlas, GDPval-AA, Humanity's Last Exam, MRCR 128k — und kommt auf Null Abweichung zur Google-Zahl. Das ist selten und verdient, festgehalten zu werden.[3]

Aber Bestätigte Einzel-Benchmarks sind nicht dasselbe wie Spitzenklasse. Derselbe Artificial-Analysis-Intelligence-Index v4.0 setzt Gemini 3.5 Flash auf Rang 8 von 148 — hinter GPT-5.5 und Claude Opus 4.7 (jeweils ~3 Punkte). Und auf SWE-Bench Pro liegt Flash bei 55,1 %, 9,2 Punkte hinter Opus 4.7 (64,3 %) — diese Pro-Zahlen sind allerdings via digitalapplied.com zusammengetragen, kein eigener Artificial-Analysis-Lauf. Bei Langkontext bricht MRCR von 77,3 % (128k) auf 26,6 % bei 1 M Tokens ein — Nadel-im-Heuhaufen über volle Kontexte ist eine echte Schwäche.[3]

Einordnung für Daniels Toolbox: Gemini 3.5 Flash ist nicht das beste Coding-Modell — es ist das beste Preis-Leistungs-Modell für agentische Schleifen, in denen Latenz und Token-Kosten dominieren (Test-Loops, Refactor-Schwärme, CI-Bots). Schwere Architektur-Aufgaben bleiben Opus/GPT-5.5; für hundert kleine parallele Agenten-Tasks ist Flash der rationalere Default. Try Über Google AI Studio / Gemini API, „high"-Thinking-Level (die Benchmark-Variante).

Managed Agents · der Agent als API-Primitive [4][5]

Die für Entwickler interessantere I/O-Ankündigung ist nicht das Modell, sondern wie Google es verpackt: Managed Agents in der Gemini-API. Ein einziger API-Call startet einen Agenten, der in einer isolierten, kurzlebigen Linux-Sandbox denkt, plant, Werkzeuge ruft, Code ausführt, Dateien verwaltet und das Web liest — du musst keine Runtime, keinen Container und keine Tool-Schleife selbst betreiben.[4]

Managed Agent: Bisher war ein „Agent" etwas, das du selbst zusammenbaust — Modell-Call, Tool-Aufruf, Ergebnis zurück ins Modell, Schleife, dazu eine Sandbox, in der der Code laufen darf. Google nimmt dir die ganze Klempnerei ab: du sagt per API „mach das", und im Hintergrund läuft eine fertige Linux-Maschine, die der Agent steuert. Wie der Unterschied zwischen einem eigenen Server (du wartest Strom, Kühlung, OS) und einer Cloud-Funktion (du lädst nur den Code hoch). Du beschreibst was, nicht wie betrieben.

Das Verhalten definierst du in Markdown — AGENTS.md und SKILL.md — und registrierst diese Dateien als Managed Agent. Der zugrunde liegende Harness (die Steuer-Schleife um das Modell) ist Googles Antigravity-Agent, gebaut auf Gemini 3.5 Flash, erreichbar über die Interactions API. Jede Interaktion erzeugt oder erbt eine Umgebung, die du im Folge-Call wiederverwendest — Dateien und Zustand bleiben erhalten, du kannst die Session fortsetzen.[5]

Bekanntes Muster Wer das AGENTS.md/SKILL.md-Schema kennt, sieht sofort: das ist dieselbe Konvention wie bei OpenAI Codex und Claude Code. Drei Anbieter konvergieren auf Markdown-definierte Agenten + gehostete Sandbox. Das ist gut für Daniel — Skills, die er für Claude Code schreibt, sind konzeptionell portierbar. Google nennt allerdings weder Preise noch Limits im Ankündigungs-Post; vor einem Produktiv-Einsatz muss man die Kostenseite separat prüfen.

  2. blog.google — „define everything in markdown files like AGENTS.md and SKILL.md and register them as a managed agent ... powered by the new Antigravity agent, built on Gemini 3.5 Flash ... via the Interactions API"; keine Preise/Limits genannt, 19.05.2026

Antigravity 2.0 · von der IDE-Hilfe zur Agenten-Plattform [6]

Antigravity war bisher Googles agentische IDE-Schicht. Mit Antigravity 2.0 (ebenfalls I/O, 19.05.) wird daraus eine eigenständige Desktop-App, die komplett um Agenten-Orchestrierung herum gebaut ist — plus eine Antigravity CLI und ein Antigravity SDK. Der Clou: CLI, SDK und Desktop-App teilen sich denselben Harness, der auch Googles eigene Produkte antreibt. Default-Modell überall: Gemini 3.5 Flash.[6]

Der strategische Punkt ist derselbe wie bei Anthropics Dynamic Workflows letzte Woche und Cognitions „orchestration layer"-These: die Orchestrierungs-Schicht über dem Modell wird zur eigenen Produktkategorie — Antigravity, Dynamic Workflows, der Codex-Service, je eine pro Anbieter. Für Daniel: die Frage ist immer weniger „welches Modell" und mehr „welcher Harness orchestriert es, und kann ich meine Skills mitnehmen".

Ehrliche Einordnung

2.0 ist Enterprise-orientiert (geplante/asynchrone Tasks, Cloud-Run-/Firebase-Integration). Für einen Solo-Entwickler ist die CLI das relevante Stück — gleicher Harness wie die Desktop-App, ohne GUI-Overhead. Ob sie an Claude Code / Codex CLI / Grok Build heranreicht, ist drei Wochen nach Launch nicht durch unabhängige Reviews belegt — Watchlist, kein heutiges Urteil.

xAI Grok Build · der vierte Spieler, ehrlich vermessen [7][8]

Während Google die Bühne hatte, ist xAI still in dieselbe Kategorie eingestiegen: Grok Build, ein terminal-natives Coding-Agent-CLI auf dem eigens dafür gebauten Modell grok-build-0.1. Start als Early Beta am 15.05. (nur SuperGrok Heavy, $300/Mo), seit dem 25.05. offen für alle SuperGrok ($30) und X Premium+ ($40) — ein steiler Preis-Sturz, der Reichweite kaufen soll.[7]

Technisch interessant: bis zu 8 parallele Subagenten, jeder in einem eigenen Git-Worktree auf einem separaten Branch; ein Orchestrator führt die Ergebnisse danach zusammen. Davor ein Plan→Search→Build-Ablauf — bei allem über einen Einzeiler schreibt Grok Build erst einen strukturierten Plan, den du Schritt für Schritt freigibst oder umschreibst. 256K Kontext, MCP nativ (GitHub, Linear, Slack ohne Reconfiguration). Zur API-Preisseite kursieren widersprüchliche Drittquellen-Zahlen — buildfastwithai.com nennt $0,20/1 M Input + $1,50/1 M Output, andere $1,00/$2,00 (der $0,20-Wert dürfte der gecachte Input-Preis sein); die offizielle xAI-Preisseite war nicht erreichbar, daher unbestätigt.[8]

Git-Worktree pro Subagent: Ein Worktree ist ein zweiter Arbeitsordner aus demselben Repo, auf einem eigenen Branch. Wenn acht Agenten gleichzeitig am selben Code arbeiten, würden sie sich sonst gegenseitig die Dateien überschreiben — wie acht Köche an einem Herd. Mit je einem Worktree bekommt jeder seine eigene Kochnische; erst am Ende wird zusammengeführt. Sauber und konfliktarm — das ist der technisch eleganteste Teil von Grok Build.

Die Benchmark-Zahl — und ihre Grenze

xAI gibt für grok-build-0.1 SWE-Bench Verified 70,8 % an — eine Anbieter-Zahl, nicht auf dem offiziellen Verified-Board (siehe Spalte rechts). Zur Einordnung: auf dem realen Board stehen GPT-5.5 bei 88,7 % und Opus 4.7 bei 87,6 % — grok-build-0.1 spielt also, wenn die eigene Zahl überhaupt hält, eine Klasse darunter. Unabhängige Reviews waren gemischt: stark bei Parallelität und Privacy, zurück bei Benchmark-Leistung und Ökosystem-Tiefe. Für Daniel: nicht heute wechseln — Grok Build ist hinter X-Premium+ eingesperrt und ökosystem-mager; die parallele-Worktree-Idee ist das, was man im Auge behält.
