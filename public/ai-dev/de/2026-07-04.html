Sonnet 5 ist der neue Default — der Trick ist der Preis, nicht der Score

Anthropic hat am 30.06.2026 Claude Sonnet 5 veröffentlicht und es sofort zum Standardmodell für Free- und Pro-Pläne gemacht — und, praktisch relevanter für uns: zum Default-Modell jeder neuen Claude-Code-Session.[1] Wer heute Claude Code startet, sitzt auf Sonnet 5, nicht mehr auf Opus. Das ist die eigentliche Nachricht: nicht „ein neues Modell", sondern „das Modell, das die meisten von uns per Default fahren, hat gewechselt".

Die Positionierung ist ehrlich formuliert und deshalb interessant: Sonnet 5 kann Pläne machen, Browser und Terminals steuern und autonom lange Loops fahren — auf einem Niveau, das bis vor Kurzem Opus-Klasse-Modellen vorbehalten war. Dazu ein natives 1-Mio-Token-Kontextfenster und „Adaptive Thinking" per Default an.[1][2] Aber: im Benchmark bleibt es unter Opus 4.8.

Anbieter-Zahlen, skeptisch lesen Auf Anthropics eigenem agentischem Coding-Benchmark (identifiziert als SWE-bench Pro) erreicht Sonnet 5 63,2 % — gegen Opus 4.8 mit 69,2 % und Sonnet 4.6 mit 58,1 %.[2][3] Das sind Anbieter-reportete Werte, nicht unabhängig gemessen: das primäre swebench.com-Verified-Board lädt seine Tabelle per JavaScript und gab beim Fetch keine Roh-Zahlen zurück, daher nur Aggregator-Cross-Checks. Ein neues Modell steht selten am Erscheinungstag schon unabhängig gelistet — behandle die 63,2 % als Herstellerangabe, bis ein neutrales Board nachzieht.

Der Sprung von Sonnet 4.6 (58,1 %) auf 5 (63,2 %) ist real, aber der Punkt ist nicht der Score. Der Punkt ist die Kosten-Leistungs-Kurve: Einführungspreis 2 $/M Input und 10 $/M Output bis 31.08.2026, danach 3 $/M und 15 $/M.[1][3] Opus 4.8 kostet 5 $/25 $. Für agentische Harnesses, die pro Task hunderttausende Token durchpumpen, ist das der eigentliche Hebel: fast Opus-Fähigkeit zum halben bis Fünftel-Preis.

Agentisches Coding / agentic Harness: Statt einer einzelnen Frage („schreib mir diese Funktion") bekommt das Modell ein Ziel und die Werkzeuge — Terminal, Dateisystem, Browser — und arbeitet in einer Schleife selbstständig: lesen, Code schreiben, Tests laufen lassen, Fehler sehen, nachbessern. Wie ein Praktikant, dem du eine Aufgabe und Schlüssel zum Werkzeugschrank gibst, statt jede Handbewegung zu diktieren. Genau diese Schleifen fressen viele Token — deshalb entscheidet der Preis pro Token, ob agentisches Arbeiten sich rechnet.

try this Wer Claude Code für lange autonome Tasks nutzt und bisher auf Opus geblieben ist: einen Sonnet-5-Durchlauf gegen deinen üblichen Opus-Durchlauf testen. Bei Refactorings und Test-Loops ist der Qualitätsunterschied oft kleiner als der Preisunterschied — und wo Sonnet 5 hakt, kannst du für den harten Teil gezielt auf Opus hochschalten.

SWE-bench Pro heute: was man kaufen kann — und was nicht

Kurze Einordnung fürs Bench-Bild, alle Zahlen als Anbieter-/Aggregator-Stand vom 04.07. gelesen: Auf SWE-bench Pro ist Opus 4.8 mit 69,2 % der höchste Score, den man aktuell tatsächlich kaufen kann.[3][4] Auf SWE-bench Verified liegt Opus 4.8 bei 88,6 %.[4] Sonnet 5 (63,2 %) reiht sich als das preislich attraktivere Arbeitspferd darunter ein.

Der höchste je berichtete SWE-bench-Pro-Wert (~80 %) stammt von einem export-beschränkten Modell, das schlicht nicht verfügbar ist — die Governance- und Exportkontroll-Geschichte dazu läuft im ai-usecases-Brief, hier nur der Verweis. Für die Praxis gilt: das beste kaufbare Modell ist Opus 4.8, das beste Preis-Leistungs-Modell wird gerade Sonnet 5.

Der „AI-Paradox": schneller tippen ≠ schneller liefern

GitLabs 2026 AI-Accountability-Report ist die ehrlichste Zahl der Woche, und sie widerspricht dem Marketing sauber. 78 % der Entwickler sagen, sie schreiben Code schneller mit KI; 79 % berichten von höherer individueller Produktivität. Und trotzdem: die Software-Auslieferung hat sich insgesamt nicht beschleunigt.[5]

Der Grund ist mechanisch und für jeden, der schon mal einen PR-Stau erlebt hat, sofort einleuchtend: Der Engpass ist nur umgezogen. 85 % stimmen zu, dass KI den Flaschenhals vom Schreiben zum Reviewen und Validieren verschoben hat.[6] Man produziert mehr Code, schneller — aber der Code muss immer noch gelesen, verstanden, getestet und verantwortet werden, und das macht weiterhin ein Mensch in Menschen-Geschwindigkeit.

KI schreibt schneller → mehr Code-Volumen → Review-Queue staut → Delivery gleich schnell

Dazu die Governance-Lücke: 80 % sagen, ihre Organisation habe KI-Coding-Tools schneller eingeführt als Richtlinien, um sie zu steuern.[7] Die Top-Barrieren beim Kontrollieren: 43 % können KI- nicht von Menschen-Code unterscheiden, 40 % kämpfen mit fragmentierten Toolchains, 39 % haben keine Herkunfts-Verfolgung für Code.[7]

Was das für dich heißt Wenn du twiceD-Projekte planst: der Produktivitäts-Gewinn aus KI-Coding ist echt, aber er landet nicht 1:1 in schnellerer Auslieferung, solange Review und Test der Engpass bleiben. Der Hebel liegt zunehmend nicht im Tippen, sondern im Review-Durchsatz — automatisierte Checks, kleinere PRs, klare Code-Herkunft. Genau da, wo die 43-%-Barriere sitzt.

Claude Code: geliebt, aber nicht dominant — der Unterschied zählt

Es kursiert die Erzählung, Claude Code habe die Konkurrenz überholt. Die Zahlen sagen etwas Präziseres — und Widersprüchlicheres. Bei Senior-Engineers ist Claude Code das „most loved"-Tool: 46 % nennen es, gegen 19 % für Cursor und 9 % für GitHub Copilot (Pragmatic-Engineer-Umfrage, Feb. 2026).[8]

Aber „geliebt" ist nicht „genutzt". Bei der Arbeitsplatz-Adoption liegt Copilot mit 29 % vorne; Cursor und Claude Code teilen sich mit je 18 % Platz zwei (JetBrains AI Pulse Survey, Jan. 2026).[9] Von einem sauberen Überhol-Vorgang kann keine Rede sein — Copilot bleibt breit installiert, JetBrains AI und andere teilen den Rest. Was stimmt: die Wachstumskurve. Claude Code sprang von 3 % (Apr. 2025) auf 18 % (Jan. 2026) — versechsfacht in neun Monaten.[9]

„most loved" vs. Adoption: Zwei verschiedene Fragen. „Most loved" heißt: von denen, die es kennen, sind viele begeistert (ein Präferenz-Signal, oft bei Early Adopters). „Adoption" heißt: wie viele es real im Job einsetzen (ein Markt-Signal, träger). Ein Tool kann von Kennern geliebt und trotzdem eine Minderheit im Alltag sein — genau das ist hier der Fall. Wie ein Restaurant mit Kult-Status, vor dem trotzdem nur eine kurze Schlange steht.

Für Daniels Toolbox heißt das: die Begeisterung der Senioren ist ein Frühindikator, kein Beweis. Wer Copilot- oder Cursor-Workflows fährt, verpasst nichts Existenzielles — aber der Trend zeigt, wohin die erfahrensten Leute ziehen.

MCP & Tooling: Claude Code stopft eine Supply-Chain-Lücke

Zwischen dem 29. und 30.06. hat Claude Code zwei Versionen ausgeliefert — v2.1.196 und v2.1.197 — und die sind keine Wartungs-Releases.[10] Neben dem Sonnet-5-Default-Flip schließt v2.1.197 eine MCP-Autorisierungs-Lücke: committete .claude/settings.json-Dateien konnten in nicht-vertrauenswürdigen Workspaces MCP-Server ohne Zustimmung starten.

MCP (Model Context Protocol): Ein Standard, über den ein KI-Coding-Tool externe Werkzeuge und Datenquellen anspricht — eine Datenbank, ein Ticketsystem, ein Browser. Ein „MCP-Server" ist so ein andockbares Werkzeug. Wie ein USB-Standard für KI-Tools: einmal die Buchse definiert, kann jeder Anbieter ein Gerät anstecken. Das Problem: wenn ein fremdes Repo die Steckerliste mitbringt, steckt es dir womöglich Geräte an, die du nicht sehen wolltest.

Genau das war die Lücke. Jetzt gilt: Claude Code spawnt MCP-Server nicht mehr automatisch aus committeten Configs in nicht-vertrauenswürdigen Workspaces. Stattdessen zeigen sie ein ⏸ Pending approval und warten auf ausdrückliche Bestätigung.[10] Ein bösartiges Repo kann also nicht mehr allein durchs Klonen einen MCP-Server bei dir hochziehen. Wer fremde Repos mit Claude Code öffnet, sollte updaten.

Dazu neu: Org-/Rollen-Default-Modelle. Enterprises können jetzt „standardmäßig Sonnet 5 oder Haiku 4.5" setzen, während Opus für explizite Auswahl erreichbar bleibt — Kostensteuerung ohne harte Sperre, neben dem bestehenden Allowlist-Mechanismus (enforceAvailableModels).[10]

Zahlen-Hygiene: drei kursierende Statistiken, geradegerückt

Drei Zahlen laufen gerade durch KI-Coding-Threads, die man nicht ungeprüft weitertragen sollte:

  • JPMorgan: die belastbare Zahl ist ~65.000 Engineers/Technologen, deren KI-Tool-Nutzung an Performance-Reviews geknüpft ist (Stand Ende März 2026), plus eine vom CIO offengelegte Produktivitäts-Steigerung von 10–20 % bei Nutzern des internen Coding-Assistenten.[13] Die kursierenden „60k Devs / +30 % Velocity" lassen sich in dieser Präzision nicht belegen — nutze 65k und 10–20 %.
  • GitHub „51 % KI-generierter Code": die 51 % für Anfang 2026 stammen nur aus einem Sekundär-Aggregator, nicht aus GitHubs eigenem Bericht — als sekundär-belegt behandeln, nicht als GitHub-Primärzahl zitieren.[14]
  • Hired.com „+340 % KI-Tool-Stellen": unbestätigt ließ sich über keine Primärquelle bestätigen und wird hier fallengelassen. Verwandte, belegte Trends existieren (KI-Skill-Postings +130–144 % YoY laut Indeed Hiring Lab) — aber die exakten 340 % nicht.

Warum das hier steht: dieser Brief ist Daniels Entscheidungs-Grundlage, keine Hype-Sammlung. Eine unbelegte Zahl weggelassen ist besser als eine falsche weitergereicht.

Es ist Samstag, der vierte Juli. Ein ruhiges Wochenende, aber die Woche hatte es in sich — und zwei Geschichten stehen sich dabei fast schön gegenüber.

Die erste: Anthropic hat am Dienstag Claude Sonnet fünf veröffentlicht und es sofort zum Standardmodell gemacht — für die Gratis- und Pro-Pläne, und, wichtiger für uns Entwickler, für jede neue Claude-Code-Sitzung. Wer heute Claude Code startet, sitzt nicht mehr auf Opus, sondern auf Sonnet fünf. Es kann Werkzeuge steuern und lange autonom arbeiten, fast auf altem Opus-Niveau, mit einem Kontextfenster von einer Million Token.

Aber im Benchmark bleibt es darunter. Auf Anthropics eigenem Coding-Test erreicht Sonnet fünf drei­und­sechzig Komma zwei Prozent, gegen Opus vier Punkt acht mit neun­und­sechzig Komma zwei. Das sind Herstellerangaben, kein unabhängig gemessener Wert — das primäre Leaderboard ließ sich nicht sauber auslesen. Der eigentliche Reiz ist nicht der Score, sondern der Preis: zwei Dollar rein, zehn Dollar raus pro Million Token, bis Ende August. Für agentische Schleifen, die Unmengen Token verbrauchen, ist das der Hebel.

Die zweite Geschichte erdet die erste. GitLab hat Zahlen vorgelegt, die dem Hype widersprechen. Acht­und­siebzig Prozent der Entwickler sagen, sie tippen schneller mit KI. Aber die Software wird trotzdem nicht schneller ausgeliefert. Der Engpass ist nur umgezogen, vom Schreiben zum Reviewen: fünf­und­achtzig Prozent bestätigen genau diese Verschiebung. Man produziert mehr Code, schneller — nur muss ihn weiter ein Mensch lesen, testen und verantworten.

Dazu die Governance-Lücke: achtzig Prozent sagen, ihre Firma habe die Tools schneller eingeführt als die Regeln dafür. Fast die Hälfte kann KI-Code nicht mehr von Menschen-Code unterscheiden. Die Lehre für die eigene Planung: der Produktivitätsgewinn ist echt, landet aber nicht eins zu eins in schnellerer Auslieferung, solange Review der Flaschenhals bleibt.

Und ein dritter Punkt, weil er gern falsch erzählt wird: Claude Code hat die Konkurrenz nicht sauber überholt. Bei erfahrenen Entwicklern ist es das beliebteste Werkzeug, sechs­und­vierzig Prozent nennen es. Aber bei der tatsächlichen Nutzung am Arbeitsplatz liegt es nur bei achtzehn Prozent, gleichauf mit Cursor und hinter GitHub Copilot mit neun­und­zwanzig. Geliebt ist nicht dasselbe wie genutzt. Was stimmt, ist die Wachstumskurve — von drei Prozent auf achtzehn in neun Monaten.

Unter der Oberfläche hat Claude Code außerdem eine Sicherheitslücke geschlossen: fremde Repositories konnten über committete Konfigurationsdateien heimlich Werkzeug-Server starten. Das geht jetzt nicht mehr ohne ausdrückliche Zustimmung. Wer fremde Projekte öffnet, sollte updaten. Bei den anderen Tools bewegt sich Cursor am meisten, mit Version zwei Punkt drei und einer iOS-Beta; Cline liefert täglich; nur Aider wirkt weiter eingeschlafen.

Nächste Woche schauen wir, ob ein neutrales Benchmark-Board Sonnet fünf endlich unabhängig listet — und ob bei den Open-Weights-Modellen wieder Bewegung kommt, die diese Woche auffällig still waren. Bis dahin: schönes Wochenende.
