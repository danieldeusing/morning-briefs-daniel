Cursor · Composer 2.5 — the first in-house frontier coding model [1]

Cursor's drop yesterday (Mon May 18) is the most-discussed tool move of the week — and not because a feature shipped, but because the model itself is in-house. Composer 2 (October) was the first try; Composer 2.5 openly positions Anysphere as a frontier provider, not just an IDE layer on top of someone else's models.

What Anysphere claims

Composer 2.5 reaches Opus-4.7 and GPT-5.5 level at "about 1/10 the cost", per the Cursor changelog — SWE-Bench Multilingual 79.8%, CursorBench v3.1 63.2%. Pricing: Standard $0.50/M in, $2.50/M out; Fast (default) $3.00/M in, $15.00/M out. Double usage limits the first week.

Skeptical reading: CursorBench v3.1 is Cursor's own benchmark. SWE-Bench Multilingual is a contamination-prone variant of the Verified set — the Pro suite would be the honest comparison, where the actual frontier models sit 15-25 points lower. The "1/10 cost" claim is plausible on Anysphere's own infrastructure but conditional: the Standard tier is cheap, the Fast tier is roughly on par with Opus pricing.

cursor.com/changelog
Try this Run Composer 2.5 in Standard tier against a manageable feature task — two hours with the tool you normally use, two hours with Composer 2.5. The honest question isn't "does it beat Opus on bench X", it's "where does it fall apart in your code with your patterns". Double limits the first week — costs you nothing but time.

OpenAI · Codex CLI rust-v0.131.0 exits alpha [2]

What's substantial

In the shadow of last week's mobile story, rust-v0.131.0 (May 18) brings the plumbing upgrade: plugin marketplace CLI with version-aware sharing and share-checkout, default-enabled plugin hooks, daemon-managed codex remote-control with enable/disable/status APIs, registry-backed remote environments. In the TUI: @ mentions now search files, directories, plugins and skills in one unified picker; data-driven service-tier slash commands; blended token-usage display; permissions/approval mode in the status line; responsive Markdown tables. Python SDK moved to openai-codex (package name).

New for daily use: codex doctor — a diagnostic command that checks auth, MCP servers, plugin state, sandbox permissions in one sweep. Sounds unspectacular, but it's exactly what used to cost 20 minutes of manual debugging on a broken session. Same-day tag: 0.132.0-alpha.1.

Agent SDK billing split, update on the June 15 cutoff

The announcement from Wed May 14 is firming up. Starting Mon June 15, Claude Agent SDK, claude -p headless mode, Claude Code GitHub Actions, and every third-party harness (Zed, OpenClaw, Conductor, Jean, Cline, OpenHands, every DIY wrapper) move out of the regular subscription pool into a separate "Agent SDK credit pool". Anthropic hasn't officially quantified the effective price range; community write-ups put it between 12× and 175× uplift per workload.

What this means for Daniel's stack if Claude Code is used natively: nothing — the interactive CLI stays in the subscription pool. What it means if your agent runs through the SDK (Zed integration, Cline with Claude backend, GitHub Actions bot): from June 15 you pay per token instead of flat subscription. Skeptical note: Anthropic might respond to OPEX-reality pressure, or competitors might respond with their own subscription models — this move isn't fully told yet.

Cline · --worktree lands in the CLI

v3.0.6 (May 17) / v3.0.7 (May 18)

Two small but practical releases. v3.0.6 brings the --worktree flag: per task a fresh git worktree is created under ~/.cline/worktrees/, cleanly isolated from the main working tree, composable with --taskId and --continue. Anyone working on three bugs in parallel without trashing their stashes finally has the clean solution instead of three repo clones. v3.0.7 is plumbing — the ChatGPT OAuth model list is synced with the Codex provider catalog, the startup roundtrip skipped. VS Code extension stays on v3.83.0 (May 13).

Anthropic SDK has had --worktree since September, Codex CLI since the mobile wave last week — Cline now closes the gap. Third out of four major harnesses to adopt this workflow convention.

github.com/cline/cline — v3.0.6
git worktree: Imagine you have a wardrobe (the repo) with outfits (branches). Normally you pull one outfit out, wear it, put it back, then pull the next. With worktree you make copies — the same repo in multiple folders, each on a different branch. Three tasks in parallel, without changing clothes.

pypi.org/project/vllm · github.com/vllm-project/vllm
Yank on PyPI: If you "yank" a package it's still installable, but it no longer shows up in a normal pip install — effectively a "don't buy this anymore" sign from the maintainer. Unlike delete (vanishes entirely), yank doesn't break existing lockfiles but does send a clear "broken" signal.
