Self-hosted Sandboxes: the agent now runs with you

At "Code with Claude" in London (May 19–20, Anthropic's first dev conference in Europe), the most important announcement for us arrived — not a model, but an architecture decision. Self-hosted Sandboxes went Public Beta: Orchestration (agent loop, context management, error recovery) stays on Anthropic's side, but tool execution moves into infrastructure you control. "The agent's code, filesystem, and network egress never leave your environment", the official docs state.[1]

Sandbox: an isolated playground where the agent runs its commands (bash, read/write files, execute code) — separate from the rest of your system. Until now, this playground was Anthropic's cloud. Now you can host it on your own server: Anthropic says "do step X", but where X runs is up to you. Like the architect sends the blueprint, but the house sits on your property.

In practice: an Environment Worker on your machine polls a work queue, claims the session, loads the skills, and executes the tool calls locally. The CLI (ant beta:worker poll) and SDK already include the worker; for stronger isolation you start a fresh Docker container per session. Why this matters: for anything that can't cross your network boundary — internal services, compliance, ZDR/HIPAA-eligibility — Managed Agents were off the table before. Now they're not.

Caveat Not everything is ready yet: "Memory is not yet supported with self-hosted sandboxes", the docs say. And Claude Platform on AWS is missing the work-list endpoint. Public Beta means Public Beta.

MCP Tunnels: private server without an open port [1][2]

The sister announcement — MCP Tunnels — is only Research Preview (access by request), but conceptually exactly what many have been waiting for. They solve the ugly problem: your cloud agent needs to reach an MCP server deep in your private network — without exposing it to the public internet.

MCP (Model Context Protocol): a USB standard for AI tools. Instead of bolting on each tool individually, the agent speaks one common protocol, and each "MCP server" (your database, your ticket system, an internal API) registers as a socket. Until now you had to make that socket reachable — so you drill a hole in the firewall. That's exactly the risk.

The tunnel flips the direction: a lightweight gateway you deploy in your network opens one single outbound connection, end-to-end encrypted — "no inbound firewall rules, no public endpoints".[1] Sandbox and tunnel are independently composable: even a session in Anthropic's cloud can reach your private MCP servers through the tunnel. My read: This is the step from "cool demo" to "can go in the enterprise network". For mandates like twiceD's with sovereignty requirements, the sandbox-plus-tunnel combo is the first time Managed Agents become even worth considering.

Antigravity 2.0: the harness becomes a product [3][4]

At Google I/O (May 19), Antigravity 2.0 arrived — Google's answer to agentic coding tools, now as a trio: a standalone Desktop App (orchestrate multiple agents, subagent workflows, scheduled background tasks), a CLI, and an SDK that exposes the same agent harness powering Google's own products. The CLI is "lightweight, high-velocity" — according to TechCrunch it's written in Go and thus snappier; the official I/O note doesn't name the language, so take the "in Go" as secondary.[3]

The engine underneath is Gemini 3.5 Flash, generally available as of May 19. On the coding/agentic benches Google itself published, Flash beats the older 3.1 Pro: Terminal-Bench 2.1 at 76.2 % (3.1 Pro: 70.3 %), MCP Atlas 83.6 %. Remarkable: Artificial Analysis independently measured these numbers — 0.0 delta to Google's May 19 claims.[4] That's rare and argues against benchmark gaming here.

Terminal-Bench: tests whether an agent solves real tasks in a real shell — install packages, fix builds, write scripts — not just isolated code snippets. Closer to your day-to-day than an abstract algorithm test.

Skepticism required regardless: On the heavier SWE-Bench Pro, Flash's lead is tiny — 55.1 % vs. 54.2 % for 3.1 Pro.[5] Which means: on real multi-file GitHub bugs, the jump is more speed (4× faster) than raw ability. For iterative agentic loops where you want many runs per minute, speed could be the win.

Gemini 3.5 Pro: confirmed, but not here

Important for setting expectations: Gemini 3.5 Pro is NOT shipped as of today (June 1). The API changelog only shows gemini-3.5-flash; the Pro model is listed as "coming soon", in limited Vertex preview. On the I/O stage, Sundar Pichai literally said "give us until next month to get it to you" — so June, no fixed date.[6]

Context: Anyone reading "Gemini 3.5 Pro beats Opus" in AI news feeds is reading speculation or Flash numbers mislabeled as Pro. What's expected is a 2-million-token context and a coding flagship to rival Opus 4.8 and GPT-5.5 — but until a dated GA entry appears, it's a promise. Watch, don't plan on it.

Deadline June 18: Gemini CLI is being shut down Heads up

Concrete and calendared: Google is migrating Gemini CLI to the new Antigravity CLI. On June 18, 2026, Gemini CLI and the Gemini Code-Assist IDE extensions will shut down — for AI-Pro, AI-Ultra, and Free-Tier users of Code Assist for Individuals.[7] Paid Standard/Enterprise licenses and GitHub integrations via Google Cloud retain access.

If this affects you If you use Gemini CLI on a consumer tier (e.g., in a script or pipeline), you have until June 18 to migrate to Antigravity CLI. Google recommends moving now; video guides are coming "in the coming weeks". Three weeks' notice — mark it on your calendar.
