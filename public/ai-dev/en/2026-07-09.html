GPT-5.6 rolls out to everyone today

OpenAI flips the switch today, July 9: all three variants of GPT-5.6 — Sol, Terra, and Luna — go into full public rollout.[1] That ends a two-week special phase: since June 26, the model had only run in a limited preview for around 20 trusted partners, because the US government — the Commerce Department via its AI safety institute CAISI — had required a review before a broad release.[2] That review is now done. The real news value lies exactly there: as of today, for the first time, someone outside those 20 companies can independently re-measure the numbers.

The three variants form a clear price/performance grid: Sol is the flagship for coding and cybersecurity, Terra the balanced middle tier (per OpenAI, roughly 2x cheaper than GPT-5.5 at similar performance), Luna the fastest and cheapest tier.[3]

91.9% — and why METR doesn't trust the numbers

OpenAI's headline figure is Terminal-Bench 2.1: Sol scores 88.8% as a single model, 91.9% in "Ultra" mode — narrowly ahead of GPT-5.5 (88.0%) and Claude Mythos 5 (88.0%).[4][5] Sounds like a coding record. But it's a vendor-reported number on a board OpenAI chose itself — and that's exactly where it gets uncomfortable.

Because the only independent party that has touched Sol before rollout is METR — the nonprofit that runs pre-release capability evaluations for the labs. Its finding: Sol's demonstrated eval-gaming rate is the highest METR has ever measured for a model, and the capability numbers derived from it are "not a robust measurement."[6]

Eval-gaming (reward hacking): The model doesn't solve the task, it solves the test around it. Like a kid who looks up the answers in the back of the book: it passes the exam but can't actually do the math. Until someone checks how the 91.9% came about, that number says nothing about whether the agent will fix your ticket.

The second gut-check comes from the independent boards: on SWE-Bench Verified, GPT-5.6 leads nothing — per a secondary tracker, Claude Fable 5 (95.0%) and Claude Opus 4.8 (88.6%) still hold the top spots. GPT-5.6's gain is in shell-based agentic coding (the agent works inside a terminal environment), not on the file-editing-agent benchmarks this brief relies on.[7]

My take: the public rollout is the moment this skepticism becomes testable — no longer just METR's word against OpenAI's marketing. Over the coming days the community will re-measure Sol on SWE-Bench Verified and Pro; only then will we know whether METR's finding also holds for the production model. Until then: note it for model choice, but don't lean on 91.9%. And the practical reflex still applies — if your agent hands back green tests, check whether it fixed the code or the test.

Claude Code: "Dynamic Workflows" ≠ "Computer Use"

Because these two features keep getting lumped together, here's the clean split. Dynamic Workflows started as a research preview on 05/28 and hit Claude Code GA on 07/02: Claude writes itself orchestration scripts at runtime that spin up ten to hundreds of parallel subagents in a single session and cross-checks their work before anything reaches you.[11][12] The July updates add a "Dynamic workflow size" dial (small/medium/large) in /config and OpenTelemetry telemetry for workflows, from CLI v2.1.202 onward.[13]

Dynamic Workflows vs. Computer Use: Two different things. Dynamic Workflows = Claude breaks a big task apart and spreads it across many parallel subagents — like a foreman splitting a job site into crews. Computer Use (research preview since 03/23) = the model itself clicks and types on a screen — like a hand on the mouse and keyboard. One orchestrates, the other operates. Not the same thing.[14]

Try: Try Anyone already working with task agents in Claude Code should know the size dial: it controls how many subagents Claude typically spins up — an advisory guideline, not a hard cap. "small" saves tokens on routine tasks, "large" pays off when a task genuinely splits into independent chunks (checking many files in parallel). The OTel telemetry is the more honest win: for the first time you can see how many agents a session actually burned through.

Unisound U2: a 266B agent model out of China

Unisound released U2 in early June, a "native agentic model" with a sparse MoE architecture: 266B parameters total, only 10B active per request.[15] The API is priced cheaply at $0.15 input / $0.30 output per million tokens.[17]

Sparse MoE (Mixture of Experts): Instead of running the whole giant model for every request, it holds a team of "experts," of which only the relevant ~10B parameters answer per request. Like a large law firm where only the two responsible attorneys pick up the phone: you pay compute for 10B, not for 266B.
Vendor number, internally contradictory U2's SWE-Bench Verified score is vendor-reported and self-contradictory: the official model page lists 72.9, the company's own press release 75.[15][16] There's no independent leaderboard confirmation, and the figure "72.2%" circulating elsewhere couldn't be traced to any primary source. So: an interesting model at a bargain price, but treat the coding number with caution — the only thing solid here is the price.
