Fable 5 is back — with a >99% jailbreak filter

The reason Claude Fable 5 is back on top of the boards this week: Anthropic returned the model to general availability again on July 1. Fable 5 originally launched on June 9, was then suspended from June 12–30 over an export-control question, and has been globally available again since July 1 across Claude Platform, Claude.ai, Claude Code, and Cowork. anthropic.com, Jul 1

The redeploy brings a new safety classifier: the jailbreak technique described in an Amazon report is, per Anthropic, blocked in over 99% of cases. If a request is blocked, the user gets a notice — and the request instead falls to Opus 4.8. anthropic.com, Jul 1

Jailbreak & classifier: A jailbreak is a trick that gets a model to bypass its own safety rules — like a password that opens a locked door anyway. A classifier is a small gatekeeper model that briefly checks each request and decides "let through / block" — the bouncer standing in front of the actual model. Anthropic sharpened the bouncer instead of rebuilding Fable 5 itself.

My take: For Daniel's day-to-day this redeploy changes little — the mechanism is the interesting part. That a blocked request silently falls back to Opus 4.8 means: anyone using Fable 5 in an agent loop should expect that, under the hood, a different and somewhat weaker model occasionally answers. Not a disaster, but a reason not to blindly assume consistent behavior.

Quick follow-up: Grok 4.5 & Muse Spark

Both were covered in full yesterday — here just the fresh caveats that sharpen the skeptical view.

Grok 4.5 (xAI, Jul 8): independent reports now flag that Grok's lead on CursorBench is contaminated — an earlier snapshot of the Cursor codebase had accidentally ended up in the training material. So the apparent coding edge there is partly measuring memorization, not skill. xAI's own marketing ("~Opus-4.7-level at 40% of the price") is unaffected by this, but exactly those coding-adjacent numbers need to be read with caution. developersdigest.tech, Jul 9

Training contamination: When the test tasks (or their solutions) leak into the training material, the model has already seen the answers beforehand. The benchmark then measures recall instead of ability — like an exam where the test-taker already knew the questions.

Muse Spark 1.1 (Meta, Jul 9): the numbers circulating — MCP Atlas 88.1, JobBench 54.7, HLE 62.1 — are all Meta's own; the primary page doesn't even state exact values inline. No independent coding benchmark (neither SWE-Bench Verified nor Pro) has listed Muse Spark so far. Price unchanged: $1.25/$4.25 per million tokens. ai.meta.com · techcrunch.com, Jul 9

My take: Try Both remain worth trying (price, context size), but yesterday's core point holds: as long as no independent coding number exists, these are promises, not results. GPT-5.6 (GA'd July 9), incidentally, still doesn't show up on Verified.

80.3% ≠ 95%: one model, two benchmark numbers

The heart of today's story. Yesterday's message was: SWE-Bench Verified has, per OpenAI itself, been considered contaminated since February 23, which is why the industry is moving to SWE-Bench Pro. Today's uncomfortable continuation — Pro is already being gamed too. Fable 5's headline number on Pro, 80.3%, comes, per Anthropic's own system card, from Anthropic's own scaffolding, not a neutral evaluation harness. Epoch AI's independent Pro evaluation of Fable 5 was still pending as of its last update (June 10). techjacksolutions.com, accessed Jul 11

Scaffolding / harness: A benchmark number never measures the model alone, but the model plus the agent loop around it — how it searches files, runs tests, retries failures. That surrounding apparatus is called scaffolding. If a vendor writes its own scaffolding, trimmed to the test, a higher number can mean "better scaffolding," not "better model." A neutral harness gives every model the same scaffolding — only then does the number actually compare models. Fable 5's 80.3% ran on Anthropic's own.
Verified vs. Pro: Two different exams sharing the family name "SWE-Bench." Verified = 500 human-reviewed GitHub bugs, since leaked into training data (contaminated). Pro = a harder, held-back set built specifically to prevent that leak. The same model scores very differently — Fable 5 gets ~95% on Verified but ~80% on Pro. Putting a Verified number next to a Pro number compares two different exams.

What we do have measured independently is the Verified leaderboard — which vals.ai ran on July 9 with its own, consistent harness. The catch: that's the very board OpenAI itself declared broken. Impressive numbers on a discredited exam.

SWE-Bench VERIFIED (not Pro!) — vals.ai, as of 2026-07-09

And the top spot is shaky: a secondary source cited Claude Mythos 5 (95.5%) in first place on July 8, with Fable 5 behind (95%); vals.ai's direct snapshot a day later has Fable 5 leading at 95.00%, with Mythos 5 missing from the excerpt. Mythos 5 is a real Claude model, not a typo — it trades first place with Fable 5 day to day. The board is volatile, not wrong. vals.ai, updated Jul 9

My take: The move from Verified to Pro was supposed to fix the contamination problem. But Pro has a second hole — vendor-owned scaffolding. Until a neutral third party (Epoch AI) runs Fable 5 on Pro with the same harness as everyone else, the 80.3% is marketing, not measurement. And the 95% on Verified sits on the board OpenAI itself discarded. Same conclusion as yesterday, just with a sharper reason: this week there's still no trustworthy cross-vendor coding number. Use Verified at most as a within-Claude comparison; wait for the independent Pro re-measurement for new models. Practically: decide this week's model choice on price, context size, and your own tests — not on board rankings that reshuffle day to day.
