GLM-5.2: an open model for $1.40 — with a number worth checking

Today's story picks up exactly where yesterday's skepticism left off. Z.ai's GLM-5.2 launched in mid-June (Jun 13); the fully MIT-licensed weights plus FP8 quantization followed on Jun 16–17; context grew to 1 million tokens (from 200K in GLM-5.1). On OpenRouter, the list price is $1.40 input / $4.40 output per million tokens (cached ~$0.26) — currently, with a temporary 70% discount, even as low as $0.42/$1.32. vercel.com, Jun 16 · apidog.com · openrouter.ai, accessed Jul 12

Open weights & MIT license: "Open weights" means: the trained model parameters are publicly downloadable — you can run the model locally or on your own hardware, without an API subscription. The MIT license is the most permissive common software license: use commercially, modify, redistribute — almost no restrictions. In practical terms for Daniel, that means no vendor lock-in, no price-per-token if he hosts it himself — the counterpoint to the closed frontier models.

The advertised coding number: 62.1% on SWE-Bench Pro — the harder, held-back benchmark set that was already a topic yesterday. Z.ai's own documentation frames this as a jump from GLM-5.1 (58.4); a citing secondary source ranks it ahead of GPT-5.5 (58.6) and behind Claude Opus 4.8 (69.2). Important: the 69.2 for Opus is Anthropic's own, self-reported figure from its System Card (Table 8.1.A, independently traceable via vellum.ai and llm-stats) — that is, vendor-reported exactly like GLM-5.2's 62.1 and not from a neutral, shared harness. Both figures are therefore reported by their respective vendor and hence not directly comparable — not a cross-vendor measurement. docs.z.ai · apidog.com · vellum.ai · llm-stats.com, accessed Jul 12

SWE-Bench Pro — vendor-/aggregator-reported, not a neutral harness · as of 2026-07-12

On agent-adjacent benchmarks, GLM-5.2 is closer: Terminal-Bench 2.1 81.0 (Opus 4.8: 85.0) and MCP-Atlas 77.0 — per apidog a "dead heat" with Opus 4.8 (77.8). On LMArena, GLM-5.2 (Max) was added to the Agent, Text, and Code arenas on Jun 16. apidog.com, accessed Jul 12

My take: Try This is the most interesting story of the week for Daniel — not because of the 62.1, but because of the combination: MIT weights, 1M context, a price that clearly undercuts the closed models. For agent loops that churn through a lot of context and where token costs explode, a self-hostable or cheaply hosted model is the real argument — the benchmark ranking is secondary and, as with yesterday, remains a vendor matter until there's an independent Pro re-measurement. In short: try it as a cheap agent worker, don't trust the 62.1.

The one bench number you can trust this week: GPT-5.6 Sol

A rare contrast to the GLM situation: GPT-5.6 Sol (max) leads the Artificial Analysis Coding Agent Index at 80.0 — a new best score, +2.8 ahead of Claude Fable 5, and per OpenAI achieves it with less than half the output tokens, less than half the time, and roughly a third lower cost. This number appears both on Artificial Analysis' own leaderboard and in OpenAI's own post — so it's primary-confirmed, unlike almost everything else this week. artificialanalysis.ai, Jul 9 · x.com/OpenAI

Coding Agent Index ≠ SWE-Bench Pro: Two different tests. Artificial Analysis' Coding Agent Index is an aggregated score across several agent tasks, run by an independent third party with a consistent setup. SWE-Bench Pro is a different bug-fixing set. GPT-5.6 Sol's 80.0 and GLM-5.2's 62.1 sit on two separate scales — they can't be directly compared.

The pricing and tier details (Sol $5/$30, Terra $2.50/$15, Luna $1/$6; Terra free for Free/Go, the higher tiers selectable freely) come from a secondary summary — OpenAI's own page returned a 403 on direct access. The pure coding number (80.0) is the solid part. datacamp.com, accessed Jul 12

My take: That an independent third party measures here with the same harness is exactly what the GLM number (and Fable 5's disputed 80.3 on Pro) lacks. Still, the standard caveat applies: an aggregated index is only as good as its task selection, and "cheaper + faster" per the vendor remains vendor framing. But if one cross-vendor number counts this week, it's this one.

Claude Code quietly switches to Opus 4.8 on cloud providers

The most practically important point for Daniel is buried in a patch release: Claude Code v2.1.207 (Jul 10) makes Auto mode available without opt-in on Bedrock, Vertex AI, and Foundry — the environment variable CLAUDE_CODE_ENABLE_AUTO_MODE is no longer needed there. At the same time, Claude Opus 4.8 becomes the default model on Bedrock, Vertex, and the Claude Platform on AWS. classmethod.jp, Jul 10 · releasebot.io

Auto mode: Instead of pinning a fixed model, Auto mode lets Claude Code itself choose which model and how much "thinking effort" to spend at each step. Like an automatic instead of a manual transmission: convenient, but you give up control over the gear. On the cloud providers, this automation is now on by default.

Anthropic says it plainly: "Because the default for these environments changes when no model is specified, configure the model explicitly if you want to stick with the previous behavior." Translation: anyone running Claude Code against Bedrock/Vertex/Foundry without pinning a model now silently gets Opus 4.8 plus Auto mode — which can noticeably shift behavior, latency, and cost. On the side, the release also fixes a bug where the terminal would freeze on very long lists/tables. classmethod.jp, Jul 10

My take: Not a new feature to celebrate, but a default change worth knowing about. For reproducible agent runs and cost control, it matters more than ever: set model and mode explicitly (--model / config), don't trust the default. Anyone who deliberately wants Opus 4.8 + automation doesn't have to do anything — but "doing nothing" is no longer neutral.

Standing Thread — Open Weights: Qwen and DeepSeek keep up the pressure

GLM-5.2 isn't alone. Qwen3-Coder-Next is an 80B MoE with only 3B active parameters, built for coding agents and local development, and is being touted with SWE-rebench Pass@5 64.6% ("#1 at release"). Together with DeepSeek-V4 (June), this marks, per aggregator round-ups, a noticeable shift in the field of local coding models. aggregator round-ups, accessed Jul 12

Mixture of Experts (MoE): Instead of activating the entire giant model for every request, an MoE has many small specialists and switches on only a few of them per token — for Qwen3-Coder-Next, 3 of 80 billion parameters. Like a large law firm where only the two lawyers responsible for a case work on it, not the whole firm. The result: large-model knowledge at the compute cost of a much smaller model.

My take: Light coverage, because the data is thin — exact release dates for Qwen3-Coder-Next and DeepSeek-V4 could only be sourced as "mid-2026"/"June," with no dated primary changelog. The solid point is the trend, not the individual number: open, agent-capable models are moving closer to the closed frontier, and the MoE approach makes them locally runnable. For a 15-year developer with an Ollama stack, that's the direction worth watching.

CodeQL 2.26.0 learns to find prompt injection in its own code

A coding-AI feature that's easy to miss: CodeQL 2.26.0 (Jul 10) brings the new query js/system-prompt-injection. It detects when unvalidated, user-supplied values flow into an AI model's system prompt — the path through which an attacker can hijack the model's behavior. The recognized AI SDK sinks were also significantly expanded: OpenAI, Anthropic, and Google GenAI, including Sora prompts, OpenAI's Realtime session instructions, Anthropic's legacy completion prompts, and GenAI's cached content/system instructions. (The pure security angle belongs in the software brief — what counts here is that the scanner now understands AI code.) github.blog, Jul 10

Prompt-injection sink: A "sink" in a security scan is the place where data arrives and can cause damage — for SQL injection, that's the database query. Here the sink is an AI model's system prompt: if unvalidated user text flows there, an attacker can overwrite the model's instructions ("ignore all previous rules…"). CodeQL now traces the path from user input to this sink and flags it.

My take: Try Directly relevant for anyone building agents or LLM features: prompt injection is the new injection class, and static analysis now catches it in CI before it goes live. Anyone already running CodeQL gets this query for free — turn it on, run it against the AI SDK code. On the side: Kotlin 2.4.0 support is included too.
