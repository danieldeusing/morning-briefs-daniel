Same model, four times the context [1][2][3]

The experimental setup is so simple you have to wonder why nobody did it this cleanly before. Take two command-line coding agents — Claude Code 2.1.207 and OpenCode 1.17.18 — force both onto the same base model (claude-sonnet-4-5), give both the same task, and count what actually goes over the wire. Result from 07/12/2026: Claude Code filled the context with roughly 33,000 tokens of system prompt, tool schemas, and injected scaffolding before the actual question even arrived. OpenCode got by with about 7,000.[1]

Harness: The model is the engine, the harness is the rest of the car — steering, transmission, brakes, dashboard. Technically, it's the layer that decides what the model even gets to see: which files, which tools, which rules, which memories from the conversation so far, and what gets thrown away once there's too much. Two cars with the same engine can drive completely differently. That's exactly what this comparison measures.

Even more revealing than the starting number is the behavior along the way. In the test run, Claude Code wrote 53,839 cache tokens across five requests, including one complete rewrite of its entire ~43k prefix mid-task. OpenCode wrote 1,003.[2]

Prompt cache and cache prefix: The always-identical start of a conversation — system prompt, tool list, project rules — doesn't need to be reprocessed on every single request. The provider caches it, like a stamp you press again and again instead of rewriting the text each time. The catch: if even one character at the start changes, the whole stamp becomes invalid and has to be re-cut — which costs extra. So rebuilding your prefix mid-run isn't an accident that happens to be expensive; it's a choice you pay for on purpose.

My reading, and I'll defend it: This isn't sloppiness, it's the bill. An agent that reorders half its context mid-task has decided that a tidy context is worth more than a cheap one. Anyone who has watched a long agent run start going in circles around minute forty knows why: the enemy isn't missing knowledge, it's a cluttered context. The 4.7x upfront cost buys tool discipline and cleanup behavior — paid for in tokens, i.e., in hard cash per run.

What the study explicitly does not show: that more overhead is automatically better. It measures tokens, not output quality. OpenCode's leaner approach is plausibly superior for short, clearly scoped tasks — less baggage, less distraction, less cost. The honest sentence is: we now have a solid measurement of the cost side and still none of the benefit side. Whoever wants to flip that has to run both tools against the same set of tasks and count success rates. That's exactly what's missing.

What the leak adds to the picture On 03/31/2026, version 2.1.88 of the @anthropic-ai/claude-code package accidentally shipped with an uncleaned Bun source map — 59.8 MB, from which ~1,900 TypeScript files spanning over 512,000 lines could be reconstructed. Anthropic classified it as a packaging error caused by human mistake, not a security incident.[3] The contents line up exactly with the measurement above: five different context-compaction strategies, a 46,000-line request engine, 29,000 lines for tool definitions alone, 14 tracked cache-break vectors, and three subagent execution models (Fork, Teammate, Worktree).[3] The fact that someone explicitly tracks fourteen ways to break the cache says more about the engineering effort here than any announcement could.

What this means in practice. Swapping models under the same harness changes less than the benchmarks promise — swapping harnesses under the same model changes more. If your agent is performing well or badly, the cause sits in the scaffolding at least as often as in the model. And, inconveniently, this layer is the actual business moat for the closed tools. That's why no model card tells you what matters here — and why a broken source map turned out more informative than two years of marketing.

The harness is also the security boundary [4]

Fitting neatly with the story above, and so it lives here instead of in the roll-call: Claude Code 2.1.214 from 07/18 is, for the most part, a permission-system repair. Among the fixes: a rule like Edit(src/**) that waved through write access to any src/ anywhere in the tree instead of only the one under the working directory; a check bypass in PowerShell 5.1 sessions; Bash checks that now, in doubt, block file-descriptor redirect forms instead of letting them through; and commands over 10,000 characters, which will from now on always prompt rather than run automatically.[4] Today came 2.1.215: /verify and /code-review no longer start on their own, only on request.

The point isn't any single gap. The point is that these decisions — is this command allowed to run without asking? — are made entirely in the harness, not in the model. A better model would not have prevented any of these six gaps. Running coding agents with elevated permissions means choosing a permission engine at least as much as a model. And that engine needs to be kept current: a list of allowed patterns you wrote weeks ago can mean something different today than it did yesterday.

MCP goes stateless in nine days [5]

This is the date to put on your calendar. On 07/28/2026, the new MCP spec goes final — per the project itself, the biggest revision since the protocol launched. The release candidate has been out since 05/21/2026, so there was plenty of lead time; anyone who hasn't looked yet still has a good week left.[5]

MCP (Model Context Protocol): A shared plug-and-socket standard between AI assistants and everything they're supposed to use — files, databases, ticketing systems, APIs. Previously, every tool wrote its own adapter for every data source; MCP is the agreement on what the plug looks like. An MCP server is then the socket for exactly one data source, and any assistant that speaks the standard fits right in.

The central change: the core becomes stateless. The initialize/initialized handshake goes away, as does the Mcp-Session-Id header — "any MCP request can land on any server instance."[5]

Stateless: Until now, an MCP server was like a phone call — you dial in, get a session number, and everything after that hangs off that one line. Anyone who wanted to spread the server across ten machines had to make sure every follow-up request landed back on that exact same machine. Stateless is like a postcard: every request carries everything it needs, and any machine can answer it. For running behind a load balancer, that's the difference between "works" and "doesn't."

Along with this, Roots, Sampling, and Logging are marked deprecated. Good news for anyone running production servers: per the spec, there are at least twelve months between deprecation and earliest possible removal.[5] So nothing breaks on 07/28 — but anyone building fresh against these three capabilities today is building into a dead end.

Why I think this is the practically most important item of the week: it's the only one with a hard date and a concrete action. Self-written MCP servers have quietly become part of many of our toolkits by now, and exactly this kind of side infrastructure is what comes back to bite you three months later. Half an hour: go through your servers, check whether any of the three deprecated areas are in use, and look at your session assumptions. If your server holds state in memory because it relies on the session number, that's where things get interesting.

PyTorch 2.13: FlexAttention comes to the Mac [6]

Try it PyTorch 2.13 shipped on 07/08/2026 and brings FlexAttention to Apple Silicon (MPS) — via hand-written Metal kernels. The number from the release post: on a shape of 1×8×32768×64 with a 256-element sliding window (i.e., 0.8% density), FlexAttention runs in ~35 ms versus ~431 ms for SDPA — a factor of ~12.3. A smaller case with length 8192 and a 64-element window comes in at ~4.15x.[6]

Attention, sparse: "Attention" is the step where every word in the text looks at every other word to decide what matters right now. At 32,000 words, that's over a billion pairs — the most expensive part of the model. But most of the time you don't actually need all of them: a word often only needs to know its few hundred neighbors. That's a sliding window, and then less than one percent of that billion is left. FlexAttention is the mechanism for writing that pattern down as a small rule and having PyTorch generate code that never bothers computing the unnecessary 99%. SDPA computes it anyway.

Why this belongs here and not in the fine print: the factor applies to the sparse case — with dense attention there's nothing to gain for free, because there's simply nothing to skip. But anyone working locally on a Mac with a long context hits exactly this case constantly, and until now that was the point where you had to reach for a remote machine with CUDA. The twelve-fold number isn't the one to expect — the four-fold figure from the smaller example is the more honest ballpark. Even that is enough to justify a weekend experiment.

Grok 4.5: nice numbers, no measurement [7][8]

xAI released Grok 4.5 on 07/08/2026, available among other places directly in Cursor. The API price is $2 per million input tokens and $6 per million output tokens, well below what competitors' top models cost.[7] But the more interesting claim isn't the price — it's the terseness: 15,954 output tokens on average per task against 67,020 for Opus 4.8 in max mode — a factor of 4.2.[8] Anyone who's ever watched a reasoning model hold a page-long internal monologue over a one-line change understands why this would be its own category of progress.

Now for the skepticism All of these numbers — 64.7% on SWE-Bench Pro for Grok 4.5, along with the comparison figures for Opus 4.8 (69.2%) and Claude Fable 5 (80.4%), as well as Terminal-Bench 2.1 (83.3% versus 78.9%) — come from each vendor's own system cards, as the summarizing source itself states.[8] These are self-reported figures, not independent measurements. A cross-check was attempted today: pulling the public SEAL leaderboard for SWE-Bench Pro at Scale AI on 07/19/2026 returned only older entries (including claude-opus-4-6, claude-opus-4-5, claude-4-5-Sonnet). Grok 4.5 was not on it. Until that's added, treat this as a vendor figure.
Why a vendor figure and a measurement are two different things: A benchmark is an exam, but whoever administers the exam has a say in the outcome. The vendor gets to choose its own harness, run multiple attempts, keep the best run, and pick which tasks to report. Like a student who practices at home, grades their own work, and then reports the score. The independent leaderboard is the exam room: same tasks, same conditions, for everyone. That's why "64.7% according to the vendor" isn't a small footnote — it's a different kind of statement entirely.

What we're leaving out today, on purpose: the bar chart. Four models with four percentage figures side by side would make a nice picture — and would visually turn four self-reported numbers into a proper measurement series. A chart says "this was compared." That's exactly not the case here. Better to leave this gap than to publish a graphic that claims more than the data supports.

One addendum: there's a reason vendors are shifting to SWE-Bench Pro at all — the older Verified set is widely considered saturated, with suspected training contamination. That is explicitly a widespread view in the field, not a finding: for that, all I've seen today is secondary commentary, no vetted primary source. Context, not a number.

It's Sunday, the nineteenth of July. The day itself is quiet, but there's a measurement on the table that flips an assumption we've been working with for two years.

We talk about models as if the model were the product. A study from the twelfth of July suggests we've been looking at the wrong half. Someone took two command-line coding tools, Claude Code and OpenCode, and forced both to run the same model underneath. Then gave them the same task, and counted what actually crossed the wire. Claude Code pushed roughly thirty-three thousand tokens of system instructions, tool descriptions, and scaffolding into the context before the actual question even arrived. OpenCode got by with about seven thousand. That's a factor of four point seven.

Quick context on what a harness is: the model is the engine, the harness is the rest of the car. Steering, transmission, brakes. It's the layer that decides what the model even gets to see, and what gets thrown away once there's too much. Two cars with the same engine can drive completely differently.

That fits with the source code that got accidentally published at the end of March. A version of Claude Code shipped with an uncleaned file that let people reconstruct roughly nineteen hundred files and over half a million lines. Anthropic called it a packaging error caused by human mistake, not a security incident. The fact that someone explicitly tracks fourteen such breakage points says more about the engineering effort in this layer than any announcement could.

The second point on the same thread: security also lives in the harness, not the model. Yesterday's release closed several gaps in permission checking at once. A permission rule for one directory was waving through write access to similarly named folders anywhere in the tree, a bypass affected PowerShell sessions, and very long commands no longer run automatically — they now ask first. No better model would have prevented any of these gaps.

The most important date is nine days out. On the twenty-eighth of July, the new Model Context Protocol spec goes final, described by the project as the biggest overhaul since launch. The core becomes stateless: the login handshake and the session number go away, and any request can land on any server instance. Three older capabilities — Roots, Sampling, and Logging — become deprecated going forward, with at least twelve months' notice before possible removal. So nothing breaks, but anyone still building fresh against them today is building into a dead end. Half an hour spent checking your own servers is well worth it.

For anyone working locally on a Mac, there's also good news. PyTorch two point thirteen, released on the eighth of July, brings Flex Attention to Apple Silicon. On very sparse attention patterns, the project measures up to a twelve-fold speedup over the previous method, and about fourfold in a smaller example. The fourfold number is the more honest ballpark, and even that's enough to justify a weekend experiment.

Finally, a word of caution about pretty numbers. Eleven days ago xAI released Grok four point five, cheap on price and supposedly four times terser than the competition. The stated figure of sixty-four point seven percent on a tough coding benchmark, though, comes from the vendor's own system card. A look at the independent public leaderboard this morning didn't show the model on it at all. That's why this issue deliberately skips the bar chart: four self-reported numbers side by side would look like a measurement series, and they wouldn't be one.

On the tools side, it's quiet. Cline shipped a new command-line version overnight, OpenHands hit four releases this week, Cursor sits unchanged at three point eleven. Aider's leaderboard is still frozen since last November, and Continue.dev has been shut down for good following its acquisition by Cursor. In the coming days, the main thing to watch is whether the final MCP spec holds to the twenty-eighth of July, and how fast the big server providers follow suit.
