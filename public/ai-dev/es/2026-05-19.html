Cursor · Composer 2.5 — el primer modelo frontier de coding hecho en casa [1]

El drop de Cursor de ayer (lunes 18/05) es el movimiento de herramienta más comentado de la semana — y no porque haya salido una feature, sino porque el modelo en sí es de la casa. Composer 2 (octubre) fue el primer intento; Composer 2.5 posiciona a Anysphere abiertamente como proveedor frontier, no sólo como capa IDE sobre los modelos de otros.

Lo que Anysphere clama

Composer 2.5 alcanza nivel Opus 4.7 y GPT-5.5 a "cerca de 1/10 del costo", según el changelog de Cursor — SWE-Bench Multilingual 79,8%, CursorBench v3.1 63,2%. Precios: Standard US$ 0,50/M entrada, US$ 2,50/M salida; Fast (default) US$ 3,00/M entrada, US$ 15,00/M salida. Límite de uso doble la primera semana.

Lectura escéptica: CursorBench v3.1 es el benchmark propio de Cursor. SWE-Bench Multilingual es una variante propensa a contaminación del conjunto Verified — el conjunto Pro sería la comparación honesta, y ahí los modelos frontier reales se ubican 15-25 puntos abajo. El claim "1/10 del costo" es plausible en la infraestructura propia de Anysphere, pero condicional: el tier Standard es barato, el tier Fast queda más o menos a la par con el precio de Opus.

cursor.com/changelog
Para probar Corre Composer 2.5 en tier Standard contra una feature task manejable — dos horas con la herramienta que usas normalmente, dos horas con Composer 2.5. La pregunta honesta no es "¿le gana al Opus en el bench X?", es "¿dónde se desbarata en tu código con tus patrones?". Límite doble la primera semana — no te cuesta nada más que tiempo.

OpenAI · Codex CLI rust-v0.131.0 sale del alfa [2]

Lo sustancial

A la sombra de la historia del móvil de la semana pasada, rust-v0.131.0 (18/05) trae la mejora de plumbing: CLI del marketplace de plugins con version-aware sharing y share-checkout, hooks de plugin habilitados por default, codex remote-control gestionado por daemon con APIs de enable/disable/status, entornos remotos backed por registry. En el TUI: menciones con @ ahora buscan archivos, directorios, plugins y skills en un picker unificado; slash-commands de service-tier orientados por datos; visualización mezclada de uso de tokens; modo de permisos/aprobación en la status line; tablas Markdown responsive. El SDK Python se mudó a openai-codex (nombre del paquete).

Nuevo para el día a día: codex doctor — un comando de diagnóstico que revisa auth, servidores MCP, estado de plugins y permisos de sandbox en una sola pasada. Suena poco vistoso, pero es exactamente lo que costaba 20 minutos de debugging manual en una sesión rota. Tag el mismo día: 0.132.0-alpha.1.

GPT-5.3-Codex pasa a ser el default de Copilot para Business/Enterprise (17/05)

En silencio pero estructuralmente importante: GitHub cambió el modelo default de todas las orgs Copilot Business y Enterprise a GPT-5.3-Codex el sábado 17/05 (antes era GPT-4.1). Es el primer modelo "LTS" de GitHub — disponibilidad garantizada hasta el 04/02/2027, multiplicador de premium-request 1×. Quien está en una org enterprise y no cambia la selección de modelo recibe un backend marcadamente distinto sin migración alguna.

Split de billing del Agent SDK, update sobre la fecha 15/06

El anuncio del miércoles 14/05 se está concretando. Desde el lunes 15/06, Claude Agent SDK, modo headless claude -p, Claude Code GitHub Actions y todos los harnesses de terceros (Zed, OpenClaw, Conductor, Jean, Cline, OpenHands, cualquier wrapper casero) migran del pool regular de suscripción a un "Agent SDK credit pool" separado. Anthropic no ha cuantificado oficialmente el rango efectivo de precio; análisis de la comunidad lo colocan entre 12× y 175× de incremento por workload.

Qué significa para el stack de Daniel si Claude Code se usa nativamente: nada — la CLI interactiva se queda en el pool de suscripción. Qué significa si tu agente corre por el SDK (integración Zed, Cline con backend Claude, bot de GitHub Actions): desde el 15/06 pagas por token en vez de suscripción flat. Nota escéptica: Anthropic puede reaccionar a la presión de la realidad de OPEX, o la competencia puede reaccionar con sus propios modelos de suscripción — este movimiento todavía no está completamente contado.

help.apiyi.com · gigazine.net

Con varios runs/día y diffs grandes caes en cifras de dos dígitos en dólares rápido — cabe en un presupuesto de equipo, pero no en un side-stack de hobby.

El cloud agent recibe modelos baratos

Botón "Fix with Copilot" en Actions que fallaron

En la página de logs de un workflow run que falló ahora aparece un botón "Fix with Copilot". El cloud agent jala el contexto de la falla, investiga, hace push de un fix a un branch nuevo, te marca para review. Sólo Copilot Business/Enterprise. Conceptualmente es exactamente el movimiento que introdujeron Codex Mobile y Claude /goal: handoffs de un clic saliendo del loop de review/aprobación. Tres frontends de lab, misma pendiente mecánica.

Movimiento de plataforma/auditoría para admins enterprise. En paralelo: remote control del Copilot CLI GA en mobile, web, VS Code — inicia sesiones en terminal, JetBrains o VS Code, sigue desde GitHub Mobile o github.com. Cline · --worktree llega al CLI

v3.0.6 (17/05) / v3.0.7 (18/05)

Dos releases chicos pero prácticos. El v3.0.6 trae la flag --worktree: por tarea se crea automáticamente un git worktree fresco bajo ~/.cline/worktrees/, aislado limpiamente del working tree principal, componible con --taskId y --continue. Quien trabaja en tres bugs en paralelo sin destrozar los stashes finalmente tiene la solución limpia en vez de tres clones del repo. El v3.0.7 es plumbing — la lista de modelos de ChatGPT OAuth está sincronizada con el catálogo de provider de Codex, el roundtrip de startup se omite. La extensión VS Code se queda en v3.83.0 (13/05).

El Anthropic SDK tiene --worktree desde septiembre, el Codex CLI desde la ola mobile de la semana pasada — Cline cierra la brecha ahora. Tercero de cuatro harnesses majors en adoptar esta convención de workflow.

github.com/cline/cline — v3.0.6
git worktree: Imagina que tienes un clóset (el repo) con conjuntos de ropa (branches). Normalmente sacas un conjunto, te lo pones, lo guardas, sacas el siguiente. Con worktree haces copias — el mismo repo en varias carpetas, cada una en un branch distinto. Tres tareas en paralelo, sin cambiarte de ropa.
