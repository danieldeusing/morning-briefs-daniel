Open weights · Qwen Conference el lunes, 3.7 aún cerrada [1]

La fecha de verdad está por delante: la Qwen Conference 2026 el lunes (26/05) es el escenario más plausible para el drop de open-weights que los usuarios locales esperan desde hace semanas. Hasta hoy el panorama está igual que ayer — así que revisé de nuevo exactamente la pregunta que se resuelve el lunes.[1]

Revisado de nuevo hoy: ninguna 3.7 abierta

huggingface.co/Qwen sigue trayendo solo las familias 3.5 y 3.6 — ningún repo 3.7, ningún GGUF, ninguna fecha. Los ítems más nuevos ahí (~11 días) son artefactos de investigación sobre los modelos base 3.x, no un modelo de chat nuevo. El Qwen 3.7-Max se reveló el 20/05 en el Apsara/Alibaba Cloud Summit, pero corre preview/API-only en chat.qwen.ai y LMArena — sin pesos abiertos, sin precio de API, sin GGUFs. Las variantes abiertas 27B/35B prometidas siguen "por venir".[1]

preview/API-only vs. pesos abiertos: "Preview" significa que puedes usar el modelo por el sitio web o un endpoint en la nube — como rentar un carro. "Pesos abiertos" significa que recibes el propio archivo del modelo y puedes correrlo en tu computadora — como comprar el carro. Para el stack local Ollama/llama.cpp de Daniel solo cuenta lo segundo; un modelo preview-only no se puede tocar localmente.

Los blogs qwenlm.github.io y qwen.ai están renderizados con JS y hoy devolvieron solo cachés viejos — un teaser previo a la conferencia no se pudo confirmar, así que no afirmo ninguno. Lo mismo con la página de Twitter (@Alibaba_Qwen): ilegible hoy, sin señal en vivo.

huggingface.co/Qwen

Qué se puede correr localmente hoy

Hasta el lunes, la generación 3.6 sigue siendo lo que de verdad se puede tocar: ollama.com/library trae qwen3.6 (27b/35b, con visión/tools/thinking), actualizado por última vez hace ~2 días. Para coding puro, el Qwen3-Coder-Next (80B-A3B, abierto, contexto de 256K) sigue siendo la referencia — pero eso no es un ítem de 48 horas, es un release de febrero; no confundirlo con la conferencia. Probar Si programas localmente y todavía estás en la 3.5: la 3.6-27b es el camino de upgrade discreto mientras la 3.7 siga cerrada.

DeepSeek · el descuento del V4-Pro se vuelve permanente [2]

De "promo" a precio de lista

El único movimiento de DeepSeek en las últimas 48 horas no es un modelo, es un precio: el descuento de 75% en el V4-Pro se habría vuelto permanente el 22/05 (cotizado en ~US$ 0,435 de entrada / US$ 0,87 de salida por 1 M de tokens). Eso hace del V4-Pro uno de los modelos de coding de frontera más baratos para cargas vía API.

Atención Las API docs de DeepSeek devolvieron 401 hoy — así que el número no viene de la fuente primaria, pero lo confirman de forma independiente varios medios establecidos. Si vas a presupuestar con base en eso, de todos modos revisa tú mismo la página oficial de precios antes de migrar. Estatus del modelo sin cambio: V4-Pro / V4-Flash, licencia MIT, desde fin de abril.

MCP · un merge de doc, por lo demás calma de fin de semana [6][7]

PR #2769 · doc alineada al schema

El repo de la spec tuvo exactamente un movimiento notable en el fin de semana: el PR #2769 ("Update Documentation to match Schema", mergeado el 22/05, 19:48 UTC). Trabajo puro de doc, sin breaking changes — la doc ahora nombra subscribe como recurso opcional de Resources, el bloque _meta obligatorio en los ejemplos de tool-call y resource links en Prompts. Después de eso, silencio: ningún PR de spec el 23 o 24/05.[6]

_meta (MCP): Una bolsa lateral en cada mensaje MCP para información extra que no es parte del contenido en sí — como la etiqueta de dirección en un paquete: no es la mercancía, sino lo que el transporte necesita. La spec ahora permite exigirlo en más respuestas.

Lado servidor y la fecha del RC

El GitHub MCP Server sigue en 1.0.5 (18/05), y el registry de MCP está sin commit nuevo desde el 14/05. La fecha fijada para el próximo release candidate de la spec sigue siendo 28/07/2026 (PR #2750), la spec publicada sin cambio en 2025-11-25.[7]

"MTP" = multi-token prediction, una técnica para predecir varios tokens de una vez.
