División de facturación Anthropic: SDK de agentes recibe su propio fondo [1]

Hoy, 15 de junio, entra en vigencia el cambio más importante del trimestre para la propia cadena de herramientas de Daniel: el uso programático de Claude se desvincula del límite de suscripción y se extrae de un pool de créditos mensual separado. Anunciado el 14 de mayo, la fecha de vigencia es ahora.[1]

Fondo de créditos vs. límite de suscripción: Hasta ahora, una suscripción Max tenía un único "vaso" por 5 horas, del cual todo bebía—el humano en la TUI tanto como un script de fondo vía claude -p. Si un agente de larga duración vaciaba el vaso durante la noche, a la mañana siguiente el trabajo interactivo también tenía límite de tasa. A partir de hoy hay dos vasos: el vaso de suscripción antiguo para trabajo interactivo (TUI, Cowork, claude.ai) y un vaso separado, medido en dólares, para todo lo programático. Los usuarios de scripts no se bloquearán mutuamente—pero el fondo de scripts es finito, no se renova el siguiente mes, y no es compartible en equipo.

Lo que se mueve al nuevo fondo: Claude Agent SDK, claude -p (CLI headless), Claude Code GitHub Actions, aplicaciones de terceros vía Agent Client Protocol.[1] Lo que permanece en límite de suscripción: Claude Code interactivo (TUI de Terminal, plugin IDE), Claude Cowork, claude.ai (Web/Desktop/Mobile).[1]

Operacionalmente para Daniel (Max 20×) $200 suena generoso, pero es por usuario (sin pool de equipo), expira mensualmente (sin acumulación), y es precio de lista. Fable 5 + Mythos 5 deshabilitados globalmente—control de exportación [2][3][4]

La nueva clase Mythos de Anthropic—Fable 5 (GA desde 09/06) y el preview Mythos 5—no están disponibles globalmente desde 12 de junio de 2026, 17h21 ET.[3] El secretario de comercio de EE.UU. Howard Lutnick emitió una directiva de control de exportación a Dario Amodei que bloquea el acceso para todo nacional extranjero—dentro y fuera de EE.UU., incluido el personal internacional de la propia Anthropic.[3] Anthropic deshabilitó el acceso para todos los usuarios porque el estado de nacional extranjero no se puede verificar confiablemente en una API pública.

En GitHub Copilot, Fable 5 fue deshabilitado el viernes 12/06, el mismo día: "effective today, access to Claude Fable 5 has been suspended across all GitHub Copilot experiences"[2]—tres días después del lanzamiento GA el 09/06.[2]

Control de exportación en un modelo de API: Clásicamente, el Departamento de Comercio de EE.UU. regula la exportación de chips, algoritmos criptográficos o software de uso dual. Aquí, por primera vez, un modelo generativo de IA se trata como tecnología de uso dual—acusación: la capacidad de "encontrar debilidades en código" es reutilizable para ataques cibernéticos. La contrapropuesta de Anthropic: la misma capacidad existe en GPT-5.5; "este estándar detendría los relanzamientos de toda la industria."[3]

Detonante, según Marktechpost: un jailbreak "narrow and non-universal" que un "socio de confianza altamente creíble" de Anthropic y del gobierno de EE.UU. demostró—la técnica le pide al modelo que escanee una base de código en busca de agujeros de seguridad.[3][4]

Sin fecha oficial de levantamiento—hasta entonces, Opus 4.8 es el mejor modelo disponible de Anthropic (88,6 % SWE-Bench Verified, consulte siguiente sección).

SWE-Bench Verified: qué permanece cuando Fable 5 no cuenta [5]

En llm-stats.com (a partir de 15/06, 99 modelos evaluados), Fable 5 lidera con 95,0 % por delante de Mythos Preview (93,9 %).[5] Ambos están suspendidos. El modelo Anthropic realmente disponible es Opus 4.8 con 88,6 %—tres lugares más abajo, pero aún claramente por delante del campo reservable. Opus 4.7 sigue con 87,6 %, Opus 4.5 con 80,9 %.[5]

SWE-Bench Verified — Top 8 modelos disponibles, a partir de 15/06/2026

Notable: la brecha entre Opus 4.8 (88,6 %) y el top no-Anthropic (DeepSeek-V4-Pro-Max + Gemini 3.1 Pro en 80,6 % cada una) es ~8 puntos porcentuales.[5] MiniMax M3 cierra con 80,5 %, Qwen3.7 Max con 80,4 %.[5]

Nota de lectura SWE-Bench Verified ≠ SWE-Bench Pro (conjunto diferente + scaffolding de proveedor; Fable 5 allí con 80,3 % según morphllm.com). La contaminación de entrenamiento nunca se descarta; Verified la filtra mucho. En la práctica: 88,6 % vs. 80,6 % no significa "nueve veces mejor", sino "en un problema de complejidad media donde Gemini falla, Opus 4.8 tiene una ventaja moderada en una corrección que pasa la prueba". La experiencia del día a día real sigue siendo la señal más confiable.

Gemini CLI → Antigravity CLI: jueves es la fecha límite [6]

Google apaga Gemini CLI el 18 de junio de 2026, 23h59 PT (vie 19/06, 08h59 CEST / 03h59 BRT).[6] El reemplazo es Antigravity CLI, disponible en general desde 19 de mayo—basado en Go (no Python), flujos de trabajo multi-agente asincrónico sin bloqueo de terminal, unificado con el arnés de escritorio Antigravity 2.0.[6]

Cambia esta semana, no en cuatro semanas. Quien tenga Gemini CLI fijado en CI o scripts locales verá "comando no encontrado" a partir del viernes. Excepción: los clientes con licencia Gemini Code Assist Standard/Enterprise mantienen su acceso—EOL afecta solo a usuarios gratuitos/inicio de sesión.[6]

Flujo de trabajo multi-agente (CLI): Un agente único hace bucle: plan → llamada de herramienta → resultado → siguiente plan. En tareas largas (indexación, ejecución de pruebas, escritura de parches), eso bloquea el terminal. Multi-agente: la herramienta principal delega a sub-agentes que se ejecutan en paralelo en segundo plano, y entrega el resultado. Nativo en Antigravity CLI; en el antiguo Gemini CLI, tenías que construir eso tú mismo.

El nivel Max es nuevo—para "flujos de trabajo de agente de alto volumen sostenido": tarifa fija de $100, presupuesto de $200 en créditos, efectivamente el doble del volumen por dólar. La justificación de GitHub: "una pregunta rápida de chat y una sesión de codificación autónoma de varias horas pueden costarle al usuario lo mismo ... el modelo de solicitud premium actual ya no es sostenible."[7] Visual Studio Magazine titulé el 27/04: "You Will Get Less, but Pay the Same Price"—los usuarios avanzados no ven aumento de volumen, solo costo efectivo más alto por ejecución.[7]

Las cargas de trabajo de agentes están explotando en quema de tokens; un límite de suscripción unificado ya no es sostenible del lado del proveedor. Para Daniel: el monitoreo del consumo ahora es higiene de herramientas.

OpenCode supera a Claude Code por estrellas [9]

El punto de datos OSS más notable de la semana: OpenCode—agente de codificación de terminal con licencia MIT—está en 172.198 estrellas GitHub y es ahora el agente de codificación de código abierto más estrellado.[9] Para comparación: Claude Code ~124.000, Gemini CLI ~105.000, openai/codex ~90.000.[9] OpenCode soporta 75+ proveedores de IA según byteiota (Claude, GPT, Gemini, Ollama)—la agnóstica de modelo es su propuesta central.

Verificación de realidad Estrellas ≠ calidad. Un revisor de medium.com comparó OpenCode en tareas idénticas: ~78 % más lento que Claude Code.[9] Voz única, sin benchmark oficial—pero se ajusta al patrón de que "75+ proveedores" a menudo significa una capa de adaptador genérico. intenta-esto desátalo en un proyecto personal—el trade-off de velocidad se nota rápido; si es insignificante, la elección de modelo es una ventaja real.

Cursor Bugbot se vuelve rápido y barato [10]

Cursor lanzó una actualización de Bugbot el 10/06: tiempo de revisión por PR de ~5 min a ~90 s, 10 % más bugs, 22 % más barato.[10] Nuevo: comando /review. El 03/06, la gestión de equipos empresariales (múltiples equipos desde una superficie, controles de seguridad/presupuesto separados) entró en GA.[10]

Menos de dos minutos significa para bots de PR: la revisión se ejecuta mientras el autor aún está en la misma pestaña—el flujo de trabajo se vuelve asincrónico. Cursor está posicionando Bugbot claramente contra las revisiones de Claude Code vía GitHub Action y CodeRabbit.

Pesos abiertos: MiniMax M3 y Qwen3-Coder-Next [11]

Dos modelos de peso abierto son dignos de mención para codificación local esta semana. MiniMax M3 (lanzamiento de junio) es, según blog HuggingFace, el primer peso abierto que combina codificación fronteriza, contexto de 1 M y multimodalidad nativa; lidera el campo de peso abierto en SWE-Bench Pro con 59,0 %.[11] Qwen3-Coder-Next es, según promptquorum.com, actualmente el mejor LLM local para codificación—puntuaciones más altas de HumanEval y SWE-Bench Verified en el espacio de peso abierto.[11]

Local vs. fronterizo 59 % en SWE-Bench Pro para un modelo abierto es progreso genuino—pero Opus 4.8 obtiene 88,6 % en SWE-Bench Verified, y estos son dos benchmarks diferentes. En la práctica: para sugerencias de código en código confidencial sin exposición en la nube, la variante local (Qwen3-Coder-Next en Ollama) se vuelve interesante; para ediciones críticas en la ruta de código de producción, Opus 4.8 sigue siendo con mucho la mejor herramienta. Los dos mundos convergen, pero aún no se han fusionado.
