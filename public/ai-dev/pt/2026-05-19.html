Cursor · Composer 2.5 — o primeiro modelo frontier de coding feito em casa [1]

O drop da Cursor ontem (segunda 18/05) é a movimentação de ferramenta mais comentada da semana — e não porque uma feature saiu, mas porque o modelo em si é da casa. O Composer 2 (outubro) foi a primeira tentativa; o Composer 2.5 posiciona a Anysphere abertamente como fornecedor frontier, não só uma camada IDE em cima dos modelos dos outros.

O que a Anysphere clama

O Composer 2.5 alcança nível Opus 4.7 e GPT-5.5 a "cerca de 1/10 do custo", segundo o changelog da Cursor — SWE-Bench Multilingual 79,8%, CursorBench v3.1 63,2%. Preços: Standard US$ 0,50/M entrada, US$ 2,50/M saída; Fast (default) US$ 3,00/M entrada, US$ 15,00/M saída. Limite de uso dobrado na primeira semana.

Leitura cética: CursorBench v3.1 é o benchmark próprio da Cursor. SWE-Bench Multilingual é uma variante propensa a contaminação do conjunto Verified — o conjunto Pro seria a comparação honesta, e nele os modelos frontier reais ficam 15-25 pontos abaixo. O claim "1/10 do custo" é plausível na infraestrutura própria da Anysphere, mas condicional: o tier Standard é barato, o tier Fast fica grosso modo em paridade com o preço do Opus.

cursor.com/changelog
Para testar Rode o Composer 2.5 no tier Standard contra uma feature task manejável — duas horas com a ferramenta que você usa normalmente, duas horas com o Composer 2.5. A pergunta honesta não é "ele vence o Opus no bench X", é "onde ele se desmonta no seu código com os seus patterns". Limites dobrados na primeira semana — não custa nada além de tempo.

OpenAI · Codex CLI rust-v0.131.0 sai do alfa [2]

O que tem de substancial

À sombra da história do mobile na semana passada, o rust-v0.131.0 (18/05) traz a melhoria de plumbing: CLI de marketplace de plugins com version-aware sharing e share-checkout, hooks de plugin habilitados por default, codex remote-control gerenciado por daemon com APIs de enable/disable/status, ambientes remotos backed por registry. No TUI: menções com @ agora pesquisam arquivos, diretórios, plugins e skills em um único picker unificado; slash-commands de service-tier orientados por dados; exibição mesclada de uso de tokens; modo de permissões/aprovação na status line; tabelas Markdown responsivas. O SDK Python migrou para openai-codex (nome do pacote).

Novidade para o dia a dia: codex doctor — um comando de diagnóstico que verifica auth, servidores MCP, estado de plugins e permissões de sandbox em uma só passada. Parece sem graça, mas é exatamente o que custava 20 minutos de debugging manual em uma sessão quebrada. Tag no mesmo dia: 0.132.0-alpha.1.

Split de billing do Agent SDK, update sobre o prazo de 15/06

O anúncio de quarta 14/05 está se concretizando. A partir de segunda 15/06, Claude Agent SDK, modo headless claude -p, Claude Code GitHub Actions e todos os harnesses de terceiros (Zed, OpenClaw, Conductor, Jean, Cline, OpenHands, qualquer wrapper caseiro) migram do pool regular de assinatura para um "Agent SDK credit pool" separado. A Anthropic não quantificou oficialmente a faixa efetiva de preço; análises da comunidade colocam entre 12× e 175× de aumento por workload.

O que isso significa para o stack do Daniel se o Claude Code for usado nativamente: nada — a CLI interativa fica no pool de assinatura. O que significa se seu agente roda pelo SDK (integração Zed, Cline com backend Claude, bot de GitHub Actions): a partir de 15/06 você paga por token em vez de assinatura flat. Nota cética: a Anthropic pode reagir à pressão da realidade de OPEX, ou concorrentes podem reagir com seus próprios modelos de assinatura — esse movimento ainda não está totalmente contado.

help.apiyi.com · gigazine.net
Atenção Quem roda um pipeline próprio do GitHub Actions com o Claude Code como reviewer deveria extrapolar o custo típico do run em tokens contra a tabela da API antes de 15/06. Com várias runs/dia e diffs grandes, você cai em valores de dois dígitos em dólar rápido — cabe num orçamento de time, mas não num side-stack de hobby.

Cline · --worktree chega no CLI

v3.0.6 (17/05) / v3.0.7 (18/05)

Dois releases pequenos mas práticos. O v3.0.6 traz a flag --worktree: por tarefa cria-se automaticamente um git worktree fresco sob ~/.cline/worktrees/, isolado limpinho da working tree principal, componível com --taskId e --continue. Quem trabalha em três bugs em paralelo sem destruir os stashes finalmente tem a solução limpa em vez de três clones do repo. O v3.0.7 é plumbing — a lista de modelos do ChatGPT OAuth está sincronizada com o catálogo de provider do Codex, o roundtrip do startup foi suprimido. A extensão VS Code fica no v3.83.0 (13/05).

O Anthropic SDK tem --worktree desde setembro, o Codex CLI desde a onda mobile da semana passada — o Cline fecha a lacuna agora. Terceiro de quatro harnesses majors a adotar essa convenção de workflow.

github.com/cline/cline — v3.0.6
git worktree: Imagine que você tem um guarda-roupa (o repo) com conjuntos de roupa (branches). Normalmente você puxa um conjunto, veste, guarda, puxa o próximo. Com worktree você faz cópias — o mesmo repo em vários pastas, cada uma num branch diferente. Três tarefas em paralelo, sem trocar de roupa.

pypi.org/project/vllm · github.com/vllm-project/vllm
Yank no PyPI: Se você "yanka" um pacote, ele continua instalável, mas não aparece mais num pip install normal — efetivamente uma placa de "não comprem mais isso" do mantenedor. Diferente do delete (some completamente), o yank não quebra lockfiles existentes mas envia um sinal claro de "quebrado".
