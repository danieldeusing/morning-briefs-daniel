Codex 0.133.0 · o Goal mode vira padrão [1][2]

Ontem o trem de alphas acelerado (0.133.0-alpha.4 às 02:02 UTC), hoje o stable aterrissado: o Codex 0.133.0 (21/05, 16:48 UTC) é o drop mais substancial do dia. O destaque não é um modelo novo, mas uma decisão de workflow: o Goal mode sai do beta e agora vem ligado por padrão — simultaneamente no app Codex, na extensão de IDE e na CLI.[1]

Goal mode: em vez de explicar ao agente, a cada passo, qual é o objetivo final, ele recebe uma meta fixada com armazenamento próprio que carrega entre os turnos e contra a qual checa o progresso. Como um bilhete adesivo com o objetivo final que o agente fica relendo — em vez de você se repetir em toda mensagem.

O que mais tem dentro

Além do Goal mode: melhor UX de remote-control (agora espera prontidão, start/stop de daemon, status da máquina), uma sandbox mais dura no Windows com perfis de permissão e uma lista de plugins ciente do marketplace. No mesmo dia veio o build 26.519 do app companheiro, com Appshots (macOS) e uso remoto de computador.[2] Contexto: essa é a resposta da OpenAI à mesma pergunta em que Claude Code e Cursor trabalham — como um agente segura a intenção ao longo de muitos passos sem o humano corrigindo o tempo todo. Que o Goal mode agora seja o padrão é a declaração de fato.

Claude Code se corrige no mesmo dia [3]

2.1.147 — os acréscimos úteis

Sobre a renomeação de ontem do /code-review, a 2.1.147 (21/05) acrescenta o substancial: uma flag --comment que posta achados de review direto como comentários inline em um PR do GitHub, além de níveis de esforço para o review. Sessões em segundo plano fixadas agora sobrevivem a idle e a um restart-in-place durante updates; o auto-updater tenta de novo em falhas transitórias e reporta códigos de erro do SO. Também um fix de paginação no MCP (respostas paginadas de tools/list só retornavam a primeira página, sumindo com os demais tools) e vários fixes de Windows/PowerShell.[3]

2.1.148 — o auto-hotfix

Poucas horas depois, já em 22/05, veio a 2.1.148 com um único fix: depois da 2.1.147 a ferramenta Bash retornava exit code 127 em todo comando — uma regressão que a própria 147 havia introduzido. Exatamente o reverso de uma cadência alta de releases: lançar diariamente (ou várias vezes ao dia) e você pega regressões diariamente. Na prática para o Daniel: pule direto para a 2.1.148, mas evite a 2.1.147 como degrau obrigatório — vale para qualquer CLI com várias tags por dia.

Benchmarks · a correção é a notícia [15][16][17][18]

Posição em 22/05/2026. Nenhum novo movimento de bench independentemente verificável — como ontem. O valor de hoje é uma correção ao próprio enquadramento de ontem.

Correção Ontem dissemos que o repo aberto do SWE-Bench estaria "congelado desde 18/11/2025". Contando os arrays resolved diretamente hoje, vê-se: a última submissão verificada de verdade foi commitada em 26/01/2026 (os runs de dezembro), o repo foi renomeado de experiments para submissions em 06/01, e o commit mais recente (29/03) foi cosmético. Logo, não "desde novembro", mas nenhuma nova submissão verificada desde meados de dezembro — efetivamente ~4 meses de parada. O topo segue em Opus 4.5 / 79,2% (396/500).

O que é verificável — e o que não é

SWE-Bench Verified (repo aberto): Opus 4.5 com 79,2%, empatado por dois agentes (live-SWE-agent / Sonar). Os 87–94% que circulam para o Opus 4.7 ou o GPT-5.5 vêm de páginas de agregadores/fornecedores, não do repo verificável.[15] O Aider Polyglot segue em 20/11/2025 (GPT-5-high 88,0%) — mais de seis meses atrás.[16] O LiveCodeBench não serve número estático via JS — por isso aqui sem valor em vez de estimado. Como fonte de preferência, a arena.ai WebDev Arena (posição em ~18/05) lidera com Opus 4.7-thinking (1567), depois Opus 4.7 (1560), Opus 4.6-thinking (1545), Opus 4.6 (1540) e GLM-5.1 (1532) — sem Gemini 3.5 Flash no top 5.[17]
