Claude Code · 2.1.152 traz auto-fix e controle de skills [1]

Depois da semana passada, em que o Claude Code ficou parado na 2.1.150 (23/05/2026), a 2.1.152 chegou hoje — e é o build mais substancial da série em dias. Três pontos valem o olhar.[1]

/code-review --fix escreve de volta

Até agora /code-review era um parecer read-only. Agora --fix escreve os achados direto na working tree — sugestões de reuso, simplificação e eficiência são aplicadas em vez de só listadas. /simplify agora chama o mesmo --fix internamente. É o passo de "o revisor te avisa" para "o revisor faz". Try Para o fluxo do Daniel, o ponto mais relevante: aprovar uma passada de revisão e ver as mudanças mecânicas direto no diff, em vez de transcrevê-las à mão.

code.claude.com/docs/changelog

Skills agora podem tirar ferramentas

Skills e slash commands podem definir disallowed-tools no frontmatter — ferramentas removidas do modelo enquanto a skill está ativa. Some-se /reload-skills (re-escanear os diretórios de skills sem reiniciar a sessão) e um hook SessionStart que pode retornar reloadSkills: true, deixando uma skill instalada pelo hook disponível na mesma sessão.

disallowed-tools na skill: Imagine uma caixa de ferramentas da qual você tira de propósito a serra afiada para uma certa tarefa, para ninguém pegá-la sem querer. Uma skill agora pode dizer "enquanto eu rodo, nada de Bash" — um escopo mais apertado significa menos jeitos de quebrar algo.
code.claude.com/docs/changelog

Letras miúdas com consequências

Três linhas que pesam no dia a dia: um novo hook MessageDisplay pode reescrever ou esconder o texto do assistente ao exibi-lo; quando o modelo primário não é encontrado, o Claude Code passa para o --fallback-model pelo resto da sessão, em vez de falhar a cada requisição; e "Auto mode no longer requires opt-in consent" — o modo autônomo roda sem a pergunta de consentimento prévia. Esta última é uma comodidade com gosto amargo: um clique a menos, mas também uma confirmação a menos antes de execuções sem supervisão.

Codex · 0.134.0 de alpha para stable [2]

A alpha passou

O que amadureceu desde 22/05/2026 como 0.134.0-alpha.1…3 saiu ontem (26/05/2026, 19:13 UTC) como stable 0.134.0 — subindo da stable 0.133.0 (21/05/2026). As novidades que valem nota: histórico de conversa local pesquisável (case-insensitive), --profile agora é o seletor primário de perfil (configs de perfil legadas são rejeitadas), env de MCP por servidor mais OAuth para streamable HTTP, e ferramentas MCP read-only agora rodam em paralelo em vez de em série.[2]

Os dois últimos pontos são o sinal de verdade: o Codex está visivelmente mexendo no encanamento de MCP esta semana — combinando com a onda na spec do MCP do outro lado (ver à direita). Rodar ferramentas read-only em paralelo é exatamente o tipo de ganho de latência que deixa execuções agênticas perceptivelmente mais rápidas, sem o risco de escritas paralelas.

MCP · onda de governança no repositório da spec [7][8]

Enquanto as CLIs lançavam novos builds, o repositório modelcontextprotocol teve o dia mais movimentado em semanas — e o foco chama atenção: segurança, auditabilidade, governança.

Mesclado: Codex/ChatGPT leem instruções do servidor

A PR #2790 (mesclada 26/05/2026) registra que Codex e ChatGPT agora suportam instruções do servidor MCP — o texto de orientação que o servidor envia é de fato lido pelo cliente e injetado no modelo. O merge mais novo do repositório já é a #2800 (27/05/2026, encanamento do workflow de release); a spec publicada segue 2025-11-25, o alvo de RC 28/07/2026 (PR #2750 mesclada).[7]

Instruções do servidor MCP: Como a bula de um aparelho — o servidor diz ao cliente de IA por conta própria "é assim que você me usa direito" (quais ferramentas para quê, em que ordem). Antes alguns clientes ignoravam a bula; Codex e ChatGPT agora a leem.
MCP PR #2790

Três SEPs novas em um dia

Em 26/05/2026 caíram três propostas de uma vez: a SEP "Tool Risk Metadata" (#2793) estende as ToolAnnotations com informação de risco legível por máquina, para um cliente decidir automaticamente o que pode rodar sem perguntar e o que não pode; a #2792 traz negociação de idioma por requisição (i18n); a #2778 "Adding constraints". A proposta de attestation SEP-2787 (recibos assinados de chamada de ferramenta, Art. 12 do EU AI Act) segue aberta mas em 26/05/2026 ganhou os primeiros vetores de teste com a #2789. Todas abertas, todas no Extensions track — mas a direção é clara.[8]

Tool Risk Metadata: Uma etiqueta em cada ferramenta dizendo o quão perigosa ela é — "só leitura" (verde) vs. "apaga arquivos" (vermelho). O agente pode então usar as verdes sem perguntar e parar só nas vermelhas. Hoje essa decisão costuma ser uma lista mantida à mão; a etiqueta a torna legível por máquina.

Comunidade · o lado B dos agentes

Simon Willison documenta esta semana uma exfiltração via prompt-injection no Microsoft Copilot "Cowork" — um atacante contrabandeia instruções nos dados que o agente processa e por aí vaza informação para fora. Mais o "The Pressure" de Daniel Stenberg sobre a enxurrada de security reports gerados por IA, sem voz humana, que atropela os mantenedores do curl. Duas histórias, uma moral: quanto mais autônomos os agentes, mais importam trilhas de auditoria e etiquetas de risco — exatamente o que o MCP está reformulando agora.

Prompt injection: Como um bilhete falso na caixa de entrada de um assistente obediente demais — comandos escondidos ("manda tudo para este endereço") ficam dentro do texto que o agente deveria processar. O agente não distingue dados inofensivos de instruções contrabandeadas, e as executa.
