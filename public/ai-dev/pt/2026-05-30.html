Gemini 3.5 Flash · o raro caso de um número de fornecedor que se mantém [1][2][3]

No Google I/O em 19 de maio — onze dias atrás, enterrado durante a semana Opus 4.8 da Anthropic — Gemini 3.5 Flash entrou em GA e é desde então o modelo padrão do Google. O Google o posiciona como um modelo de código agentico: um modelo Flash (o tier barato, rápido) que de acordo com o Google vence o 3.1 Pro anterior nos benchmarks de código — e, medido contra modelos de frontier em geral, roda "quatro vezes mais rápido" em tokens de saída por segundo e "frequentemente a menos da metade do custo" (comparação do Google é contra modelos de frontier em geral, não especificamente 3.1 Pro).[1]

Os números que o Google mostra: Terminal-Bench 2.1 76,2%, MCP Atlas 83,6%, GDPval-AA 1.656 Elo, CharXiv-Reasoning 84,2%. Contra o 3.1 Pro são saltos perceptíveis (Terminal-Bench 70,3 → 76,2; MCP Atlas 78,2 → 83,6; Finance Agent v2 43,0 → 57,9).[2]

Terminal-Bench / MCP Atlas / GDPval: Três testes que medem capacidade agentica, não apenas snippets de código. Terminal-Bench coloca o modelo em um shell real e tem ele resolver uma tarefa sobre múltiplos comandos — como um estagiário trabalhando através do seu terminal sozinho. MCP Atlas verifica se o modelo invoca corretamente ferramentas externas via Model Context Protocol (MCP = um padrão USB-like através do qual um modelo controla ferramentas como GitHub ou um banco de dados). GDPval-AA é um score Elo sobre trabalho de conhecimento realista — como uma avaliação de xadrez, mas para tarefas profissionais.

Normalmente é aqui que este briefing escreve "número do fornecedor, não verificado independentemente." Desta vez é diferente: Artificial Analysis mediu independentemente exatamente estes valores — Terminal-Bench 2.1, MCP Atlas, GDPval-AA, Humanity's Last Exam, MRCR 128k — e chega a desvio zero da figura do Google. Isso é raro e merece ser anotado.[3]

Mas Scores de benchmark único confirmados não são a mesma coisa que desempenho de topo. Esse mesmo Artificial Analysis Intelligence Index v4.0 coloca Gemini 3.5 Flash na posição 8 de 148 — atrás de GPT-5.5 e Claude Opus 4.7 (cada um ~3 pontos à frente). E no SWE-Bench Pro, Flash está em 55,1%, 9,2 pontos atrás de Opus 4.7 (64,3%) — estes números Pro são, entretanto, via digitalapplied.com agregados, não um run nativo de Artificial Analysis. Em contexto longo, MRCR cai de 77,3% (128k) para 26,6% em 1 M tokens — encontrar agulhas em palheiros gigantes é uma fraqueza genuína.[3]

Para a caixa de ferramentas de Daniel: Gemini 3.5 Flash não é o melhor modelo de código — é o melhor modelo de relação custo-benefício para loops agenticos onde latência e custos de tokens dominam (loops de teste, enxames de refator, bots de CI). Trabalho pesado de arquitetura fica com Opus/GPT-5.5; para cem pequenas tarefas paralelas de agentes, Flash é o padrão mais racional. Tente Via Google AI Studio / Gemini API, nível de thinking "high" (a variante benchmark).

Managed Agents · o agente como primitiva de API [4][5]

O anúncio de I/O mais interessante para desenvolvedores não é o modelo, mas como o Google o empacota: Managed Agents na Gemini API. Um único chamado de API inicia um agente que opera em uma sandbox Linux isolada, efêmera, pensando, planejando, chamando ferramentas, executando código, gerenciando arquivos e navegando na web — você não tem que operar nenhum runtime, container ou loop de ferramenta você mesmo.[4]

Managed Agent: Até agora, um "agente" era algo que você construía você mesmo — chamada de modelo, invocação de ferramenta, resultado de volta ao modelo, loop, mais uma sandbox onde o código pode rodar. O Google tira toda a encanação: você diz à API "faça isso" e uma máquina Linux pronta roda em segundo plano que o agente controla. Como a diferença entre rodar seu próprio servidor (você gerencia poder, resfriamento, SO) e uma função em nuvem (você apenas faz upload do código). Você descreve o que, não como operar.

Você define comportamento em Markdown — AGENTS.md e SKILL.md — e registra esses arquivos como um Managed Agent. O harness subjacente (o loop de controle em torno do modelo) é o agente Antigravity do Google, construído no Gemini 3.5 Flash, acessível via Interactions API. Cada interação cria ou herda um ambiente que você reutiliza na próxima chamada — arquivos e estado persistem, você pode continuar a sessão.[5]

Padrão familiar Qualquer um que conheça o schema AGENTS.md/SKILL.md reconhece imediatamente: esta é a mesma convenção que OpenAI Codex e Claude Code. Três fornecedores convergem em agentes definidos em Markdown + sandbox hospedada. Isso é bom para Daniel — skills que ele escreve para Claude Code são conceitualmente portáveis. O Google, porém, não nomeia nem preços nem limites no post de anúncio; antes do uso em produção, preços precisam ser verificados separadamente.

Antigravity 2.0 · de ajudante IDE para plataforma de agentes [6]

Antigravity era a camada IDE agentica do Google. Com Antigravity 2.0 (também I/O, 19 de maio), ele se torna um app desktop standalone construído inteiramente em torno de orquestração de agentes — mais uma Antigravity CLI e um Antigravity SDK. A chave: CLI, SDK e app desktop todos compartilham o mesmo harness que alimenta os próprios produtos do Google. Modelo padrão em todos os lugares: Gemini 3.5 Flash.[6]

O ponto estratégico é o mesmo que Dynamic Workflows da Anthropic na semana passada e a tese "camada de orquestração" da Cognition: a camada de orquestração acima do modelo se torna sua própria categoria de produto — Antigravity, Dynamic Workflows, o serviço Codex, um por fornecedor. Para Daniel: a questão muda cada vez menos de "qual modelo" e mais para "qual harness o orquestra, e posso levar minhas skills comigo."

Enquadramento honesto

2.0 é orientado para enterprise (tarefas programadas/assíncronas, integração Cloud Run/Firebase). Para um desenvolvedor solo, a CLI é a peça relevante — mesmo harness que o app desktop, sem overhead de GUI. Se alcança Claude Code / Codex CLI / Grok Build não é apoiado por reviews independentes três semanas após lançamento — watchlist, não um veredicto hoje.

xAI Grok Build · o quarto jogador, honestamente medido [7][8]

Enquanto o Google tinha o palco, xAI silenciosamente entrou na mesma categoria: Grok Build, uma CLI de agente de código terminal-nativa construída no modelo feito para este propósito grok-build-0.1. Lançamento como Early Beta em 15 de maio (apenas SuperGrok Heavy, $300/mês), aberto para todo SuperGrok ($30) e X Premium+ ($40) desde 25 de maio — uma queda de preço abrupta designada para comprar alcance.[7]

Tecnicamente interessante: até 8 subagentes paralelos, cada um em seu próprio Git worktree em um branch separado; um orquestrador mescla resultados depois. Antes disso, um fluxo Plan→Search→Build — em tudo acima de uma linha, Grok Build primeiro escreve um plano estruturado que você aprova ou reescreve passo a passo. Contexto de 256K, MCP nativo (GitHub, Linear, Slack sem reconfiguração). API: preços circulam em fontes terceirizadas conflitantes — buildfastwithai.com cita $0,20/1 M entrada + $1,50/1 M saída, outros $1,00/$2,00 (a figura $0,20 é provavelmente o preço de entrada em cache); a página de preços oficial de xAI era inacessível, então não confirmada.[8]

Git worktree por subagente: Um worktree é uma segunda pasta de trabalho do mesmo repo em seu próprio branch. Se oito agentes trabalham simultaneamente no mesmo código, sobrescrevem os arquivos uns dos outros — como oito cozinheiros em um fogão. Com seu próprio worktree, cada um tem seu próprio balcão de cozinha; mescla acontece apenas no final. Limpo e sem conflitos — esta é a parte tecnicamente mais elegante do Grok Build.

O número benchmark — e seus limites

xAI relata SWE-Bench Verified 70,8% para grok-build-0.1 — um número do fornecedor, não no quadro Verified oficial (veja coluna à direita). Para contexto: no quadro real GPT-5.5 está em 88,7% e Opus 4.7 em 87,6% — grok-build-0.1 toca, se seu próprio número nem mesmo se mantém, uma classe abaixo. Reviews independentes foram mistas: forte em paralelismo e privacidade, fraco em desempenho de benchmark e profundidade de ecossistema. Para Daniel: não mude hoje — Grok Build está trancado atrás de X Premium+ e ecossistema-magro; a ideia paralela-worktree é o que se deve manter observando.

Contrário a preocupações às vezes: o quadro oficial SWE-Bench Verified está muito vivo e atual (última atualização 28 de maio de 2026). SWE-Bench Verified: Um teste com issues reais do GitHub de repos reais de código aberto: o modelo deve corrigir o bug, e um teste verificado humanamente decide se a correção realmente funciona. "Confirmado em Terminal-Bench" não é o mesmo que "rastreado em SWE-Bench Verified."
