Kimi K3: um modelo aberto na ponta da arena

A Moonshot AI lançou o Kimi K3 via aplicativo e API em 16/07 — um modelo com 2,8 trilhões de parâmetros e 1 milhão de tokens de contexto, descrito pelos criadores como seu "primeiro modelo aberto de 3 trilhões". O que chama atenção é a colocação: na Arena.ai Frontend Code Arena, o K3 está em 1º lugar com 1.679 de Elo, à frente do Claude Fable 5 (1.631) e do GPT-5.6 Sol (1.618). Nos sete domínios de frontend da arena, ele lidera em seis; só em "Gaming" o Fable 5 fica à frente. Para dar a dimensão do salto: o antecessor Kimi K2.6 estava, no mesmo ranking, apenas na 18ª posição (1.515).

winzheng.com · tomshardware.com · simonwillison.net · Dados de 22/07/2026
Elo de arena e MoE: Uma "arena" não mede com perguntas de teste, e sim com votação popular: dois modelos resolvem a mesma tarefa, um humano escolhe o resultado melhor — como um torneio de degustação às cegas, do qual sai uma pontuação de Elo parecida com a do xadrez. Isso diz qual saída as pessoas preferem, não se o código realmente roda num repositório de verdade. "MoE" (mistura de especialistas) é o truque por trás dos 2,8 trilhões de parâmetros: em vez de um cérebro gigante único, há muitos especialistas, e um roteador só aciona uns poucos deles por token — enorme em disco, mas viável na conta. Só que: todos os parâmetros ainda precisam caber na memória. Continua sendo um modelo de servidor, não de notebook.

Vencedor no frontend, meio de tabela no trabalho agêntico

Duas ressalvas antes de você coroar o K3 como novo rei. Primeira: os pesos ainda não saíram. A Moonshot promete o lançamento dos pesos abertos para 27/07 — até lá, "aberto" é um anúncio, não algo que você possa hospedar por conta própria; hoje só dá para usar via API da Kimi e via Kimi Code CLI. Segunda, e mais importante: a arena mede frontend pelo gosto. No Coding Agent Index da Artificial Analysis — que mede trabalho real de agente em várias etapas (ler arquivos, editar, rodar testes) — o K3 fica na 5ª posição com 57 pontos, no meio do pelotão de ponta fechado: GPT-5.6 Sol 61, Fable 5 59, Grok 4.5 58, K3 57, Opus 4.8 55. No Terminal-Bench v2 ele marca 84%.

Essa é a leitura honesta: na frente na arena de gosto voltada a frontend, empatado com — não à frente de — os modelos fechados no trabalho pesado de agente. testar Se você está de olho num modelo aberto para código, o K3 se torna o candidato para testar você mesmo a partir de 27/07; até lá, vale mais um teste via API no seu próprio repositório de frontend do que confiar na posição na arena.

Preços de pico/fora de pico: Como a tarifa de energia fora de ponta, só que para a API. Em vez de um preço fixo por milhão de tokens o dia inteiro, as requisições custam mais nos horários de pico e menos fora deles — assim o fornecedor equilibra a carga dos servidores. Na prática: jobs em lote que não têm pressa podem ser agendados para as janelas mais baratas, cortando o preço de saída praticamente pela metade.

Os preços reportados

Um modelo de faixa horária como esse deve chegar junto com o lançamento. Os números reportados para o V4-Pro giram em torno de US$ 0,87 por milhão de tokens de saída fora do horário de pico e US$ 1,74 no horário de pico; para o V4-Flash, US$ 0,28 e US$ 0,56. O relatório aponta 09h–12h e 14h–18h como o horário de pico. Até lá, são valores de referência, não base para cálculo.

MCP fica sem estado — a maior reforma desde o lançamento

Em 28/07 sai a especificação final do MCP — o release candidate já está disponível publicamente desde 21/05, com SDKs beta para Python/TS/Go/C#. Segundo o projeto, é "a maior revisão do protocolo desde o lançamento". A reforma central: o protocolo passa a ser sem estado — o handshake de inicialização anterior e o Mcp-Session-Id deixam de existir. Chegam o MCP Apps (servidores podem entregar sua própria UI) e uma extensão Tasks para tarefas mais longas. São removidas as capacidades Roots, Sampling e Logging. Ou seja: o RC e os SDKs beta já foram lançados — 28/07 é a data de lançamento da versão final (o carimbo de versão datado da especificação), não a data do RC.

blog.modelcontextprotocol.io · Dados de 22/07/2026
Sem estado e handshake: Um "handshake" é a rodada de apresentações no início de uma conexão — cliente e servidor combinam quem faz o quê, e guardam isso na memória durante a sessão. "Sem estado" quer dizer: não existe mais essa sessão guardada na memória. É a diferença entre uma ligação de telefone, em que os dois lados ficam na linha o tempo todo, e cartões-postais, cada um completo por si só. A vantagem: servidores sem estado escalam mais fácil e sobrevivem a reinicializações, porque não há memória de sessão para perder. O custo: quem mantém servidor MCP hoje vai precisar de ajustes na hora da virada.

Por que isso importa para você: o MCP hoje está por trás de quase toda ferramenta de agente que você usa — é o plugue padrão pelo qual o Claude Code e companhia acessam ferramentas externas e fontes de dados. Uma reforma desse tamanho significa que servidores próprios ou hospedados vão precisar se atualizar mais cedo ou mais tarde, e que as capacidades removidas (como o Sampling) podem quebrar integrações existentes. Não há urgência hoje — mas é uma data para anotar na agenda do fim de semana.
