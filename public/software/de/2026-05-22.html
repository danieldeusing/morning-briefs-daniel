Docker — drei Wege von Container zu Host-Root über docker cp

Das ist der Pflicht-Patch der Woche, und er sitzt direkt in Daniels Stack. Docker Engine 29.5.1 (18.05) schließt drei Lücken, die alle denselben Kern haben: ein bösartiger Container kann beim harmlos wirkenden docker cp aus dem Container ausbrechen und auf das Host-Dateisystem durchgreifen, weil die Kopier-Logik mit Host-Root-Rechten läuft. CVE-2026-41567: die Entpack-Binaries (xz, unpigz) werden über den PATH im Container aufgelöst — ein untergeschobenes Binary läuft dann als Host-Root. CVE-2026-42306: ein TOCTOU-Race lenkt einen Bind-Mount auf einen beliebigen Host-Pfad um. CVE-2026-41568: ein TOCTOU-Symlink-Swap legt beliebige Dateien/Verzeichnisse auf dem Host an. Docker stuft alle drei als „critical" ein; die zugehörigen GHSA-Records vergeben High/High/Medium [1].

TOCTOU (Time-of-check / Time-of-use): Wie wenn der Türsteher deinen Ausweis prüft, du ihn danach blitzschnell gegen einen gefälschten tauschst, und der Stempel auf den falschen geht. Das Programm prüft einen Pfad, der Angreifer ändert ihn im Sekundenbruchteil dazwischen, und die privilegierte Aktion trifft das untergeschobene Ziel — hier: einen Host-Pfad statt des erwarteten Container-Pfads.

29.5.2 (20.05) ist ein Nachbrenner: 29.5.1 hatte eine Regression eingeführt, bei der docker cp mit Bind-Mounts über In-Container-Symlinks (/var/run → /run) mit „mkdirat: file exists" scheiterte. Wer also schon auf 29.5.1 gesprungen ist, nimmt direkt 29.5.2 mit [1].

Was heute tun Engine-Version prüfen (docker version) und auf 29.5.2 heben. Der Angriff setzt einen kompromittierten oder feindlichen Container voraus, der ein docker cp mitschneidet — wer fremde Images oder PR-getriggerte CI-Container laufen lässt, ist das primäre Ziel. Bis zum Patch keine docker cp-Operationen gegen ungetrustete Container fahren.

OpenJDK — JDK 27 verdichtet sich, drei JEPs vor dem Target

Die Early-Access-Linie steht bei Build 22 (von ~17 am Mittwoch). Drei JEPs gingen Proposed to Target, mit Review-Ende am 19.05: JEP 523 (G1 als Default-GC in allen Umgebungen), JEP 534 (Compact Object Headers per Default — die Promotion von JDK 25s JEP 519) und JEP 537 (Vector API, 12. Inkubator). Neu auf Candidate: JEP 538 (PEM-Encodings für Krypto-Objekte). RDP1 (Feature-Freeze) forkt am 04.06, GA im September. Ehrlichkeits-Flag: dass die drei JEPs von Proposed to Target tatsächlich auf Targeted gekippt sind, ließ sich auf der (403-enden) openjdk.org-Primärseite nicht final bestätigen — Review-Fenster geschlossen, Targeting erwartet [1][2].

Compact Object Headers: Jedes Java-Objekt trägt einen kleinen Verwaltungs-Header mit sich. JDK 25 hat ihn von 12–16 auf 8 Byte geschrumpft — wie wenn jedes Paket im Lager ein kleineres Etikett bekommt: bei Millionen Objekten spart das spürbar Heap und damit GC-Pausen (~15 % weniger GCs in SPECjbb2015). JEP 534 macht das Verhalten in JDK 27 zum Default statt zur Opt-in-Flag.

TC39 #114 — using ist jetzt ECMAScript

Das Amsterdam-Plenary (gehostet von JetBrains) ist Donnerstag zu Ende gegangen; die Stage-Änderungen stehen im tc39/proposals-Repo, die narrativen Notes folgen erst. Headline: Explicit Resource Management — die using- und await using-Deklarationen, die viele aus TypeScript kennen — hat Stage 4 erreicht und ist damit Teil von ECMAScript 2026. Ebenfalls auf Stage 4: Atomics.pause (Spin-Loop-Hinweis für Shared-Memory) und Joint Iteration (Iterator.zip). Auf Stage 3 rückten u. a. Iterator-Chunking/-Includes/-Join und ein standardisierter error.stack-Accessor. Decorators kamen nur von 2 auf 2.7 — der alte Vorbehalt hält: keine der großen Engines will als Erste ausliefern, der Zeitplan bleibt weich [1][2].

using / Explicit Resource Management: Wie try-with-resources in Java, nur für JavaScript: schreibst du using file = openFile(), wird die Ressource automatisch geschlossen, sobald der Block endet — egal ob normal oder per Exception. Schluss mit vergessenen finally { close() }-Blöcken.

Kommentar — die Grenzen, denen wir vertrauen

Drei der heutigen Top-Stories sind dieselbe Geschichte in drei Schichten. Der Container soll den Host nicht sehen — docker cp und der runAsNonRoot-Bypass zeigen, dass die Isolations-Wand dünner ist als das Mental-Modell. Das ORM soll SQL-Injection unmöglich machen — die Drupal-Lücke zeigt, dass „nur auf Postgres" reicht, um diese Annahme zu kippen. Und die SLSA-Provenance soll Lieferketten-Vertrauen beweisen — Shai-Hulud zeigt seit zwei Wochen, dass ein gekaperter Build dieses Vertrauen erbt. Das Muster: nicht die Krypto oder das Feature ist kaputt, sondern die Grenze, an der wir aufhören nachzudenken.

Für Daniels Stack heißt das pragmatisch und überschaubar: Docker auf 29.5.2 heben und keine docker cp gegen ungetrustete Container; falls Postgres-Drupal irgendwo läuft, heute patchen; npm-Lockfiles und Maintainer-Kompromittierungen weiter im Auge behalten. Die gute Nachricht kommt aus derselben Branche, die sich gerade häutet: using ist jetzt ECMAScript, JDK 27 macht Compact Object Headers zum Default, Node hat seinen LTS-Patch. Am Montag liefert die EU mit CADA den nächsten Souveränitäts-Baustein — und dann sehen wir, ob aus dem geleakten US-Cloud-Absatz echter Gesetzes-Text wird [1].

  1. europarl.europa.eu/legislative-train

Es ist Freitag, der zweiundzwanzigste Mai. Der rote Faden heute ist unangenehm konkret: die Grenzen, denen Entwickler vertrauen, lecken gerade alle gleichzeitig — bei der Container-Isolation, bei der Datenbank und in der Lieferkette.

Die wichtigste Geschichte sitzt direkt im Docker-Stack. Docker Engine neunundzwanzig Punkt fünf Punkt eins schließt drei Lücken, über die ein bösartiger Container beim eigentlich harmlosen Kopier-Befehl docker cp aus dem Container ausbrechen und zu Host-Root eskalieren kann — einmal, weil die Entpack-Programme über den Pfad im Container aufgelöst werden, zweimal über ein Timing-Race, das einen Mount-Punkt auf das Host-Dateisystem umlenkt. Docker nennt alle drei kritisch. Eine Etage tiefer patcht containerd am selben Tag einen Bypass, durch den ein Container, der ausdrücklich nicht als root laufen soll, es trotzdem tut. Der Rat für heute: Engine auf neunundzwanzig Punkt fünf Punkt zwei heben, denn die erste Korrektur hatte eine Regression — und keine docker-cp-Operationen gegen Container fahren, denen man nicht traut.

Die zweite Schicht ist die Datenbank. Der am Mittwoch angekündigte Drupal-Release ist da: eine unauthentifizierte SQL-Injection im Core, die ausschließlich PostgreSQL-Installationen trifft. Drupal selbst bewertet sie mit dreiundzwanzig von fünfundzwanzig, die amerikanische Schwachstellen-Datenbank dagegen nur als mittel — beide Werte gehören genannt. Und seit heute werden laut Drupal Exploit-Versuche in freier Wildbahn beobachtet. Wer Postgres-Drupal betreibt, patcht heute.

Die dritte Schicht ist die Lieferkette. Der selbstreplizierende Wurm aus dem letzten Brief, Shai-Hulud, ist nicht abgeklungen, sondern eskaliert.
