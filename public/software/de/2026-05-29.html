Exchange-OWA-Frist heute — und immer noch kein Patch

Die CISA-KEV-Frist für die On-Prem-Exchange-Lücke CVE-2026-42897 läuft heute, Fr 29.05, ab. Microsoft hat die aktive Ausnutzung am 14.05 bestätigt, CISA am 15.05 in den Katalog aufgenommen und 14 Tage zur Mitigation gegeben — das Fenster schließt jetzt [1].

XSS in OWA: Outlook Web Access ist Microsofts Web-Frontend für Exchange. Cross-Site-Scripting heißt: der Angreifer schmuggelt fremden JavaScript-Code in deinen authentifizierten OWA-Tab, der dann mit deinem Login läuft. Praktisch: präparierte Mail kommt rein, du öffnest sie in OWA, sein Code klaut Session-Cookies, kopiert das Postfach oder legt Mail-Regeln an. Der Server wird nicht angefasst — die Lücke missbraucht den Browser, den du schon authentifiziert hattest.

Das Unangenehme: kein Patch im klassischen Sinn. Microsoft verteilt nur über den Exchange Emergency Mitigation Service eine URL-Rewrite-Maßnahme „M2.1.x" [2]. EMS ist auf Subscription Edition und 2019 per Default aktiv; bei 2016 + älteren CUs muss man händisch nachhelfen. Wer EMS aus Compliance-Gründen abgeschaltet hat (mehr Behörden, als man denkt), ist heute formal nicht mitigiert. CVSS 8,1 [3]. Für deutsche Mittelständler mit hybridem Exchange: jetzt das EMS-Log prüfen, nicht nächste Woche.

Gitea: vier Jahre lang private Container-Images für Anonymous-Reqs

Gitea, die selbstgehostete Alternative zu GitHub Enterprise/GitLab, hat am 27.05 CVE-2026-27771 bestätigt und mit 1.26.2 geschlossen. CVSS 8,2. Das Problem: die eingebaute Container-Registry respektierte das private-Flag in der UI, lieferte aber an den Registry-Endpoints (GET /v2/<name>/manifests/<tag>, /blobs/<digest>) Manifeste und Blob-Layer auch anonymen Requests aus. Repo-Name + Tag kennen reichte — ohne Auth, ohne Konto, ohne Token [4].

Container-Registry: Der Paketmanager für Docker-Images — wie npm für Node, nur für ganze fertige Server-Setups. Du baust lokal (docker build), pushst in die Registry, der Server zieht per docker pull. „Private" Registry = nur Auth — in den Image-Layern sitzen oft Source, .env-Files, eingebaute API-Keys, AWS-Credentials. Fehlt der Auth-Check, ist das schlimmer als ein offener Git-Repo: nicht nur Code, sondern das gebaute, lauffähige Geheimnis-Bündel.

Die Lücke existiert fast vier Jahre. Exposition: >30.000 Instanzen in 30+ Ländern, Top-5 China, USA, DE, FR, UK [5][6]. Forgejo hat den gleichen Registry-Code geerbt und ist mitbetroffen. Entdeckt im April von „NoScope", einem autonomen KI-Pentesting-Agenten — Vorgeschmack auf die nächsten 12 Monate [7]. Mitigation: Upgrade ≥1.26.2; Notnagel [service].REQUIRE_SIGNIN_VIEW = true [8]. Für Solo-Devs / Mittelständler mit BSI-Sovereignty-Reflex ist Gitea/Forgejo der naheliegende selbstgehostete Git+CI+Registry-Stack — wer Kunden bedient: Upgrade jetzt, Registry-Log-Audit (welche Anonym-IPs haben welche Manifests gepullt?), Secret-Rotation.

C# bekommt einen Rust-Anstrich — pointer types raus aus „unsafe"

Microsofts .NET-Team hat am 26.05 auf dem .NET-Blog die größte Memory-Safety-Reform für C# seit Jahren skizziert. PM Richard Lander erklärt in „Improving C# Memory Safety": das unsafe-Keyword wird umdefiniert — nach Rust-Vorbild, ohne den GC aufzugeben [9].

„unsafe" in einer GC-Sprache: C# räumt Speicher normalerweise per Garbage Collector auf. Sobald du rohe Pointer benutzt (für nativen C-Code, Hardware), hebelst du diesen Schutz aus — das macht das Keyword unsafe. Rust ist präziser: nur die wirklich gefährliche Operation (Pointer dereferenzieren, also tatsächlich auf die fremde Adresse zugreifen) muss in einen Unsafe-Block; mit dem Pointer zu rechnen ist safe.

Genau dorthin will Microsoft auch mit C#. Pointer-Typen + Pointer-Arithmetik werden künftig safe, nur das Dereferenzieren bleibt unsafe. Effekt: FFI- und Interop-Wrapper werden lesbarer, der gefährliche Schritt sticht heraus statt mit harmlosen Berechnungen im selben Block unterzugehen — dieselbe Begründung wie bei Rust [10].

Roadmap: Preview in .NET 11, Produktion in .NET 12 / C# 16, opt-in. Alter Code läuft unverändert; Runtime-Libraries werden aber opted-in sein [11]. Passt ins größere Bild: Microsoft übersetzt seit 2024 intern Windows-C/C++ nach Rust; jetzt zieht .NET die Schraube in der eigenen Sprache an. Für Java/Spring-Devs Beobachtungs-Punkt; mit .NET-/Unity-Berührung relevanter — das Modell wandert diesen Sommer in die .NET-11-Previews.

npm „staged publishing" — der menschliche 2FA-Riegel ist GA

GitHubs Antwort auf TanStack: seit 22.05 GA. Statt direktem Publish auf npmjs.com landet das Tarball in einer Stage-Queue — ein Maintainer muss mit 2FA explizit freigeben. Erfordert npm CLI 11.15.0+ und npm stage publish [17].

Trusted Publishing / OIDC: Eine CI (z. B. GitHub Actions) authentifiziert sich per kurzem OIDC-Token gegen npm und pusht automatisch ein neues Paket — ohne Klick. Bequem, aber: wenn der Angreifer die CI kapert (Pwn-Request, Cache-Poisoning), pusht die CI sein vergiftetes Paket mit gültiger Signatur. So lief TanStack. Staged Publishing kehrt das um: die CI lädt hoch, online geht es erst nach einem zweiten, menschlichen 2FA-Schritt.

„Proof of presence" auch für CI/CD-Publishes. Caveat: verteidigt gegen den TanStack-Pfad (Token-Klau), nicht gegen den Nx-Pfad (kompromittiertes Maintainer-Konto). Und greift nur, wenn Maintainer es aktivieren — Defaults bleiben vorerst.

Es ist Freitag, der 29. Mai, und der Tag bündelt zwei sehr unterschiedliche Sorten von „seit Jahren offen". Erstens läuft heute die CISA-Frist für Microsofts Exchange-OWA-Zero-Day ab, CVE 26-42897. Den Patch dazu gibt es bis heute nicht — Microsoft hat lediglich eine temporäre URL-Rewrite-Mitigation namens M-2-Punkt-1 über den Emergency Mitigation Service verteilt. Wer als deutscher Mittelständler hybrid Exchange fährt, prüft heute das EMS-Log, nicht nächste Woche.

Zweitens hat Gitea, die selbstgehostete Git-Alternative, am Mittwoch eine Lücke geschlossen, die fast vier Jahre lang private Container-Images für anonyme Web-Requests ausgeliefert hat. Die eingebaute Registry hat das Privacy-Flag in der UI respektiert, aber an den Image-Endpoints nicht durchgesetzt. Wer Repo-Name und Tag kannte, bekam die fertigen Images samt eingebackener Secrets — ohne Konto, ohne Token. Sicherheitsforscher schätzen rund dreißigtausend internet-exponierte Instanzen, schwerpunktmäßig in China, den USA und Deutschland. Auch der Fork Forgejo ist betroffen. Fix ist Gitea-Version eins Punkt sechsundzwanzig Punkt zwei. Bemerkenswertes Detail: gefunden hat den Bug ein autonomer KI-gestützter Pentesting-Agent namens NoScope, im April dieses Jahres.

Drittens, deutlich ruhiger, aber langfristig groß: Microsofts .NET-Team hat auf dem dotnet-Blog die größte C-Sharp-Memory-Safety-Reform seit Jahren skizziert. Das unsafe-Schlüsselwort wird Rust-inspiriert neu definiert. Mit Pointern zu rechnen wird künftig safe, nur das Dereferenzieren, also der tatsächliche Speicherzugriff, braucht weiterhin einen unsafe-Block. Preview in .NET 11, Produktion in .NET 12 und C-Sharp 16. Opt-in — alter Code läuft unverändert weiter.

Oracle hat gestern die erste monatliche Critical Security Patch Update ausgeliefert, fünfunddreißig Patches, neuer Monatsrhythmus auf den dritten Dienstag. Begründung: KI-gestützte Schwachstellen-Discovery hat das alte Quartalsfenster überholt. Wer eine Oracle-DB on-prem fährt, plant jetzt für zwölf statt vier Patch-Wochenenden im Jahr.

Ein CI-Token darf weiterhin pushen, aber das Paket geht erst online, wenn ein Mensch mit Zwei-Faktor-Authentifizierung freigibt. Voraussetzung ist npm-CLI elf Punkt fünfzehn.
