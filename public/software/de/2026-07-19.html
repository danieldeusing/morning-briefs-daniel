Drei npm-Einbrüche, drei verschiedene Türen

Sonntag, kein frisches Advisory in den beobachteten Quellen — also die Bilanz einer Woche, die es in sich hatte. In elf Tagen wurden drei npm-Pakete gekapert: @injectivelabs/sdk-ts (08.07), jscrambler (11.07), @asyncapi/* (14.07). Einzeln liefen die hier schon durch. Zusammen ergeben sie ein Muster, das mir wichtiger erscheint als jeder Einzelfall: jeder benutzte einen anderen Ausführungs-Pfad. Und ausgerechnet am 08.07, dem Tag des ersten Einbruchs, hat npm v12 allowScripts standardmäßig abgeschaltet — eine Abwehr, die genau einen der drei Wege blockiert hätte. Kein Argument gegen die Änderung; eines dagegen, sie für erledigt zu halten. Dazu: warum der „Rekord-Patchday" zu rund drei Vierteln aus Chromium besteht, ein SharePoint-Score, bei dem Microsoft 5,3 und die NVD 9,8 sagen, und die NIS-2-Registrierung in zwölf Tagen.

  • Infra Global npm v12 ist seit 08.07 draußen: allowScripts standardmäßig aus, --allow-git und --allow-remote auf none. Blockiert Install-Skripte — nicht Code, der beim import läuft.

  • Sec USA SharePoint-Score, zwei Meinungen: CVE-2026-56164 — Microsoft rechnet CVSS 5,3 (Medium), die NVD 9,8 (Critical). Gleicher Angriffsvektor, komplett andere Wirkungs-Annahme. Steht seit 14.07 im CISA-KEV.
  • Sec Global Der Rekord ist zum Teil Buchhaltung: Von den 621 CVEs im Juli-Patchday entfallen laut ZDI rund 480 auf Chromium/Edge. Bleiben ~141 echte Microsoft-Bugs, davon 63 Critical. Outlets melden 569, 570, 621 und 622 — alle rechnen anders.

Drei Einbrüche, drei verschiedene Türen

Drei gekaperte npm-Pakete in elf Tagen sind unangenehm, aber nicht ungewöhnlich — 2026 läuft das im Wochentakt. Interessant wird es nebeneinandergelegt: keiner der drei Angreifer benutzte denselben Ausführungs-Pfad. Wer sich auf eine einzelne Abwehrmaßnahme verlässt, hätte diese Woche zwei Mal danebengegriffen.

Ausführungs-Pfad: Damit ein bösartiges Paket Schaden anrichtet, muss sein Code irgendwann laufen. Dafür gibt es mehrere Momente — beim Installieren, beim Importieren im eigenen Code, beim Start eines CLI-Kommandos. Wie ein Einbrecher, der zwischen Haustür, Kellerfenster und Garagentor wählt: Ein zusätzliches Schloss an der Haustür hilft nur, wenn er die Haustür nimmt.

08.07 — Injective SDK: Keys im HTTP-Header

Über einen kompromittierten GitHub-Zugang wurde @injectivelabs/sdk-ts@1.20.21 veröffentlicht (mehrere Zehntausend Downloads/Woche — ~50.000 laut The Hacker News, >30.000 laut Datadog). Wichtig für die Abwehr: Der Code lief weder als Install-Skript noch beim import. Datadog zeigt zwei Hooks direkt in den Key-Ableitungs-Funktionen PrivateKey.fromMnemonic() und PrivateKey.fromHex() — sie feuern erst zur Laufzeit, wenn eine Wallet geladen wird. Abgegriffene Mnemonics und Private Keys gingen base64-kodiert im X-Request-Id-Header ausgehender Anfragen hinaus — ein Feld, in das kaum jemand schaut, weil es normalerweise nur eine Korrelations-ID trägt. Injective fand den Commit und lieferte 1.20.23 nach rund 49 Minuten aus.[1][2][24]

11.07 — jscrambler: der Angreifer wechselt mitten im Angriff die Tür

Die bösartige jscrambler@8.14.0 ging um 16:12:40 BST live, Sockets Research-Team sah sie sechs Minuten später; rund 1.500 Downloads. Fünf Versionen waren betroffen (8.14.0, 8.16.0, 8.17.0, 8.18.0, 8.20.0), sauber ist 8.22.0. Der bemerkenswerte Teil steckt in der Mitte: Die frühen Versionen liefen über einen preinstall-Hook auf dist/setup.js. Die späten ließen das Install-Skript weg und injizierten eine selbstausführende Funktion in den Paket-Hauptcode — die beim import oder CLI-Aufruf zündet und an jedem Scanner vorbeiläuft, der nur Install-Skripte prüft.[3][4]

14.07 — AsyncAPI: Import-Zeit, sauber getaktet

Fünf kompromittierte Versionen über vier Pakete (@asyncapi/specs 6.11.2-alpha.1 und 6.11.2, generator@3.3.1, generator-components@0.7.1, generator-helpers@1.1.1). Der Einstieg war kein vertrauter Maintainer, sondern ein externer Angreifer über einen CI-Pwn-Request: Ein Workflow (manual-netlify-preview.yml) lief auf pull_request_target, checkte aber den Code des Pull Requests aus — also fremder Code im Kontext des Basis-Repos, mit Zugriff auf die Secrets. Über PR #2155 (05:08 UTC, 14.07) fiel so ein Token des asyncapi-bot-Servicekontos.[25][26] Microsofts Rekonstruktion liest sich wie ein Release-Plan: erste Neu-Veröffentlichung gegen 07:10 UTC, die stabile Version mit byte-identischer Payload um 08:30:09 UTC, erster nachweislicher Download um 08:49:22 UTC. Auch hier läuft der Code beim Import; npm install --ignore-scripts war wirkungslos.[5]

npm v12 hat die richtige Tür zugemacht — es sind nur mehrere

Am 08.07, demselben Tag, an dem der Injective-Backdoor live ging, ist npm v12 erschienen. Die wichtigste Änderung: allowScripts steht standardmäßig auf aus. npm install führt preinstall, install und postinstall aus Abhängigkeiten nicht mehr aus, solange man es nicht ausdrücklich erlaubt — inklusive der impliziten node-gyp-Rebuilds. Auch --allow-git und --allow-remote stehen jetzt auf none.[6]

Richtige Entscheidung, spät gekommen. Aber die Wochenbilanz ist unbequem: Der neue Default hätte einen Teil eines einzigen Vorfalls gestoppt — die frühen jscrambler-Versionen. Injective und AsyncAPI liefen komplett an Install-Skripten vorbei, und der jscrambler-Angreifer wechselte mitten im Angriff selbst dorthin. Die Schutzmaßnahme wurde nicht überwunden; die Angreifer sind umgezogen.

Install-Skripte blockiert → Payload wandert in den Hauptcode → Ausführung beim import

Die Konsequenz ist unspektakulär, aber konkret: --ignore-scripts ist Grundhygiene, keine Verteidigung. Was hilft, ist die Zeit-Achse — Lockfiles mit exakten Versionen und eine Karenzzeit, bevor man ein frisches Release übernimmt. Sechs Minuten Entdeckungszeit nutzen nur dem, der nicht in den ersten sechs Minuten installiert.

Für Daniels Stack Der Angular-/TypeScript-Zweig ist der exponierte Teil: Ein npm ci in einer GitHub-Action zieht denselben Abhängigkeits-Baum wie lokal, nur unbeaufsichtigt und oft mit Registry-Tokens im Environment. Die jscrambler-Malware zielte auf AWS-/GCP-/Azure-Credentials, Krypto-Wallets und Konfigurationen von KI-Coding-Assistenten[7] — genau auf das, was auf einer Dev-Maschine herumliegt. Maven und pip haben strukturell dasselbe Problem.

SharePoint: Microsoft sagt 5,3, die NVD sagt 9,8

Update seit 16.07 Im Brief vom Donnerstag stand, der Papier-Score für CVE-2026-56164 liege mit 5,3 „eher im Mittelfeld". Das war Microsofts Zahl. Die NVD hat inzwischen ihre eigene Bewertung veröffentlicht — und kommt auf 9,8, Critical. Das ist keine Rundungsdifferenz, das sind zwei verschiedene Welten.[8]

Beide sind sich beim Angriffsvektor völlig einig: AV:N/AC:L/PR:N/UI:N — über das Netz, geringe Komplexität, keine Rechte, keine Nutzer-Interaktion nötig. Der Streit liegt ausschließlich bei der Wirkung. Microsoft setzt C:N/I:L/A:N an: keine Vertraulichkeits-Verletzung, geringe Integritäts-Auswirkung. Die NVD setzt C:H/I:H/A:H — alles hoch.

CVSS: Ein Punktesystem von 0 bis 10, das eine Schwachstelle nach Angriffsweg und Schadenshöhe bewertet. Wie ein Restaurant-Guide: Die Sterne stehen fest, aber wer sie vergibt, entscheidet mit. Der Hersteller bewertet oft die Lücke in seiner eigenen Komponente, ein externer Analyst das, was ein Angreifer damit am Ende erreicht.

Praktisch ist der Streit egal, und das ist der Punkt. Die Lücke steht seit dem 14.07 im CISA-KEV-Katalog — am selben Tag, an dem der Patch erschien —, US-Behörden mussten bis zum 17.07 nachziehen. Betroffen sind SharePoint Server 2019 (< 16.0.10417.20175), Subscription Edition (< 16.0.19725.20434) und Enterprise Server 2016 (< 16.0.5561.1001).[9] Die Zero Day Initiative formuliert es trockener, als ich es könnte: „patch it now and worry about the score later".[10]

Meine Lesart: Ein KEV-Eintrag ist ein empirischer Befund, ein CVSS-Score eine Modell-Annahme. Wo beide auseinanderlaufen, gewinnt der Befund. Der Score taugt zum Sortieren, nicht zum Entscheiden.

Der „Rekord-Patchday" ist zu drei Vierteln Chromium

Der Juli-Patchday wurde als größter aller Zeiten durchs Netz gereicht — und die Schlagzeilen widersprechen sich munter: 622 bei The Hacker News, 621 bei Security Affairs, 570 bei BleepingComputer, 569 in Tenables eigener Zählung. Wer eine einzelne dieser Zahlen als Fakt zitiert, zitiert eine Zählregel.[11][12]

Die Zero Day Initiative löst es auf: Von den 621 CVEs entfallen rund 480 auf Chromium und das Chromium-basierte Edge — Bugs, die eine andere CNA vergeben hat und die Microsoft nur mitführt. Bleiben etwa 141 genuine Microsoft-Lücken. Nach Schweregrad: 63 Critical, sechs Moderate, eine Low, der Rest Important.[10]

CNA: Die Stelle, die eine CVE-Nummer vergeben darf — meist der Hersteller selbst. Wie beim Einwohnermeldeamt: Zieht jemand von einer Gemeinde in die andere, taucht er womöglich in beiden Statistiken auf. Übernimmt Microsoft eine Chromium-Lücke in sein eigenes Edge, erscheint sie in Microsofts Update-Guide, obwohl Google sie vergeben hat.

Das entschärft die Zahl nicht — 141 Bugs mit 63 Critical sind reichlich, und die ZDI schreibt zu Recht, der Jahresstand liege über jedem Gesamtjahr der letzten zwei Jahrzehnte. Aber wer ein Patch-Fenster begründen muss, kommt mit „141 eigene, davon 63 kritisch, zwei aktiv ausgenutzt" weiter als mit einer Zahl, bei der die Fachpresse um fünfzig auseinanderliegt.

AD FS: der zweite Zero-Day, der leiser blieb

Neben SharePoint stand am 14.07 CVE-2026-56155 auf der Liste — Rechte-Eskalation in Active Directory Federation Services, CVSS 7,8 (AV:L/AC:L/PR:L/UI:N), aktiv ausgenutzt und von Microsoft bestätigt. Ursache ist eine zu grob gesetzte ACL auf den DKM-Schlüsseln; wer sie ausnutzt, landet auf Administrator-Rechten.[13] Der lokale Angriffsvektor macht sie unauffälliger als die SharePoint-Lücke — sie ist der zweite Schritt einer Kette, nicht der erste. Läuft nicht in Daniels Stack, gehört aber in jedes Windows-Domänen-Gespräch dieser Woche.

Browser-Woche: Firefox mit PoC, Chrome mit Ozone

Firefox 152.0.6 kam am 14.07 mit dem Advisory MFSA 2026-67 und schließt zwei als Critical eingestufte Lücken, CVE-2026-15718 und CVE-2026-15719. Für beide existiert öffentlich zugänglicher Exploit-Code; Mozilla ist nach eigener Aussage keine aktive Ausnutzung bekannt.[14][15] Ein Teil der Fachpresse hat das als „Zero-Day" überschrieben — das ist es nicht, solange niemand Nutzung in freier Wildbahn belegt. Veröffentlichter PoC heißt „die Uhr läuft", nicht „es ist zu spät".

Google hat am 16.07 ein Stable-Update für Desktop veröffentlicht: 150.0.7871.124/.125 für Windows und macOS, 150.0.7871.124 für Linux, mit 15 Sicherheitsfixes. Zwei davon sind kritische Use-after-free-Fehler in Ozone, Chromes Grafik-/Fenster-Abstraktion unter Linux: CVE-2026-15764 und CVE-2026-15765.[16][17]

Use-after-free: Das Programm gibt einen Speicherbereich frei, greift aber danach noch einmal darauf zu. Wie wenn du deinen Schlüssel im Hotel abgibst, das Zimmer neu vergeben wird — und du trotzdem noch aufschließt. Wer schnell genug neu einzieht, bestimmt, was dort steht, wenn das Programm nachschaut.

Beide Updates sind Neustart-Sachen, kein Drama — und ein guter Teil der 480 Chromium-CVEs aus dem Absatz oben stammt aus genau dieser Pipeline.

NIS-2: zwölf Tage, und die „Frist" ist keine

Der härteste Termin für einen deutschen Leser steht am 31.07.2026. Bis dahin duldet das BSI verspätete Registrierungen nach dem NIS-2-Umsetzungsgesetz — der reguläre Stichtag war der 06.03.2026. Der Stand ist ernüchternd: Von rund 29.500 erwarteten Einrichtungen hatten sich bis zum ursprünglichen Termin weniger als die Hälfte registriert, bis Ende Mai waren es rund 18.500.[18]

Der Teil, der in der Berichterstattung meist untergeht: Diese Nachfrist ist keine gesetzliche Fristverlängerung, sondern eine behördliche Duldung. Die Pflicht besteht seit März — wer sie versäumt hat, ist bereits säumig und verlässt sich darauf, dass das BSI nicht sanktioniert. Juristisch etwas anderes als „ich habe noch Zeit bis Ende Juli".

NIS-2: Eine EU-Richtlinie, die Unternehmen in kritischen Sektoren zu Mindeststandards bei IT-Sicherheit und zur Meldung von Vorfällen verpflichtet — in Deutschland seit Dezember 2025 in Kraft. Der erste Schritt ist banal: sich beim BSI melden und sagen „uns gibt es". Wie die Anmeldung beim Einwohnermeldeamt — nichts passiert dadurch, aber wer es nicht tut, hat später ein Problem.

Der Bußgeldrahmen reicht bis 500.000 €, bei bestimmten Verstößen bis 10 Mio. €, dazu kommt persönliche Haftung der Geschäftsleitung.[18] Auch für nicht direkt Betroffene relevant: Registrierte Einrichtungen reichen ihre Sicherheitsanforderungen die Lieferkette hinunter — der nächste Ausschreibungs-Fragebogen fragt danach.

CRA: der Termin nach dem Termin

Bewusst nicht in der Sieben-Tage-Spur, aber der nächste Block nach NIS-2: Der Cyber Resilience Act macht am 11.09.2026 die Meldepflichten scharf — 24 Stunden Frühwarnung, 72 Stunden Vollmeldung für aktiv ausgenutzte Schwachstellen und schwere Vorfälle. Vollständige Anwendung am 11.12.2027.[19]

Die Kommission pflegt dazu eine FAQ-Seite, die laufend nachgezogen wird — das zuletzt auf der Seite selbst ausgewiesene Datum ist der 23.04.2026; ein in Such-Snippets kursierendes Juli-Datum konnte ich am Dokument nicht bestätigen. Vor September lohnt dort ein erneuter Blick, denn genau diese FAQ definiert, wer als „Hersteller" gilt — für einen Ein-Personen-Softwarebetrieb die entscheidende Abgrenzungsfrage.

Stack-Wartung, kurz

PostgreSQL 19 Beta 2 (16.07)

Die Aufräum-Beta, Schwerpunkt auf den beiden großen Neuerungen des Zyklus: mehrere Fixes für das SQL/PGQ-Property-Graph-Feature und die temporale Syntax FOR PORTION OF. Dazu eine Regression in vacuumdb --analyze-in-stages bei partitionierten Tabellen und ein Autovacuum-Bug, bei dem der Multixact-Age-Score unendlich werden konnte.[20] GA ist für Sept./Okt. angepeilt — also: testen, nicht migrieren.

Angular 22.0.7 / 22.1.0-next.6 (16.07)

Wartungs-Release mit einem Sicherheits-Anteil: ReDoS-Prävention bei der Regex-Prüfung von Sourcemap-URLs in platform-browser. Die Vorgänger-Version 22.0.6 vom 08.07 brachte die „safe property lookups"-Härtung im Router.[21]

ReDoS: Ein regulärer Ausdruck, der bei einer geschickt gebauten Eingabe exponentiell viele Kombinationen durchprobiert und dabei einen CPU-Kern blockiert. Wie ein Türsteher, der bei jedem Gast alle Gästelisten-Kombinationen durchgeht: Ein Name mit vielen Bindestrichen legt den Eingang lahm.

Tailscale 1.98.9 (14.07)

Sechs Sicherheitspunkte, TS-2026-004 bis -009: Unix-Socket-Forwarding in Tailscale SSH respektiert jetzt Symlink-Rechte, der Serve-Socket-Proxy ist auf root beschränkt, Serve und Funnel fangen einen Pfad-Walk ab, der bei nicht-absoluten Pfaden eine CPU festnageln konnte, und SSH weist rein numerische sowie mit Bindestrich beginnende Benutzernamen zurück.[22] Für ein Tailnet mit aktivem Serve/Funnel die relevanteste der drei Zeilen hier.

Es ist Sonntag, der neunzehnte Juli. In den beobachteten Quellen gab es über das Wochenende kein neues Advisory, also gibt es heute keine Eilmeldung, sondern die Bilanz einer Woche, in der drei npm-Pakete gekapert wurden.

Am achten Juli traf es das Injective-SDK. Ein kompromittierter GitHub-Zugang, eine vergiftete Version, mehrere Zehntausend Downloads pro Woche. Der Schadcode lief weder beim Installieren noch beim Importieren, sondern hing sich an die Funktionen, die den privaten Schlüssel ableiten — er feuerte also erst, wenn jemand eine Wallet lud. Dann stahl er die Schlüssel und schmuggelte sie in einem HTTP-Header namens X-Request-Id nach draußen — einem Feld, in das normalerweise niemand hineinschaut. Das Team fand den Commit und lieferte nach knapp fünfzig Minuten eine saubere Version nach.

Am elften Juli war jscrambler dran. Socket entdeckte die bösartige Version sechs Minuten nach der Veröffentlichung. Spannend ist, was der Angreifer danach tat: Die ersten Versionen liefen über ein Installations-Skript. Als das auffiel, ließ er das Skript weg und versteckte den Schadcode direkt im Hauptcode des Pakets, wo er beim Importieren zündet. Er hat den Schutz nicht geknackt, er ist einfach umgezogen.

Am vierzehnten Juli folgte AsyncAPI. Der Angreifer kam von außen: Er schickte einen Pull Request, den eine schlecht konfigurierte GitHub-Action mitsamt Zugriff auf die Geheimnisse des Projekts ausführte. Microsoft hat die Zeitachse rekonstruiert: Die stabile vergiftete Version ging um halb neun UTC live, neunzehn Minuten später lud sie das erste Projekt herunter. Auch hier lief der Code beim Importieren, nicht beim Installieren.

Und genau am achten Juli, dem Tag des ersten Einbruchs, ist npm Version zwölf erschienen. Die wichtigste Neuerung: Installations-Skripte aus Abhängigkeiten laufen nicht mehr automatisch. Das ist die richtige Entscheidung. Aber von den drei Vorfällen hätte sie nur einen Teil eines einzigen gestoppt. Die Lehre für die eigene Praxis: Das Blockieren von Installations-Skripten ist Grundhygiene, keine Verteidigung. Was hilft, ist Zeit — feste Versionen im Lockfile und ein paar Tage Abstand, bevor man ein frisches Release übernimmt.

Zweite Geschichte, ein Streit um eine Zahl. Für die SharePoint-Lücke mit der Kennung CVE zwei tausend sechsundzwanzig, sechsundfünfzig tausend einhundertvierundsechzig vergibt Microsoft den Schweregrad fünf Komma drei, also Mittelfeld. Die amerikanische Schwachstellen-Datenbank NVD kommt auf neun Komma acht, also kritisch. Beim Angriffsweg sind sich beide einig, sie streiten nur über den Schaden. Die Zero Day Initiative bringt es auf den Punkt: erst patchen, dann über den Score nachdenken. Die Lücke wird aktiv ausgenutzt, und dieser Befund schlägt jede Modellrechnung.

Dazu passt die dritte Geschichte. Der Juli-Patchday von Microsoft wurde als Rekord gefeiert, mit Zahlen zwischen fünfhundertneunundsechzig und sechshundertzweiundzwanzig, je nachdem, wen man liest.
