Security & supply chain

Cisco SD-WAN CVE-2026-20182 — CVSS 10.0, in the wild

Rare enough that CISA adds an edge auth bypass with the CVSS ceiling score to the KEV catalog — and this one comes with explicit attribution. CVE-2026-20182 in Cisco Catalyst SD-WAN Controller and Manager lets an unauthenticated remote attacker obtain administrator privileges. Cisco itself attributes exploitation "with high confidence" to the UAT-8616 cluster — the same group that weaponized CVE-2026-20127. CISA added the CVE to KEV on 2026-05-14; the FCEB patch deadline was 2026-05-17 [1][2].

Daniel's stack isn't directly affected — but DACH clients with Cisco SD-WAN edges definitely are. The more interesting takeaway is topological: SD-WAN controllers tend to sit in the management VLAN with weak segmentation to production. When the controller falls, in the worst case it takes the route control of the entire site mesh with it. CVE-2026-20133, -20128, -20122 from the April KEV batch are in the same path — attackers are chaining them [3].

pgvector 0.8.2 — integer underflow in parallel HNSW

pgvector has graduated from weekend hobby to default vector index in Postgres stacks — anyone running their own RAG has it in the cluster. CVE-2026-3172 sits in the parallel HNSW index build: an integer underflow produces an out-of-bounds read into backend memory. An authenticated DB user can use that to leak data from other relations, or crash the server. CVSS 8.1. Affected 0.6.0–0.8.1, fixed in 0.8.2.

HNSW (Hierarchical Navigable Small Worlds): Imagine looking up a word in a book. Instead of reading every page, you have an index that gets you roughly to the right section — and there, local signposts to neighbouring words. HNSW is exactly that for vector search: multiple layers, the top one coarse (few nodes, long jumps), the bottom one fine. Result: approximate nearest neighbour in milliseconds instead of seconds. The parallel variant builds the index across multiple workers — and in pgvector 0.6+ the size arithmetic was going through an unsigned integer that wrapped to negative.

Practically: any Postgres cluster with pgvector active and multi-tenant DB users (i.e. the default for SaaS) is exposed. Upgrading to 0.8.2 is a this-week task; cross-checked against NVD [1][2].

Spring AI 1.0.7 / 1.1.6 / 2.0.0-M6 — three memory CVEs at once

If anyone is putting Spring AI into a production path seriously, the 2026-05-08 release wave is required reading. Three CVEs close gaps that arise structurally in memory-based chat agents when the API is used carelessly. CVE-2026-41712: the default `DEFAULT_CONVERSATION_ID` meant that chat memory without an explicit user scope leaked across sessions — the classic "user A sees user B's answer" in PoC stacks. CVE-2026-41713: an attacker can craft prompts that land in conversation memory and get interpreted on a later turn as "instruction" — memory poisoning. CVE-2026-41705: filter injection in the Milvus vector store connector.

The fix isn't just an upgrade — the PromptChatMemoryAdvisor API has been deprecated; the replacement forces an explicit conversation-ID parameter. Anyone with Spring AI code from Q1 2026 fixes not just the build but the threat model. Spring AI 2.0 is announced for 2026-05-28 [1][2][3].

Threat-model note Memory poisoning isn't a code bug; it's an architecture assumption. Anyone using chat memory as storage for "what the user said" is treating user input as trusted system prompt. That breaks systematically as soon as a tool-use in the same turn reads the memory. The fix API enforces conversation-ID scoping — that fixes cross-user leak. The instruction-from-memory problem remains an architecture question.

Firefox 151 — Container Style Queries cross-browser-stable

Mozilla rolls Firefox 151 as stable today, and the most important item isn't in the top marketing line: Container Style Queries are now stable in a third major browser. What does that mean? A component can react to the value of a custom property on its container — not only its size. Until now, theming had to be modeled via ancestor selectors (".dark .card" styles). From today, "the container has `--theme: dark` set" is enough — theme switching without JS, without class toggling, locally per subtree. Also: `CSSContainerRule.conditions` as a new property, multiple conditions per rule.

Three more items that matter in production: the Document Picture-in-Picture API (DOM content in a floating always-on-top window — Chrome has had it since ~24, Firefox is catching up), Fullscreen Keyboard Lock (browser games can capture Ctrl-W), and local profile backups on Linux and macOS with cross-platform restore. Firefox 151 is also the first stable build that, per Mozilla, carries preparation for the new "Firefox Home" — the larger UI redesign lands with 152 [1][2].

TC39 114 in Amsterdam — Temporal lands for real

Three days of TC39 plenary in Amsterdam starting today, hosted by JetBrains. The biggest movement actually happened before the meeting: Temporal advanced to Stage 4 at the March 113 meeting — meaning it lands in ECMAScript 2026 and implementations can ship to production. Firefox and Safari have implementations in the browser already; V8 is the remaining layer. Source Phase Imports for static worker-module analysis is progressing. Concrete stage advances from the current 114 meeting will be published in the notes over the coming days [1].

Temporal: JavaScript's Date object is one of the most notorious pieces of the language's standard library — months numbered from 0, mutations instead of immutability, no clean timezone handling. Temporal replaces it: immutable values, clean separation between Instant (a point in time), PlainDate (a calendar day), and ZonedDateTime (point in time + timezone). JavaScript code now writes `Temporal.PlainDate.from('2026-05-19').dayOfWeek` instead of `new Date('2026-05-19').getDay() + 1`.

M&A & OSS politics

Anthropic ↔ Stainless — the SDK workshop falls

Anthropic officially confirmed the Stainless deal at >$300M on 2026-05-18. Stainless is the small NYC shop (founded 2022 by Stripe alumnus Alex Rattray) that has, for three years, automatically generated and maintained the official SDKs for OpenAI, Google, Cloudflare, and Runway from their OpenAPI specs — Python, TypeScript, Go, Java. Anthropic is winding down the hosted Stainless products and keeping the tech in-house. Existing customers keep their generated SDKs and full modification rights.

This is the sharpest platform move in the AI world in May 2026 — and not one that makes "new model" headlines. Reading: Anthropic isn't buying the competition out; it's buying the shared infrastructure of the competition. OpenAI and Google have, as of today, an internal tooling hole in the SDK pipeline — three to six months until replacement solutions are in place. For SDK consumers (i.e. all developers) the layer becomes slowly more fragmented, not more uniform [2].

SDK lock-in effect Stainless generates SDKs from OpenAPI specs — the specs still belong to the respective vendors, so OpenAI/Google remain fully consumable API-wise. What's missing is the auto-updated cross-language SDK layer. If they build it themselves, it takes months; if they outsource to Speakeasy / orval / openapi-generator, they lose the tight coupling to proprietary API features (streaming conventions, tool-call encoding, etc.) that Stainless modeled along the way.

Commentary — the layer under the model

Three May moves show the same pattern: the competition is no longer playing only on the model layer. First: Anthropic buys the competition's SDK generator workshop — the layer that no one has on a conference slide because it produces no benchmark score. Second: Spring AI abandons its memory API default in the same breath, because the data-flow layer under the LLM call is the actual risk layer, not the model itself. Third: Google I/O tonight will talk a lot about Gemini, but the commercially decisive slide is the one with the MCP logo — whoever controls the tool schema controls what an agent can do, not just what it says.

What does that mean for Daniel's stack DNA? Java-Spring-Postgres-Caddy stacks have, in the AI wave so far, the virtue that the model stays an externally called service, not an architectural component. Anyone building Spring AI into the stack has had a new API to learn since yesterday, three CVEs to review, and a threat model that wasn't there before. Pragmatic path: HTTP calls into the AI provider, conversation state explicitly in your own Postgres schema, no memory-advisor magic. That isn't "behind the times" — that's the architecture that survives an audit.
