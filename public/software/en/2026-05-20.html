Mini Shai-Hulud — the worm that forges real signatures

This is the most consequential story of the week, and it makes no keynote. On May 11 (between 19:20 and 19:26 UTC) a self-replicating worm published 84 malicious artifacts across 42 @tanstack/* npm packages — published by TanStack's own legitimate release pipeline under its trusted OIDC identity, after attacker code hijacked the runner mid-workflow. Within 48 hours the campaign grew to 172 packages across 403 versions on npm and PyPI — among them Mistral, Guardrails AI, UiPath, OpenSearch, Bitwarden CLI, SAP. CVE-2026-45321, CVSS 9.6, confirmed by NVD [1][2].

SLSA provenance: Like a sealed delivery slip that cryptographically proves a package was really built by the genuine CI pipeline — the industry's usual answer to supply-chain attacks. The scary part: the worm didn't crack the signature, it hijacked the pipeline itself so it signed the malware. Sigstore verified the build process correctly — what SLSA never guaranteed is that the code being built is safe. First npm attack with valid SLSA build-L3 provenance.

The worm steals GitHub PATs, npm tokens, SSH keys, cloud and K8s credentials, plus .env files, persists via VS Code and Claude Code hooks, and spreads itself through the victims' repos. Attributed to "TeamPCP," the group said to have compromised the Trivy scanner in March and the Bitwarden CLI in April. Fallout: TanStack is drawing the consequence and weighing making PRs invitation-only going forward — a drastic narrowing of contribution that shows how deep the incident cuts [3][4].

Check today Daniel's stack is exposed (GitHub Actions + npm + Claude Code). Three checks: (1) grep workflows for pull_request_target with checkout of PR code, and pin Actions by SHA instead of by tag. (2) Review ~/.claude and VS Code hooks for unknown entries. (3) Run npm audit signatures + diff lockfiles against the compromised versions, then rotate PATs/npm tokens if workflows ran during the window. SLSA verification alone would not have protected here.

Linux "Copy Fail" — container isolation breaks, deadline already gone

Right in Daniel's Docker/Linux path: CVE-2026-31431 ("Copy Fail," CVSS 7.8) sits in the kernel's authencesn crypto template — an unprivileged user can write four controlled bytes into the page cache of any readable file and escalate that to root. What's unsettling is the triviality: a 732-byte Python script, running unchanged on every major distribution since 2017. The bug is nine years old. Per Theori it also works as a container-escape primitive and hits K8s nodes, because the page cache is shared host-wide. In KEV since May 1, the FCEB deadline (May 15) has passed. Fix in kernel 6.18.22 / 6.19.12 / 7.0 [1][2].

Exchange CVE-2026-42897 — and the scoring dispute

CISA added an on-prem Exchange XSS to the KEV catalog on May 14 (FCEB deadline May 29): a crafted email can run arbitrary JavaScript in the OWA browser context. What's interesting is the scoring conflict: NVD rates it 6.1 MEDIUM (scope changed, C:L/I:L), Microsoft itself 8.1 HIGH (C:H/I:H). Cite both values, don't claim either as "the" score — exactly the cross-check the research discipline demands. Exchange 2016/2019 are affected; on-prem Exchange is still plentiful in the DACH mid-market [1].

OpenJDK — JDK 27 EA and post-quantum for TLS

The active early-access line is JDK 27 (build ~17); RDP1 (feature freeze) forks on Jun 4, GA in September. The only officially targeted JEP so far: JEP 527 (post-quantum hybrid key exchange for TLS 1.3) — three hybrid schemes (X25519MLKEM768, SecP256r1MLKEM768, SecP384r1MLKEM1024), active since EA build 6. It builds on JEP 496 (ML-KEM) from JDK 24 [1][2].

Post-quantum hybrid key exchange: During the TLS handshake, browser and server agree on a secret session key. "Hybrid" means they use two methods at once — a classical one and a quantum-resistant one. Even if a future quantum computer cracks the one, the connection stays secure. Same motive as the OpenSSF Sigstore topic below — the industry hardening its crypto before the quantum threat is real.

CRA — reporting obligations from September, retroactive

The Cyber Resilience Act reporting obligations take effect from Sep 11, 2026: actively exploited flaws and severe incidents must be reported — early warning within 24h, full notification within 72h, final report no later than 14 days after a fix is available. The catch for Daniel and anyone shipping software: the obligation applies retroactively to every product placed on the market before Dec 11, 2027 (Art. 69(3)). A product shipped in 2019 that's still in use and has an actively exploited flaw is covered. Full CRA applicability then follows on Dec 11, 2027 [1].

Commentary — trust is the supply chain

The Shai-Hulud worm is so unpleasant precisely because it didn't exploit a hole — it forged the proof of trust itself: valid SLSA provenance, the cryptographic receipt that five years of supply-chain security has been sold as the answer. The lesson isn't "SLSA is broken," it's subtler: SLSA proves that the pipeline built it, never what it built. Hijack the pipeline and you inherit its trust. That's exactly why the quantum hardening of Sigstore (tomorrow at OpenSSF) and JEP 527 in the JDK are only half the battle — they protect the signature, not the build context around it.

For Daniel's stack DNA — GitHub Actions, npm, Java/Spring, Postgres, Caddy/Tailscale, Claude Code — the pragmatic takeaway: scope CI by the same least-privilege principle as everything else. Banish pull_request_target from the workflows or fence it hard, pin Actions by SHA, version and review hook directories, keep tokens short-lived. Notable here: the worm persisted via the same agent-tooling hooks (VS Code, Claude Code) that are supposed to bring productivity — the tooling layer is also a persistence vector. (This week's AI agent launches themselves — Google I/O — belong topically in the ai-dev brief; here only their attack-surface side counts.) Three control points for today: don't miss the Drupal window tonight, grep workflow triggers for pull_request_target, and check the Postgres clusters against the May 14 patch level.
