Docker — three ways from container to host root via docker cp

This is the must-patch of the week, and it sits right in Daniel's stack. Docker Engine 29.5.1 (May 18) closes three flaws that share one core: a malicious container can break out during the innocuous-looking docker cp and reach the host filesystem, because the copy logic runs with host-root privileges. CVE-2026-41567: the decompression binaries (xz, unpigz) are resolved via the PATH inside the container — a planted binary then runs as host root. CVE-2026-42306: a TOCTOU race redirects a bind mount to an arbitrary host path. CVE-2026-41568: a TOCTOU symlink swap creates arbitrary files/directories on the host. Docker rates all three "critical"; the corresponding GHSA records assign High/High/Medium [1].

TOCTOU (time-of-check / time-of-use): Like a bouncer checking your ID, you swapping it for a fake one in a flash, and the stamp landing on the wrong card. The program checks a path, the attacker changes it in the split second between, and the privileged action hits the planted target — here, a host path instead of the expected container path.

29.5.2 (May 20) is the follow-up: 29.5.1 introduced a regression where docker cp with bind mounts traversing in-container symlinks (/var/run → /run) failed with "mkdirat: file exists." So if you already jumped to 29.5.1, take 29.5.2 straight away [1].

What to do today Check your engine version (docker version) and move to 29.5.2. The attack requires a compromised or hostile container that catches a docker cp — anyone running foreign images or PR-triggered CI containers is the primary target. Until patched, don't run docker cp against untrusted containers.

containerd — runAsNonRoot was a request, not a lock

One floor down, same day: CVE-2026-46680 (GHSA-fqw6-gf59-qr4w, CVSS v4 7.3) in containerd. User-ID handling can be tricked so that a container meant to run not as root via runAsNonRoot does so anyway. That defeats one of the standard hardening guarantees Kubernetes pod-security profiles rely on. Affected: 1.7.27–1.7.31, 2.0.4–2.0.8, 2.1.0–2.2.3 and 2.3.0-beta–2.3.0; fixed in 2.3.1 / 2.2.4 / 2.0.9 / 1.7.32 (May 20). Correction to Wednesday's brief: the CVE in this wave is 2026-46680, not the spdystream number cited there [1].

Shai-Hulud Wave 5 — the worm rolls through AntV

The self-replicating npm/PyPI worm from the last brief hasn't faded — it's escalating. On May 19 (01:39–02:06 UTC) a fifth wave hit the AntV ecosystem: through a compromised atool maintainer account, ~639 malicious versions across 323 packages were published in about half an hour — roughly 7.6× the TanStack wave. Affected are packages with millions of weekly downloads (echarts-for-react, size-sensor, @antv/g2, @antv/g6). Worse: the fully weaponized worm code was posted publicly to GitHub on May 12 (since removed) — the technique is no longer exclusive to the originating group, copycats are the new risk. It remains the first documented case of malicious npm packages carrying valid SLSA provenance [1][2].

OpenJDK — JDK 27 firms up, three JEPs near targeting

The Early-Access line is at build 22 (up from ~17 on Wednesday). Three JEPs went Proposed to Target, with the review window ending May 19: JEP 523 (G1 as the default GC in all environments), JEP 534 (Compact Object Headers by default — the promotion of JDK 25's JEP 519), and JEP 537 (Vector API, 12th incubator). Newly at Candidate: JEP 538 (PEM encodings of cryptographic objects). RDP1 (feature freeze) forks on Jun 4, GA in September. Honesty flag: that the three JEPs flipped from Proposed to Target to actually Targeted couldn't be finally confirmed on the (403-ing) openjdk.org primary page — review window closed, targeting expected [1][2].

Compact Object Headers: Every Java object carries a small bookkeeping header. JDK 25 shrank it from 12–16 to 8 bytes — like giving every package in the warehouse a smaller label: across millions of objects that frees real heap and so cuts GC pauses (~15% fewer GCs on SPECjbb2015). JEP 534 makes that the default in JDK 27 instead of an opt-in flag.

TC39 #114 — using is now ECMAScript

The Amsterdam plenary (hosted by JetBrains) wrapped Thursday; the stage changes are in the tc39/proposals repo, with the narrative notes still to come. Headline: Explicit Resource Management — the using and await using declarations many know from TypeScript — reached Stage 4 and is now part of ECMAScript 2026. Also at Stage 4: Atomics.pause (a spin-loop hint for shared memory) and Joint Iteration (Iterator.zip). Advancing to Stage 3 were iterator chunking/includes/join and a standardized error.stack accessor, among others. Decorators moved only from 2 to 2.7 — the old caveat holds: none of the major engines wants to ship first, so the timeline stays soft [1][2].

using / Explicit Resource Management: Like try-with-resources in Java, but for JavaScript: write using file = openFile() and the resource is closed automatically when the block ends — whether it exits normally or by exception. No more forgotten finally { close() } blocks.

Commentary — the boundaries we trust

Three of today's top stories are the same story in three layers. The container shouldn't see the host — docker cp and the runAsNonRoot bypass show the isolation wall is thinner than the mental model. The ORM should make SQL injection impossible — the Drupal flaw shows "only on Postgres" is enough to topple that assumption. And SLSA provenance should prove supply-chain trust — Shai-Hulud has shown for two weeks that a hijacked build inherits that trust. The pattern: it's not the crypto or the feature that's broken, but the boundary where we stop thinking.

For Daniel's stack that means something pragmatic and manageable: move Docker to 29.5.2 and run no docker cp against untrusted containers; if Postgres-Drupal runs anywhere, patch it today; keep watching npm lockfiles and maintainer compromises. The good news comes from the same industry that's molting: using is now ECMAScript, JDK 27 makes Compact Object Headers the default, Node has its LTS patch. On Monday the EU ships the next sovereignty piece with CADA — and then we'll see whether the leaked US-cloud paragraph becomes real legal text [1].
