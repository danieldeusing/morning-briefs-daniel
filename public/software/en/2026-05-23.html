Drupal SQLi — from "reported" to "officially confirmed exploited"

Yesterday's top story took the most serious escalation step overnight: CVE-2026-9082 (SA-CORE-2026-004) was added to CISA's KEV on May 22 — the US agency's catalog of vulnerabilities it has evidence are being actively exploited. So yesterday's "exploit attempts being observed" became an official "exploited in the wild." The wrinkle relevant to Daniel still holds: the unauthenticated SQL injection in Drupal's database abstraction hits PostgreSQL installations only (MySQL/MariaDB/SQLite stay clear). It was discovered by Michael Maturi of Google/Mandiant; several security firms now report automated attacks against thousands of sites [1][2].

CISA KEV (Known Exploited Vulnerabilities): A list kept by the US cyber agency of flaws with real evidence they're being attacked right now — not just theoretically dangerous. Like the difference between "this door has a weak lock" and "this door got broken into yesterday." US federal agencies must patch KEV entries on a deadline; for everyone else, the list is the best prioritization signal.

The score spread is unchanged: Drupal's own model rates the flaw 23/25 ("highly critical"), while NVD assigns just 6.5 MEDIUM — both numbers deserve a mention, and the KEV listing is the decisive signal. Fixed in 11.3.10/11.2.12/11.1.10/10.6.9/10.5.10 plus hand-patches for EOL branches. If you run Postgres-backed Drupal anywhere: today, not Monday [3][4].

npm Staged Publishing — the registry finally builds a safety net

This is the day's most important positive story, and it sits right in Daniel's toolbox. On May 22 npm made Staged Publishing generally available. Until now: anyone with a token (or an OIDC trusted-publishing flow in CI) pushed a package tarball straight into the live registry — and millions of installs pulled it seconds later. That exact path was the entry point of the past few weeks. Now a tarball published via CI/OIDC lands first in a staging queue: a human maintainer with 2FA must explicitly approve it (npm stage publish) before it becomes installable. The automated build and the human publish decision are now separated [1].

OIDC trusted publishing: Instead of storing a long-lived secret token in CI (which an attacker can steal), the CI pipeline re-proves its identity on every run via a short-lived, signed ticket — like a day pass instead of a permanent key. Good against token theft, but worthless if the pipeline itself is hijacked — which is exactly the gap the human approval step now closes.

Alongside it, npm 11.15.0 added four install-time switches that control where dependencies are even allowed to come from: --allow-file (local paths/tarballs), --allow-remote (URLs incl. https tarballs), --allow-directory (local directories) — next to the existing --allow-git. Each accepts all (default) or none. If you run a lockdown CI, you can now simply switch off non-registry sources. The GitHub changelog itself names no specific attack as the trigger — but the Shai-Hulud / TanStack wave context is obvious [1].

Shai-Hulud — no new wave, but the registry cleaned house

The self-replicating npm worm from recent briefs did not follow up with a Wave 6 in the 24h window — the last confirmed wave remains the AntV burst of May 19 (~639 versions / 323 packages). The real response story: npm invalidated all granular write tokens that could bypass the 2FA requirement, platform-wide (May 19), and GitHub keeps disabling the malicious packages under its terms of use. Together with Staged Publishing above, this is a coherent hardening line: fewer long-lived keys, more human approval points. If you still use classic publish tokens in CI, now's the time to check whether they're still valid [1].

Deno 2.8.0 — the day's biggest release

On the quiet release side, Deno 2.8.0 (May 22, 10:51 UTC) is the most substantial drop in the window. Four new subcommands: deno ci (CI workflow), deno pack (produce an npm tarball), deno transpile, and deno bump-version (version management with workspace support); deno audit fix becomes an alias for deno audit --fix. On the package side: the catalog: protocol for centrally maintained dependency versions across a workspace, a hoisted node_modules linker mode, --prod (skips dev deps and @types), and --os/--arch for cross-platform npm installs. The unprefixed default in add/install now points to the npm registry. Under the hood: TypeScript 6.0.3, V8 14.9, plus broader Node compat (node:wasi, node:cluster on Unix) [1].

catalog: protocol: In a monorepo with many packages, you don't want to pin a library to the same version in every package.json separately. A catalog is a central list saying "this version of X applies everywhere" — change the entry, all packages follow. It removes the tedious by-hand version-syncing.

OpenBSD 7.9 — post-quantum TLS reaches a stock system

The 60th OpenBSD release (May 19) matters to Daniel less as an OS than as a crypto signal: it bundles LibreSSL 4.3.0, which supports an ML-KEM768/X25519 hybrid keyshare in TLS — i.e. post-quantum key exchange in the standard library of a mainstream system. Plus OpenSSH 10.3 with hardened validation of shell metacharacters in usernames and corrected setuid/setgid handling for scp downloads as root. Smaller but neat system changes: the scheduler can exclude slower cores on heterogeneous CPUs, and root can no longer bypass bpf(4) BIOCLOCK [1].

ML-KEM / post-quantum hybrid: A large quantum computer may one day retroactively crack today's key exchanges ("capture now, decrypt later"). ML-KEM is a quantum-resistant scheme; "hybrid" means it runs in addition to classic X25519 — if one breaks, the other holds. Like two independent locks on the same door.

pgBackRest — how a critical backup tool nearly went orphaned

The OSS story that touches Daniel as a Postgres user: pgBackRest, the de facto standard tool for Postgres backup and restore, came close to stalling. Creator and lead maintainer David Steele had maintained it for 13 years as principal architect at Crunchy Data — until Snowflake bought Crunchy Data in June 2025 and left the tool without dedicated funding. Now a sponsor coalition has stepped in: AWS, Percona, Supabase, pgEdge, Tiger Data (Timescale) and Eon.io jointly fund Steele's time for bug fixes and feature work, plus Percona engineering to onboard a second maintainer. The project no longer hangs on a single sponsor [1][2].

Why it matters It's the week's supply-chain theme from the other side: not a malicious maintainer, but a missing one. Critical infrastructure resting on a single, unpaid person — that's the same single-point-of-failure class as a hijacked npm account, just with the sign reversed.

It's Saturday, May twenty-third. The news day itself is quiet, but the week's two through-lines keep moving — the supply chain under fire, and the supply chain fighting back.

The biggest move is an overnight escalation. The counterweight comes from npm itself. The registry made staged publishing generally available. Until now, a package published from a build pipeline went live instantly; now it lands in a queue first, and a human with two-factor confirmation has to approve it before it becomes installable. That automatic publish path was the entry point of the past few weeks. Four new switches also let you control whether dependencies are allowed to come from non-registry sources at all. And fittingly: the self-replicating worm Shai-Hulud added no new wave, but npm invalidated every token that could bypass the two-factor requirement, platform-wide.

On the release side, Deno two point eight is the day's biggest drop: four new subcommands, a catalog protocol for centrally maintained dependency versions, and TypeScript six under the hood. OpenBSD seven point nine brings something bigger than a mere system update: its bundled crypto library, LibreSSL, now supports post-quantum key exchange in TLS — quantum-resistant encryption reaching the standard system of a mainstream operating system.

The open-source story of the day shows the same supply-chain question from the other side. pgBackRest, the most important backup tool for PostgreSQL, nearly stalled: its sole maintainer lost funding after Snowflake bought his employer. Now a coalition of six companies — including Amazon, Percona and Supabase — funds his work jointly. Not a malicious maintainer, then, but a missing one — the same class of single weak point, just with the sign reversed.

Among the big releases, it's otherwise quiet: Node, Angular, Spring and the browsers unchanged, the announced Quarkus Final still not here. In the Java world, JDK twenty-seven is targeting the Vector API, with feature freeze on June fourth. And the European sovereignty machine is warming up: on Monday the EU Commission adopts the Cloud and AI Development Act. In Germany, the losing bidders withdrew their challenges against the big cloud contract, and in Brazil, President Lula moved oversight of digital platforms to the data-protection authority by decree.

The common thread holds: it isn't the feature that breaks, but the boundary where we stop thinking — whether an attacker overruns it or a defender goes missing. On Monday we'll see whether the leaked US-cloud paragraph in CADA becomes real legal text, and whether Quarkus finally ships. Until then.
