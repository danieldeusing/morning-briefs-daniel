Laravel-Lang — own the git tag, own the package

The weekend's top story doesn't sit in npm but in Composer, the package manager of the PHP/Laravel world. On May 22 a single attacker hijacked four widely used packages of the Laravel-Lang organization — laravel-lang/lang, /http-statuses, /attributes and /actions. The twist: he didn't publish a new malicious version that a maintainer might have noticed. He rewrote every existing git tag (the flagship laravel-lang/lang alone has 502 tags) so the tags pointed at a commit in a fork outside the organization. The rewrites started at 22:32 UTC and were done by midnight — roughly 700 historical versions across the four packages [1][2].

Git tag rewrite + Packagist: A git tag is a sticky note with a name (say "v3.0") on a particular state of the code. Packagist (the Composer registry) checks which code a tag is stuck to and publishes that as a release. The attacker moved the notes — the name stayed, the contents were swapped. Like the "organic" label someone peels off and sticks on a different jar: the wording is right, the contents aren't.

The real harm is in the payload: the commits add a src/helpers.php to Composer's autoload.files — the list of files loaded automatically on every vendor/autoload.php, i.e. on every startup of a Laravel/Symfony app. The credential stealer grabs EC2 metadata (169.254.169.254), Kubernetes tokens, HashiCorp Vault, Jenkins secrets and browser passwords. If you pulled any of these packages on or after May 22, 22:32 UTC via composer install, treat every reachable secret as compromised and rotate it [3][4].

Megalodon — 5,561 repos in six hours, straight through CI

If Laravel-Lang shows Composer is attackable too, Megalodon goes a step further: the campaign hit GitHub itself, at the CI/CD pipeline layer. On May 18, in a six-hour window (11:36–17:48 UTC), an automated attacker pushed 5,718 malicious commits into 5,561 repositories. It was disguised behind forged bot identities like build-bot, ci-bot or pipeline-bot — names nobody looks at twice in a commit history [1][2].

CI/CD workflow injection: A CI pipeline is the robot that automatically builds, tests and deploys on every push — with access to all the project's secrets. Smuggle in a malicious workflow file and that robot will spill the keys on the next push. Like quietly slipping the building's master-key janitor a second job.

The workflows contain base64 bash that exfiltrates CI secrets, cloud credentials, SSH keys and OIDC tokens to a C2 server. Two variants: the mass variant adds a workflow that fires on every push; the targeted one replaces existing workflows with workflow_dispatch triggers — dormant backdoors fired later via the API. Real fallout: the npm package @tiledesk/tiledesk-server (2.18.6–2.18.12) carried the targeted variant forward because the maintainer published from a compromised environment [3].

Drupal SQLi — "confirmed exploited" turns measurable

The PostgreSQL-only SQL injection from the start of the week (CVE-2026-9082, SA-CORE-2026-004) now has numbers. Since disclosure on May 20, Imperva reports over 15,000 attack attempts against nearly 6,000 distinct sites in 65 countries; the leaders are the US at 61.8%, then Singapore (6.6%) and Australia (6.3%), with about half the attacks targeting gaming and financial sites. Important framing from Imperva itself: this is still mostly reconnaissance — scanners mapping vulnerable Postgres-Drupal installs and testing exploits, not yet wholesale takeovers [1][2].

The score spread persists: Drupal rates it "highly critical," the NVD only 6.5 MEDIUM — the KEV entry (since May 22) is the signal, not the argument over the number. FCEB agencies must patch by May 27; a good rule of thumb for everyone else. If you run Postgres-Drupal, there's nothing to decide here — only to do [3].

The supply chain strikes back — npm, pip, provenance

npm Staged Publishing remains the biggest lever

npm's Staged Publishing (GA since May 22) separates the CI publish from a human 2FA approval — a tarball published via a pipeline lands in a queue first, before it becomes installable. Exactly the gap Megalodon and Shai-Hulud exploit (hijacked pipeline → instantly live package) now gets a human checkpoint. Plus the four --allow-* switches (npm 11.15.0) against non-registry sources [1].

pip cooldowns — simple, big effect

A fitting counterpart from the Python camp, available since pip 26.1 (late April): dependency cooldowns. The --uploaded-prior-to flag (e.g. P5D) stops freshly published — and thus potentially freshly hijacked — versions from going straight into the build. Per the cited analysis, a 7-day cooldown would have stopped 8 of 10 analyzed attacks before reaching the end user. No silver bullet, but cheap [2][3].

Shai-Hulud — no new wave

The self-replicating npm worm didn't add a new wave in the window; the last confirmed one remains the AntV burst of May 19. Worth noting only: the family now forges Sigstore provenance at runtime — the signature layer is no longer a free pass [4].

Linux kernel — fresh stable batch, with Fragnesia behind it

On the infra side, the news that touches Daniel's own stack: on May 23 a whole set of stable and longterm kernels shipped — 7.0.10 as the current stable, plus the longterm branches 6.18.33, 6.12.91, 6.6.141, 6.1.174, 5.15.208 and 5.10.257. This is routine cumulative maintenance, not an emergency release; if you run stable branches, you pull it on schedule [1].

The more serious story runs in the background, though: Fragnesia (CVE-2026-46300, CVSS 7.8), a local privilege escalation in the "Dirty Frag" family, found by William Bowling (Zellic). It sits in the XFRM / ESP-in-TCP subsystem and lets an unprivileged local user modify read-only files — including setuid binaries — via page-cache manipulation, becoming root. Distributions have been patching since mid-May (AlmaLinux, Ubuntu/Canonical, CloudLinux); the quick reboot-free mitigation is to unload or denylist the esp4, esp6 and rxrpc modules where they aren't needed [2][3].

Page-cache LPE: The kernel caches frequently read files in RAM (the page cache) so not every access hits the disk. "Dirty Frag" & relatives trick it into modifying that cached copy of a file that is supposed to be read-only — e.g. a program that runs with root rights. You don't forge the original in the safe, but the working copy on the desk that everyone uses.
Research note The May 23 stable batch (kernel.org) and Fragnesia are sourced separately; that these specific releases name CVE-2026-46300 explicitly in the changelog is not individually verified here — the Fragnesia fixes have been rolling out via distro kernels since mid-May.

Two 10.0s and a stubborn NGINX case

LiteSpeed cPanel plugin CVE-2026-48172 — actively exploited

Top score, and already in the wild: a logged-in cPanel user can run scripts as root via the Redis enable/disable feature of the LiteSpeed User-End cPanel Plugin — CVE-2026-48172, scored CVSS 4.0 10.0 by the CNA (MITRE) (NVD assessment still pending), published May 20. Versions before 2.4.5 are affected; fix in 2.4.5/2.4.7. For Daniel pure awareness value (no cPanel/LiteSpeed in the stack), but an instructive pattern: a supposed convenience feature becomes the root path [1][2].

Cisco Secure Workload CVE-2026-20223 — 10.0, but not (yet) exploited

Also 10.0, different profile: an unauthenticated REST API flaw in Cisco Secure Workload lets an attacker read and configure across tenant boundaries as Site Admin. Fix in 3.10.8.3/4.0.3.17, SaaS already patched, no workaround; Cisco reports no exploitation. Here the class matters, not the alarm [3].

CADA — Brussels gets serious on Monday

The Cloud and AI Development Act is still on the agenda of the Commission meeting for Monday, May 27, as part of the tech-sovereignty package — bundled with the Chips Act 2, an open-source strategy and an "AI-in-energy" roadmap. The goal is unchanged: triple EU datacenter capacity, EU-wide eligibility rules for cloud providers, a unified cloud-procurement policy for public administration [1].

Digital sovereignty: The question of who can pull the plug in an emergency. If government data sits on a US cloud, the provider is subject to US law — a foreign government could demand access or cut the service. "Sovereignty" means keeping critical data under your own legal jurisdiction. Like a safe-deposit box at your own bank versus one in a country whose laws you don't get to set.

The contentious question still sits in the leaked draft (CNBC, May 7): it considers restricting the use of US cloud for sensitive government data (health, finance, justice) — explicitly not a ban, not for the private sector. What's fresh is the diplomatic headwind: a US State Department cable from February instructs diplomats to oppose foreign data-sovereignty moves — so Monday's adoption lands in a heated environment. Anyone serving public-sector contracts in the EU reads the real text on Monday [2].

EU regulation clock — DMA, CRA, the next deadlines

Two dates close enough to know. DMA: the statutory review (COM(2026) 178, Apr 28) holds its conclusion that it's "premature" to extend messenger interoperability to social networks — no new build mandate. Unaffected: Apple's interop deadline on June 1 (notifications, proximity pairing, audio switching). CRA (Cyber Resilience Act): the next hard dates are the notification of conformity assessment bodies on June 11, 2026 and — the one that matters more to software makers — the start of the reporting obligations via the ENISA platform on September 11, 2026 (24-hour early warning / 72-hour notification / 14-day final report). The latter is weeks out, so it belongs on the radar, not in the 7-day track [1][2].

Brazil — the fight over ANPD oversight escalates

Follow-up to Lula's platform decrees of May 20: the opposition filed several PDLs (legislative-decree projects) in the lower house (Câmara) to suspend the two decrees 12.975 and 12.976. The charge: they impose "unprecedented" monitoring, removal and risk-mitigation duties on platforms and expand the powers of the data-protection authority ANPD without an adequate legal basis. A weighty counter-voice: Renata Mielli, coordinator of the internet steering body CGI.br, calls the ANPD oversight role "coherent." The decrees take effect 60 days after publication — so the political fight runs in parallel with a ticking deadline [1].

PDL (Projeto de Decreto Legislativo): An instrument in the Brazilian Congress that lets parliament overturn an act of the executive (here: a presidential decree). Not a new law, but a veto tool against government regulations. Like a supervisory board pulling back a management decision by majority vote.

In parallel, the ANPD's own staffing deadline keeps ticking: by May 31 the competition for 200 permanent posts must be authorized (Lei 15.352/2026), the edital projected for ~Sep 24, the exams for early 2027. You don't get oversight from a decree alone — you also need the people to do it [2].

Comment — it was never an npm problem

Anyone who read the past few weeks as "npm has a security problem" corrects that this week. Laravel-Lang hit Composer, Megalodon hit GitHub Actions — neither needed a malicious new version. They hijacked the layer underneath: the git tag Packagist trusts blindly, and the CI bot identity nobody checks in a commit history. The pattern isn't the ecosystem but the implicit trust assumption — "this tag points at the code the maintainer meant," "this commit came from the pipeline."

That's exactly where the good news lands: Staged Publishing puts a human between pipeline and live package; cooldowns buy time until a hijack is noticed; and the fact that Shai-Hulud now forges provenance only means the signature too is a measure, not a proof. For Daniel's stack: patch Postgres-Drupal today (KEV deadline May 27). Check whether CI pipelines verify tag/commit provenance at all — and whether a cooldown fits the build policy. Pull the Linux stable batch, unload esp4/esp6 if unused. It's Sunday, the twenty-fourth of May. The official security feeds are quiet — the catalog of actively exploited flaws hasn't had a new entry since Friday. But the week's big theme switched territory overnight.
