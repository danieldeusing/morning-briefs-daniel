Three Supply Chains in One Day — the KEV Batch Entry

Yesterday's CISA-KEV move is the most unusual of the week: catalog version 2026.05.27 (released 05.27, 17:08 UTC) adds not one but three vulnerabilities — and all three are not classic web server or auth bugs, but supply-chain incidents: poisoned installers, poisoned npm packages, poisoned IDE extensions. Same campaign, three different entry points [1].

Supply-Chain Attack: Instead of hacking your server directly, the attacker poisons something you install yourself — a library from the package manager, an editor plugin, a vendor installer. You build in the backdoor yourself because it comes from a source you already trust. Like when a plumber doesn't kick down your door, but slips a manipulated drawer into the furniture store, which you unsuspectingly assemble in your living room.

The three entries in detail:

1. CVE-2026-8398 · Daemon Tools Lite. Deadline Sat 05.30 — only three days. NVD rates 9.8 (v3.1) or 9.3 (v4.0). Official Windows installers from daemon-tools.cc were trojanized between 04.08 and 05.05.2026 according to Kaspersky — versions 12.5.0.2421 through 12.5.0.2434 write three signed binaries (DTHelper.exe, DiscSoftBusServiceLite.exe, DTShellHlp.exe) with valid developer signature of AVB Disc Soft plus a payload that downloads shell commands via HTTP GET. Several thousand infection attempts in over 100 countries; victim distribution (Russia, Brazil, Turkey, Spain, Germany, France, Italy, China) and artifact analysis suggest a Chinese-speaking actor. Fix: 12.6.0.2445 (or newer) from 05.05 [2][3].

2. CVE-2026-45321 · TanStack. Deadline Wed 06.10. CVSS 9.6. On 05.11.2026 between 19:20 and 19:26 UTC — six minutes — 84 malicious versions across 42 @tanstack/* packages were published to the npm registry, cleanly signed via GitHub Actions OIDC Trusted Publisher setup. The attacker chained three known vulnerability classes: a pull_request_target "Pwn Request" misconfiguration, cache poisoning across the fork/base trust boundary of GitHub Actions caches, and runtime memory extraction of the OIDC token from the Actions runner process. Result: malware under trusted identity, with valid SLSA Build Level 3 attestations. Actor name: TeamPCP. The campaign — "Mini Shai-Hulud" — subsequently compromised over 170 npm and 2 PyPI packages (Mistral AI, UiPath, OpenSearch, Guardrails AI) [4][5].

3. CVE-2026-48027 · Nx Console. Deadline Wed 06.10. VS Code extension Nx Console (over 2.2 million installations) was published with a malicious version 18.95.0 in Visual Studio Marketplace on 05.18.2026 for ~11–18 minutes (Nx postmortem ~11 min / 12:47 UTC; GitHub advisory ~18 min / 12:48 UTC, in Open VSX Marketplace nearly 36 minutes). Impact nonetheless ≥6,000 auto-updates. The payload harvested credentials from 1Password vaults, Claude Code configs, npm, GitHub, and AWS. This made 3,800 private GitHub repos exfiltrable — confirmed victims: GitHub Internals, OpenAI, Grafana Labs, Mistral AI. Update to Nx Console ≥18.100.0, remove persistence artifacts, rotate every secret reachable from that machine [6][7].

The connection is the unsettling part: Nx Console was compromised via GitHub credentials stolen seven days earlier by the TanStack incident. One supply-chain compromise fed the next. The publicly released Mini Shai-Hulud tools have been on GitHub since 05.12 — the toolchain knowhow is no longer just TeamPCP's [5].

What Daniel Does Daemon Tools doesn't matter to him (no Windows). But: every still-running CI job that pulled a @tanstack/* package between ~05.11 and ~05.18 should be treated as potentially compromised host and workflow tokens rotated. If Nx Console extension ran on his Mac: check VS Code Marketplace, enforce version ≥18.100.0, reset 1Password vault locks, audit GitHub PATs/OIDC trusts. That's the honest answer this week, not "patch tonight."

npm Hits the Brakes — "Staged Publishing" Is GA

Straight from the same story, but with its own half-life: GitHub shipped staged publishing for npm to General Availability on 05.22.2026. This is the structural answer to the TeamPCP worm — because it exploited exactly the two doors that were open before: a CI token that could publish without human approval, and Trusted Publishing with OIDC, which gives "provable CI provenance" but not "provable human presence" [1][2].

Mechanics: instead of publishing directly, npm publish or the new npm stage publish lands in a staging queue, which a maintainer must explicitly unlock with 2FA — even in non-interactive CI. Requires npm CLI ≥11.15.0. Registry recommendation: combine staged publishing with OIDC Trusted Publishing — presence and provenance [3].

To be honest for a solo setup like Daniel's this changes little — he consumes npm, doesn't publish. But every library he pulls now potentially runs through a 2FA gate, lowering the odds of another 84-versions-in-6-minutes incident. What it doesn't do: enforce provenance at install time. npm can verify signed attestations but doesn't block if one's missing. That's the next frontier [1].

NGINX "Rift" — Missing from Catalog, Day Fifteen

Update since Wed KEV 2026.05.27 adds three supply-chain gaps — and leaves a webserver 9.2 still outside. CVE-2026-42945 (NGINX "Rift") has been patched since 05.13 (Stable 1.30.1, Mainline 1.31.0), public PoC on GitHub since 05.15, and exploitation attempts observed by VulnCheck since 05.16; SecurityWeek, Bleeping Computer, and The Register confirm on 05.18 a PHP webshell drop on compromised hosts. That's fifteen days after patch — and the catalog still doesn't know about it [1][2].

Affected: every NGINX OSS version 0.6.27 through 1.30.0, NGINX Plus R32 through R36, any downstream compiling in ngx_http_script.c (including Ingress NGINX Controller through v1.15.1). Trigger is a rewrite/if/set directive with an unnamed capture group ($1, $2) behind a ? — no auth needed, a single HTTP request suffices. Not relevant for Daniel's Caddy; already relevant for any EU/BR hosting on Kubernetes with Ingress NGINX. The KEV gap remains: active exploitation reported, not in catalog — so prioritization doesn't follow wild-exploit observation one-to-one [3].

7-Zip 26.01 — An NTFS Bag with 8.8

From the heise channel and highly mid-market-relevant: CVE-2026-48095 in 7-Zip, a heap buffer write overflow in the NTFS archive handler. CVSS 8.8 (v3.1, GitHub Security Lab, found by Jaroslav Lobačevski). The bug lives in CInStream::GetCuSize(): a crafted NTFS image can force the shift exponent to 32, triggering C++ undefined behavior and shrinking _inBuf to a single byte. The following read writes attacker data into this mini buffer, corrupts the stream object's v-table — code execution. Tricky: the file doesn't need a .ntfs extension — 7-Zip's signature fallback also routes .7z, .zip, .rar into the NTFS parser if other handlers fail. PoC (gen_ntfs_sparse.py) is published. Fix: 7-Zip 26.01 from 04.27.2026 [1][2].

For most devs this is phishing ammunition — a ZIP that should look harmless to open has code-execution potential. The heise article ran in the English channel with 05.27 date; the update itself is a month old. Anyone running 7-Zip on Windows workstations has a straightforward patch [1].

Commentary — Supply Chains as Primary Attack Surface

Yesterday's thread was "who controls the stack." Today it runs one layer deeper: how does foreign code enter the stack. The KEV Triple is symptomatic — not a server was hacked but three distribution paths: a vendor installer with valid signature, an npm Trusted Publisher, a marketplace extension with 2.2 million installations. Classic webserver hardening helps against none of them. npm's answer — human-with-2FA in every publish path, even with OIDC — is structural, slow-acting; CISA's answer — three entries in one day — is fast.

For Daniel's stack today concretely: nothing urgently to patch (no cPanel/LiteSpeed, no Daemon Tools, no TanStack in tree, no Nx Console extension installed) — but the real homework is different: If the next poisoned library lands in a CI job, what happens? If you can't answer that — which tokens are rotatable, which build artifacts are reproducible-built, what do the audit logs show — start today. And the weekly clock: tomorrow LiteSpeed deadline, Saturday Daemon Tools, Monday DMA-Apple and Spring Train, Wednesday CADA — and Thursday JDK 27 forks.

It's Thursday, May 28th. The catalog of actively exploited vulnerabilities made an unusual move last evening — adding three supply-chain incidents all at once. That's the top story of the day.

These three entries aren't classic server bugs but three different ways to smuggle poisoned code into other stacks. First: Daemon Tools Lite. Official Windows installers from the vendor website were trojanized between April 8th and May 5th, according to Kaspersky — with valid digital signatures from the developer. That's the sinister part: if you pulled the software from the real source, you got a backdoor payload delivered with it. Thousands of infections in over a hundred countries, including Germany and Brazil. CISA set an unusually short deadline for federal agencies: three days, through Saturday.

Second: TanStack. On May 11th, between 7:20 and 7:26 p.m. coordinated time — six minutes — eighty-four malicious versions across forty-two TanStack packages went live on the npm registry. Cleanly signed through GitHub's Trusted Publisher setup. The group is called TeamPCP and their campaign is called Mini Shai-Hulud — a worm that jumps from one compromised package to the next. And third, in the same family: the Visual Studio Code extension Nx Console. It appeared in the marketplace with a malicious version on May 18th for eighteen minutes — long enough. About thirty-eight hundred private GitHub repositories got exfiltrated as a result, including code from OpenAI, Grafana, and Mistral.

The dark connection is that the second supply-chain breach enabled the third: a developer token stolen in the TanStack incident gave the attackers access to the Nx publishing pipeline seven days later. One supply-chain compromise fed the next. For Daniel, no direct action — he's got neither Daemon Tools nor TanStack nor Nx Console running — but the honest homework is different: if a poisoned library lands in a CI job next time, what happens? Which tokens can you rotate, which build artifacts are traceable?

npm gave a structural answer to exactly these stories. Since May 22nd, "staged publishing" is generally available: every publish — even from CI — lands in a queue that a human has to unlock with two-factor authentication. That closes exactly the door TeamPCP came through — an automated pipeline that could publish without human approval.

Today, Oracle also starts its new patch cadence. The first monthly Critical Security Patch Update arrives today, with thirty-five patches across multiple product areas. Oracle's quarterly updates used to be the standard — this compresses it to monthly. For Daniel it's a case study, but a clear signal: even the most conservative enterprise vendor is speeding up.

One old wound stays open: the NGINX vulnerability called Rift, with a score of 9.2, was patched May 13th, actively exploited since May 16th — and as of yesterday's catalog, still not listed. That's fifteen days between wild exploits and the KEV entry. CISA's prioritization clearly doesn't track threat observation one to one.

Europe gets dense next week. Monday, Apple's interoperability deadline under the Digital Markets Act closes; Wednesday, the Commission takes its fourth shot at the Tech Sovereignty Package — with the Cloud and AI Development Act at its center, which would keep EU states away from U.S. cloud providers for sensitive data. And the Cyber Resilience Act clock ticks: starting September 11th, vulnerability reporting becomes mandatory for networked product makers, and the BSI has the lead role.

In Brazil, the data protection authority ANPD is still waiting for approval for two hundred permanent positions, deadline effectively tomorrow Friday. Tomorrow is also the LiteSpeed deadline from yesterday's catalog. Saturday is Daemon Tools, Monday is the Spring release train, Thursday JDK 27 forks. The day itself is modest — the weekly clock is not.
