Exchange-OWA deadline today — still no patch

The CISA KEV deadline for the on-prem Exchange gap CVE-2026-42897 expires today, Fri 05-29. Microsoft confirmed active exploitation on 05-14, CISA added it to the catalog on 05-15 and gave 14 days to mitigate — the window closes now [1].

XSS in OWA: Outlook Web Access is Microsoft's web frontend for Exchange. Cross-Site Scripting means: the attacker smuggles foreign JavaScript code into your authenticated OWA tab, which then runs under your login. In practice: a crafted email arrives, you open it in OWA, and their code steals session cookies, copies your mailbox, or sets up mail rules. The server goes untouched — the bug hijacks the browser you already authenticated to.

The catch: no patch in the classical sense. Microsoft is only distributing a URL-rewrite mitigation called "M2.1.x" via the Exchange Emergency Mitigation Service [2]. EMS is active by default on Subscription Edition and 2019; on 2016 and older CUs you have to patch it manually. Anyone who turned off EMS for compliance reasons (more government agencies than you'd think) is technically unmitigated today. CVSS 8.1 [3]. For German mid-market shops running hybrid Exchange: check the EMS log today, not next week.

Gitea: four years serving private container images to anonymous requests

Gitea, the self-hosted alternative to GitHub Enterprise/GitLab, confirmed CVE-2026-27771 on 05-27 and closed it with 1.26.2. CVSS 8.2. The issue: the built-in container registry respected the private flag in the UI, but delivered manifests and blob layers to the registry endpoints (GET /v2/<name>/manifests/<tag>, /blobs/<digest>) even to anonymous requests. Knowing the repo name and tag was enough — no auth, no account, no token [4].

Container registry: The package manager for Docker images — like npm for Node, except for entire pre-built server setups. You build locally (docker build), push to the registry, and the server pulls via docker pull. "Private" registry means auth only — in the image layers live source code, .env files, baked-in API keys, AWS credentials. If auth checks fail, it's worse than an open Git repo: not just code, but the built, running bundle of secrets.

The bug existed for almost four years. Exposure: >30,000 instances in 30+ countries, top 5 China, USA, Germany, France, UK [5][6]. Forgejo inherited the same registry code and is also affected. Discovered in April by "NoScope", an autonomous AI pentesting agent — a preview of the next 12 months [7]. Mitigation: upgrade to ≥1.26.2; stopgap [service].REQUIRE_SIGNIN_VIEW = true [8]. For solo devs and mid-market shops with a BSI-sovereignty reflex, Gitea/Forgejo is the obvious self-hosted Git+CI+registry stack — if you serve customers: upgrade now, audit the registry logs (which anonymous IPs pulled which manifests?), rotate secrets.

C# gets a Rust coat — pointer types out of "unsafe"

Microsoft's .NET team sketched on 05-26 in the .NET blog the largest memory-safety overhaul for C# in years. PM Richard Lander explains in "Improving C# Memory Safety": the unsafe keyword is being redefined — after Rust's model, without dropping the GC [9].

"unsafe" in a GC language: C# normally cleans up memory via garbage collection. The moment you use raw pointers (for native C code, hardware), you disable this safety net — that's what the unsafe keyword does. Rust is more precise: only the truly dangerous operation (pointer dereferencing, i.e. actually reading from a foreign address) goes in an unsafe block; arithmetic with the pointer stays safe.

That's exactly where Microsoft wants C# to go. Pointer types and pointer arithmetic will become safe; only dereferencing stays unsafe. Effect: FFI and interop wrappers become more readable, the dangerous step stands out instead of hiding among harmless calculations in the same block — the same logic as Rust [10].

Roadmap: preview in .NET 11, production in .NET 12 / C# 16, opt-in. Old code runs unchanged; runtime libraries will be opted-in [11]. Fits the bigger picture: Microsoft has been translating Windows C/C++ to Rust internally since 2024; now .NET is tightening the screw on its own language. For Java/Spring devs, a point to watch; more relevant if you touch .NET/Unity — the model lands in .NET 11 previews this summer.

npm "staged publishing" — the human 2FA gate is GA

GitHub's answer to TanStack: GA since 05-22. Instead of straight publish to npmjs.com, the tarball lands in a staging queue — a maintainer must explicitly approve with 2FA. Requires npm CLI 11.15.0+ and npm stage publish [17].

Trusted Publishing / OIDC: A CI (e.g. GitHub Actions) authenticates with a short-lived OIDC token to npm and pushes a new package automatically — no click. Convenient, but: if an attacker compromises the CI (pwn request, cache poisoning), the CI publishes their poisoned package with a valid signature. That's how TanStack happened. Staged publishing flips it: the CI uploads, but it goes online only after a second, human 2FA step.

"Proof of presence" even for CI/CD publishes. Caveat: defends against the TanStack path (token theft), not the Nx path (compromised maintainer account). And only works if maintainers enable it — defaults stay put for now.
