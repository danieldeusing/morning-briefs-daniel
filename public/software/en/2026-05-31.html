The AI-tooling layer is now an attack surface

A quiet patch day, a loud rest of the month. Microsoft's May patch day was the first in nearly two years without actively exploited zero-days — and yet May was the most intense month yet for the layer around it: AI-agent tools shipped insecurely, and npm packages distributing malware with valid provenance. The model isn't the problem; the software that packages and ships it is.

PraisonAI ships its API with auth disabled — and was scanned within hours [1][2]

AI-agent framework: A library for building small programs that execute tasks autonomously — "read this email, summarize it, file a ticket". Think of it like giving an intern a to-do list (`agents.yaml`) and watching them work through it.

With PraisonAI, that intern's front door was wide open. The bundled legacy API server (an old Flask backend) has AUTH_ENABLED = False and AUTH_TOKEN = None hardcoded — the auth check always gives the green light. Anyone reaching the server over the network can read the configuration via /agents and execute the stored workflows via /chat, no token required.[1]

This is CVE-2026-44338, rated 7.3 (High) under CVSS 3.1. Affected are all Python package versions from 2.5.6 through 4.6.33; fixed in 4.6.34.[1] The real kicker is the speed: within three hours and 44 minutes of the advisory's publication (May 11, 13:56 UTC), a scanner called "CVE-Detector/1.0" was already knocking on the vulnerable endpoint; the first targeted request landed at 17:40 UTC.[2]

Context Why is this in the software brief, not the ai-dev brief? Because the hook isn't the model — it's the software delivery: an insecure default in a shipped server. This is a classic "insecure by default" bug — the same kind we've seen for years with MongoDB/Elastic/Redis-without-auth, only this time in the AI-tooling layer. If your agent can execute what's in `agents.yaml`, an open API isn't a data leak — it's remote code execution through the back door.

Azure AI Foundry: critical privilege escalation, fixed cloud-side [3][4]

CVE-2026-35435 hits the M365 "published agents" in Azure AI Foundry. Faulty access control (CWE-284) allows an unauthenticated attacker from the network to escalate privileges — from a low-privilege role up to control of AI resources and potentially the M365 environment behind it. Microsoft rates the bug as critical with CVSS 8.6.[3]

Two things matter and are being conflated online. First: The bug has not been exploited in the wild — the ZDI review of the May patch day explicitly states "has not been exploited", and the CVSS vector carries E:U (exploit code unproven). But Microsoft's outlook is "Exploitation More Likely", the highest tier short of confirmed exploitation.[3] Second: Microsoft has fixed the hole in its own cloud infrastructure — no customer action required.[4]

Source hygiene Some aggregators claim CVE-2026-35435 is "already actively exploited". That contradicts both ZDI and Sophos's patch day count ("Exploit detected: 0") and the E:U vector — and it's not on any primary source. We rate the flaw as serious, but not actively exploited.

  2. msrc.microsoft.com (JS-rendered) · Cloud-fix note via redpacketsecurity.com.

May patch day: 132 CVEs — and for the first time in years, no zero-day [5][6]

Microsoft's May patch day brought 132 CVEs (Sophos count including Edge/Mariner; CrowdStrike reports 130, ZDI reports 138 new CVEs — the difference is the usual question of whether Chromium/Mariner entries are counted), with 29 marked critical.[5] What was actually remarkable is what wasn't there: not a single actively exploited zero-day and nothing publicly known beforehand — the first exploit-free patch day since around June 2024.[6]

That's a rare breathing room, not a reason to let your guard down: 14 CVEs are CVSS 9.0 or higher, many of them remote code execution through Office, SharePoint, Azure, and graphics components. But for once, the calendar is setting the patch pace, not an attacker.

npm supply chain: a worm with a genuine authenticity seal [7][8]

SLSA provenance: A cryptographically signed certificate of origin for a software package — "this package was built from exactly this repo, by exactly this build pipeline". Designed as an authenticity seal against tampering. Like a customs stamp on a package: confirms the supply chain, not the contents.

The Mini-Shai-Hulud campaign (actor "TeamPCP") released 84 malicious versions across 42 @tanstack/* packages on May 11 between 19:20 and 19:26 UTC — including @tanstack/react-router with over 12.7 million downloads per week.[7] The entry point was the GitHub Actions pipeline: a pull_request_target "pwn request", cache poisoning across the fork-to-base trust boundary, and reading an OIDC token directly from the runner process.[8]

What's new and troubling: it's the first documented case where an npm worm has produced validly attested SLSA Build Level 3 provenance for malware packages. The Sigstore attestations are correct — they properly attest that the packages were built from `release.yml` on `refs/heads/main`. But SLSA doesn't attest that the workflow was authorized, ran on a protected branch, or was triggered by the right commit.[8]

Lesson Provenance ≠ trustworthiness. Anyone writing "has valid SLSA attestation" as a free pass in their pipeline has moved the problem, not solved it. Consequence for your own repos: avoid `pull_request_target` or run it without secrets/write tokens, don't share GH Actions cache across fork boundaries, tie release workflows to protected branches + manual approval.

node-ipc and 14 typosquat packages: the credential thieves are back [9][10]

Two more waves frame the TanStack story. On May 14, three tampered versions of node-ipc appeared (9.1.6, 9.2.3, 12.0.1) — a foundation library with over 10 million weekly downloads — each carrying identical 80-KB credential-stealer payloads.[9] On May 28, a single actor under the fresh alias "vpmdhaj" published 14 typosquat packages mimicking OpenSearch, Elasticsearch, DevOps, and env-config libraries in a four-hour window.[10]

Typosquatting: A malicious package with a name dangerously similar to a real one (`opensearch-setup` instead of the genuine package). Like a counterfeit bill that looks nearly identical to the real thing — hit one wrong key while typing or trust copy-paste, and you install the fake.

The payload runs during `npm install` via lifecycle hooks and grabs AWS credentials (including EC2 Metadata Service and Secrets Manager across 16+ regions), HashiCorp Vault tokens, GitHub Actions tokens, and npm publish tokens.[10] Steal → move sideways into the cloud → compromise the next maintainer accounts: same worm mechanism.

  2. microsoft.com/security — typosquat campaign (vpmdhaj, 14 packages).

PAN-OS GlobalProtect: VPN auth bypass actively exploited, now in KEV [11][12]

CVE-2026-0257 lets a remote, unauthenticated attacker forge "authentication-override" cookies and slip into the VPN unauthorized through the GlobalProtect gateway. The flaw only triggers in a non-standard configuration (the override certificate is shared with the HTTPS service) — Palo Alto rates it as "medium".[11]

In practice it's not: Rapid7 observed active exploitation since May 17 (first wave from Vultr IPs, second on May 21), and a PoC script is public. CISA added it to the Known-Exploited-Vulnerabilities catalog on May 29, with remediation deadline June 1 for federal systems.[12] An auth bypass on an internet-facing VPN appliance is exactly the kind of initial-access vector you don't leave on the table — regardless of what CVSS says.

  1. security.paloaltonetworks.com — vendor advisory.
  2. rapid7.com (exploit timeline + KEV) · bleepingcomputer.com.

PostgreSQL patch roundup: 11 CVEs, and a good reason to kill MD5 auth [13]

On May 14, PostgreSQL 18.4, 17.10, 16.14, 15.18, and 14.23 shipped — closing 11 security flaws and over 60 bugs.[13] Three deserve a closer look:

  • CVE-2026-6478 — a covert timing channel when comparing MD5-hashed passwords lets attackers reconstruct credentials. scram-sha-256 (default in all supported releases) is not affected — so if you're still running MD5 auth, you now have a solid reason to switch.
  • CVE-2026-6479 — unbounded recursion in the SSL/GSS handshake permits sustained DoS for anyone reaching the AF_UNIX socket.
  • CVE-2026-6476 — SQL injection via subscription name in pg_createsubscriber.

Standard patch roundup: roll it out when the maintenance window fits, no rush — but the MD5 story is the nudge you've been putting off for years.

  1. postgresql.org — release announcement + CVE list, May 14, 2026.

CRA reporting obligation kicks in September: the 24-hour timer starts ticking [14]

Not a 7-day deadline, but it's drawing closer and affects anyone bringing connected products to the EU: from September 11, 2026, the reporting obligation of the Cyber Resilience Act takes effect. For actively exploited flaws and serious incidents: early warning within 24 hours, full report within 72 hours, final report within 14 days — reported once via the central CRA platform.[14]

CRA (Cyber Resilience Act): EU regulation requiring "connected product" makers — from smart lightbulbs to commercially supported libraries — to meet minimum security and report flaws. Like product liability, but for cybersecurity.

In Germany, the reporting goes to the BSI or CERT-Bund. For solo developers and small teams selling commercial components, that means: set up processes now, not in September — a 24-hour timer isn't a deadline you improvise.

It's Sunday, May thirty-first. The day itself is quiet, but the week before brought significant news — and the theme is clear: the software layer around AI models has become an attack surface. Not the model, but the server that packages it, and the package that ships it.

Start with PraisonAI, a framework for AI agents. Its bundled legacy API server had authentication completely disabled by default — anyone reaching it over the network could execute the stored workflows without a password. The flaw is called CVE two thousand twenty-six, forty-four thousand three hundred thirty-eight, rated seven point three. The really frightening part is the speed: within just under four hours after publication, the first scanner was already knocking on the open door.

A second AI flaw comes from Microsoft: Azure AI Foundry, a critical privilege escalation rated eight point six. Be careful with the headlines here — some aggregators say it's already being exploited. That's wrong. The Zero Day Initiative review explicitly says it hasn't been exploited, and Microsoft already fixed it in its own cloud anyway, no customer action needed.

Which flows seamlessly into Microsoft's patch day: May brought one hundred thirty-two flaws, twenty-nine of them critical — but for the first time in nearly two years, not a single actively exploited zero-day. A rare breathing room.

The loudest part of the week was npm's supply chain. TanStack attackers released eighty-four malicious versions across forty-two packages — and for the first time with valid SLSA provenance. That's the authenticity seal for software origin, and this time the seal was correct — for malware. The lesson, Bloomberg-style: origin doesn't prove trustworthiness. Add a tampered node-ipc version with ten million weekly downloads, plus fourteen misspelled packages on May twenty-eighth specifically hunting cloud and pipeline credentials.

And to concrete patching: the PAN-OS VPN flaw from Palo Alto has been actively exploited since mid-May, a working attack script is circulating, and US authorities CISA added it to their catalog on May twenty-ninth — with a deadline of June first. PostgreSQL released a patch roundup on the fourteenth with eleven flaws, including a reason to finally ditch old MD5 authentication.

In the background, the EU timer is ticking: starting September eleventh, the CRA reporting obligation takes effect, with twenty-four hours for early warning. Anyone bringing commercial software to the EU should set up the process now.

Tomorrow the CISA deadline for the Palo Alto flaw expires — that's the next date to watch. Otherwise things should stay quiet until the next npm wave arrives.
