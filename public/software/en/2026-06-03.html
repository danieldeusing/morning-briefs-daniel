Build 2026: Windows becomes a proper Unix box [1][2]

The headline-grabbing AI keynote at Microsoft Build (June 2–3) is the domain of the ai-dev brief — here only what devs get directly from tools beyond AI agents. And this time it's unusually tangible.

Coreutils for Windows is now generally available: over 75 Linux standard commands — ls, cp, mv, rm, cat, grep, find — run natively on Windows, without WSL and without a VM. Built from the open-source project uutils, a reimplementation of GNU Coreutils in Rust.[1] Technically a single coreutils.exe with NTFS hardlinks (ls.exe, cp.exe …) pointing to it.[2] For anyone shuttling between Linux servers and a Windows laptop, this removes a daily friction point in the shell.

Coreutils: These are the small foundational commands you use on Linux to navigate folders, copy files, and filter text — like the basic utensils in a kitchen. Until now the same toolkit had different names and lived in a different drawer on Windows; now the Linux toolkit lives right there in the Windows kitchen.

WSL containers & dev configs: less setup friction [2]

Second solid Build point: WSL containers — a built-in way to create, start, and manage Linux containers via familiar CLI and API directly from Windows; public preview, according to Microsoft, "coming soon".[2] That's the container engine a bit deeper in the OS, without the detour through a separate desktop app.

Plus Windows Developer Configurations: from fresh machine to work-ready environment "in minutes", with pre-configured tools — WSL, PowerShell 7, Git, GitHub CLI, VS Code, Python.[2] For Daniel's stack (Java/Spring, Angular, Python, Docker), that means less manual follow-up when setting up, more reproducible starting point. No miracle, but saves real hours on every new device.

Post-quantum crypto lands in Windows TLS stack [2]

Third Build point from the trust angle: Windows expands post-quantum cryptography with PQ-hybrid key exchange in the TLS stack and in certificate services.[2] "Hybrid" means: a classical and a quantum-resistant algorithm run together, so the connection stays secure even if one of them later becomes crackable.

"Harvest now, decrypt later": Attackers intercept encrypted traffic today and store it — hoping to decrypt it in ten years with a quantum computer. Like someone stealing a locked letter and squirreling it away until they eventually have the right key. PQ crypto makes sure the letter stays locked.

Practically no action pressure today, but the direction is set — TLS libraries and configurations will see hybrid suites as default over the next few years. Anyone running their own TLS endpoints (Caddy, reverse proxies) should keep this on the radar.

Chrome 149 stable: CSS gap decorations & scroll promises [3][4]

Chrome 149 hit the stable channel on 06.02.2026 (Android, ChromeOS, Linux, macOS, Windows).[3] Relevant for frontend work: CSS gap decorations — gaps in grid/flexbox can finally be styled directly (new properties like column-rule-inset, row-rule-visibility-items), analogous to column-rule in multi-column layout.[4]

Plus scroll promises: element.scrollTo() now returns a promise — an old gap that lets you wait for a scroll animation to end without timer gymnastics. Also, a page with an open WebSocket can now enter the back/forward cache because the socket closes proactively on entry. Small bits individually, but exactly the platform gaps that Angular/React devs otherwise bridge with polyfills.

npm "Miasma": Red Hat packages as worm vector [6][7]

On 06.01.2026, Wiz Research discovered a supply-chain compromise in the npm namespace @redhat-cloud-services: at least 32 packages, totaling 96 versions, contained unauthorized changes not matching the source repos.[6] The trigger was a compromised GitHub account of a Red Hat employee; the attacker pushed malicious GitHub Actions workflows into three RedHatInsights repos.[7]

The malware — dubbed "Miasma", a variant of the Mini-Shai-Hulud family — ran via a preinstall script: the payload starts automatically on npm install, before actual app code and without visible warning. It collects GitHub tokens, SSH keys, and cloud credentials (new in this variant: GCP and Azure identities).[6] Each infection gets a uniquely encrypted payload — hash-based IOCs are only good per package version.

preinstall script: npm is allowed to run a script ahead of time during package installation — meant for setup work. Like a piece of furniture that, when unpacked, automatically dispatches a helper. If the package is malicious, foreign code runs on your machine before you've even imported a single line of it.
Heads up If you use @redhat-cloud-services packages: audit affected versions, rotate GitHub tokens/SSH keys/cloud credentials, inspect CI/CD environments. Generally, npm install --ignore-scripts as default protects against exactly this class.

33 fake npm packages via dependency confusion [8]

Nearly simultaneously (05.28 to 05.29), Microsoft Threat Intelligence uncovered a second campaign: 33 malicious npm packages in three waves, distributed across nine organizational scopes, impersonating real internal company namespaces — classic dependency confusion.[8] Behind three maintainer aliases (mr.4nd3r50n, ce-rwb, t-in-one) sits, according to shared C2 infrastructure, a single actor.

Noteworthy: The packages were still in reconnaissance mode — they gathered system info, hostnames, environment variables, and developer context, with a built-in kill switch for full exploitation in follow-up attacks. So intelligence gathering before the actual strike.

Dependency confusion: Companies often use internal packages with their own names. If someone uploads a same-named package publicly, the package manager can accidentally fetch the public (malicious) one instead of the internal one. Like a delivery driver taking an impostor package with your name on it.
Mitigation Microsoft advises: scope-lock internal registries (no fallback to public npm), disable post-install scripts, block egress to C2 domains, inspect CI/CD logs. For your own projects: scoped packages and an upstream registry proxy.

CISA KEV: two fresh entries on 06.02 [9][10]

CISA added two actively exploited vulnerabilities to the KEV catalog on 06.02.2026:[9]

  • CVE-2022-0492 — Linux kernel, cgroups v1: a logic bug in release_agent permits privilege escalation and potentially container escape. Affects the older, but still widely deployed cgroup-v1 architecture.[10] An old vulnerability (2022) now apparently actively abused again — patch container hosts or audit cgroup v2.
  • CVE-2025-48595 — Android framework, integer overflow: local privilege escalation without user interaction. Google speaks of "limited, targeted exploitation" — the pattern often tied to spyware vendors.
CISA KEV catalog: A list of vulnerabilities proven to be exploited in the wild. U.S. federal agencies must close them by a deadline — for everyone else it's a prioritized "patch these first" because attackers are actually using them.

EU regulation: CRA reporting duty starting 09.11 [13]

Standing thread because the deadline is approaching: The reporting obligations of the EU Cyber Resilience Act (CRA) take effect as of 09.11.2026. Makers of "products with digital elements" must report actively exploited vulnerabilities and serious incidents — early warning within 24 hours, full report within 72 hours — via the CRA Single Reporting Platform that ENISA is currently building.[13] The main CRA duties take effect 12.11.2027, but the reporting obligation is the first hard deadline.

CRA / SBOM: The CRA requires, among other things, a machine-readable bill of materials for software components (software bill of materials) — like an ingredients list on a package, only for all libraries in the product. Without a mature SBOM process, the reporting deadlines are hard to meet.

Relevant for Daniel/twiceD once a product with digital elements enters the EU market: plan SBOM pipeline and a reporting process early, not scrambling in September.
