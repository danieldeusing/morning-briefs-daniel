Miasma: a worm in Red Hat's npm packages — and why "Trusted Publishing" didn't stop it [1][2][3]

On June 1st, 2026, multiple security teams independently reported a fresh supply-chain incident in the npm namespace @redhat-cloud-services: 32 packages were re-published in 96 versions with a credential stealer.[1] These packages together pull roughly 80,000–117,000 downloads per week — they sit inside Red Hat's Hybrid Cloud Console frontend libraries.[2]

Supply chain attack: Instead of attacking your app directly, someone poisons an ingredient you were going to use anyway — here an npm package that thousands of projects auto-pull. Like someone tampering with flour at the mill: every baker who buys it bakes poisoned bread without knowing.

The really interesting part is the how. The attacker didn't get in via a stolen npm token, but through a compromised GitHub account of a Red Hat employee — and then used OIDC / Trusted Publishing via GitHub Actions to pose as a legitimate publisher.[3] This very token-less model was celebrated as the hardening after the Shai-Hulud waves in 2025. It eliminates long-lived tokens — but if the identity anchor (the GitHub account) falls, so does the trust hanging on it.

OIDC / Trusted Publishing: Instead of a permanent password ("npm token"), your CI pipeline proves its identity anew per release — like a day pass instead of a permanent key. If someone steals the day-pass issuer (the GitHub account), they still get valid passes.

Miasma is a direct descendant of the Mini-Shai-Hulud worm from spring (which previously hit SAP and TanStack packages). What's new and striking: each infection encrypts its payload individually instead of copying itself byte-for-byte — that severely complicates signature-based detection.[4] The malicious code runs in the preinstall phase, scans the machine and reachable cloud for credentials, and re-publishes itself via every package that can reach the stolen tokens.

Red Hat's line: The compromised versions have been removed from npm; the publishing pipeline for console.redhat.com strips install scripts before deployment anyway; customers face "no action needed".[5] That's credible for the hosted console — but anyone pulling the packages directly from npm should check lockfiles and rotate CI secrets.

Heads up Not a "patch tonight" alarm — but a good reason to audit preinstall/postinstall scripts from dependencies in your own repos and make --ignore-scripts and pinned lockfiles a habit. The attack surface is the tooling layer, not a single CVE.

The uncomfortable punchline: GitHub itself concedes that Trusted Publishing in its current form does not prevent Shai-Hulud-like attacks — the token-less model is "an important step," not a final answer.[6] As of late 2025, the stricter regime was already in place anyway: 7-day max lifetime for write tokens, FIDO 2FA instead of TOTP, deprecation of classic tokens.[7] Since May 20, 2026, new Trusted Publisher configs must also explicitly name at least one allowed action.[8] All that helps — but doesn't close the "compromised developer account" hole.

CISA adds two actively exploited bugs to KEV list — container escape included [9][10][11]

On June 2nd, CISA added two vulnerabilities to the Known-Exploited-Vulnerabilities catalog, both confirmed actively exploited in the wild.[9]

CISA KEV catalog: A list from the US cyber agency of flaws that are demonstrably actively attacked — not "could theoretically," but "is happening right now." US agencies must close them within a deadline; for everyone else, the list is the best prioritization signal for what's actually burning.

CVE-2022-0492 (Linux kernel, CVSS 7.0) sits in cgroup_release_agent_write() in the cgroups v1 subsystem: due to insufficient authentication, a local attacker can bypass namespace isolation, escalate privileges, and in the worst case break out of a container onto the host — root included.[10] For Daniel's Docker/containerd world the more relevant of the two: an old bug (2022), but container escapes on the KEV list are the signal to check kernel and runtime versions.

CVE-2025-48595 (Android Framework, integer overflow, CVSS 8.4) affects Android 14–16, enables code execution plus privilege escalation, and per Google is under "limited, targeted exploitation".[11] The BOD-22-01 deadline for agencies already passed on June 5th — anyone managing mobile fleets should enforce the June patch level.

Days later (06-03), CVE-2026-45247 was added — a CVSS 9.8, unauthenticated RCE via PHP object deserialization in "Mirasvit Full Page Cache Warmer" from the Magento/Adobe Commerce ecosystem (affects versions < 1.11.12).[12] Agency deadline was 06-06 — Magento operators should treat this urgently, not defer.

EU CRA: the reporting clock is ticking louder [13]

Standing thread, but increasingly relevant: From September 11th, 2026 onwards, the reporting obligations of the EU Cyber Resilience Act (CRA) go into effect — the rest of the law follows only at end of 2027.[13]

Cyber Resilience Act (CRA): An EU law that requires software vendors (including small ones!) to secure their products over their lifetime and report incidents. Like product liability for code: whoever brings "products with digital elements" to the EU market must answer for their security.

Concretely from September: if there's an actively exploited flaw or serious incident, the vendor must give a 24-hour early warning, a 72-hour full report, and no later than 14 days after a fix is available a closure report — all via the central Single Reporting Platform, addressed to the responsible CSIRT and in parallel to ENISA. For organizations of twiceD's size, that means: a lean, documented incident process must be in place by summer. The platform itself is supposed to be operational by the deadline.

PostgreSQL 19 Beta 1: property graphs, REPACK, and finally parallel autovacuum [14]

For anyone running Postgres as default database (that's Daniel): On June 4th, PostgreSQL 19 Beta 1 shipped.[14] GA is expected for September/October 2026 — beta means: test, don't run in production.

The highlights that actually change things in practice:

  • SQL/PGQ property graph queries — graph queries directly in SQL, no separate graph DB needed.
  • REPACK — a new command to reorganize bloated/inflated tables without VACUUM FULL lock pain.
  • Parallel autovacuum — cleanup now runs across multiple workers; noticeable on large tables.
  • ON CONFLICT DO SELECT — on upsert get the colliding row back instead of a second roundtrip.
  • Native JSON for COPY TO and a WAIT FOR LSN guard against stale reads on replicas.
Autovacuum: Postgres marks deleted/changed rows as "dead" first and cleans them up later in the background — like a restaurant clearing tables after guests leave. Where it used to run single-threaded, it would block on large tables; now multiple workers clean up at once in parallel.

It's Monday, June eighth. The news day itself is quiet — but the supply-chain saga in the npm ecosystem has a new chapter, and that's the story of the day.

On June first, a worm called Miasma re-published itself across thirty-two npm packages in the Red Hat Cloud Services namespace — ninety-six versions, together between eighty and one hundred seventeen thousand downloads per week. The worm steals credentials, runs right at install time, and encrypts itself differently with each infection so antivirus scanners have a harder time spotting it. The real gut-punch: the attacker came in through a hijacked GitHub account and leveraged exactly Trusted Publishing — the very mechanism supposed to make stolen tokens unnecessary. Red Hat has removed the affected versions and says customers face no action needed. GitHub itself admits that the new protections still don't prevent Shai-Hulud-style attacks.

If you have your own projects, don't read this as panic — read it as a prompt: audit install scripts from your dependencies, pin your lockfiles, rotate CI secrets. The attack surface is the tooling layer, not a single vulnerability.

The more important one for the container world is an old Linux kernel bug from twenty twenty-two that lets you break out of a container onto the host. The other hits the Android Framework in versions fourteen through sixteen. The government deadline already passed on June fifth.
