The biggest Patch Tuesday since 2017 — record volume, but no fresh in-the-wild zero-day [1][2]

Yesterday was Microsoft's monthly Patch Tuesday, and it broke every previous record. The Zero Day Initiative — whose analyst has counted every month since 2017 — tallies 208 CVEs; the previous high stood at 177 the year before. The mainstream security press counts slightly more conservatively ("~198–200 holes", excluding Microsoft's own duplicate edge entries), but either way: this is the fattest monthly round the program has ever seen.[1][2]

What's remarkable is the risk structure underneath — and it's more reassuring than the number suggests. Three of the holes were publicly known in advance (zero-days in the strict sense), but according to Microsoft at patch time not actively exploited: a BitLocker bypass through physical access (CVE-2026-50507), an HTTP/2 resource exhaustion in HTTP.sys that can take down a server (CVE-2026-49160), and a privilege escalation through link-following in the Collaborative Translation Framework (CVE-2026-45586). Additionally, the Defender hole already reported in May and exploited since then rolls into the cumulative June engine: CVE-2026-41091 (CVSS 7.8, SYSTEM privileges).[1][3]

Heads up No "patch tonight" alarm. The record count is more a signal that Microsoft's internal fuzzing and bug bounty program find a lot — not that 200 doors suddenly opened. For Daniel's stack (Windows build runners, .NET, Visual Studio are in the list), that means: normal rollout rhythm, no emergency. The Defender EoP was due for US agencies anyway by 06/03.

Chrome V8 zero-day CVE-2026-11645: live in the wild, fifth this year — Chrome 149 patches [4][5][6]

This is the bug that really demands speed. Google has in Chrome 149 (builds 149.0.7827.102/.103 for Windows/macOS, .102 for Linux, rolled out 06/08–09) closed an out-of-bounds read/write hole in the JavaScript engine V8 — and confirmed that "an exploit for it exists in the wild".[4]

V8 / Out-of-bounds: V8 is the engine that executes every line of JavaScript in the browser — including code from any website. "Out-of-bounds" means: code reads or writes accidentally past the allowed memory region, like reaching into a neighbor's shelf in a high cabinet. Someone who controls that deliberately can run code in the browser — and just visiting a weaponized page is enough.

CVSS 8.8. It's the fifth in-the-wild exploited Chrome zero-day this year. The BSI has rated the hole as high-risk and warned of active attacks. In practice: Chrome (and every Chromium derivative — Edge, Brave, Vivaldi) checks and forces restart so the update actually takes effect.[5][6]

Cisco Catalyst SD-WAN Manager: root hole without patch, already exploited [7][8]

A command injection in the Cisco Catalyst SD-WAN Manager CLI (CVE-2026-20245, CVSS 7.8) allows an attacker with netadmin rights to upload a weaponized file and run arbitrary commands as root. Cisco is already seeing limited exploitation — including cases where configuration changes were pushed to edge devices. It's the seventh SD-WAN zero-day of the year at Cisco. As of now there's neither patch nor mitigation; CISA listed the hole on KEV yesterday.[7][8]

Heads up The issue here is the lack of mitigation. Until a fix arrives: lock down access to the Manager web/CLI interface strictly to trusted management networks and audit netadmin accounts. A "requires netadmin" hole becomes critical the moment an attacker reaches the management segment through another route.

"Miasma" worm in Red Hat's npm: signed attestation didn't block the pipeline hijack [9][10][11]

A lesson in the supply chain as an attack surface — and exactly the software-engineering second-order effect that belongs in this brief, not in an AI brief. Since 06/01, 96 versions across 32 packages under the npm namespace @redhat-cloud-services have been poisoned with malware called "Miasma" (together ~117,000 downloads/week), a variant of the known "Shai-Hulud" worm. The poisoned package carries a preinstall script that on install immediately runs an obfuscated payload, steals developer and cloud credentials (GitHub Actions tokens, AWS/GCP/Azure keys, Vault, kubeconfig, SSH keys, all .env), and tries to spread to further packages on the victim's system.[9]

The real pivot point for Daniel's world: the code was smuggled in via a compromised GitHub account of a Red Hat employee with orphan commits bypassing multiple branch protections — and published through GitHub Actions OIDC. That is: the CI/CD pipeline itself was compromised, not a stolen npm token. That means the beautiful new world of signed provenance attestations falls flat: the artifact was "genuinely signed" from the real pipeline run — only the run itself was hijacked. Meanwhile, 57 more packages have been affected.[10][11]

Provenance attestation: A cryptographic "origin seal" that proves which repo and which build run a package came from. Meant as an authenticity badge — but a seal only proves where something was built, not whether the build robot was kidnapped during construction.
Heads up Check lockfiles against the published IOCs, pin/remove affected @redhat-cloud-services versions, and rotate all credentials that were on a machine that saw a fresh npm install of these packages in the last few days.

PostgreSQL 19 Beta 1 is here — property graphs, ON CONFLICT DO SELECT, online checksums [12][13]

On 06/04, the first beta of PostgreSQL 19 landed — this is the moment when it pays to test a copy of your own database with it, long before GA arrives in the fall. The highlights for Daniel's stack:

  • SQL/PGQ property-graph queries: graph queries directly in SQL, without a separate graph database on the side.
  • ON CONFLICT DO SELECT: atomic fetch-or-create in a single statement — sidesteps the classic upsert race condition.
  • Parallel autovacuum and faster inserts: less maintenance stall on large tables.
  • Online checksums: data checksums can be toggled on/off while the cluster is running, no restart needed.

Two defaults have flipped, things to watch on upgrade: JIT is now off by default (was a frequent source of surprise latency), TOAST compression now uses lz4 by default, and RADIUS authentication is removed. Beta means: for testing, not production — but that's exactly what it's for.[12][13]

EU CRA gets a turn of the screw tomorrow: conformity body rules take effect 06/11 [14]

The Cyber Resilience Act ramps up in stages, and tomorrow is a milestone: starting 06/11/2026, the rules from Chapter IV governing the designation of conformity assessment bodies take effect — national authorities can now designate the bodies that certify high-critical products. For Daniel as a solo/small developer, this isn't an action item yet, but it's the lead-in to what really bites in the fall.

The date to actually mark: 09/11/2026. That's when the Article 14 reporting duties kick in — manufacturers of products with digital elements must actively report exploited vulnerabilities and incidents, with 24 hours for early warning and 72 hours for full disclosure. If you ship software with "digital elements" in the EU, you should have your reporting process in place by then.[14]

It's Wednesday, June tenth. Yesterday was Microsoft's Patch Tuesday, and it broke a record — it's the largest monthly security round the program has ever seen. Today is mostly about security: a record patch day, a Chrome emergency, an unpatched Cisco bug, and a worm deep in the npm supply chain.

First Microsoft. The Zero Day Initiative counts two hundred and eight vulnerabilities in this round; the old record was one hundred and seventy-seven. Sounds like an alarm, but it's not. Three of the holes were publicly known in advance — a BitLocker bypass, an HTTP two denial of service hole, and a privilege escalation — but at patch time not actively exploited. The one Defender hole that really is being attacked had already been known since May. For Daniel's build runners, that means: normal rollout, no drama.

The bug that really deserves speed is in the browser. Google has in Chrome one hundred and forty-nine closed a hole in the JavaScript engine V8 — the number is CVE twenty twenty-six, eleven-six-four-five. Google confirms that an attack is already running in the wild, the severity is eight point eight, and it's already the fifth exploited Chrome zero-day this year. The BSI explicitly warns. Just visiting a weaponized website can be enough — so update Chrome and all Chromium variants and restart.

Then Cisco. In the Catalyst SD-WAN Manager there's a hole through which an attacker can run commands as root. It's already being exploited, in some cases even pushing configurations to edge devices — and there's no patch or mitigation yet. The only thing that helps is locking down management access tight until Cisco ships.

The lesson of the day comes from the supply chain. Since June first, ninety-six versions across thirty-two packages in Red Hat's npm namespace have been poisoned with a worm called Miasma. What's interesting isn't the theft of credentials, but how it happened: via a hijacked GitHub account and the build pipeline itself. So the package was cleanly signed — from the real run — only the run was kidnapped. The beautiful new world of signed provenance proofs didn't stop it. Whoever installed these packages should check lockfiles and rotate keys.

Two quieter pieces to close. PostgreSQL nineteen has landed as a first beta — with graph queries right in SQL, an atomic fetch-or-create, parallel autovacuum, and checksums you can flip while running. Now's exactly the time to test it on a copy. And in Europe, the Cyber Resilience Act gets a turn of the screw tomorrow; the date that really matters is September eleventh, when the twenty-four and seventy-two hour reporting duties kick in.
