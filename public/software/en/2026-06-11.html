Three exploited holes in the KEV — and twice the vendor says "no patch"

On 06-09, CISA added three actively exploited vulnerabilities to its KEV catalog.[1] What's unusual: two have no fix — the vendors consider it more dangerous than the hole itself.

KEV catalog: CISA's list of vulnerabilities proven to be exploited right now. Think of it like a wanted poster with "last spotted robbing the bank" — not a theoretical risk, but a door someone is already walking through. US agencies must close them by the deadline (here: June 23); for everyone else, the list is the sharpest free prioritization filter that exists.

Arista EOS CVE-2026-7473 (CVSS 6.9): Only affects switches configured as tunnel endpoints (decap-groups, GRE, VXLAN) — then the tunnel protocol type is not validated and unintended tunnel traffic gets through. Arista plans no patch because a fix would break existing configs; mitigation is ACLs on upstream devices or the affected switch.[2]

Cisco Catalyst SD-WAN Manager CVE-2026-20245 (CVSS 7.8): A locally authenticated attacker can execute arbitrary commands as root via a crafted file. Here too: neither patch nor mitigation. True, it needs netadmin rights — but you get those via stolen credentials or older Cisco holes (CVE-2026-20182, -20127).[3]

The third case, the Chromium V8 hole, has its own section next. For Daniel's infrastructure stack, the key takeaway is this: "no patch" doesn't mean "no problem", it means "defense shifts to network segmentation and ACLs". SD-WAN manager interfaces don't belong on the open network, period.

  2. securityweek.com — Arista CVSS 6.9, no patch planned
  3. thehackernews.com — Cisco SD-WAN CVSS 7.8, RCE as root

Chrome V8 zero-day [4][5]

The third KEV entry of the day is closer to Daniel's day-to-day: CVE-2026-11645 (CVSS 8.8), an out-of-bounds read in V8, Chrome's JavaScript and WebAssembly engine. An attacker can execute code in the sandbox via a crafted HTML page; Google confirms an "exploit in the wild" but stays silent on details. It's the fifth actively exploited Chrome hole this year.

The fix ships in Chrome 149.0.7827.103 (Windows/Mac) or 149.0.7827.102 (Linux) — and in all Chromium descendants (Edge, Brave, Electron apps). If you run Electron-based tools, track their Chromium versions: the hole travels with them. Pro tip: restart the browser so the update takes effect — an open session keeps running on the vulnerable engine.

  2. theregister.com — fifth exploited Chrome hole in 2026

Patch Tuesday: 200 flaws, one Exchange spoofing hole already burning

Microsoft's June round (06-09) is big: 200 flaws, 33 rated "critical" (28 RCE), and six zero-days — five disclosed in advance, one already exploited.[6] The exploited case is CVE-2026-42897, a spoofing vulnerability in Exchange Server: a crafted email can execute arbitrary JavaScript in the browser context when opened in Outlook Web Access under certain conditions.

For Daniel as a Spring/FastAPI user with no on-premise Exchange, that's mostly context — but the five pre-leaked holes (two BitLocker bypasses, one HTTP.sys DoS via "HTTP/2 bomb", one CTFMON privilege escalation) are a useful signal: Windows clients in mixed environments come up on the next maintenance window.

Context Patch Tuesday is news, not an alarm. The one actively exploited hole tops the list; the rest is routine maintenance, not "patch tonight".

CRA: the real deadline is September, not 2027 [15]

Standing thread, today with a concrete date: on the EU Cyber Resilience Act, many assume 12-2027 is the deadline — full vendor obligations don't kick in until then. But the reporting mandate starts 09-11-2026: actively exploited holes and serious incidents must be reported via a central ENISA platform, with early warning in 24h, report in 72h, final report within 14 days.

Anyone shipping "digital elements" to the EU market like twiceD does should have SBOMs and a vulnerability-management process by then — without them you can't meet the 72-hour deadline. For a solo shop it's not a crisis, but it's worth setting up cleanly before the clock starts.

  1. herodevs.com · ec.europa.eu — reporting mandate from 09-11-2026 (24h/72h/14d)

The npm supply-chain wave hits Red Hat — 32 packages, a 4.29-MB dropper

On 06-01, a compromise of the @redhat-cloud-services npm packages went public. The attack — Microsoft calls it "Miasma" — came via a compromised GitHub account of a Red Hat employee and trojaned 32 packages across more than 90 versions, totaling ~80,000 downloads/week.[7]

preinstall hook: an npm package can register a script that runs automatically before the package is even installed. It's meant for setup tasks — like furniture that assembles itself as soon as the box hits the hallway. Here, the hook pulled in a heavily obfuscated 4.29-MB stealer and searched for CI secrets, AWS/GCP/Azure/Kubernetes/Vault/npm tokens.

The good news: Red Hat's release pipeline strips install scripts before shipping them to console.redhat.com — and during the compromise window there was no console release. But pulling directly from npm? That's where infection happened. Anyone with @redhat-cloud-services packages in the frontend should check lockfiles and rotate CI tokens.[8]

  1. microsoft.com — 32 packages, >90 versions, 4.29-MB dropper
  2. access.redhat.com — RHSB-2026-006, console unaffected

…and before that, TanStack: same worm, same mechanism

Red Hat is not an outlier but a continuation of the "mini-Shai-Hulud" worm that already hit TanStack packages on 05-11: 84 malicious artifacts across 42 packages in the @tanstack namespace, CVE CVE-2026-45321 with CVSS 9.6.[9] Entry was a chain of three GitHub Actions weaknesses — a PR triggered a pull_request_target workflow, the cache was poisoned with a rogue pnpm store, and the legitimate release pipeline published with TanStack's own OIDC identity.

Across npm and PyPI combined, the wave hit 170+ packages with over 518 million cumulative downloads.[10] Daniel uses TanStack components (Query, Router) in the React space — versions from the May window should be checked. But the real lesson lies in the attack vector: not a maintainer password falling, but the CI pipeline being hijacked and self-signing. A trusted signature ≠ trusted content when the build runner is compromised.

  2. orca.security — 170+ packages, 518 million downloads, GitHub Actions chain

JFrog's annual report: why the vulnerability flood is AI-driven and exploding

The thread connecting both incidents comes from JFrog's "State of the Union 2026" (released 05-20): 48,000 new CVEs in the year (+20%), malicious npm packages +451%. The finding most interesting for Daniel is the class shift: SQL-injection CVEs +445%, the injection class (CWE-74) overall +3,110%.[11]

JFrog's thesis: AI assistants generate decades-old, well-understood vulnerability classes faster than manual review can catch them — injection is back because the generator writes it that way. At the same time, the report deflates the panic: of 248 high-profile CVEs examined, only 11.9% were actually exploitable.[12] The lesson isn't "scan more", it's "prioritize better" — and don't commit generated code unreviewed.

Angular 22: the "Signal-first" era is official [13]

For Daniel as an Angular developer, this is the week's most important release. Angular 22 (06-03) is a consolidation release — experiments are over: the app is now zoneless by default, OnPush is the default change-detection strategy, Signal Forms graduate from experimental to stable, and Vitest replaces Karma/Jasmine as the default test runner.

Zoneless: Angular used to track changes automatically via the Zone.js library, re-checking everything whenever anything changed — like a janitor who walks the whole building every time there's a noise. With signals, every value tells you when it changes, and only the affected part rerenders. Less magic, less overhead, more precise.

In practice: migrating from an older version means planning for OnPush-by-default and the loss of Zone.js — both can affect components that relied on the old "re-check everything" behavior. The Vitest switch is the easier part: faster test runs, a modern ecosystem.[13]

Docker Desktop plugs a container-escape hole

For Daniel's Caddy/Tailscale/Docker setup on ddmini/ddStudio, "copy.fail" is the relevant item — container isolation is exactly the security guarantee you build on. Time to update Docker Desktop.

It's Thursday, June eleventh. Two days after June Patch Tuesday, the real story isn't a big release — it's an uncomfortable truth from Washington: CISA added three actively exploited holes to its mandatory list on Monday, and two of them have no patch.

It hits network gear. Arista switches and Cisco's SD-WAN manager — the vendors say flat out: we're not shipping a fix because it would break existing configs. Instead, you defend with access lists and network segmentation. Agencies have until June twenty-third to do it, a good deadline for everyone else too. The message underneath: no patch doesn't mean no problem. It means defense shifts to network design.

The third hole is closer to home: a zero-day in V8, Chrome's JavaScript engine. An attacker can run code from a crafted web page, and Google confirms an exploit is already circulating. That's five exploited Chrome holes just this year. The fix is in version 149.0.7827.103 — and in all Chromium offshoots, including Electron apps. Restart your browser once, or the open session keeps running on the old engine.

Microsoft's Patch Tuesday on the ninth was large: two hundred flaws, six zero-days, one already burning — a spoofing hole in Exchange that can run JavaScript from a crafted email. For someone without their own Exchange, that's mostly context. But a signal for the next Windows maintenance window.

The second big thread is the supply chain. After TanStack in May, it's Red Hat this time: a worm called Miasma trojaned thirty-two npm packages across more than ninety versions. The trick was a preinstall hook — a script that runs before the package even installs — dragging in a four-point-two-nine megabyte credential stealer that vacuumed up cloud and CI tokens. Good news: Red Hat's own pipeline strips those scripts, the web console wasn't hit. But anyone pulling straight from npm checks lockfiles now and rotates tokens.

Both incidents share the same mechanism, and that's the real lesson: it wasn't a password that fell. It was the build pipeline that got hijacked and self-signed. A trusted signature says nothing about trusted content when the runner is compromised.

JFrog gives the reason in its annual report: forty-eight thousand new vulnerabilities in the year, up twenty percent — and SQL-injection holes went up four-fold. The theory: AI assistants write old, well-understood error classes faster than humans can catch them in review. But the report also tempers the panic: of just over two hundred high-profile holes, only about twelve percent were actually exploitable. Better prioritize, don't scan more.

Two good-news releases. Angular twenty-two commits to the signal era: zoneless as standard, OnPush as standard, Signal Forms stable, and Vitest replaces the old test runner. And PostgreSQL nineteen entered its first beta — general availability expected in fall, a good time to test the beta against a staging database.

In the background, the clock is running on Europe's Cyber Resilience Act: unlike what many think, the first hard deadline isn't twenty twenty-seven. It's September eleventh this year — from then on, actively exploited holes must be reported within twenty-four hours.
