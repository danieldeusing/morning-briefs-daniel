The largest Patch Tuesday ever — featuring a wormable kernel hole [1][2]

Microsoft's June cumulative patch from June 9 is off the charts: Trend Micro's Zero Day Initiative counted 208 CVEs from Microsoft alone — the largest single Patch Tuesday since the program started in 2003, beating the previous record of 177 (2025). Add in Chromium and other bundled third-party components, and the month hits 571 CVEs.[1] This is not a "click and move on" situation — it's a real planning job for anyone running Windows servers.

The one hole that stands out: CVE-2026-45657, a use-after-free in the Windows kernel when processing TCP/IP traffic. CVSS 9.8, explicitly rated by ZDI as wormable — no login, no user interaction, the attack can propagate itself across the network.[2] The profile recalls EternalBlue, the hole behind WannaCry. Multiple researchers expect the gap between patch and working public exploit to be measured in days rather than weeks.

Wormable: A "normal" vulnerability needs someone to click it or actively exploit it. A wormable flaw is like a cold in an open office: one infected machine infects its neighbors on its own, who infect theirs — without anyone doing anything. This property is what made WannaCry so devastating in 2017.

Currently being actively exploited is a different bug from the pile: CVE-2026-41091, a privilege escalation in Microsoft Defender (CVSS 7.8) — it elevates an attacker who already has a foothold to full machine control.[2] Not a direct hit to Daniel's Linux/container world, but if you have Windows hosts in reach — even just a build-runner VM — don't defer the kernel fix to the next maintenance window.

Spring dumps its full portfolio patch train — and Daniel's stack is in it

The usually quiet Spring release train turned into a security convoy this week: between June 8 and 12, about a dozen advisories stacked up across the portfolio — Framework, Boot, Cloud, Integration, Security, Statemachine, WS.[3] For anyone running Java/Spring, this is the patch priority of the week, not the kernel.

The sharpest entries: In Spring Statemachine, CVE-2026-41862 (HIGH) allows deserializing persisted state-machine contexts via Kryo without a class allowlist — according to spring.io, "can lead to remote code execution," the classic deserialization path to code execution. In Spring Integration, the remote file synchronizer writes files without path canonicalization to arbitrary locations (CVE-2026-40987, HIGH) — a textbook path-traversal bug.[3]

Deserialization: A program saves an object as a byte sequence (serialization) and rebuilds it later (deserialization). If strict checks aren't applied to which classes are allowed, an attacker can slip in a crafted byte sequence — like a kit that secretly starts a second machine when assembled. "Class allowlist" means: only accept explicitly approved components.

Add denial-of-service in Spring Framework via versioned static resources (CVE-2026-41842; affects 5.3, 6.1, 6.2, and 7.0 lines) and unbounded SpEL cache (CVE-2026-41851), a header-trust issue in Spring Cloud Gateway (CVE-2026-47825, HIGH), plus various medium-severity fixes in Boot, Security, and WS.[4] Pragmatically: check which of these modules you're running, and upgrade selectively to the patched versions — blindly bumping the whole stack is rarely needed.

On the side, for the ai-dev beat In the same batch is CVE-2026-47835 in Spring AI: special characters in vector-store metadata filters can force arbitrary queries in Elasticsearch/OpenSearch/GemFire. This is a real injection class at the RAG layer — but the model-side piece belongs in the ai-dev brief.

"Miasma": the npm/PyPI worm now chewing through Red Hat scopes

Since June 1, a new wave of the Shai-Hulud supply-chain worm has been running, internally called "Miasma" (npm) or "Hades" (PyPI). The tally grows daily: SecurityWeek counts over 100 affected packages across npm and PyPI; in the Miasma variant alone, there were at least 57 npm packages and over 300 malicious versions by June 5.[5] The most spectacular landing: in the first wave, 32 packages in the @redhat-cloud-services scope (Red Hat's hybrid-cloud console ecosystem) were published with malicious versions via compromised repos.

The mechanism is the real lesson: Miasma sneaks in via crafted binding.gyp files during npm install, Hades via *-setup.pth at Python startup — hooks that run automatically on install or import. The payload scans the local system and connected cloud services (AWS, GCP, Azure, Kubernetes) for credentials, API keys, and tokens, exfiltrates them, and uses the stolen access to spread further via the victim's repos.[5]

A new twist, handle with care Multiple trackers (including Socket) report an AI component: some packages planted .cursorrules and CLAUDE.md files with zero-width Unicode-hidden instructions, so an AI coding assistant would "follow" them when opening the project and run a supposed security scan that leaks secrets. This claim stands only in aggregator sources so far and is unconfirmed by SecurityWeek — noted as an early signal, not as established fact. The confirmed core remains classic install-hook theft.

Concretely for anyone with npm or pip lockfiles: enforce --ignore-scripts in CI installs, audit lockfiles for the affected versions, and rotate CI/cloud tokens that have passed through fresh installs since early June. Those tokens are the prize.

Ivanti Sentry: bullseye CVE in the KEV list, agency deadline tomorrow

CISA added CVE-2026-10520 to the Known-Exploited-Vulnerabilities catalog on June 11 — an OS command injection in Ivanti Sentry (before R10.5.2/R10.6.2/R10.7.1) with a maximum score of CVSS 10.0. A remote, unauthenticated attacker achieves root RCE on publicly exposed instances in certain configurations.[6] The agency deadline is tight — patch by June 14.

The backstory is interesting: the Shadowserver Foundation documented that many internet-exposed Sentry gateways were compromised shortly after patches became available — even though Ivanti initially saw no signs of active exploitation.[6] Sentry is Ivanti's mobile/VPN gateway, typically sitting right at the network edge. Not a direct hit to Daniel's stack, but the pattern "edge appliance + unauthenticated RCE + immediately exploited" is reliable enough now that every such box deserves its own patch track.

Docker Engine 29: BuildKit escape and AuthZ bypass fixed

The Docker Engine 29 line shipped two security fixes relevant to container builders. CVE-2026-33747: in BuildKit, an untrusted frontend could write files outside the BuildKit state directory — an escape from the intended workspace, fixed in BuildKit 0.28.1. Plus CVE-2026-34040, an authorization bypass in AuthZ plugins that can circumvent authorization plugins under certain conditions.[7]

Also bundled: containerd 2.2.2 and Go runtime 1.25.8. For Daniel's Docker/CI practice: if you run builds from not-fully-controlled Dockerfiles or frontends (multi-tenant CI, third-party PRs), upgrade the Engine to the 29 fix level — the BuildKit issue is exactly where "untrusted build input" meets "shared builder" that it's dangerous.

Python 3.15: Feature freeze — what lands in October

With Beta 1, Python 3.15 has been in feature freeze since May 7; from now on, bugfixes only, final release on October 1.[8] So what's coming is locked in — and some of it is pleasantly practical: a frozendict as a builtin (immutable dict, finally without a third-party library), clean cancellation of asyncio TaskGroups, thread-safe iterator helpers that replace a lot of Queue boilerplate, and a builtin sentinel object (PEP 829).[9]

Feature freeze: From this point forward, no new features are accepted — the version is "content-complete," only bugs get fixed. Like the moment when you stop changing the recipe and start tasting.

For FastAPI/Python services, nothing that forces action immediately — but frozendict and more robust TaskGroup cancellation are exactly the kind of quality-of-life improvements to pick up come October.

CRA: less than 100 days to mandatory reporting

Yesterday was the quiet June 11 milestone of the EU Cyber Resilience Act topic here (the conformity assessment body rules). Today the follow-up: it's now less than 100 days until the operationally toughest piece — the Article 14 mandatory reporting requirement starting September 11, 2026, the obligation to actively report exploited holes and major incidents to ENISA or the CSIRT within tight timeframes.[10]

The catch: the central reporting platform (Single Reporting Platform) is not yet live, according to the current status. ENISA plans to deliver registration guidance, training materials, and dry runs in June; the platform is supposed to be operational by September 11, with a test phase before then.[10] For a solo shop like twiceD, not yet acute pressure — but if you bring "digital-element products" to the EU market, now is the time to think through your incident-reporting process, not in September when the 24/72-hour window gets tight.

RoguePlanet remains open [11]

As a reminder from this week's thread: the Defender zero-day "RoguePlanet" — the race condition that elevates a local user to SYSTEM — has yet to get either a CVE or a fix even after Patch Tuesday. Researcher "Nightmare Eclipse" published it nearly simultaneously with June's patch day, hours after Microsoft fixed two other Defender holes.[11] That means: Defender remains an open attack surface for local privilege escalation. Standard hygiene — accounts without admin rights, EDR telemetry sharp — remains the only brake.

It's Saturday, June 13. The release calendar is quiet, but the security week was one of the heaviest in recent times — and it carries into the weekend.

The biggest story comes from Microsoft. The June cumulative patch is the largest in Patch Tuesday's history: the Zero Day Initiative counted two hundred eight holes from Microsoft alone; the previous record was one hundred seventy-seven. With third-party components like Chromium, the month reaches five hundred seventy-one. The one hole that stands out has the identifier CVE-2026-45657 and sits in the Windows kernel, in the handling of network traffic. It has a severity of nine point eight and is rated as wormable — it can spread itself across the network without anyone clicking on anything. The profile is reminiscent of EternalBlue, the flaw behind WannaCry. Researchers expect a working exploit to emerge in days rather than weeks.

For Java developers, though, Spring is the real patch priority this week. The otherwise quiet Spring release train dumped about a dozen security advisories between June 8 and 12 across the entire portfolio — Framework, Boot, Cloud, Integration. Particularly serious: a deserialization flaw in Spring Statemachine that can lead to code execution, and an error in Spring Integration that allows writing files to arbitrary locations. If you use Spring, you should check which modules are affected and upgrade selectively to the patched versions.

A third story shows how supply-chain threats continue to escalate. Since early June, a new wave of the Shai-Hulud worm, internally called Miasma, has been circulating. Over one hundred packages in npm and PyPI ecosystems are affected, including thirty-two packages in a Red Hat scope. The malware runs automatically at install, searches the system and connected cloud services for credentials, steals them, and spreads itself using the stolen access. There are also reports of an AI component where hidden instructions in config files could compromise a coding assistant — but that claim stands only in individual sources and is not fully confirmed.

There was also a bullseye flaw in Ivanti Sentry, the mobile and VPN gateway: a remote attacker without authentication achieves root access, with a severity of ten out of ten. CISA added the flaw to the list of actively exploited vulnerabilities on June 11 and gave agencies a deadline of June 14.

On the quieter side: the Docker Engine 29 line shipped two security fixes, an escape from the build directory and a bypass of authorization plugins. And Python 3.15 reached its feature freeze — it brings a built-in immutable dictionary and cleaner cancellation of asyncio tasks, among other things. The final release comes on October 1.

For the EU Cyber Resilience Act, the countdown is on: less than one hundred days until the hard mandatory reporting requirement on September 11. Interestingly, the EU's central reporting platform hasn't been set up yet.

On the horizon for next week: on Wednesday, Node opens its scheduled security-release window across multiple version lines. And whether a public exploit emerges for the wormable kernel bug will be the question of the week.
