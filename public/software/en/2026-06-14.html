Chrome and Edge update for the weekend: actively exploited V8 hole [1]

The one piece of maintenance that can't wait for the next maintenance window: Google released a security update for Chrome on June 8th that closes CVE-2026-11645 — a memory hole (out-of-bounds access) in the JavaScript engine V8 with CVSS 8.8. Google confirms that the hole is already actively exploited; that's why it qualifies as a zero-day.[1] It's patched in Chrome versions 149.0.7827.102/.103.

Because Microsoft Edge runs on the same Chromium/V8 base, Edge is also affected and gets the update through its own channel.[1] Both browsers typically update themselves — but the update only takes effect after a browser restart. That's exactly what needs to happen today: open "About Chrome" or "About Edge," check the version, and restart.

Zero-Day & Sandbox: A "zero-day" is a hole that's being exploited before a patch is widely rolled out — defenders got zero days of lead time. Browsers normally isolate each webpage in a "sandbox," a sealed-off box, so a malicious site can't reach the rest of your computer. An engine hole like this is dangerous because it can be the first step to breaking out of that box — which is why a simple browser restart packs so much punch here.

June Patch Tuesday echoes: on-prem Exchange and an HTTP/2 DoS [2][3]

Microsoft's batch patch from June 9th was one of the largest ever — Tenable and BleepingComputer count around 200 CVEs.[2] Two entries deserve a second look over the weekend because they hit typical mid-market infrastructure.

First, CVE-2026-42897 in on-prem Microsoft Exchange (CVSS 8.1 per Microsoft; NVD: 6.1): an XSS vulnerability in Outlook on the Web that was already on the KEV list before the patch — meaning it was demonstrably exploited.[3] Anyone still running their own Exchange server should definitely have applied the June update. Second, the "HTTP/2 Bomb" bug CVE-2026-49160 in HTTP.sys (CVSS 7.5): a denial-of-service that cripples the Windows HTTP stack via crafted HTTP/2 traffic patterns — relevant for anyone running IIS or Windows-hosted services directly on the net.[2] Plus a privilege escalation in Windows text input (CTFMON, CVE-2026-45586, CVSS 7.8).

For Daniel's Linux/container world, no direct stack impact — but the pattern holds: Windows hosts in reach (even if it's a build-runner VM) belong on the June update before public exploits lower the bar for attackers.

  2. securityaffairs.com

PostgreSQL 19 Beta 1: graph queries come to the core [4]

The week's bright release news: PostgreSQL 19 Beta 1 landed on June 4th and now stakes out the feature set that will ship at GA target in September/October.[4] Three points stand out.

First, SQL/PGQ: property graph queries directly in SQL — you can query relationships ("who's connected to whom") as a graph without spinning up a separate graph database. Second, parallel autovacuum: Postgres' cleanup mechanism can finally run multiple workers in parallel, which noticeably speeds up maintenance on large tables. Third, around 2× faster inserts with foreign keys — a detail that makes real impact in write-heavy applications.[4]

Autovacuum: Postgres marks deleted/updated rows as "dead" and sweeps them up later in the background — that's autovacuum. On large tables, this housekeeping used to be done by one worker; now multiple can sweep in parallel, so the table gets cleaned up faster.

For Daniel's stack, nothing ready for production yet — a beta doesn't belong in production. But anyone planning a new project can already experiment with the graph features against the beta. Additionally, the security release from May 14th (11 CVEs, CVSS up to 8.8) still applies to all supported branches — unlike the beta, that should definitely be applied.

  1. postgresql.org

npm/PyPI supply chain: the wave continues [5]

Continuation from yesterday: The Shai-Hulud/"Miasma" campaign against npm and PyPI hasn't died down.

Since early June, over 100 packages across npm and PyPI have been hit by a supply-chain campaign; the most spectacular impact remains the 32 packages in the @redhat-cloud-services scope, whose malicious versions combined racked up roughly 80,000–117,000 weekly downloads.[5] No CVE has been assigned yet — the threat runs through package registry metadata, not a classic software hole.

What to do stays the same and applies to anyone with npm or pip lockfiles: check your lockfiles against the versions reported in the advisories, force npm install --ignore-scripts in CI pipelines, and rotate all CI and cloud tokens that have run through fresh installs since early June. Sources: Wiz and SecurityWeek carry the affected package lists.

  1. wiz.io · securityweek.com

EU Cyber Resilience Act: next building block activated [6]

The CRA switches on in stages, and on June 11th the next block came into force: rules for notified conformity assessment bodies. In plain terms: member states must now designate their notifying authorities — the bodies that will later be allowed to check whether a "product with digital elements" is CRA-compliant.[6]

For a solo shop like twiceD, that's not an action trigger yet, but it's the runway to the operationally toughest part: the reporting obligation starting September 11th, 2026, the duty to report actively exploited holes within tight timeframes. Anyone bringing software to the EU market should think through their own reporting and compliance process now — the infrastructure behind it is just being built.

  1. digital-strategy.ec.europa.eu
