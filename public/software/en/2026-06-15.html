Check Point VPN: critical auth bypass, actively exploited [1][2]

The week's loudest story: CVE-2026-50751 is an authentication bypass in the IKEv1 code path of Check Point's VPN stack with CVSS 9.3 (Critical).[1] Affected are Remote Access VPN, Mobile Access/SSL VPN, and Spark Firewall in version branches R80.20.X through R82.10 — four of these branches (R80.20.X, R80.40, R81, R81.10) are formally end-of-support, but still in heavy use in many enterprise networks.[1]

More important than the score is what lies behind it: Rapid7 and Help Net Security confirm active exploitation since May 7, 2026, with a spike in early June, attributed to a Qilin ransomware affiliate — dozens of organizations reportedly already hit.[2] Bundled with it is CVE-2026-50752 (CVSS 7.4), a related man-in-the-middle weakness in the same IKEv1 path for site-to-site VPN.[1]

IKEv1 & auth bypass: IKE ("Internet Key Exchange") is the handshake language in which VPN endpoints greet each other and negotiate a shared secret before the actual tunnel opens. An auth bypass means: the attacker can skip this greeting step — like a bouncer who stops checking IDs. Whoever reaches the gateway IP can log in without authorization actually being validated.

For Daniel as a solo shop without Check Point boxes, no direct to-do — but the story matters because customer environments will have them. Anyone touching Check Point infrastructure even tangentially should now ask for the patch status and check whether the gateway is reachable from the internet; Germany's BSI flagged it as "high" on 6/9.[3]

Shai-Hulud, Miasma, Hades — and npm pulls the install hook [4][5]

The structurally most important story of the day has two halves that belong together: an ongoing supply-chain attack, and the platform response that goes live in four weeks.

Half one is the campaign. SecurityWeek counts 471 malicious artifacts across npm and PyPI for the linked waves Shai-Hulud, Miasma, and Hades; the suspected "TeamPCP" group has been active since September 2025, tooling source code was leaked in May 2026.[4] Concretely: On June 1, the npm wave hit 32 packages in the @redhat-cloud-services scope, totaling 57 npm packages by June 5 with over 300 malicious versions — payloads abuse binding.gyp to bypass standard controls and steal AWS, GCP, Azure, and Kubernetes credentials. On June 8, the second Hades wave hit PyPI with another ~29 packages that execute via setup.pth at Python startup and pull down a Bun runtime.[4]

Heads up If you maintain npm or pip lockfiles: check against the versions listed in the advisories, force npm install --ignore-scripts in CI, and rotate all CI and cloud tokens that have passed through fresh installs since early June.

Half two is the answer. GitHub announced that npm v12 in July 2026 will not execute preinstall/install/postinstall hooks from dependencies by default, unless explicitly allowed; additionally, git deps and remote-URL deps will be blocked by default.[5] This closes exactly the vector that Shai-Hulud, the eslint-config-prettier takeover, and the Toptal "Picasso" campaign used for persistence. Aikido quantifies the problem: in 2025, roughly 455,000 malicious packages were published exploiting the install-script vector.[6]

postinstall hook: When you run npm install foo, the package foo traditionally gets to run a small script directly — perhaps to compile a native binary. This exact mechanism became the favorite backdoor: a malicious package doesn't need you to call it; the script runs while you're installing. npm v12 makes this opt-in instead of default.

Prep now: npm upgrade -g npm to at least v11.16.0, then npm will warn you on every install about the breaking changes — time to see which package in your tree still relies on install scripts before the hard cutover in July.[5] This isn't purely an AI-tooling wave, but notably the defenders observed that attackers are further automating their tooling pipeline — persistence and spread are now self-replicating.

June Patch Tuesday aftermath: Kernel and HTTP.sys RCE [8][9]

Microsoft's batch patch on June 9 came with 198 CVEs (32 critical, 166 important) and was, per Tenable, the largest single Patch Tuesday delivery in program history — other sources count higher depending on methodology (BleepingComputer ~200, ZDI/Security Affairs ~208).[8] Two stand out, both with CVSS 9.8:

CVE-2026-45657 is a use-after-free in the Windows Kernel allowing remote, unauthenticated code execution with SYSTEM privileges — the worst-case profile of a server patch.[9] CVE-2026-47291 is an integer overflow in HTTP.sys, the Windows kernel driver that receives HTTP traffic directly: also remote, unauthenticated, RCE.[9] Both hit every Windows host directly connected to the network — whether IIS, WCF, or custom HTTP listener.

Add three Hyper-V guest-escape CVEs (CVE-2026-47652, -45641, -45607) as critical RCEs allowing a compromised VM to jump to the host — its own chapter for multi-tenant hosts.[9] Plus a cluster of 11 Remote Desktop Client CVEs (CVSS 7.5–8.8), including the "exploitation more likely" flagged CVE-2026-42985.[8] Three of the simultaneously disclosed zero-days (BitLocker CVE-2026-50507, HTTP.sys DoS CVE-2026-49160, CTFMON EoP CVE-2026-45586) were already publicly known; on top comes CVE-2026-41091 (Microsoft Defender Elevation-of-Privilege, CVSS 7.8), an actively exploited zero-day that Microsoft already patched out-of-band on 5/19–5/20 via Defender engine 1.1.26040.8 and now formally releases with the June batch — this one is the decision-critical one.[8]

For Daniel's Linux/container world, no direct stack touch — but the pattern is clear: build-runner VMs, Windows test hosts, and anything with IIS/WCF involvement belongs on the June baseline before exploit code circulates publicly.

CISA KEV from June 9: Chrome, Arista, Cisco SD-WAN [10]

The KEV list is the pragmatic filter when you need prioritization: what's on it is proven exploited. On June 9, CISA added three entries — the FCEB remediation deadline is 6/23, but the signal applies outside federal agencies too.[10]

First, CVE-2026-11645 in Google Chrome V8 (CVSS 8.8) — fifth actively exploited Chrome zero-day in 2026, patched in 149.0.7827.102/.103.[10] If you haven't forced the Chrome or Edge restart yet, now's the time. Second, CVE-2026-7473 in Arista EOS (CVSS 6.9) — router/switch OS in many data centers and hyperscaler facilities — and CVE-2026-20245 in Cisco Catalyst SD-WAN Manager (CVSS 7.8).[10] For the network side of your stack, a clear directive to check maintenance status of your WAN stretch.

Angular 22: Signal Forms, Zoneless, OnPush — all production-stable [11][12]

On the release side, the biggest frontend news of the week: Angular 22 went GA on June 3 — and the Angular team calls it themselves a consolidation release: the experiments are done.[11] Signal Forms, the Resource API, and Angular Aria are stable; the Zoneless architecture is production-stable; OnPush becomes the standard change detection strategy.[11]

Zoneless & OnPush: Angular traditionally listened to every mouse click, timer, and promise (via the zone.js library), then blanket-checked the entire component tree for changes — safe, but overreactive. Zoneless throws out this blanket suspicion; OnPush additionally says: check a component only if its inputs actually changed. Result: less render work, measurably significant in larger apps.

For Daniel as an Angular-experienced dev, v22 is the release where starting a new project, you don't have to justify why you're going Signal/Zoneless — it's now the documented path.[12] The tradeoff: toolchain floor rises (Node 20+). Migrating existing apps: move to 22 first, then step-by-step enable OnPush and Zoneless — not all at once.

PostgreSQL 19 Beta 1: graph queries, parallel autovacuum, async I/O [13]

PostgreSQL 19 Beta 1 landed on June 4, GA expected September/October.[13] Four highlights worth remembering.

First, SQL/PGQ: property graph queries directly in SQL — you can query relationships as a graph without standing up a separate graph DB next to Postgres. Second, parallel autovacuum: the new autovacuum_max_parallel_workers setting lets multiple workers clean simultaneously — genuinely noticeable on large tables.[13] Third, asynchronous I/O that auto-scales via io_min_workers/io_max_workers. And fourth, ~2× faster inserts with foreign keys, plus a nice touch: ON CONFLICT DO SELECT as an upsert variant, and SNI support via a new pg_hosts.conf so multiple TLS certificates can coexist on one Postgres port.[13]

Watch for: the beta already shows early MD5-deprecation warnings on password login — if you still use MD5 auth, plan the migration to SCRAM now.[13] A beta doesn't belong in production; but developing against it (especially the graph features) makes sense for new projects.

  1. postgresql.org

EU Cyber Resilience Act: next building block live [14]

The CRA phases in stepwise, and on June 11 the next block took effect: rules for notified conformity assessment bodies. The EU Commission confirms that member states may now name their notifying authorities — the bodies that will later audit whether a "product with digital elements" meets CRA requirements.[14]

Operationally real starting September 11, 2026: the reporting obligation for actively exploited vulnerabilities, with the tight timelines the CRA defines. Full CE marking requirement: December 11, 2027. Enforcement range, worth repeating: up to €15 million or 2.5% of global annual revenue.[15]

For a solo shop like twiceD, no immediate pressure; but if you're a maker of "Important" or "Critical" products, now's the time to secure a slot with a notified body — capacity is expected to tighten significantly by late 2027.[16]
