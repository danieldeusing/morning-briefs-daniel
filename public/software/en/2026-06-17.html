June Patch Tuesday: the largest since program launch — with a burning OWA hole

This is the headline of the day, and it's so for two reasons. First, the sheer volume: Microsoft closed ~198 CVEs in June (by Tenable's count; BleepingComputer says 200) — per Tenable, "the largest Patch Tuesday release since the Patch Tuesday program began".[1] Of those, 32 are rated critical. The distribution is predictable: 31.8% of the holes are elevation-of-privilege, 27.3% remote code execution.[1]

Second, and this is the part that really matters today: Among the three publicly known zero-days is one already under active exploitation in Outlook Web Access. CVE-2026-42897 (CVSS 8.1) is a spoofing bug arising from a cross-site scripting flaw in OWA; it affects Exchange Server 2016, 2019, and Subscription Edition.[2] Microsoft delivered security updates on June 9 for Exchange SE RTM, Server 2019 (CU14/CU15), and Server 2016 (CU23).[3]

XSS (Cross-Site Scripting): Picture a bulletin board where anyone can pin up a note — and someone pins a note that doesn't say "hello," but instead an instruction that the janitor automatically executes when walking past. In XSS, an attacker smuggles code into a web page (here: the webmail inbox), which the victim's browser then executes as legitimate. In the OWA context, that means: crafted content in the mailbox can act in the victim's name.

The other two zero-days are less acute, but named: CVE-2026-45586 (CTFMON EoP, CVSS 7.8) allows escalation to SYSTEM privileges[1], and the "bit war" bug CVE-2026-50507 (CVSS 6.8) lets an attacker with physical access bypass BitLocker device encryption — PoC exists but no wild exploitation yet.[1] Plus the DoS classic CVE-2026-49160 ("HTTP/2 Bomb," CVSS 7.5), where tiny data amounts force the server into disproportionately large memory allocations.[1]

Heads up No blanket alarm over 198 CVEs — but priority is clear: Exchange OWA first, because actively exploited. Anyone running Exchange on-prem should roll out the 06/09 update first; the rest of the batch follows based on exposure.

TanStack npm incident: this time the pipeline got hijacked, not a password stolen

On May 11, a supply-chain attack hit the TanStack ecosystem: 42 packages in 84 compromised versions.[4] The incident carries the ID CVE-2026-45321 with a CVSS score of 9.6.[5] Snyk and Heise classify it under the "mini-Shai-Hulud" pattern — self-replicating npm worms.

What's technically remarkable is the vector: The packages were not published by an attacker who had stolen credentials. They were published from TanStack's legitimate release pipeline using its trusted OIDC identity — after attacker-controlled code hijacked the CI runner mid-workflow.[4] The malware then exfiltrated AWS IMDS credentials, GitHub tokens, and SSH keys.

OIDC in CI/CD: Instead of putting a long-lived password in your build pipeline, the pipeline proves its identity like a freshly issued day pass (OIDC token) that the registry accepts briefly. Sounds safer — and it is against stolen passwords. But if the attacker is in the build run, they just use the real, valid day pass while it's good. That's exactly what happened here: no stolen key, but a hijacked machine using its own legitimate key.

To check if you're affected: @tanstack/query*, table*, form*, virtual*, store and the meta-package @tanstack/start (not the start-* subpackages) were not compromised.[4] It fits the bigger picture: ReversingLabs reports a 73% increase in malicious open-source packages for 2025 versus the prior year.[6]

Heads up Homework: check lockfiles against the affected version windows — and take the lesson that OIDC pipeline trust is only as good as the runner's integrity. Short-lived tokens protect against theft, not against a machine hijacked mid-run.

Chrome V8: fifth actively exploited zero-day of the year [7][8]

Google has patched another browser zero-day. CVE-2026-11645 is an out-of-bounds read/write in the V8 JavaScript engine; Google confirms that "an exploit for CVE-2026-11645 exists in the wild".[7] Fixed in Chrome 149.0.7827.102/.103, rolled out from June 8/9 onward. Google doesn't assign a numerical CVSS score; the hole is rated as "high severity." It's the fifth Chrome zero-day exploited in the wild since the year began — and the second to hit V8.[8] CISA added it to the KEV on June 9.

Out-of-Bounds Read/Write: Like a book with numbered pages — the code is supposed to access only pages 1 through 100, but accidentally reads or writes page 101 or −1. In V8 (the engine that runs JavaScript in the browser), that means: crafted JavaScript on a web page can access foreign memory — the first step to escaping the browser.

In the same June 9 KEV batch were also Arista EOS (CVE-2026-7473) and Cisco SD-WAN (CVE-2026-20245).[9] Handy note: Chrome just needs a restart for the update to kick in — many installs otherwise wait days in the background.

Microsoft Work IQ API GA: a new API layer for enterprise agents

Since June 16, the Work IQ API endpoints are generally available.[10] The AI promise is not what matters here — that's ai-usecases territory — but the platform/API surface opening to dev teams. Work IQ provides 10 generic M365 tools (mail, calendar, files, people, chat, sites) and speaks three protocols: A2A, MCP (newly reshaped remote server), and REST.[11]

Two details matter for the software-engineering side. First, billing: consumption-based via Copilot Credits and explicitly independent of M365 Copilot licensing.[12] That moves an API hookup from license negotiation into a usage-based cost model — relevant for anyone pricing an M365 integration. Second, governance: a Rego-based policy engine, every tool invocation is logged and evaluated, plus rate-limiting and real-time compliance enforcement.[13]

Rego / Policy-as-Code: Instead of baking access rules into app code, you write them as a separate, readable rule language (Rego, known from Open Policy Agent) — like a separate rulebook at the door that decides "allowed in / not allowed" on every request. The upside: rules are versionable, auditable, and changeable without redeploying the app.

Work IQ was announced at Microsoft Build 2026 (June 2), and the preview has now moved to GA.[14] Frame of reference: for Daniel's world, this is less an AI topic than a new piece of enterprise platform engineering — a standardized, billable, policy-controlled bridge to M365 data that you can hang services on.

Android June patch level: actively exploited hole — and a numbers tangle

The June Android bulletin closes officially 96 distinct CVEs across patch levels 2026-06-01 and 2026-06-05, 11 of them critical.[15] Key here: CVE-2025-48595 (high, elevation-of-privilege, no user interaction) affects Android 14, 15, 16, and 16-QPR2 and sits under "limited, targeted exploitation" per Google.[15] CISA added the hole to the KEV on June 2.[16]

A note on diligence: several outlets (including SC World, The Hacker News) cite 124 vulnerabilities — notably more than the bulletin's 96.[17] The gap is unexplained; plausible is that the higher count includes Pixel-specific or partner/vendor patches not in the main AOSP bulletin. When in doubt, the primary source wins — so 96 — with the caveat that media counts differently.

On the side, but handy: the June system update brings a Credential Exchange Standard that makes passwords and passkeys portable between third-party managers and Google Password Manager.[18]

PostgreSQL security line: 11 CVEs, three at CVSS 8.8

The PostgreSQL group released versions 18.4, 17.10, 16.14, 15.18, and 14.23 (May 14) — 11 security holes and over 60 bugfixes.[20] Three of the CVEs sit at CVSS 8.8 and hit all versions 14–18: CVE-2026-6473 (integer wraparound in memory allocation), CVE-2026-6475 (file overwrite via pg_basebackup/pg_rewind), and CVE-2026-6637 (stack buffer overflow plus SQL injection in the refint module).[21] A schedulable minor update — no drama, but three 8.8s in one batch is good reason not to push the next maintenance window.

Python 3.15 Beta 2: faster JIT, Lazy Imports by default

On June 2, Python 3.15.0 Beta 2 landed.[22] The JIT compiler gets faster: 8–9% geometric mean on x86-64 Linux versus the standard interpreter, 12–13% on AArch64 macOS versus the tail-calling interpreter.[22] Content highlights: PEP 810 (explicit lazy imports for faster startup), PEP 814 (frozendict as built-in), PEP 661 (sentinel as built-in), and PEP 686 (UTF-8 as default).[23] The next beta (3.15.0b3) is scheduled for June 23, the RC for August 4.[24]

It's Wednesday, June 17, and the day is dominated by security updates — led by a Patch Tuesday that broke records.

Microsoft closed roughly one hundred ninety-eight vulnerabilities in June — per Tenable, the largest Patch Tuesday since the program began. Thirty-two of them are critical. The really urgent part, though, is a single hole: a cross-site scripting flaw in Exchange's webmail, called CVE-2026-42897, with a severity score of eight point one. It's already under active attack and affects Exchange 2016, 2019, and the Subscription Edition. Microsoft shipped the patch on June 9. If you run Exchange yourself, that specific update should go in first.

The second big story comes from open-source supply chain. The TanStack project had forty-two packages in eighty-four versions compromised, with a severity of nine point six. Here's what's different: nobody stole a password. The attackers hijacked the actual, trusted release pipeline while it was running, and published the poisoned packages using TanStack's own valid identity. The lesson is uncomfortable: short-lived tokens protect against theft, but not against a machine taken over mid-run.

Google had to patch too. The JavaScript engine in Chrome contained a bug that lets code access foreign memory — and an exploit for it already exists in the wild. It's the fifth such Chrome zero-day this year. Restarting the browser is all it takes for the update to take effect.

Microsoft also flipped the switch on a new platform layer: Work IQ is generally available as of June 16. What's interesting here isn't the AI promise, but the straightforward technology — ten standard tools for Microsoft data, three protocols, consumption billing, and its own rule-based permission system. For development teams, it's mainly a new piece of enterprise infrastructure you can build services on top of.

On the lighter side: PostgreSQL put out a security line with eleven holes, three of them severity eight point eight. Python 3.15 is in its second beta, with eight to nine percent faster just-in-time compilation and lazy imports. And Android shipped its June patch level — officially ninety-six vulnerabilities, though the media report one hundred twenty-four; one of them is being actively exploited.

From Europe: the Cyber Resilience Act took a step on June 11, but the real hard stage with reporting obligations doesn't come until September. And there's a date to mark on the stack calendar: June 30 is when Spring Framework 6.2 loses free support. If you're on it, you need to move to version 7.0.
