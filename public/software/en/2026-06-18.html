Microsoft closes 206 CVEs — the largest Patch Tuesday ever

June's patch day is the heaviest Microsoft has ever shipped: 206 CVEs in one go,[5] 33 of them rated critical (28 RCE, 4 privilege escalation, 1 information disclosure).[1] Six of the flaws are zero-days — five previously disclosed, one actively exploited.[1]

The exploited one is CVE-2026-42897, a spoofing/XSS flaw in Exchange Server (Outlook Web Access), CVSS 8.1.[2] It affects Exchange 2016/2019/SE, not Exchange Online; triggered via a specially crafted email to a user.[3] If you still run on-premises Exchange, this is your top priority — the flaw was already in CISA's KEV before the patch day.

Watch for The highest rating in the batch goes to CVE-2026-48567 in Azure HorizonDB at a flat CVSS 10.0 — but "requires no customer action," Microsoft fixed it server-side.[4] More practical for your own hosts is CVE-2026-44815 in the Windows DHCP Client (CVSS 9.8, unauthenticated, no user interaction).[4] Among the disclosed zero-days, CVE-2026-50507 stands out — a BitLocker bypass with proof-of-concept code already circulating.[5]

Why the record? Security researchers point to AI-assisted vulnerability discovery — both in Microsoft's own hardening and from reporters.[6] My take: 206 isn't a panic number, it's a tooling signal. The machine finds more, so the machine patches more — the number will likely become the new baseline over the next few months rather than an outlier. Prioritize rather than patch everything: Exchange first, then the unauthenticated RCEs.

Oracle: a PeopleSoft 9.8 ran as zero-day for two weeks — and the second monthly CPU brings 245 fixes

Oracle's June update has two stories. The loud one: CVE-2026-35273 in PeopleSoft Enterprise PeopleTools (8.61/8.62), an RCE without login and no user interaction, CVSS 9.8.[7] The group ShinyHunters exploited it between May 27–June 9, 2026 as a zero-day — before Oracle even issued a warning on June 10.[8] Tally: around 300 vulnerable instances across 100+ organizations in two weeks; University of Nottingham had 40 GB of personal data and student billing records exfiltrated.[9]

The second story is routine business, only bigger: Oracle's June CPU from June 17 delivers 245 new patches, ~120 rated critical and 100 remotely exploitable without authentication.[10] Largest single family is Fusion Middleware with 106 patches (53 of them remotely unauthenticated).[10] The PeopleSoft flaw was closed out-of-band on June 10 and now rolled into the CPU.

Heads up The pattern is the real signal: an unauthenticated 9.8 RCE in an ERP/campus platform, exploited before the vendor even responded. If you run PeopleSoft, check now for MeshCentral agents masquerading as "Microsoft Azure" services and for credentials stolen from psappsrv.cfg.[11]

Spring Boot 4.1.0 — and the 3.5 clock is ticking

Spring Boot 4.1.0 shipped on June 10: first native Spring gRPC support, SSRF mitigation in the HTTP client (via InetAddressFilter), OpenTelemetry updates, Jackson improvements, and Log4j file rotation — plus all fixes from 4.0.7.[12] Solid, no drama.

The date worth circling: Spring Boot 3.5 reaches OSS end-of-support on June 30, 2026.[13] If you're still on 3.5 and don't pay for commercial support, you have twelve days to plan your migration path. Not an emergency — but the day you get your last free security fixes for the line.

88 minutes: how an entire npm scope falls via a forgotten credential

Overnight on June 17 (01:12–02:39 UTC) an automated campaign backdoored 142 packages in the @mastra/* scope plus standalone packages mastra and create-mastra.[14] This stings because @mastra/core alone pulls roughly 4 million downloads per month — the entire scope aggregates "tens of millions."[15]

Supply-chain attack: Instead of attacking your server directly, the attacker poisons a package you automatically pull in during builds — like a saboteur who doesn't break into your kitchen but taints an ingredient at the wholesale market that thousands of kitchens buy. You install it unsuspecting.

The entry wasn't a hack in the classic sense, but a dead credential: the contributor account "ehindero" still had publish rights in the scope — and npm doesn't auto-revoke scope publish rights for inactive users.[16] An old, never-revoked maintainer credential was all it took to push to every package. The payload came via a typosquatted clone called easy-day-js@1.11.22 (mimicking dayjs): a postinstall hook disabled TLS (NODE_TLS_REJECT_UNAUTHORIZED=0), fetched a RAT from 23.254.164[.]92:8000, and installed a wallet stealer for MetaMask, Phantom, and Solflare.[16] The tradecraft overlaps with Sapphire Sleet / BlueNoroff — North Korea's crypto-theft operation — according to Microsoft.[14]

Quick check Run npm ls easy-day-js across your projects — that's the fastest way to spot the compromise.[14] And lesson for anyone managing an npm scope: actively remove inactive maintainers, because the platform won't do it for you.

This is exactly the kind of story that belongs in this brief: not that the Mastra AI framework is the issue, but that agent tooling layers have become supply-chain attack surfaces. No CVE has been assigned to the incident yet.

CISA KEV: LiteSpeed cPanel flaw — government deadline today

CISA added CVE-2026-54420 to its KEV directory on June 15: a flaw in the LiteSpeed WHM plugin (< v2.4.8), CVSS 8.5, that grants root privileges to a user with FTP or web shell access.[17] Root cause is improper symlink handling — a privesc vector that particularly hurts on shared hosting with CloudLinux/CageFS.[17] Fix: upgrade to LiteSpeed WHM Plugin v5.3.2.1 (with cPanel plugin v2.4.8). For FCEB agencies, the patch deadline is today, June 18.[17]

In the same KEV batch: CVE-2026-20262 in Cisco Catalyst SD-WAN Manager (CVSS 6.5, directory traversal, deadline June 29).[17] For Daniel, the LiteSpeed flaw is more practically relevant: if you use LiteSpeed/cPanel at a hoster, check your plugin version.

Red Hat "Miasma": npm namespace hijacked, credential worm — but customers can relax

As early as June 1, the @redhat-cloud-services npm namespace was compromised (RHSB-2026-006): frontend libraries for Hybrid Cloud Console.[18] The path went through a compromised GitHub Actions OIDC token leading to unauthorized commits in the RedHatInsights GitHub organization.[18] The injected worm "Miasma" hunts credentials for GitHub, npm, AWS, Azure, GCP, HashiCorp Vault, and Kubernetes — and disguises its exfil traffic as api.anthropic.com/v1/api (no real endpoint).[19] Red Hat removed the compromised versions from npm; per the advisory, no customer action is needed.[18]

Twice in just over two weeks, then, the same mechanism — hijacked CI/maintainer identity, worm in the OSS supply chain. That's not coincidence, that's the season's attack pattern.

Azure Linux 4.0 in Public Preview — Microsoft's first universal server Linux

At Build on June 2, Azure Linux 4.0 entered Public Preview: Fedora-derived, RPM-based — and the first universal Microsoft server Linux (earlier versions were container-only).[20][21] In benchmarks, Microsoft reports 12% more requests/sec than the standard Ubuntu Server image in the NGINX test, plus 5–15% more network throughput/disk IOPS versus Ubuntu 24.04.[22] GA is tentatively aimed for end of 2026.[22]

Worth noting for Daniel's Docker/Linux DNA as a base-image option in Azure — but preview stays preview: don't put it in production until GA and real load-test data exist. That Microsoft is betting on Fedora rather than rolling its own line is the really remarkable statement here.

EU regulation: BSI launches CRA conformity bodies, next hard deadline in September

The BSI published the framework for designating conformity assessment bodies (CABs) under the Cyber Resilience Act on June 11 — as of June 2026, the BSI designates the third parties authorized to certify CRA compliance.[23]

CRA (Cyber Resilience Act): EU law requiring digital products to be secure out-of-the-box and report vulnerabilities — the "TÜV mindset" for software and networked devices. Conformity bodies are the independent auditors who issue the seal.

Next deadline with teeth: as of September 11, 2026, reporting obligations kick in for exploited vulnerabilities and serious incidents.[24] General CRA obligations land December 11, 2027. If you sell products into the EU, spin up your vulnerability-reporting workflow now, not in August.

It's Thursday, June eighteenth. And it's one of those patch days where the supply chain, not the single error, is the real story. Three major security stories running in parallel — and all three are about not attacking the server, but attacking what's on the way to the server.

Starting with Microsoft. June's patch day is the largest ever: two hundred six vulnerabilities at once, thirty-three of them critical. Six are so-called zero-days, flaws that were known before the patch arrived — and one of them is actively exploited. It's a spoofing flaw in Exchange Server, in Outlook Web Access. If you still run your own Exchange, you should patch that first. Security researchers, by the way, credit the record number to AI-assisted vulnerability discovery — the machine finds more, so the machine patches more. That's more the new normal than an outlier.

The second story is the most spectacular: overnight on the seventeenth, an entire npm package scope, Mastra, fell in just eighty-eight minutes. One hundred forty-two packages got a backdoor. And the entry wasn't hacking, it was a dead credential: an old contributor account still had publish rights that nobody ever revoked. npm doesn't auto-revoke those for inactive users. A single forgotten access card was enough to write to every package. The malicious code came through a typosquatted clone called easy-day-js, disabled encryption, and installed a stealer for crypto wallets. Microsoft sees overlap with BlueNoroff, the North Korean cryptocurrency theft operation. Anyone using Mastra should run npm ls easy-day-js across their projects.
