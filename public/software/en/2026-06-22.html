Oracle PeopleSoft: two-week zero-day, 100+ organizations exposed [1][2][3]

This is the story of the week, and it's ugly. CVE-2026-35273 in Oracle PeopleSoft Enterprise PeopleTools is a remote code execution vulnerability with CVSS 9.8 — reachable over HTTP, no login, no user click.[1] The severity scale doesn't go much higher.

Remote Code Execution (RCE): A vulnerability that lets an attacker run arbitrary code on a foreign server over the network — as if they were sitting at the keyboard. "Without authentication" means they don't even need a password.

The real scandal is the timing. Google Mandiant attributes the attacks to group ShinyHunters (UNC6240) and dates active exploitation to May 27 through June 9. But Oracle didn't publish its out-of-band advisory until June 10 — so the vulnerability was a genuine zero-day the whole time, with nobody able to patch.[2] Result: 100+ organizations compromised, roughly 68% from the higher-education sector.

Zero-Day: A vulnerability that attackers know and are exploiting before the vendor has a fix — defenders had "zero days" of advance notice. Like a burglar who already picks the lock while the locksmith still doesn't know it's broken.

Nested in Oracle's regular Critical Patch Update from June 16: 245 security patches, 243 individual CVEs, 122 rated as critical (just under half).[3] Anyone running PeopleSoft should first check for signs of exploitation — patching alone isn't enough here, because the door stood open for two weeks.

Shai-Hulud Returns: the worm eating the supply chain [4][5][6]

If the Oracle case is the loud break-in, this one is the slow seep. A new wave of the self-replicating supply-chain worm Shai-Hulud (internally "Miasma") has compromised 100+ npm and PyPI packages since early June.[4] On June 1 alone, it hit 32 packages from Red Hat's hybrid cloud console ecosystem; by June 5, it was 57+ npm packages across 300+ malicious versions.

Supply Chain Attack: Instead of hitting your server directly, the attacker poisons a package you pull normally via npm install. Like someone spiking an ingredient that many bakeries buy — nobody attacks the bakery, they all bake in the poison anyway.

The worm part is what's alarming: the payload runs via a preinstall hook, searches the build environment for CI/CD and cloud credentials, exfiltrates them — then uses them to write itself into more packages. A PyPI variant called "Hades" hides in *-setup.pth files and starts on every Python process.[5]

What's technically truly alarming is the TanStack wave from May: CVE-2026-45321 (CVSS 9.6) — 84 malicious versions across 42 @tanstack/* packages, published in under six minutes. All of them with valid SLSA build-level-3 provenance.[6]

SLSA Provenance: A kind of authenticity seal for software: a signed proof that a package really came from the stated build pipeline and wasn't tampered with in transit. That seal was genuine here — the attacker took over the pipeline itself, so the signature was correct but worthless.

The lesson is uncomfortable: provenance attestation protects against tampering in transit, not against a compromised maintainer account or a hijacked build pipeline. Anyone running CI caches and OIDC tokens — basically every modern JS/Python project — should audit token scopes, freeze lockfiles, and not auto-merge new versions blind.

Spring CVEs keep coming — and now Spring AI's turn [7][8]

CVE-2026-47835 (CVSS 8.6, HIGH) in Spring AI allows query injection: special characters can force arbitrary queries into vector stores for Elasticsearch, OpenSearch, and GemFire. Affects 1.0.x and 1.1.x; fixed in 1.0.9 and 1.1.8.[7] AI is just window dressing here — the bug is classic input validation in the data layer, our territory.

This fits the pattern: HeroDevs counted 30 CVEs in the Spring ecosystem for March/April 2026 alone, including 4 critical and 10 high.[8] Anyone running Spring stacks should not put dependency alerts on "defer" right now.

CISA KEV +3: countdown runs through tomorrow [9]

On June 9, US cyber agency CISA added three vulnerabilities to its catalog of actively exploited gaps (KEV). The federal patch deadline is tomorrow, June 23 — formally only for US agencies, but the best free priority signal everyone else gets.[9]

CISA KEV: A public catalog from US cyber agency of vulnerabilities actively being exploited right now. Other CVE lists say "could be dangerous"; KEV says "is being weaponized against people right this moment". That's why the hard deadline.
  • CVE-2026-11645 — Chromium V8: out-of-bounds read/write flaw that executes code in the sandbox via a crafted HTML page. Hits Chrome, Edge, and Opera. Updating the browser is enough.
  • CVE-2026-20245 — Cisco Catalyst SD-WAN Manager: authenticated local attacker can execute commands as root via a crafted file.
  • CVE-2026-7473 — Arista EOS: incomplete tunnel comparison. The kicker: Arista is not planning a patch — a fix would break existing configurations, according to the vendor. Mitigation instead of update, the uncomfortable exception.

The first two are routine patch work. The Arista case shows how KEV listing and vendor reality can diverge: listed, exploited, no patch planned — anyone running EOS needs to read Arista's workaround notes.

Android tightens developer identity [10][11]

Google has been rolling out a new system service this month that installs automatically on most Android devices — the precursor to "Android Developer Verification".[10] From September 30, 2026 onward, apps in Brazil, Indonesia, Singapore, and Thailand must register with a verified developer identity, enforced across seven stores: Google Play, Samsung Galaxy Store, Xiaomi GetApps, HONOR, OPPO, vivo, and Transsion's Palm Store.

Developer Verification: Android wants to know who is behind an app before it can be installed — even for sideloading outside the Play Store. Like showing ID at the door: not every app has to go through the Play Store bouncer, but every developer has to verify themselves first.

For the tooling side: a Developer-ID-Status API arrives in July (checks if a package is registered), a Console API in August that plugs directly into CI/CD pipelines — plus "Limited Distribution" accounts (up to 20 devices, no ID required).[11] Anyone delivering Android builds automatically should flag August on the calendar; after that, the verification check belongs in the pipeline.

NIS2: Germany lags — new BSI deadline July 31 [12]

The numbers are sobering: by end of May, only around 18,500 of approximately 29,000 NIS2-mandated facilities had registered with the BSI.[12] The original deadline (March 6) is long past; the BSI extended on June 18 to July 31, 2026. Non-registration faces fines up to €500,000.

NIS2: An EU directive that requires more companies in more sectors to maintain minimum cybersecurity standards and file breach reports. For many mid-market IT service providers, it concretely means: register, document security controls, report incidents — or pay.

For German software shops in B2B: even if you're not directly covered, customers are now asking for NIS2 compliance as a contract prerequisite. The registration logjam shows how much work is still ahead.

PostgreSQL 19 Beta 1: the week's good news [13]

PostgreSQL 19 Beta 1 landed June 4; the community targets September/October for GA.[13] This round's highlights are operational improvements you'll actually feel in daily work:

  • Parallel Autovacuum — the cleanup routine can now use multiple workers (autovacuum_max_parallel_workers). Less queuing on large tables.
  • REPACK CONCURRENTLY — reorganize tables without exclusive lock. The painless replacement for VACUUM FULL or the external pg_repack.
  • SQL/PGQ Property-Graph Queries — run graph queries directly in SQL, no separate graph database.
  • Logical replication without restart and ON CONFLICT DO SELECT.

None of this is flashy — these are exactly the friction points where Postgres operations hurt. Beta means: test, yes; production, no. But the test window is open now, and feedback to the community helps.

Container & Runtime: runc, Docker Desktop, Node.js [14][15][16]

Three minor but relevant shifts in the infra layer:

runc container escape — three gaps (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881) enable breakout under certain mount configurations. Fixed in runc 1.2.8 / 1.3.3 / 1.4.0-rc.3; active exploitation not yet reported, and they're not on CISA KEV.[14] Bring them along on the next image refresh.

Docker Desktop — CVE-2026-8936 (VM panic via grpcfuse) and CVE-2026-31431 (unprivileged container → host root via page cache) are patched. Also, extensions are now disabled by default; settings warn explicitly that extensions run with host privileges and aren't audited by Docker.[15]

Node.js 25 hit end-of-life on June 1 — no more security fixes. The current release is 26.3.1; the active LTS line remains 24.x.[16] Anyone still on 25 should plan the upgrade.
