Miasma / Shai-Hulud: a worm living in your coding hooks [1][2][3]

This is the story of the day, and it hits uncomfortably close to your workstation. Over 100 packages in npm and PyPI have been compromised — security researchers count a total of 471 malicious artifacts (57+ npm packages in 300+ poisoned versions, plus 37 PyPI wheels).[1] Entry point was an early-June Red Hat npm scope (@redhat-cloud-services). The distinctive and dangerous part: this is a self-replicating worm, not a one-time theft.

Supply-chain worm: Imagine someone poisons an ingredient at the wholesale bakery, not a single cake. Everyone who buys the ingredient bakes the poison in — and their kitchen starts poisoning other ingredients. Here the "ingredient" is an npm/PyPI package that runs code on install without asking, then copies itself to other packages from your machine.

The real hammer is the persistence vector: the worm injects itself into the SessionStart hooks of 13 AI coding tools — including Claude Code, GitHub Copilot, and Gemini CLI.[2] Meaning: not just one build is affected, but every new coding session re-runs the malicious code. It forges SLSA provenance signatures to get past npm audit, uses GitHub for command and control, and carries a DEADMAN_SWITCH that wipes the developer's machine if the token is revoked. No CVEs yet — this runs completely outside traditional vulnerability accounting.

SLSA provenance: a digital certificate of origin for a software package — like an authenticity seal saying "this build really comes from this repo". The worm prints the seal itself, so the check looks "authentic" and waves it through.

Technically elegant and precisely why it's nasty is the "Phantom Gyp" technique: a 157-byte binding.gyp file triggers code execution during npm install itself — before preinstall/postinstall scripts that security tools normally watch.[3] Anyone relying on lifecycle-script scanning is looking in the wrong place.

Heads up If you use Claude Code, Copilot, or Gemini CLI: freeze lockfiles (npm ci instead of npm install), check your tools' SessionStart hooks for unexpected entries, and cross-reference installed versions against published IoC lists. The AI tooling layer itself is the attack surface here — the model assessment of affected coding tools is in the ai-dev brief; here the supply-chain mechanism is what counts.

Context: this is not isolated — it's the second wave from the same group (TeamPCP / UNC6780). In May a "Mini-Shai-Hulud" ran through 42 @tanstack/* packages — 373 malicious package-version entries across 169 npm names, affecting Mistral AI and UiPath among others.[4] If TanStack is in your stack (React Query, Router), check history there too.

UniFi: three 10.0 flaws, chainable to full takeover

This one may hit your home or lab network. Ubiquiti UniFi OS has three CVSS-10.0 vulnerabilities — CVE-2026-34908 (Improper Access Control), -34909 (Path Traversal), and -34910 (Command Injection) — that chain together to complete, unauthenticated takeover of any exposed UniFi device.[5] Active exploitation confirmed; CISA added the flaws to the KEV catalog on 06-23.

Affected devices include UDM, UDM-Pro, UDM-SE, and UDM-Pro-Max before 5.1.12, UniFi Express before 4.0.14, and the UNAS line before 5.1.10. The patch has been available since UniFi OS 5.0.8 (05-21).

Heads up If your UniFi management interface is reachable from the internet: patch now and put the web UI behind VPN/Tailscale, not exposed to the net. CVSS 10.0 + KEV + "chainable to full takeover" is the combination where you don't wait for the maintenance window.

Security long tail: Splunk, KDDI, Cisco, Arista [6][7][8][9]

Splunk Enterprise (CVE-2026-20253, CVSS 9.8): unauthenticated RCE via a PostgreSQL sidecar endpoint with no auth controls. Affects 10.2.0–10.2.3 and 10.0.0–10.0.6; fixes (10.2.4 / 10.0.7) came on 06-10, active exploitation confirmed since 06-17 — researchers counted 20 attacking IPs. CISA KEV deadline was 06-21.[6]

KDDI Japan: a third-party software flaw exposed potentially 14.22M email credentials (addresses + passwords) across six ISPs — STNet, KDDI Web Communications, JCOM, Chubu Telecom, Nifty, and Biglobe. Detected 06-17, disclosed 06-23.[7] Again the pattern: a shared third-party system as a single point of failure.

Cisco keeps adding: CVE-2026-20245 (Catalyst SD-WAN Manager, CVSS 7.8, privilege escalation to root) is the seventh SD-WAN zero-day of the year and was reportedly exploited months before the patch; CISA deadline was 06-23.[8] Plus a Unified CM auth-bypass flaw (CVE-2026-20230, CVSS 8.6) with public PoC. On 06-09, Cisco, Chrome, and an Arista EOS flaw (CVE-2026-7473) hit the KEV list — the Arista flaw's CVSS score has not yet been confirmed from primary sources.[9]

OpenAI × Broadcom "Jalapeño": the first custom inference ASIC [10][11][12]

OpenAI and Broadcom unveiled "Jalapeño" on 06-24 — reportedly OpenAI's first custom-built inference chip, an ASIC rather than a GPU.[10] The official openai.com page was not accessible to us (403); specs are cross-checked via CNBC and other secondary sources.

ASIC vs. GPU: A GPU is a Swiss Army knife — does many things okay. An ASIC is a specialty tool that does one thing really well and way cheaper and more power-efficient. "Inference" is running the trained model, so generating answers — the part that runs around the clock at OpenAI and costs money.

According to reports the die is reticle-sized (as large as one chip mask can make it), made at TSMC — the node is reportedly N3 (though that came from a TrendForce advance note from January 2026 and is not explicitly confirmed in the 06-24 announcement).[12] Developed in nine months; engineering samples are already running GPT-5.3-Codex-Spark in the lab at target frequency and power. Target: roughly 50% cheaper inference per watt vs. current state-of-the-art GPUs, rollout end of 2026, aiming for gigawatt datacenters with Microsoft and partners.

Why this matters even if you don't build chips: if OpenAI shifts inference from Nvidia GPUs to its own silicon, it reshapes the cost curve for AI inference over time — and thus the prices of the APIs half your tooling stack sits on. Model capability details live in the ai-dev brief; here the hook is infrastructure.

Qualcomm acquires Modular: a strike at CUDA [13][14]

Same day (06-24), Qualcomm acquired the AI infrastructure firm Modular for around $3.92B, all-stock, closing in H2 2026 subject to regulatory approval.[13] Modular builds the language Mojo and the MAX inference engine — both explicitly hardware-agnostic, runnable across CPU/GPU/NPU/ASIC without rewriting.

CUDA lock-in: CUDA is Nvidia's software layer — GPUs are essentially unusable for AI without it, like a socket that only fits Nvidia plugs. Once your AI system is built on CUDA, you're stuck. Modular's promise: a universal adapter running the same code on any hardware.

The interesting part is personnel: Modular was founded in 2022 by Chris Lattner — creator of LLVM and Apple's Swift — and Tim Davis.[14] Qualcomm CEO Cristiano Amon frames the purchase as a bet that "the future belongs to developer-friendly, horizontal platforms running across diverse compute environments." For you as a backend or infrastructure person, the compelling question is whether Mojo graduates from a Python superset promise to a real ecosystem — so far it's toolchain, not mainstream.

SpaceX becomes a GPU landlord [15][16]

A shift that slips past notice: SpaceX (which acquired xAI in February) leases its Colossus datacenters in Memphis onward as a compute bank. Google pays per SEC filing $920M/month over 32 months for 110,000 Nvidia GPUs — nearly $30B through June 2029.[15] On 06-22 came another deal with Reflection AI: $6.3B total, $150M/month starting 07-01 for access to Nvidia's GB300 chips.[16]

The point for developers: compute becomes a tradeable commodity, leased to the highest bidder — Colossus runs over 2 gigawatts. When a rocket manufacturer becomes the AI industry's largest GPU landlord, it says more about scarcity of power and silicon than about rockets.

Microsoft Patch Tuesday: 208 CVEs, a new record [17][18]

June Patch Tuesday (06-10) is the largest monthly release since 2017 with 208 CVEs — including 37 Critical and three publicly disclosed zero-days.[17] Two flaws stand out, both CVSS 9.8, both remote and requiring no user interaction:

  • CVE-2026-45657 — Windows kernel RCE via TCP/IP handling, code execution at SYSTEM level. ZDI calls the profile "similar to EternalBlue/WannaCry" — potentially wormable. Microsoft itself rates exploitation as "less likely."[18]
  • CVE-2026-47291 — HTTP.sys RCE. Systems with default MaxRequestBytes are not affected; a registry workaround exists until patching.

Translation: no reason to panic, but the kernel bug is exactly the kind that becomes mass-exploited in six months if unpatched machines pile up. If you run Windows Server infrastructure, prioritize 45657.

Releases: Spring Boot 3.5 EOL, Docker 29.6 [19][20]

For your Java/Spring stack a hard deadline: OSS support for Spring Boot 3.5 ends 06-30 — after that no open-source patches ship on Maven Central.[19] Spring Boot 4.1.0 shipped 06-10 (Java 17 minimum, supports 17–26, supported through 07-31-2027). If you're on 3.5, plan the move now — either to 4.x or, with commercial support, stay on 3.5.

Docker Engine 29.6.0 (06-18) brings two security fixes: CVE-2026-8936 (VM panic from unbounded recursion in the grpcfuse kernel module) and CVE-2026-31431 (privilege escalation from unprivileged container user to root, a backported upstream kernel patch).[20] Side note: Docker Scout view is out, RHEL 8 support drops.

EU CRA crystallizes — and Secure Boot ticks [21][22]

Standing thread on EU regulation: The Cyber Resilience Act is gaining momentum. On 06-11 the BSI formally notified conformity assessment bodies (CABs) and published procedures. The date that matters for you is autumn: starting 09-11-2026, the reporting requirement for vulnerabilities and incidents kicks in for all manufacturers of networked products — via the centralized EU reporting platform.[21] If you ship hardware-with-software or networked devices, build the reporting process now.

Small but dated today: a Secure Boot KEK certificate expires 06-25. Devices continue to function normally but lose future early-boot protection updates.[22] No press release — just another item on the long list if you own firmware or boot chains.
