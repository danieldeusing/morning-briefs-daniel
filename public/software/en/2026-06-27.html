OpenAI builds silicon: the "Jalapeño" inference chip

On 06-24, OpenAI and Broadcom unveiled "Jalapeño" — OpenAI's first custom chip, purpose-built for the inference load of its services.[1] It's a massive, reticle-sized ASIC in TSMC's 3nm process with eight HBM stacks and a systolic array architecture; the tape-out ran in about nine months — reportedly the fastest ASIC development cycle yet for high-performance semiconductors, accelerated by OpenAI's own models in the design.[2]

Inference ASIC: "Inference" is running a finished, trained AI model — what happens when you ask ChatGPT a question (not training). An ASIC is a chip that does only one job, but does it extremely fast and power-efficient. Like a bread machine instead of a universal kitchen: makes only bread, but cheaper and better than the oven. "Reticle-sized" means: as big as the exposure mask in chip manufacturing physically allows — at the upper limit of what a single die can be.

The reported headline is ~50% lower inference cost per token versus mainstream AI GPUs.[3] Caveat: This number does not appear in OpenAI's official statement — it comes from secondary reporting. OpenAI's only firm primary claim is "significantly better performance-per-watt". Treat the 50% as unconfirmed marketing order-of-magnitude, not measured fact.

Timeline: prototype deployment by end-2026, full production ramp 2027–2028; Broadcom talks of gigawatt-scale datacenters with Microsoft and other partners starting 2026. Microsoft reportedly will take around 40% of initial production.[4]

Why it's here (not in the ai-dev brief) The hook is hardware infrastructure, not a new model. For you as a dev, the signal is: the economic gravity of inference is shifting from "rentable Nvidia GPU" to "vertically integrated custom silicon". That pressure will eventually hit API prices — and the market power of the single GPU vendor the whole industry depends on.
Sources: techcrunch.com [1] · tomshardware.com [2] · macgpu.com [3] · venturebeat.com [4]

"Miasma": an npm worm that forges valid signatures

On 06-01, a self-propagating npm worm called "Miasma" was discovered: through a compromised GitHub account of a Red Hat employee, malicious commits were pushed to the namespace `@redhat-cloud-services` — 32+ packages, totaling around 80,000 downloads/week. The payload steals cloud credentials and CI/CD secrets on `install`.[5]

What's truly disturbing isn't the theft, but the trust bypass: the attacker built GitHub Actions workflows that requested an OIDC identity token via `id-token: write`, minted npm publish tokens from it, and released the packages with valid SLSA provenance attestations.[6]

SLSA provenance: a cryptographic "birth certificate" for a package — it proves which pipeline and which commit an artifact was built from, so you spot imposters. Like an authenticity seal on a medicine bottle. The problem here: if the thief has access to the seal machine itself (the compromised CI account), they stamp genuine seals on counterfeit goods. The signature was technically correct — the trust anchor behind it was not.

The payload derives from "Mini Shai-Hulud" malware (TeamPCP, open-sourced 05-12) and adds new collectors for GCP and Azure identities; each infection gets a uniquely encrypted payload, which defeats static signatures.[7] Red Hat officially tracks it as `RHSB-2026-006`.[8]

Lessons for your own setup Provenance attestation is not a silver bullet — it's only as trustworthy as the identity account issuing it. Concretely: minimize `id-token: write` in workflows, grant OIDC permissions per-job not globally, tie npm publish to protected environments with reviewer gates, and treat install scripts (`--ignore-scripts` where possible) as suspicious.
Sources: wiz.io [5][6] · orca.security [7] · access.redhat.com [8]

BSI: AI turns weaknesses into attack paths faster now

On 06-22, the BSI warned that AI systems are now capable enough to "identify, analyze, and convert software vulnerabilities into usable attack paths in a short time".[9] This isn't hype — it's a sober threat-model update: the window between patch release and widespread working exploit is shrinking — exactly the quiet zone where patching used to be safe. Read the UniFi and Splunk KEV additions this week through that lens, and patch urgency just got harder again.

CISA KEV this week: UniFi triple-10, Splunk, Lantronix

On 06-23, CISA added several actively exploited vulnerabilities to its KEV catalog. Source note: the KEV page itself returned 403 on fetch — the additions were cross-checked consistently across four independent outlets (SecurityWeek, Cybersecurity News, Cybernews, SecurityAffairs).[10][11][12]

CISA KEV: the US Cyber agency's "Known Exploited Vulnerabilities" catalog — a list of flaws with active attacks running now. Like a government recall list for products where accidents are already happening. US federal agencies have binding patch deadlines; for everyone else, "in KEV" is the strongest signal available: this is being actively exploited right now.

Ubiquiti UniFi OS — Triple CVSS-10

Three flaws each scoring the maximum CVSS 10.0: CVE-2026-34908 (Improper Access Control, unauthenticated system changes), CVE-2026-34909 (Path Traversal, OS file access), and CVE-2026-34910 (Improper Input Validation, unauthenticated OS command injection).[10][12] Chained, 34908 + 34909 → 34910 form an unauthenticated remote code execution. KEV-added 06-23, federal deadline 06-26 (already past for government). Fix: UniFi OS Server v5.0.8 (patched 05-21) — the vulnerability has been closed for over a month, but missing from many devices.

Splunk Enterprise — pre-auth RCE

CVE-2026-20253, CVSS 9.8 (CWE-306): unauthenticated attackers can create or truncate arbitrary files on the host via the PostgreSQL sidecar service — the path to RCE. A public PoC has circulated since 06-13.[13] Affected: 10.2.0–10.2.3 and 10.0.0–10.0.6. Fix: 10.4.0, 10.2.4, or 10.0.7.

Lantronix EDS5000

CVE-2025-67038, CVSS 9.8, code injection with root-level OS command execution — also added to KEV on 06-23.[11]

Sources: securityweek.com [10] · securityaffairs.com [11] · cybernews.com [12] · thehackernews.com [13]

Docker: AuthZ plugin bypass (CVE-2026-34040)

A flaw relevant to your stack outside KEV: CVE-2026-34040, CVSS 8.8, an incomplete fix for CVE-2024-41110. Request bodies over 1 MB silently get truncated before the AuthZ plugin sees them, while the daemon continues processing — letting attackers bypass authorization checks and potentially take over the host.[14] Fix: Docker Engine 29.3.1, Docker Desktop 4.66.1. If you run AuthZ plugins for multi-tenant Docker, you should audit specifically.

Context: runC container escapes Meanwhile, three runC escape flaws (CVE-2025-31133/52565/52881) keep circulating actively — disclosed Nov 2025, exploited in 2026, affects Docker + containerd + Kubernetes. Patched versions: runC 1.2.8, 1.3.3, 1.4.0-rc.3.[15]

Browser engines: V8 zero-day, Chrome 150, Firefox 152

Chrome had an in-the-wild zero-day: CVE-2026-11645, an out-of-bounds read/write in V8 (Chrome's JS engine), letting a crafted HTML page run code inside the browser sandbox. In KEV since 06-09, fixed in Chrome 149.[16] Follow-up patch 149.0.7827.196/197 (06-23) also closed CVE-2026-13025 (DevTools sandbox-escape helper) plus 18 more fixes. Chrome 150 stable is due 06-30 — notable: an IndexedDB backend on SQLite and CSS system color support (AccentColor). Firefox 152 has been stable since 06-16; 152.0.2 (06-23) fixes crashes on Intel Raptor Lake CPUs.[17]

ARD: an open "robots.txt for the agent era"

On 06-17, the Agentic Resource Discovery spec (ARD) was published under Apache 2.0 — backed by 11 companies: Google, Microsoft, GitHub, Hugging Face, NVIDIA, Salesforce, Cisco, Databricks, GoDaddy, ServiceNow, and Snowflake.[18] ARD defines a well-known manifest (`ai-catalog.json`) plus a registry API and answers a single question: "Which resource helps with this task?"

ARD / ai-catalog.json: think of `robots.txt` — the tiny file a website uses to tell search crawlers what they're allowed to index. ARD is the same idea, but for AI agents instead of search engines: a domain publishes an `ai-catalog.json` listing what tools, data, and services it offers — and agents discover them machine-readably, rather than hard-coding them.

Why it's here (Software) not in the AI brief: ARD is an interop standard — a registry and discovery layer — not a model or agent tool. Reference implementations already exist: GitHub's "Copilot Agent Finder" and Hugging Face's "Discover Tool" index ARD-compliant resources.[19] No spec version number was findable; the repo is at `github.com/ards-project`.

EU CRA: less than 100 days to reporting deadline

The Cyber Resilience Act is ticking: beginning 09-11-2026, the reporting obligation kicks in for incidents and actively exploited vulnerabilities — less than 100 days away.[23] The earlier deadline to name notifying authorities (06-11) has already passed; ENISA is offering dry-run support in June.

CRA reporting obligation: whoever sells "products with digital elements" in the EU must now report actively exploited vulnerabilities and serious incidents to ENISA/CSIRT within tight deadlines — like the breach-notification requirement under GDPR, but for security flaws. For solo developers and small shops, that means: a documented incident-reporting process becomes compliance requirement, not optional nice-to-have.
