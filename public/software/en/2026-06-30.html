NIS2 in Germany: 62 % unregistered — and oversight tightens [1][2]

In the background of the patch wave, Germany's NIS2 rollout is underway. For Daniel as an EU service provider, this is the most relevant "software policy" topic, even though twiceD as a small vendor likely falls below the thresholds — the supply chain still catches it.

NIS2: An EU directive that mandates minimum cybersecurity standards — like a TÜV requirement, but for the IT security of companies deemed "critical" or "essential" to society. Germany is implementing it via the new BSI Act.

The scope has grown from roughly 4,000 legacy KRITIS operators to about 29,500 entities — a sevenfold jump. The problem: by the BSI registration deadline in March — extended through July 31, 2026 in practice — 62 % of affected firms had not yet even registered. The BSI oversight phase with proactive audits for "essential" entities begins in the second half of 2026; those deemed "essential" risk fines up to €10M or 2 % of global revenue. Practically for Daniel: customers will increasingly demand security certifications from small suppliers too — the obligation cascades down.

Complementing this, Chapter IV of the EU Cyber Resilience Act (conformity assessment body designation) has been in effect since June 11; reporting and vulnerability obligations follow from September 11, 2026, full applicability December 2027.[3]

Splunk: First bug ever on the CISA KEV list — and it's a 9.8 RCE [4][5][6]

In all these years, no Splunk bug ever appeared on CISA's list of actively exploited flaws. CVE-2026-20253 is the first — and with force: CVSS 9.8, pre-auth remote code execution.

The mechanism is instructive: Splunk Enterprise runs a PostgreSQL sidecar whose endpoint was accessible without authentication. Through it, an attacker without login could create or truncate arbitrary files — the lever to code execution.

PostgreSQL sidecar: A helper container that runs alongside the main app and provides it with a database — like a copilot holding the map. Here, the passenger door was unlocked: whoever got in could write files at the wheel.

Affected are Splunk Enterprise 10.2.0–10.2.3 and 10.0.0–10.0.6. A public proof-of-concept landed on June 12, confirmed wild exploitation followed on June 18 — six days. CISA added the bug to the KEV list on June 18 with an FCEB deadline of June 21. Fix: 10.2.4 or 10.0.7. Anyone running Splunk belongs in patch review these days.

Check Point VPN: IKEv1 legacy, exploited by Qilin — and BSI warns [7][8][9]

CVE-2026-50751 (CVSS 9.3) lurks in the deprecated IKEv1 protocol of Check Point's remote-access VPN: an auth bypass that lets a VPN session establish without any authentication. This is the classic "deprecated but still active" trap.

IKEv1: The old handshake protocol where two VPN endpoints swap keys before encrypting. Superseded by IKEv2 for years, but often still enabled for backward compatibility — like a second, rusty door lock you never removed.

On June 9, the BSI issued a "high" level warning — actively exploited flaw, VPN sessions without authentication. Rapid7 attributes the activity to a Qilin ransomware affiliate, with observed access as early as May 7 and uptick in early June; the exact count of affected organizations stood at "dozens" in early June. A related flaw in the same IKEv1 path, CVE-2026-50752 (CVSS 7.4), allows man-in-the-middle on site-to-site VPNs. If you don't need IKEv1, disable it — that's the most effective mitigation alongside the patch.

Ubiquiti UniFi OS: Three tens, chained to root [10][11]

If you run a UniFi gateway or UniFi OS server — and many homelabs and small offices do — you should be on 5.0.8 today. Three CVEs each rated CVSS 10.0 (CVE-2026-34908 access control, -34909 path traversal, -34910 command injection) can be chained: unauthenticated remote code execution with full root privileges. CISA added all three to the KEV list on June 23; the patch (UniFi OS Server 5.0.8) has been available since May 21 — the lag between fix and KEV addition shows how long unpatched systems stayed exposed.

"Prompts are the new malware" — Five Eyes warn, CrowdStrike counts [12][13][14][15]

This is the story software owns: not the next coding model, but the attack surface that AI pulls into every stack. On June 22, the Five Eyes (USA, UK, Australia, New Zealand, Canada) released a rare joint threat assessment: coming AI models will ramp up offensive hacking capabilities — on a timescale of "months, not years." The recommendations are unremarkable and exactly right: invest in defense, patch legacy systems, limit access to critical assets.

CrowdStrike lays down the numbers: in their 2026 report, "prompts are the new malware" — attackers injected malicious prompts into GenAI tools at over 90 organizations in 2025. Average "breakout time" fell to 29 minutes (fastest case observed: 27 seconds), AI-assisted attacker operations rose 89 % year-over-year.

Prompt injection: An attacker hides instructions in data that an AI agent reads anyway — an email, a document, a website. The agent can't cleanly separate "data" from "command" and executes the hidden instruction. Like a forged memo slipped into the intern's inbox saying "wire this account" — and the intern does it.

OWASP now maps prompt injection to six of its top ten categories for agentic applications (2026); per the same OWASP tracking (via helpnetsecurity), 73 % of production AI deployments showed prompt injection weaknesses. For Daniel's practice: if an agent calls tools or reads external content, input is an attack vector — least privilege on tool access and a human gatekeeper on irreversible actions are no longer optional.

The supply-chain vector migrates into the AI-tooling layer [15]

Part of the same story: a concrete incident in March 2026 when the PyPI package LiteLLM was backdoored — about 47,000 downloads in a three-hour window, and the dependency chain hit CrewAI, DSPy, and Microsoft GraphRAG. Exactly the building blocks people use today to build agents. In OWASP's tracking of agentic app repos, Claude Code ranks second with 22 security advisories (behind n8n with 57) — no scandal, just a measure that these tools are now seriously being probed for attacks. Lesson: the agent-tooling layer is a real supply chain — pins, lockfiles, and provenance belong there as much as in the app stack.

WordPress: 1.2M sites backdoored via compromised CDN scripts [16][17]

Classic supply-chain attack, but with an instructive path: between June 12–14, Awesome Motive's OptinMonster, TrustPulse, and PushEngage plugins were backdoored via compromised CDN scripts — roughly 1.2M sites affected. The entry wasn't via the plugin itself but through a flaw in the UpdraftPlus plugin on Awesome Motive's own WordPress site; attackers leveraged that to steal a CDN API key and alter the delivered JavaScript files. The malware created a rogue admin (developer_api1) and installed a self-hiding backdoor plugin. If you run one of those three plugins, audit admin accounts and installed plugins.

Microsoft Patch Tuesday: Record with 206 CVEs [18][19]

On June 9, Microsoft delivered its largest patch Tuesday ever: 206 CVEs, of which roughly 39 critical (exact count varies by source) and at least three zero-days. The most interesting is CVE-2026-49160 — DoS against large web servers via HTTP.sys, technique called "HTTP/2 bomb." Plus a CTFMON privilege escalation (CVE-2026-45586) and a BitLocker bypass via physical attack (CVE-2026-50507, CVSS 6.8). Distribution: 65 EoP (32 %), 55 RCE (27 %), 29 information disclosure (13 %). No patch-tonight alert — but the HTTP/2-bomb vector is relevant for anyone running exposed HTTP/2 endpoints.

Spring 7.0.8 / 6.2.19: 17 CVEs — and a migration signal [20][21]

For Daniel's Java/Spring DNA, the release of the week: Spring Framework 7.0.8 and 6.2.19 (June 8) close 17 CVEs. The spring.io notes list the CVE IDs but no individual CVSS scores — so no score quote from me, just the substance: arbitrary method invocation in SpEL expressions (CVE-2026-41852), SSRF via UriComponentsBuilder (-41854), unsafe deserialization via Jackson JMS converter (-41855), and multipart request smuggling (-41853). The real message hides in the footnote: 6.2.19 is "very likely the last OSS release of the 6.2.x generation." That's the official EOL signal — anyone on 6.2.x should plan migration to 7.0.x (Java 17 minimum, support through June 30, 2027) now.
