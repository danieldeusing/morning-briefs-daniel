SharePoint RCE: CISA overrides Microsoft's risk call [1]

Today's story is less about the hole than the contradiction behind it. CISA has added CVE-2026-45659, a remote code execution in on-premises SharePoint (insecure deserialization), to its KEV catalog after confirming active exploitation. Affected: SharePoint Server Subscription Edition, 2019, and Enterprise 2016. The kicker: when Microsoft patched the bug in May, it downrated real-world exploitation as "Less Likely"—CISA has now reversed that call.[1]

KEV catalog: The Known Exploited Vulnerabilities list from US authorities (CISA) is like a wanted poster for security holes that attackers are already using in the real world. If a bug is on it, it's not theoretical anymore—someone is exploiting it today. For US federal agencies, patching becomes mandatory with a deadline.

What makes the hole especially nasty is the low bar: according to The Register, any account with "Site Member" permissions plus valid credentials is enough to run arbitrary code—no admin needed.[1] Under Binding Operational Directive 26-04, agencies must patch by 2026-07-04. For anyone running on-premises SharePoint, that's the signal to finally apply the May patch—not because I'm raising an alarm, but because it's being exploited in the wild.

libssh2: two pre-auth holes, one PoC, no release [2][3]

libssh2 sits as an SSH client library in countless tools and language bindings—anywhere a program connects over SSH without invoking the OpenSSH client directly. Two newly disclosed bugs hit all versions ≤ 1.11.1, and both fire before authentication.

  • CVE-2026-55200 (CVSS 9.2)—Out-of-bounds write in transport.c. A malicious SSH server can inject code into a client. The critical detail: a public proof-of-concept is already circulating.[2]
  • CVE-2026-55199 (CVSS 8.2)—Out-of-bounds read plus infinite loop in the SSH_MSG_EXT_INFO handler (packet.c), making it a pre-auth denial-of-service via CPU exhaustion.[3]
Heads up The fixes are merged to master (commits 7acf3df / 1762685), but there's no tagged official release yet. If you have libssh2 vendored or pull it via distro packages, there's nothing finished to deploy right now—if you connect outbound to untrusted SSH servers (CI runners, scrapers, deploy tools), keep the master fixes on your radar.

A sourcing note of our own: the 9.2 for CVE-2026-55200 comes from the official GitHub Advisory database; the 8.2 for the DoS bug is currently only secondhand-sourced (Hacker News snippet, cross-checked against the sibling entry)—so read that number with a caveat until the official CVE entry catches up.

npm worm "Miasma": self-propagating via binding.gyp [4]

The week's most interesting software-engineering incident is a supply-chain worm in the Shai-Hulud line. It has infected 57 npm packages in hundreds of malicious versions—and unlike the usual suspects, it doesn't spread via lifecycle scripts but through the binding.gyp file that node-gyp processes when compiling native modules.[4]

binding.gyp / node-gyp: Some npm packages contain native C/C++ code that must be compiled during installation. node-gyp is the build tool, and binding.gyp is its recipe. Imagine ordering furniture and the assembly instructions themselves run a command when you open the box—that's where the worm hooks in.

The mechanism is textbook vicious: node-gyp's command expansion runs code during `npm install`, the worm lists all packages by the maintainer via registry.npmjs.org, downloads each target, injects its own binding.gyp plus index.js, and republishes. Biggest victims by reach: @vapi-ai/server-sdk (~86,500 downloads/week) and ai-sdk-ollama (~36,900/week).[4]

Context That AI SDK packages rank among the biggest targets is no accident—the agent and tooling layer around AI is a fast-growing, often freshly-maintained attack surface. But the root problem remains software—install-time code execution—not the model behind it. Practical hardening: `--ignore-scripts` doesn't cut it here because the vector is node-gyp command expansion, not a lifecycle hook. Pin lockfiles, scrutinize new major bumps in native packages.

M365 password spray: 81 million attempts, MFA bypassed [5]

Huntress documented an aggressive password-spray campaign against Microsoft 365 environments: over 81 million login attempts between June 12 and 26, 2026 (peak on the 22nd), with 78 compromised accounts across 64 organizations at the end.[5]

Password spraying: Instead of hammering one account with many passwords (which triggers lockouts), the attacker tries one common password against many accounts—like a burglar testing a master key on every door on the street instead of trying all keys on one door.

The real pain lies in the mechanism: attackers used the ROPC OAuth flow, which sends the password directly to the /token endpoint—without an interactive MFA prompt. Even if Conditional Access requires MFA, the second factor doesn't trigger on this flow. Countermeasures: block ROPC entirely via Conditional Access policy, disable legacy auth.[5]

curl shuts the bounty door for a month [7]

Daniel Stenberg is pausing curl's HackerOne report form for all of July—2026-07-01 00:00 CEST through 2026-08-03 09:00 CEST, dubbed "summer of bliss" in the blog post. Reason: plain overload. The rate of incoming security reports is currently 4–5× above 2024 and double 2025, averaging more than one report per day.[7]

That's remarkably honest for a core infrastructure project: better to close the door and work through the real cases carefully than drown in noise (much of it questionable "AI-slop" reports). Collateral damage: curl 8.22.0 is sliding to 2026-09-02. No security issue—just a release waiting its turn.

Standing thread: CRA disclosure duty looms

A heads-up, even though it falls outside the 7-day window: the EU Cyber Resilience Act activates its disclosure duty for vulnerabilities and incidents on 2026-09-11—the actual product security requirements don't kick in until 2027-12-11. Meaning: if you bring software to market in the EU, you'll need to disclose actively exploited holes to ENISA/CSIRT within tight timeframes starting in September. For small teams (Daniel-size) this is the moment to think through your incident reporting chain before you have to improvise under time pressure.[8]
