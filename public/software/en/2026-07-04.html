An emergency weekend: three actively exploited holes

Adobe, Citrix, and Microsoft each closed a flaw between June 30 and today in products that sit at the center of many corporate networks — and all three were already under attack before most admins even saw the patch. That's the real point of today: not the number of CVEs, but the window between disclosure and exploit, now measured in hours. I'll walk through all three — what it is, who's affected, and what to do without getting lost in ops details.

Adobe ColdFusion — CVSS 10.0, the maximum [1][2][3]

On 06/30, Adobe published the emergency bulletin APSB26-68 — Priority Rating 1, the top tier — closing eleven flaws.[1] The most severe is CVE-2026-48282, a Path-Traversal (CWE-22) that leads to arbitrary file write and thus code execution. The CVSS score: 10.0 — the theoretical maximum, almost never seen.[2]

Path-Traversal: Picture a doorman who's only allowed to fetch files from a specific room. A path-traversal is the trick of handing him a note saying "fetch me the file two doors over, then down three flights" — he faithfully follows the directions right out of the allowed room. With CVE-2026-48282, the attacker lands outside the web directory and gets to write there — and whoever can write can execute code.

Affected are ColdFusion 2025 (Update 9 and older) and ColdFusion 2023 (Update 20 and older); fixes are in Update 10 and Update 21 respectively.[3] And this is where it gets serious: The hole came within hours of disclosure under active exploit, one observed attempt traced to an IP in India.[3] ColdFusion has been a recurring CISA-KEV theme in prior years — anyone still running it knows the pattern. If a ColdFusion instance is reachable from the internet, the patch window is effectively closed.

Heads up CVSS 10.0 means: no user account needed, no interaction, remote, full impact. Publicly reachable 2025/2023 installs need Update 10 / 21 immediately — and if in doubt, check whether anything's already been written.

Citrix NetScaler — "CitrixBleed," round three [4][5][6]

On 06/30, Citrix closed six bugs in NetScaler ADC/Gateway, including CVE-2026-8451 (CVSS 8.8): insufficient input validation leading to a memory overread when the appliance is configured as a SAML Identity Provider.[4] An unauthenticated attacker can thus read fragments of the appliance's memory — same class as the notorious "CitrixBleed" hole.

Memory Overread: Like reading aloud from a note without stopping at the end of the sentence, just keeping going with whatever happens to be lying nearby on the table — notes that aren't yours. Here a custom-built XML/SAML parser doesn't stop at a null byte, a `>`, or a matching quote; it doesn't stop at spaces or newlines either.[6] So it "keeps reading" into adjacent memory — which might hold session tokens, credentials, or other secrets.

Vulnerable are 14.1 before 14.1-72.61 and 13.1 before 13.1-63.18 (plus corresponding FIPS/NDcPP builds).[5] The pace is troubling here too: Security firm Lupovis reported active exploitation less than 24 hours after release.[4] NetScaler boxes typically sit at the network edge as VPN/gateway gatekeepers — exactly the place where a memory leak hurts most.

SharePoint — KEV deadline expires today [7]

The US Cyber Agency CISA has added CVE-2026-45659 — an RCE hole in Microsoft SharePoint Server, CVSS 8.8, arising from deserialization of untrusted data — to its Known-Exploited-Vulnerabilities catalog (KEV). For US federal agencies (FCEB), the patch deadline is 07/04/2026 — today.[7]

CISA KEV: A wanted list maintained by the US Cyber Agency for holes that are demonstrably being exploited in the wild. Being added is the signal "this is no longer theory, it's happening now" — and it sets agencies a hard patch deadline. For everyone else, the KEV is the best free prioritization list out there.

Deserialization in short: the program takes stored data and reconstructs live objects from it; if that data is tampered with and awakened unchecked, it can hold malicious code — a recurring pattern in SharePoint. Anyone running SharePoint on-prem should be aware of this deadline, agency mandate or not.

Source note The CISA KEV page itself was unreachable today on direct fetch (HTTP 403). The KEV listing and the 07/04 deadline are thus cross-checked via trade press and carried as reported — not confirmed from primary source.

npm v12: Supply chain gets locked down by default [8][9]

The biggest OSS move of the week isn't a CVE, it's a principle shift: npm v12 — per GitHub announcement of 06/09, planned for July 2026 — turns off three risky defaults.[8] npm install will no longer run install scripts (preinstall/install/postinstall) from dependencies, no longer resolve Git dependencies, and no longer fetch remote-URL tarballs (e.g., https packages) — each only with explicit permission (--allow-git, --allow-remote, or a project allowlist).[9]

Install-script attack: Many npm packages are allowed to run a small script automatically during install — meant e.g. for compilation. It's also the doorway: a malicious (or hijacked) package can let arbitrary code run during a bare npm install, before you've used a single line. Like new furniture that walks through your house on unpacking. npm v12 locks that door — it only opens if you explicitly turn the key.

This is the direct, overdue response to the wave of self-replicating supply-chain worms in recent months. Good to know: the changes are already testable today — behind warnings in npm 11.16.0, so build pipelines can be tested before the jump.[8]

The catch Security researchers point out that v12 does not solve the bigger problem: takeover of a maintainer account. If a legitimate account is compromised, malicious code ships as a regularly signed release — then script-blocking doesn't help, because the package is "trusted." Hardening installation ≠ hardening identity.[10]

Let's Encrypt ends tlsclient — Chrome splits the PKI [11]

On 07/08, Let's Encrypt sunsets its tlsclient ACME profile — the profile used to issue certificates for client authentication (mTLS). The reason isn't a flaw at Let's Encrypt, but a mandate from Chrome's Root Program: Google requires by June 2026 that TLS client and server auth be split into separate PKIs.[11]

mTLS & separate PKIs: In standard HTTPS, only the server proves itself. In mTLS, the client also shows an ID — useful for service-to-service traffic. Until now, the same certificate "authority" (PKI) could issue both types of ID. Chrome wants them split, so a problem in one world doesn't poison the other — like separate ID-issuing authorities for a passport and a work badge. Upshot: for mTLS, you'll need your own private CA instead of Let's Encrypt going forward.

In practice: if you've used Let's Encrypt for mTLS client certs, you need to switch to a private CA by 07/08 — for plain server certs (regular HTTPS), nothing changes. For most setups this is a sideshow, for zero-trust / service-mesh architectures it's a real migration point.

One more thing, same thread: Let's Encrypt is shortening certificate lifetimes. The tlsserver profile has been able to optionally issue 45-day certs since 05/13; the classic default profile moves to 64 days by 02/10/2027 and to 45 days by 02/16/2028.[12] Anyone still renewing by hand should move to automation now.

PostgreSQL: security thread in the rearview [14]

Standing thread — EU: CRA reporting mandates approach [15]

The Cyber Resilience Act (CRA) isn't a deadline this week, but it's on every networked-product maker's radar: Starting 09/11/2026, mandatory reporting kicks in for flaws and incidents via a central platform — first report within 24 h, follow-up within 72 h, final report no later than 14 days after the security patch.[15]

Cyber Resilience Act (CRA): An EU law that mandates "security by design" for everything with digital elements — from smart door locks to a library in an app. The part coming into force now: if an exploited hole becomes known, you must report it to authorities, and fast. Like a reporting requirement for certain diseases — except you're reporting software flaws. If you ship into the EU, start practicing the 24-hour reflex now, not on 09/11.

For solo devs and small DE teams: prepare a lean reporting workflow before the first incident starts the clock. The combination this week — holes exploited in hours — plus a 24-hour reporting mandate from September makes clear where we're headed.
