A No-Warning PoC Dump Leaves Two Critical Flaws Wide Open

In late June, someone ignored the industry's rulebook: an anonymous researcher going by the handle "bikini" published working exploit prototypes for Docker, Gitea, libssh2, PHP, and Firefox — without notifying a single vendor beforehand. Two of the findings are severe, and one still has no fixed version today. In parallel, the kernel closed a flaw that had sat in the code since August 2010 and lets guests escape their VM onto the host. No alarm mode — but a good day to ask: who's allowed to start CI workflows at your org?

  • Sec Infra Gitea act_runner: CVE-2026-58053, NVD score 9.9 critical. Anyone allowed to run a workflow on a Docker runner lands on the host as root. NVD lists no fixed version.
  • Sec OSS libssh2: CVE-2026-55200 — fix exists only as upstream commit 7acf3df, still no tagged release (latest remains 1.11.1). A client-side flaw — the malicious SSH server is the one attacking you. Score divergence: 9.2 (CVSS 4.0) vs. 8.3 (NVD, CVSS 3.1).
  • Sec Infra "Januscape": CVE-2026-53359, a 16-year-old use-after-free in KVM's shadow MMU. Guest-to-host escape on Intel and AMD. Fixed in the kernel since July 4 — don't forget companion CVE-2026-46113.
  • Sec USA CISA KEV: SharePoint RCE CVE-2026-45659 on July 1, three more on July 7 — including Langflow CVE-2026-55255 (8.4 per NVD vs. 9.9 per secondary trackers).

"Exploitarium": a dump that breaks the rules of the game [1][2]

Between June 27 and July 1, 2026, an anonymous researcher going by the handle "bikini" filled a public repository with working proof-of-concept exploits — for Docker, Gitea, libssh2, PHP, Firefox, OpenVPN, 7-Zip, VLC, and Ghidra.[1] No vendor was notified beforehand: no embargo, no deadline, no patch window.

Coordinated disclosure: Someone who finds a flaw normally tells the vendor first, typically giving them 90 days. It's like discovering your neighbor's lock is broken — you knock on their door instead of putting it in the city paper. Exploitarium skipped the knock.

What's notable is that no one can pin down a clean number: heise described "just under two dozen" projects on July 1,[1] while byteiota counts 130+ PoCs across 22 projects.[2] The repo kept growing while the articles were being written. There's no reliable "as of today" total — anyone who quotes one has simply picked their favorite.

My take: in the short term, the dump hurts defenders more than it helps attackers. The more interesting structural lesson is that the more serious finding sits in a tooling layer almost no one inventories as attack surface — the CI runner.

libssh2 CVE-2026-55200 — and why "9.2" and "8.3" are both correct [3][4]

ssh2_transport_read() was missing an upper bound on packet_length. Per NVD, it's an out-of-bounds write, affecting all versions through 1.11.1, fixed with commit 7acf3df — which as of today (2026-07-08) has landed in no tagged release; the newest upstream tag remains 1.11.1, and distributions ship patched 1.11.1 builds.[3][4]

Score divergence NVD itself carries both scores: 9.2 (critical) and 8.3 (high). Not a contradiction, but a standard difference: 9.2 is a CVSS 4.0 score, 8.3 a CVSS 3.1 score.[3] If you prioritize by threshold, know which scale your scanner speaks.

More important than the score is the vector. NVD's 3.1 vector ends in UI:R. So this is a client-side flaw: it's not your SSH server being attacked, but your curl, your Git, your PHP — the moment it connects to a server an attacker controls.[3] No credentials are needed: the corruption happens while reading the transport packet, before any authentication.

Transitive dependency: You never installed libssh2, and yet it's in your house. curl uses it for SCP/SFTP, Git for git clone ssh://…, PHP for ssh2_*. Like sugar in a jarred sauce: not on the front of the label, but in there. The question isn't "do I use libssh2?" but "what in my image links libssh2?".

The uncomfortable consequence: outbound SSH connections are attack surface. A CI job that clones from a compromised remote is enough.[4] Check with ldd $(which curl), then rebuild base images.

Gitea: 9.9 in the runner, header spoof in the server [5][2]

act_runner CVE-2026-58053 — no fixed version

The most severe finding in the dump. NVD assigns 9.9 (critical), vector CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H.[5] Gitea's act_runner, with the Docker backend, passes a workflow's container.options string straight through to the job container's HostConfig. If it says privileged: false, only the privileged flag gets switched off — --pid=host, --cap-add, or --security-opt travel along unchanged.[5] Anyone allowed to run a workflow gets host namespaces and far-reaching capabilities this way.

No fix in sight Affected is act through 0.262.0 inclusive; NVD lists (as of Jul 8) no fixed version.[5] Mitigation is organizational, not technical: who's allowed to start workflows on your Docker-backed runners? For self-hosted runners on public repos, the answer is often "anyone with a pull request".

CVE-2026-20896 — auth bypass, but fixed

Separately, a confirmed authentication bypass: any source IP could gain full admin rights via the HTTP header X-WEBAUTH-USER: admin.[2] One header, no password. Fixed in Gitea 1.26.3 / 1.26.4 — unlike the runner issue, there's actually an update here.

  1. nvd.nist.gov — CVE-2026-58053 (modified Jun 30, 2026)

Docker cp: reported, unconfirmed, no comment [2]

The same dump includes a PoC against docker cp, validated against Docker CLI/Engine 29.6.0 (API 1.55), PoC dated June 23.[2] Described deliberately in broad strokes: a process inside the container can race a host-side copy operation over target-path resolution, so that a path controlled by the container ends up outside the requested destination.

Two honest caveats: there's no CVE, and Docker Inc. had not commented as of research time. This is a disclosed PoC, not a vendor-confirmed vulnerability with a severity rating — anyone who runs docker cp automated against containers populated by others should keep it in mind.

"Januscape": a 16-year-old VM escape [6][7]

CVE-2026-53359 is a use-after-free in KVM's shadow paging. Per NVD: when a shadow page is freed after a memslot is deleted, an rmap entry survives, and later memory accesses dereference a freed page.[6] The vulnerable code traces back to a kernel commit from August 1, 2010.[7] Because it sits in the shared x86 shadow MMU, it hits Intel and AMD equally.

Guest-to-host escape: A VM is a guest in the hypervisor's house and is never supposed to leave its room. An "escape" is when the guest walks through the wall into the living room and finds the keys to every other room there. For cloud neighbors, that wall is the entire security model.
Score pending There's no consolidated CVSS: NVD lists (as of Jul 8) no score of its own,[6] Red Hat's advisory gives a base of 7, secondary trackers list 7.8 (CVSS 3.0).[7] All three figures circulate in parallel. For prioritization, what matters anyway is whether you run other people's code in VMs on your hardware.

Where /dev/kvm is world-writable — RHEL, for instance — reporting says an unprivileged local user is enough to get root on the host.[7] That elevates the flaw from a "cloud-provider problem" to a "multi-user machine problem".

Fixed on July 4, 2026 in the stable series 7.1.3, 6.18.38, 6.12.95, 6.6.144, and 6.1.177.[7] Important caveat (as of 2026-07-08): NVD lists no fix for the 5.15.x and 5.10.x series — if you run 5.x, check with your distributor whether the backport landed rather than assuming you're patched. The related CVE-2026-46113 (KVM shadow-paging UAF, same file) affects series below the July stables; no reachable primary source claims the two patches are strictly coupled.

CISA KEV: four adds in one week [8][9]

KEV is the catalog of vulnerabilities for which CISA has evidence of active exploitation — not a risk forecast, but "this is happening right now". On July 1, the SharePoint Server RCE CVE-2026-45659 was added (CVSS 8.8, deserialization of untrusted data, patched via an out-of-band update in late May); the deadline for US federal agencies expired on July 4.[8] On July 7, CVE-2026-48908 (JoomShaper SP Page Builder, unrestricted file upload), CVE-2026-55255 (Langflow, authorization bypass), and CVE-2026-56290 (Joomlack Page Builder, improper access control) followed.[8]

Sourcing note cisa.gov and thehackernews.com returned HTTP 403 on every direct fetch attempt. The KEV figures come from the indexed text of the CISA alert page plus confirmation via NVD — not from a directly loaded cisa.gov document. CVE IDs and Langflow details are checked against NVD; the KEV-add data is not.

Langflow deserves a second look. NVD describes an IDOR vulnerability in the /api/v1/responses endpoint: any authenticated user can execute someone else's flow by writing that flow's ID into the request. Fixed in Langflow 1.9.1 per the GitHub advisory; NVD's CPE data lists <1.9.2 as affected.[9] And another score divergence: the GitHub CNA (mirrored on NVD) lists 8.4 (high), vector CVSS:3.1/AV:N/AC:H/PR:L/UI:N/S:C/C:H/I:H/A:L; secondary trackers report the same CVE as 9.9 (critical).[9] I'm citing both — the difference is visibly in AC:H and A:L (Scope is S:C in both).

Docker Compose 5.3.1 — and what was in 5.3.0 [10]

Compose 5.3.1 came out yesterday and is a pure maintenance release (CI and dependency bumps).[10] The actual feature landed five days earlier: 5.3.0 from July 2 brings native init containers.[10]

Familiar from Kubernetes, previously only faked in Compose via depends_on contortions: an init container runs to completion before the service starts — migrations, waiting on Postgres, volume permissions. Anyone who baked wait-for-it.sh into their entrypoint can now express that declaratively. With the reconciliation engine from 5.2.0,[10] Compose is moving toward "little Kubernetes for one machine".

npm v12: install scripts finally off [12]

The three breaking changes from GitHub's changelog of June 9:[12] allowScripts is off by default — npm install no longer runs preinstall, install, and postinstall of dependencies. --allow-git and --allow-remote default to none: Git or remote-URL dependencies must be explicitly allowed.

postinstall script: A package is allowed to run an arbitrary command on your machine while installing — meant for "compile the native extension". Like a furniture box that starts walking around your apartment the moment you open it. Almost every npm supply-chain worm rides on this: installing means executing code, not just downloading it.

The trigger is obvious. Per reports, the Mastra AI incident happened around June 17 — a stolen maintainer account, 140+ packages in the @mastra scope republished with a malicious dependency, attributed by Microsoft's threat intel to the Sapphire Sleet group (BlueNoroff/APT38); the sixth notable npm poisoning in eleven weeks, preceded by Bitwarden CLI, SAP CAP, TanStack, and AntV. The attribution and count come from secondary reporting — context, not proof.

v12 is slated for July 2026, not out yet as of today.[12] npm 11.16.0 and newer already warn about the spots that will break — a look at the CI logs on Saturday tells you what's going to cost you in August. My opinion: this should have arrived in 2018. That it took state actors to flip a default is the real story.

Jamf AI Governance: endpoint policy for AI tools [13]

Generally available since June 30: an OS-level visibility and policy layer for AI usage across Mac fleets. It detects AI tools, agents, and LLM runtimes on the device ("shadow AI discovery"), lets you set policy on them, and logs for audits; initially covering Claude Code, Claude Desktop, and OpenAI Codex.[13] Jamf says it manages over 35 million devices across 78,000+ organizations.[13]

Why this runs here and not in the ai-dev brief: the hook is endpoint management, not the model. From an IT standpoint, a coding agent with shell access is a process with network and filesystem rights — and until recently, it appeared in no inventory at all. Jamf cites Gartner on AI governance spend of $492 million in 2026 and over $1 billion by 2030; organizations with deep AI integration reported an incident 40% more often.[13] Both figures come from the Jamf press release, not from a Gartner report we reviewed directly.

EU: Action Plan on Cybersecurity & AI [14]

Yesterday, July 7, the Commission unveiled its "Action Plan on Cybersecurity and Artificial Intelligence." It explicitly builds on existing law: the AI Act, the Cyber Resilience Act, the NIS2 directive, and the Cyber Solidarity Act.[14] In substance, it covers an EU-owned AI evaluation capacity and a secure testing platform together with ENISA.

What's not in it: concrete implementation deadlines — notable, since the plan explicitly refers to the CRA, whose reporting obligations become mandatory in September 2026. An action plan without a date is a statement of intent. For anyone shipping software into the EU, the CRA deadline remains the date that counts.

Confidential Computing: the location guarantee is gone [15]

Research published on July 4 describes a "diversion attack" against attested TLS: a connection meant for a specific server can be silently redirected to a different, compromised machine running identical software — without the client noticing.[15] Affected: Meta's WhatsApp Private Processing, Edgeless Systems Contrast, and Cocos AI (v0.4.0–0.8.2), plus a PoC from the CCC Attestation SIG.

Remote attestation: Before you hand a stranger's server your secrets, its CPU shows you a signed photo: "Exactly this code is running, in this encrypted enclave." Like a photo ID — it's genuine, the code checks out, it just doesn't say which person is holding it up.

The researchers' recommendation is uncomfortable: drop attestation inside the handshake, switch to post-handshake attestation; a complete fix within today's model may not be possible at all.[15] Not an operations issue, but a procurement one: anyone being sold "Confidential Computing" as a compliance argument should ask when attestation actually happens.

  1. theregister.com — Confidential computing broken (Jul 4, 2026)

Microsoft loses used-license dispute [16]

The UK Court of Appeal yesterday rejected Microsoft's appeal against a ruling that allows ValueLicensing to resell used on-prem licenses; the claim puts damages at £270 million.[16] At its core, this is the old question of whether a vendor can dry up the secondary market for perpetual licenses to push customers into subscriptions. For European IT buyers, it's a real bargaining position.

It's Wednesday, July 8th, and today's news is overshadowed by something that happened at the end of June: an anonymous researcher going by the handle bikini simply dropped a whole collection of working exploit prototypes onto the internet. For Docker, for Gitea, for libssh2, for PHP, for Firefox. Without telling a single vendor beforehand.

That breaks a rule the industry has lived by for decades. Normally you tell the vendor and give them around ninety days. Here, there was no knock on the door. And nobody can say for sure exactly how many exploits we're talking about. heise wrote on July first about just under two dozen projects; byteiota counts a hundred and thirty prototypes across twenty-two projects. The repository kept growing while the articles were being written. There's no solid overall number.

Two of the findings are genuinely serious. The worse one sits in Gitea, specifically in act_runner — the thing that runs your continuous integration jobs. The American vulnerability database NVD rates it nine point nine out of ten. Anyone allowed to start a workflow on a Docker-backed runner can ride it straight to root on the host, even with privileged mode turned off. And the uncomfortable part: there's still no fixed version. The only real countermeasure is organizational — you have to ask who's actually allowed to start workflows at your shop. For self-hosted runners on public repos, the answer is often: anyone who opens a pull request.

The second finding involves libssh2, a small library almost nobody installs on purpose and yet almost everybody has in the house. curl uses it, Git uses it, PHP uses it. Here's a detail that gets lost in a lot of the coverage: this is a client-side flaw. It's not your SSH server under attack, it's your Git, the moment it connects to a server an attacker controls. A CI job that clones from a compromised remote is enough. That's fixed in version one-point-twelve-point-zero.

One more thing about libssh2, because it explains the confusion you'll see in ticketing systems. NVD itself carries both severities: nine point two and eight point three. That's not a contradiction — nine point two is calculated under the newer CVSS version four standard, eight point three under version three point one. If you prioritize by threshold, you need to know which scale your scanner is speaking.

The second big story is a kernel bug called Januscape. The vulnerable code traces back to a commit from August first, two thousand ten. Sixteen years. It sits in KVM's memory management — the virtualization layer in the Linux kernel — in a part shared by Intel and AMD alike. A guest can break out of its virtual machine and land on the host. On distributions like Red Hat Enterprise Linux, reports say even an unprivileged local user is enough to get root that way.

The stable kernels were fixed on July fourth. Important detail: NVD lists no fix for the five point fifteen and five point ten series. If you still run those, ask your distributor rather than assuming you are patched. Alongside the actual escape fix, you need the companion carrying the identifier CVE twenty twenty-six, forty-six thousand one hundred thirteen. If you only backport one of them, you're not done. And there's no consolidated severity score here either: NVD hasn't rated it at all yet, Red Hat calls it a seven, secondary trackers say seven point eight.

There's good news out of the supply-chain world. npm version twelve turns off install scripts by default. A package can no longer just run arbitrary code on your machine the moment you install it. Dependencies pointing at Git or arbitrary remote addresses will also need explicit permission going forward. It's the late but correct answer to one supply-chain poisoning after another over eleven weeks. My take is blunt: this should have happened back in 2018. That it took state-backed attackers to flip a default setting is the real story here.

Quickly through the rest. Docker Compose five-point-three finally brings native init containers — steps that run cleanly to completion before the actual service starts. The European Commission unveiled an action plan on cybersecurity and artificial intelligence yesterday, without a single concrete deadline. And researchers have shown that confidential computing can prove which code is running, but not on which machine it's running. A fix within today's model may not even be possible.
