GitLost: a polite issue reads your private repo [1][2]

On July 6, Noma Security disclosed a critical prompt injection in GitHub's new Agentic Workflows — the layer where an AI agent (Copilot or a connected model) autonomously responds to issues, reads code, and writes comments.[1] The attack requires no login at all: an attacker opens a crafted issue in a public repo of the same organization. The agent reads it, follows the hidden instructions — and posts content from a private repo back as a public comment.[1]

Prompt injection: An AI agent doesn't reliably distinguish between "an instruction from my operator" and "text I happen to be reading." Like an overeager intern who follows every note left on his desk — including one buried between the lines of a customer email. The attacker simply writes the instruction into data the agent processes anyway.

The real lesson is in the bypass. GitHub had built in safeguards specifically meant to reject exfiltration prompts like this one. According to Noma, the word "Additionally" placed before the injected instruction was enough: instead of refusing, the model rephrased its output and quietly slipped past the guard.[1] The Register puts it dryly — the agent leaks private repos "when asked nicely."[2] Responsibly reported to GitHub; no fix or documentation update confirmed as of publication.

Why this runs here and not in the AI brief: the hook isn't the model, it's the tooling layer. An agent that executes issue text and holds cross-repo permissions is, from a security standpoint, an unauthenticated code path with read access to your crown jewels. My take: guardrails that fall to a single filler word aren't a security boundary, they're a politeness filter. The boundary that actually holds is the classic one: which repos can the agent even see? As long as the answer is "everything in the org," every public corner is an entry point.

81 million login attempts against Microsoft 365 [3][12]

Huntress analyzed a broad password-spray campaign: over 81 million login attempts between June 12 and 26, 2026, over the course of which 78 Microsoft accounts across 64 organizations were compromised.[3] The traffic came from an IPv6 range Huntress attributes to LSHIY LLC (AS32167); the abuse report sent to the operator initially went unanswered — but per Huntress' own update, LSHIY suspended the responsible user on July 2 for a terms-of-service violation, after which the spraying from that IP range stopped.[3][12]

Password spraying: Not trying a thousand passwords against one account (that trips lockouts fast), but one popular password against thousands of accounts. Like a burglar who doesn't drill at one door but tries "1234" once on every apartment's lock in the building — it fits somewhere, and no single door raises an alarm.

The single figure is less striking than the trend behind it: per Huntress, organizations now average 1,964 failed logins per tenant per month — a more than 155-fold increase in spray activity.[3] In practice that means anyone relying on "strong passwords" has already lost. What matters is MFA coverage without gaps, Conditional Access rules that block anomalous geo/ASN patterns, and shutting down legacy auth endpoints that bypass MFA entirely. Not a "patch tonight" item — more a "review your login telemetry this week" one.

PostgreSQL JDBC 42.7.12: a silent downgrade in the login [4]

The PostgreSQL JDBC team released security update 42.7.12 on July 6, closing CVE-2026-54291 — a silent downgrade of channel-binding authentication. Affected are driver versions 42.7.4 through 42.7.11, and only when channelBinding=require is explicitly set.[4]

SCRAM channel binding: At login, the client doesn't just prove it knows the password (SCRAM) — it also ties the login to the exact TLS connection it's currently on. Like a wax seal that glues letter and envelope together — slip the letter into a different envelope along the way, and the seal breaks. That's exactly what prevents a man-in-the-middle. The sealed variant is called SCRAM-SHA-256-PLUS; the unsealed one is just SCRAM-SHA-256.

The bug allows exactly that seal to disappear: an attacker in a TLS-MITM position presents a certificate whose signature algorithm has no known tls-server-end-point binding hash — for instance Ed25519, Ed448, or a post-quantum scheme. The driver then silently falls back from SCRAM-SHA-256-PLUS to the unbound SCRAM-SHA-256, even though the user explicitly demanded the opposite via require.[4] The fix is 42.7.12.

For Daniel's stack — Spring/JPA on PostgreSQL — this is the most relevant item of the week. Anyone who set channelBinding=require in the JDBC URL (the whole point of which is MITM protection) has been relying on a promise the driver quietly dropped for modern EdDSA/PQC certificates. Exploitability is narrow (the attacker needs a MITM position), but the trust breach is fundamental: a require that can be downgraded to optional isn't one.

Node.js fights "AI slop" — with AI [5][13]

The Node.js Foundation is drowning in fabricated vulnerability reports — but two facts often lumped together need to be kept separate. First: the project paused its HackerOne/IBB bug bounty program back in April 2026 — per its own announcement, solely because external funding through the Internet Bug Bounty program ran out; AI isn't mentioned there at all.[13] Second, and independent of that: AI-generated fake reports keep straining the remaining triage of the Security Working Group; at its meeting on July 7, the WG weighed, of all things, AI pre-sorting of incoming reports as a countermeasure.[5]

AI slop: A flood of plausible-sounding but worthless AI output. For security reports: one that looks like a real vulnerability — CVE-style wording, code snippets, a severity rating — but isn't. Like a stack of convincing counterfeit bills: every single one has to be checked by hand, and that's exactly the burden.

Initiator Rafael Gonzaga sums up the motive: submitters care "more about the fame of appearing as an author on an official CVE advisory than about the money" — and: "We are still overwhelmed."[5] On the scale of the problem, industry figures are circulating that I'm citing as context, not as evidence: HackerOne cites 60–80% invalid submissions platform-wide, Bugcrowd cites over 500 additional reports per week in 2025 — both secondary aggregation, not pulled from primary reports.

The punchline is bitter: an OSS project is fighting AI's side effect with more AI, because human triage doesn't scale. That's the real software-engineering second-order effect of the agent era — not new features, but a new class of maintenance burden landing on volunteer shoulders.

PostgreSQL ecosystem: a dense release week [6]

A lot shipped simultaneously around the database core this week (all July 5–8). The most relevant item for container setups: CloudNativePG 1.30.0 introduces a new DatabaseRole feature — roles and permissions declared as a Kubernetes resource instead of via init SQL.[6]

Alongside it: Barman 3.19.0/3.19.1 (backup & recovery), the PostgreSQL ODBC driver REL-18_00_0002 — now on OpenSSL 3.5.5 —, pg_ivm 1.15 (incrementally updated materialized views), Autobase 2.9.0, plus two smaller tools: pglayers (50+ extensions as prebuilt Docker layers) and the PostgreSQL Data Unloader (PDU).[6] No single blockbuster, but the signal is healthy: the Postgres ecosystem is maintaining itself, not just counting the core. Most worth a look for Daniel: DatabaseRole in CNPG, if roles have so far been managed via migration scripts.
