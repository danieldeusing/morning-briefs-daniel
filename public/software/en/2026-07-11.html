TypeScript 7.0: the compiler is now native — and ten times faster

Today's story is a release many have been waiting more than a year for: TypeScript 7.0 shipped final on July 8 — a "10x faster native port of TypeScript," as Microsoft calls it. The point isn't a new type syntax, but the machine underneath: the compiler and the language service were rewritten from TypeScript to Go. That changes the performance class, not just the decimal point.

Native compiler port: Until now, the TypeScript compiler itself was written in TypeScript and ran as JavaScript on Node — a language that's interpreted at runtime and, at its core, uses a single thread. Go, by contrast, compiles directly to machine code and can put many cores to work in parallel. Picture it this way: you've translated the same cookbook from a language the cook stumbles through into their native tongue — and put several cooks in the kitchen on top of that. Faster on both counts.

The numbers are correspondingly clear: Microsoft cites "speedups typically … between 8x and 12x on full builds." The most vivid example is the editor: in the VS Code repository, "open file with an error to first error shown" used to take around 17.5 seconds — with TypeScript 7 it's "under 1.3 seconds," more than 13x faster. For Daniel's Angular- and FastAPI-adjacent TS projects, that's not a nice-to-have; it's noticeably less waiting on every save and every CI run.

What that means in practice: the language itself stays compatible — it's the same type checker, just fast. For a migration, it's worth checking the migration notes (a few exotic API usages and Node-internal hooks can behave differently now that the core is Go instead of JS), but the everyday setup — tsc, editor integration, tsconfig — keeps working unchanged. Realistically: keep the 6.x line running in parallel for a while, throw 7.0 at one project's own CI, and measure the time saved.

Source: devblogs.microsoft.com

Gitea Docker trusts any IP by default (CVSS 9.8)

The toughest CVE of the week hits self-hosters: CVE-2026-20896, CVSS 9.8 (critical), published July 3. Affected are the Gitea Docker images up to and including version 1.26.2. Per NVD, these images set REVERSE_PROXY_TRUSTED_PROXIES=* as the default — and that asterisk is exactly the problem.

Reverse-proxy auth header: When Gitea runs behind a reverse proxy that handles login, the proxy tells Gitea via an HTTP header (e.g., X-WEBAUTH-USER): "This request comes from user X." Gitea is only supposed to trust that header if it really comes from its own proxy — hence the list of trusted proxy IPs. If that list is just *, Gitea trusts the header from any source IP. Like a bouncer who lets in anyone who claims "my colleague already checked me" — no matter whether that colleague even exists.

So if header authentication is enabled, anyone who can reach the Gitea instance directly can impersonate any user — including admin — just by setting X-WEBAUTH-USER. Security press (including The Hacker News) reports early scans for the flaw; that exploitation signal couldn't be cross-checked against a primary telemetry source and should be read as a secondary indicator, not a confirmed campaign. To do: update the image and restrict REVERSE_PROXY_TRUSTED_PROXIES explicitly to the real proxy IP (or internal subnet) — not to *. Anyone not using header auth at all isn't affected, but should still set the value cleanly.

Source: nvd.nist.gov

Chrome: critical Views use-after-free — now with details

Chrome's second July update, which was still light on details yesterday, is now clear. The Stable update from July 7 (Chrome Releases) bumps desktop Chrome to version 150.0.7871.114/.115 and, per Forbes, closes CVE-2026-15129, a use-after-free in the Views component — Chrome's UI rendering layer — rated critical. Via crafted web content, that opens "opportunities for remote code execution." No exploit in the wild is known so far.

This is the concrete spelling-out of what was noted here yesterday only as "another Stable update, details pending" — and not to be confused with the 382-fix package from June 30 (that was version .46/.47). Handling as always: auto-update is enough, but check rather than assume it went through — force a Chrome restart if the version isn't on .114/.115 yet.

Sources: chromereleases.googleblog.com · forbes.com

CISA KEV: two fresh additions, deadline Monday

Fresh as of yesterday: on July 10, CISA added two more actively exploited flaws to the KEV catalog — CVE-2026-56291 in Balbooa Forms and CVE-2026-48939 in iCagenda, both Joomla extensions with the same pattern: unrestricted upload of dangerous file types. The patch deadline for US federal agencies is July 13, i.e. Monday. What's newsworthy here is less the individual plugin flaw than the listing itself: a KEV entry means "this is actually being exploited right now" — for everyone else, a clear "fix this one first." Anyone running Joomla with one of these components should get the update done over the weekend.

For context: the previous quartet from July 7 (led by the ColdFusion path traversal, plus SharePoint on-prem RCE, Langflow, and JoomShaper) had its deadline yesterday — that ran in yesterday's brief. Today it's just about the two new names.

Source: cisa.gov (KEV catalog)

EU: cybersecurity & AI action plan — and the AI Act clock

The standing EU regulation thread gets a new chapter: on July 7, the European Commission presented the Action Plan for Cybersecurity and Artificial Intelligence (COM(2026) 577) — nine measures addressing two directions at once: making AI usable as a tool for defense, and securing AI as an attack surface. Concretely planned, among other things, is a joint test platform run by ENISA and the Joint Research Centre (JRC) with "cyber ranges," where operators of critical infrastructure can test AI systems before putting them into production — targeted for Q4 2026.

Cyber range: A sealed-off training ground inside a data center — a realistic replica of real systems where you can play out attacks and defenses without breaking anything real. Like a flight simulator, just for IT security.

In parallel, the AI Act clock keeps running: starting August 2, 2026, the Commission can enforce against providers of AI models with systemic risk — with fines of up to 3% of global annual revenue or €15 million. For Daniel and twiceD, this isn't a direct call to action (it hits model providers), but it sets the frame within which AI features will be procured and documented going forward. The date is deliberately not in the 7-day timeline — it's three weeks out — but it belongs on the roadmap.

Sources: digital-strategy.ec.europa.eu · Factsheet

BSI A5: an audit catalog for trustworthy AI — as OSCAL

Fitting the EU line, Germany delivers a concrete tool: on July 6, the BSI published the A5 "Community Draft" — a modular audit catalog for structurally assessing the trustworthiness of AI systems. What's interesting for developers is the format: the catalog appears not only as a PDF, but also in OSCAL.

OSCAL: A machine-readable format for security and compliance catalogs, driven by the US NIST. Instead of looking up requirements in a PDF, a tool can read them directly and check them against your system — compliance as a file rather than a document.

The comment period runs until August 31, 2026 (via form to aisecurity@bsi.bund.de). Software relevance over AI hype: the interesting part isn't the "AI" label, but that a government agency is shipping an audit catalog in machine-readable form — exactly the kind of building block from which automated compliance checks can later be built. (Date via trade press/BSI notice; high confidence, but not quoted verbatim from a direct fetch.)

Source: heise.de

npm v12 — the reminder

For completeness, since it's landing this month: per the GitHub changelog, npm v12 is "estimated … July 2026" and flips three defaults to safe — no install/preinstall/postinstall scripts from deps, no git deps, and no remote-URL deps without explicit approval. This was covered in depth in yesterday's brief; today just the reminder to prepare an allow list, before upgrading, for the few packages that genuinely need scripts.

Tailscale: nested group sync

From the Tailscale changelog: group sync with Microsoft Entra ID and Google Workspace now also accounts for members of nested groups (July 8), and owners can switch their tailnet's identity provider directly from the admin console (July 6). Not a security fix, but a real convenience for anyone with an IdP-bound ACL model.

Caddy CVE-2026-45692 (older, but in the stack)

For Daniel's reverse proxy, an addendum: CVE-2026-45692 (published June 23 — not, as an AI summary initially claimed, July 7; the NVD primary source states June 23). From 2.4.0 through 2.11.2, the authorization layer and the /config path traversal layer disagree on which object a path points to. Fixed in 2.11.3. The severity is disputed: NIST scores it 3.8 (low), GitHub's CNA scores it 5.4 (medium) — still worth checking your version for an exposed admin API endpoint.

Sources: github.blog · tailscale.com · nvd.nist.gov
