jscrambler npm package hijacked: five malicious releases in three hours

The freshest hit landed yesterday, Saturday July 11: the npm package jscrambler was compromised. According to security press (The Hacker News, building on Socket), the version released on Jul 11, 8.14.0, carried a preinstall hook that drops and executes a native binary — one build each for Windows, macOS, and Linux. Over roughly three hours, the attacker pushed five malicious releases (8.14.0/8.16.0/8.17.0/8.18.0/8.20.0). The payload: a Rust-written infostealer that grabs Chrome/Brave/Edge/Chromium profiles, the Bitwarden browser extension, and Steam sessions.

Infostealer: A program whose only job is to quietly copy stored passwords, browser cookies, crypto wallets, and session tokens and send them to the attacker. Like a burglar who doesn't break anything — he just photographs every key and password lying around and leaves.

The instructive detail: At 8.18.0, the attacker moved the dropper out of the preinstall hook and directly into the shipped files dist/index.js and dist/bin/jscrambler.js — specifically to fool scanners that only look at install scripts. This means the malware no longer runs at install time, but at import or execution time. This exact trick will matter shortly when we assess npm v12. (The release list and the dropper move come from security-press summaries — the Socket and THN detail pages blocked direct access, so this should be read as secondhand, not primary telemetry.)

Source: thehackernews.com

Injective SDK backdoored: wallet keys from 18 packages

A few days earlier, on Wednesday July 8, the same attack class with a harder target. @injectivelabs/sdk-ts — the TypeScript SDK for the Injective blockchain — had a backdoor slipped in via a compromised GitHub developer account. Version 1.20.21 quietly captured wallet recovery phrases (seed phrases) and private keys. Scope per BleepingComputer and StepSecurity: 18 affected packages, around 50,000 downloads per week, live for under an hour, no damage done. Fixed in 1.20.23 — anyone pulling the Injective SDK should update.

Seed phrase: The 12–24 words that serve as the master key to a crypto wallet — whoever has them owns the wallet, no password needed. Like a master key to a safe that happens to fit on a postcard: once stolen, everything inside is exposed forever.

The contrast with jscrambler is the point: two different entry points, the same outcome. jscrambler came via a hijacked publish account with install-script and code payloads; Injective via a hijacked GitHub account that committed the backdoor directly into the SDK source code. Both bypass the simple maxim "just don't run install scripts" — the malicious code sits in the code itself, which is meant to run anyway.

Sources: thehackernews.com · bleepingcomputer.com

What npm v12 stops — and what it doesn't

npm v12 flips three defaults to secure this month — no preinstall/install/postinstall scripts from dependencies, no git deps, and no remote-URL deps without explicit opt-in (GitHub changelog). The mechanics ran in detail in the Jul 10 brief; today's fresh, honest question is: would this have stopped this weekend's two cases?

Install hook → npm v12 blocks · Code in dist/ → keeps running

Partially. jscrambler's first releases used a preinstall hook — npm v12 would have blocked that by default. But at 8.18.0 the dropper moved into dist/index.js, i.e. code that runs when the package is imported — npm v12 does nothing there. And the Injective backdoor sat in the SDK source code from the start, executed at runtime as intended — likewise untouched by the install-script lockdown.

Bottom line, with an opinion: npm v12 raises the floor and kills the cheapest attack, the install-script drive-by. But the payload simply relocates to import- and runtime code. Necessary, not sufficient. The real lines of defense stay the same: pin versions, lockfile plus integrity hashes, check provenance/attestation — and don't blindly bump just because a new version is out.

PostgreSQL JDBC 42.7.12: silent channel-binding downgrade (CVE-2026-54291)

Right in Daniel's stack (Java/Spring on Postgres): the security release 42.7.12 of the PostgreSQL JDBC driver (released June 29, reported in postgresql.org news on Jul 6) closes CVE-2026-54291. The core issue: connections with the setting channelBinding=require could be silently downgraded from SCRAM-SHA-256-PLUS (with channel binding) to plain SCRAM-SHA-256 (without) — undermining exactly the protection that had been explicitly requested. Affected versions are 42.7.4 through 42.7.11; severity is rated medium.

Channel binding: When your app logs into Postgres over TLS, channel binding ties the password handshake to that exact encrypted connection — an attacker sitting in the middle who relays your login to a different server can't reuse it. Like a signature valid only on this one sealed envelope, not on a copy. A "silent downgrade" means the client believes it got the protection — but never did.

Anyone who set channelBinding=require against MITM attacks should move to 42.7.12; otherwise the hardening may have been running for nothing. From the Postgres ecosystem this week also came CloudNativePG 1.30.0 (new DatabaseRole resource), pg_ivm 1.15, and Barman 3.19.0/3.19.1 — not a security fix, but worth a look for anyone running Postgres in Kubernetes.

Sources: jdbc.postgresql.org · postgresql.org

GitHub CodeQL 2.26.0: Kotlin 2.4.0 — and a query against prompt injection

CodeQL is GitHub's static code-analysis engine — the machine behind Code Scanning. Version 2.26.0 (Jul 10) now supports Kotlin up to 2.4.0 and brings — the interesting part for a security tool — a new query, js/system-prompt-injection. It detects when untrusted user input flows into an AI model's system prompt, and adds new prompt-injection sinks for the OpenAI, Anthropic, and Google GenAI SDKs. This clearly belongs in the software brief: it's a scanning feature, not a model launch.

Prompt-injection sink: If your app builds user input into the instructions it gives an AI model, a malicious user can smuggle in commands that hijack the model ("ignore the above, do X"). A CodeQL "sink" is a known-dangerous endpoint the scanner watches; the new query tracks whether untrusted data reaches the system-prompt sink — the same taint-tracking idea used for SQL injection, applied to LLM calls.

The takeaway: the more applications wire LLMs into their backend, the more prompt injection becomes an AppSec class like XSS or SQLi — and now mainstream SAST tooling is starting to model it. For anyone already using code scanning, this arrives for free with the CodeQL update.

Source: github.blog

Microsoft Foundry: hypervisor-isolated sandboxes for agents (GA)

As an infra note, cleanly scoped: Microsoft Foundry's "hosted agents" reached general availability in July (byteiota.com). Every session gets its own hypervisor-isolated sandbox — dedicated compute, memory, and file system — and it's framework-agnostic: LangGraph, Claude Agent SDK, OpenAI Agents SDK, or a custom Dockerfile all run. Pricing: $0.0994 per vCPU-hour plus $0.0118 per GiB-hour, with scale-to-zero when idle.

The software-relevant angle here is deliberately not "which agent framework" (that's ai-dev/ai-usecases territory — the details live there), but the isolation model: anyone running untrusted, LLM-generated code needs a hypervisor boundary per session — the same problem gVisor or Firecracker solve for multi-tenant compute. The fact that the big clouds now ship this as a managed default is the real infra story.

Source: byteiota.com

Langflow: higher CVSS ≠ most exploited

Follow-up to the Jul 7 KEV listing (which ran in the Jul 10 brief): the fresh nuance on CVE-2026-55255 in Langflow is the rating itself. NVD rates the IDOR flaw at CVSS 8.4 (high), the vendor at 9.9 (critical) — a methodology divergence. And despite the lower NVD figure, per Sysdig it's been actively exploited since Jun 25. Lesson: don't triage purely by NVD base score — KEV membership beats CVSS score.
