Ubiquiti: 25 UniFi flaws in one shot — one at CVSS 10.0

Today's top story comes from a corner of the network that sits in countless homes and offices: Ubiquiti disclosed 25 vulnerabilities across the entire UniFi ecosystem in Security Advisory Bulletin 066 — per the security press, seven critical and 18 high. The vendor page (community.ui.com) renders client-side only and wasn't readable via direct fetch; the figures below are cross-checked against NVD (as the primary record) plus securityonline.info and BleepingComputer, all of which agree on the 25-CVE / CVSS-10 headline.

The top offender is CVE-2026-50746, rated by NVD at CVSS v3.1 10.0 (critical) — the unbeatable maximum. It's an unauthenticated command injection in the UniFi Connect application (affecting all versions up to but not including 3.4.20). NVD's wording: "A malicious actor with access to the network could exploit an Improper Access Control vulnerability found in UniFi Connect Application to execute a Command Injection on the host device." The CVE record was published in NVD on Jul 2; press coverage started on Jul 7.

Command injection: An app takes an input and passes it on to the operating system's command line without properly cleaning it up. An attacker then smuggles in their own command instead of "just data" — and the device executes it with its own privileges. Like a waiter who carries a slip with your order to the kitchen without checking whether it also says "…and open the safe" at the bottom. "Unauthenticated" means: you don't even need a password for this, just network access.

Four more flaws sit right below it at CVSS 9.9: CVE-2026-54402 (UniFi OS), CVE-2026-55115 (UniFi Protect), CVE-2026-50747 (SQL injection in UniFi Talk), and CVE-2026-50748 (command injection in UniFi Access). That's the real weight of this bulletin — not a single hit, but a cluster of five at the top of the scale spread across multiple products.

Who's affected and what to do: Per BleepingComputer, Censys counts more than 100,000 UniFi OS instances reachable from the public internet, about 50,000 of them in the US — so the attack surface is real and large. The good news: fixes are ready across the entire line — UniFi Connect from 3.4.20, Talk from 5.2.2, Access from 4.2.29, Network from 10.4.57, Protect from 7.1.83, UniFi OS from 5.1.19, Protect Floodlight from 1.13.6. So far there's no public proof-of-concept and no confirmed active exploitation — but with a 10.0 and five-digit exposure, waiting is the risk, not the patch. Anyone leaving UniFi OS management reachable from the network should additionally put it behind a VPN or firewall.

Sources: nvd.nist.gov · securityonline.info · bleepingcomputer.com

ModSecurity 3.0.16: WAF bypass via parser differential (CVE-2026-52747)

An instructive bug in a protection tool itself: ModSecurity, the widely used open-source web application firewall, closed two gaps with 3.0.16. The more important one is CVE-2026-52747 — rated at CVSS 8.6 by the security press (cybersecuritynews.com, published Jul 10). The engine silently strips embedded line breaks (\r\n or \n) from non-file form fields before they land in the ARGS and ARGS_POST variables. Result: the firewall inspects a different string than the application behind it later processes.

Parser differential: Two programs read the same raw data differently. The firewall sanitizes the input and sees something harmless; the app behind it reads the original and sees the attack. Like a doorman who reads a note through frosted glass and deems it harmless — inside, the same note is read sharply. It's exactly in this gap between two readings that the attacker slips past the rule.

The second, more minor flaw (CVE-2026-52761) is a 32-bit truncation bug from a misapplied sizeof() on a pointer — on 32-bit systems only 4 bytes are processed instead of the full length. Both are fixed in 3.0.16. Context: WAFs are a layer, not a replacement for clean input validation in the app itself — this bug is the reminder why. (The figures come from the trade press; the details should be read against the vendor fix in 3.0.16, not primary telemetry.)

Source: cybersecuritynews.com

OpenSSH 10.4: two silent path traps removed, post-quantum edges in

Straight into Daniel's day-to-day infrastructure (SSH is everywhere): OpenSSH 10.4 was released on July 6. Two of the fixes address file-transfer traps that are easy to overlook. First, sftp: downloading via sftp host:/path . could let a malicious server cause the file to land in an unexpected location instead of the target directory. Second, scp: when copying between two remote endpoints (remote-to-remote), a malicious server could write files into the parent directory of the actual target — a directory escape.

On top of that, a subtle bug in sshd/internal-sftp: long command lines were silently truncated after the 9th argument — if a security-relevant option sat at position 10 or later, it simply got dropped. Sounds cosmetic, but it's exactly the kind of detail that quietly undermines an intended restriction.

Post-quantum signature (composite): Today's key schemes like Ed25519 could eventually be broken by future quantum computers. A "post-quantum" signature uses math that would hold up even then. "Composite" means: the new scheme (ML-DSA 44) is combined with the proven old one (Ed25519) — both would have to break for the signature to fail. Like two locks of different types on one door: as long as only one is picked, the door stays shut.

10.4 builds in exactly this combined PQC signature experimentally — keys are generated with ssh-keygen -t mldsa44-ed25519. Important: it's not the default, explicitly flagged as experimental, meant for trying out, not for production. But it's the tangible start of quantum migration in a tool that practically every server operator touches daily.

Source: openssh.org

CISA KEV: SharePoint RCE actively exploited, plus a run of CMS plugins

SharePoint CVE-2026-45659 — actively exploited

The heaviest new entry in the CISA KEV catalog is a SharePoint flaw: CVE-2026-45659, a deserialization vulnerability in SharePoint Server, added on Jul 1 per the CISA alert due to confirmed active exploitation. Affected are Subscription Edition, Server 2019, and Enterprise Server 2016 (per The Register). Direct fetching of the cisa.gov alert page was blocked (403); the date and CVE ID are cross-checked via the same site's search index plus the trade press — solid, but should be read as a secondary layer.

Deserialization: A server receives data in packed form and "reconstructs" it into objects in memory. If it trusts that data blindly, an attacker can send a crafted package that executes code as it's unpacked. Like a furniture package with assembly instructions: if the instructions say "drill a hole in the wall here" and the person assembling it follows blindly, the damage is done.

CMS plugin upload flaws on a near-weekly cadence

Around it, CISA added a run of smaller but actively exploited CMS plugin flaws: on Jul 7, CVE-2026-48908 (JoomShaper SP Page Builder), CVE-2026-55255 (Langflow), and CVE-2026-56290 (Joomlack Page Builder); on Jul 10, CVE-2026-48939 (iCagenda) and CVE-2026-56291 (Balbooa Forms). The pattern throughout is insecure file uploads in Joomla/web builders. For the KEV selection, the simple rule applies here: if a CVE is in the catalog, it's really being exploited — regardless of how mild the base CVSS score sounds.

Sources: thehackernews.com · theregister.com

Cloud security beat: Microsoft's SFI numbers and Azure Defender GA

Microsoft SFI — a progress report with actual numbers

Microsoft's July progress report on the Secure Future Initiative (Jul 10) is interesting because it quantifies its own attack-surface reduction instead of just asserting it: 99.97% of internal user/device pairs use phishing-resistant MFA, more than 732,000 resources disconnected from public access, 1.4M unused apps decommissioned, cross-boundary credential isolation at 98.7%, and 83% of build pipelines blocked by default from reaching unapproved package endpoints. Microsoft targets quantum-safe migration of critical products "by 2029".

Phishing-resistant MFA: Two-factor login that can't be phished — instead of a code you could type into a fake site, a hardware key (FIDO2/passkey) cryptographically verifies that the real domain is asking. Like a key that only fits the original lock, no matter how convincing the fake looks.

Azure Defender for Cloud — three GA milestones

From the July release notes: since Jul 1, new container security features, Kubernetes misconfiguration enforcement in Defender for Containers, and discovery/posture for serverless container workloads are all generally available. On Jul 5 came a deprecation: the plan enablement API now blocks onboarding to five retired Defender plans. Together with the SFI report, that's this week's "cloud provider security" thread.

And Google Cloud/Apigee? Quiet this week

In the interest of honesty: there's no new CVE on the Apigee side this week. The latest Bulletin GCP-2026-041 (Jun 24) is an all-clear — Google clarifies that Apigee Edge for Private Cloud is not vulnerable to two log4j CVEs (despite scanner false positives). No reason to spin a story out of that.

Sources: microsoft.com · learn.microsoft.com

For anyone selling software with "digital elements" in the EU, that's the next hard deadline — still eight weeks out, but not one that handles itself.

JVM & browser horizon
