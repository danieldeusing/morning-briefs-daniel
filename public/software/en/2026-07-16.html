CISA KEV: six actively exploited flaws in 48 hours

Between July 14 and 15, CISA added six new vulnerabilities to the KEV catalog. Not a single theoretical find among them — all with evidence they're currently under attack.

KEV (Known Exploited Vulnerabilities): CISA's list of flaws confirmed to be exploited in the wild. US federal agencies must patch them by a deadline; for everyone else, a KEV entry is the strongest "this isn't hypothetical anymore" signal there is — priority, not a panic button.

Oracle E-Business Suite — CVE-2026-46817 (CVSS 9.8) is the heaviest item: an unauthenticated HTTP takeover in the File Transmission component of Oracle Payments, affecting versions 12.2.3 through 12.2.15. CISA added it to the list on July 15; per BleepingComputer, the first attacks were already observed on June 27. Oracle had shipped the fix back in the May CPU — anyone who hasn't applied it has been exposed for weeks.

SonicWall SMA1000 comes as a pair: CVE-2026-15409, an unauthenticated SSRF, plus CVE-2026-15410, an OS command injection for the authenticated admin. Remediation deadline for US agencies: July 17.

SSRF (Server-Side Request Forgery): You get the server to make network requests on your behalf — to addresses you could never reach yourself, like internal admin endpoints or cloud metadata. Like asking the building superintendent to unlock a basement door for you that you'd never be able to reach on your own.

Two came out of Redmond: Microsoft SharePoint Server — CVE-2026-56164, missing authentication for a critical function (network-based privilege escalation), deadline also July 17. Interesting: the July Patch Tuesday base score sat around a middling 5.3 — on the KEV, "actively exploited" outweighs the paper score. Plus Microsoft AD FS — CVE-2026-56155, local privilege escalation via overly coarse access control, deadline July 28.

For Daniel's stack None of this runs directly in your environment — no Oracle EBS, no SonicWall, no AD FS. The point is the prioritization: when a flaw lands in the KEV, someone has already weaponized it. That catalog is exactly the filter you use to decide which CVE out of the daily flood needs attention right now.

Progress ShareFile: shut down the server first, ask questions later

On July 13, Progress sent its on-prem ShareFile customers an unusually blunt instruction: "IMMEDIATE ACTION REQUIRED: You must manually shut down the server hosting your Storage Zone Controllers." The reason given was a "credible external security threat" — initially without details. The situation has since been resolved: on July 14, Progress confirmed a highly critical path-traversal flaw in the Storage Zones Controller (versions 5.x and 6.x), shipped the fixes 5.12.5 and 6.0.2 that same day, and lifted the shutdown instruction — access is back. Progress is deliberately withholding the CVE number for about two weeks so customers can patch before the details go public. Progress stresses there's "no indication of unauthorized access to ShareFile customer accounts or data."

Honest read: the emergency shutdown with no CVE wasn't a false alarm after all — a real, now-confirmed flaw was behind it. Progress pulling the plug first and following up with the fix and CVE number on a delay is the price of giving customers time to patch before disclosure. Given Progress' track record (MOVEit, 2023), the fast, hard-line move was warranted. Anyone using ShareFile purely as cloud SaaS was, per Progress, never affected; this was specifically about the self-hosted Storage Zone Controllers, which now belong on 5.12.5 or 6.0.2.

LegacyHive: a Windows zero-day with no CVE, Microsoft stays quiet

On July 14, an actor going by "NightmareEclipse" — per The Register, a repeat offender when it comes to leaking Microsoft zero-days — published a PoC for "LegacyHive": a local privilege escalation in the Windows User Profile Service (profsvc) that works via loading registry hives. The exploit specifically targets the usrclass.dat hive and requires additional credentials — so it's not remote-access-out-of-nowhere, but a building block for the second stage after an initial foothold.

What stands out is less the severity (LPE, with preconditions) than the pattern of the disclosure: a named, known leaker, no CVE number, no confirmed out-of-band fix — exactly the signature that often precedes a Patch Tuesday fire drill. Worth flagging, not worth losing sleep over tonight.

Tailscale 1.98.9: six bulletins — and one of them is in your stack

Tailscale cleared out six security bulletins at once with 1.98.9 (TS-2026-004 through TS-2026-009). For Daniel this isn't a news curiosity, it's maintenance work: Tailscale runs in his own setup.

The sharpest is TS-2026-009: Tailscale SSH accepted usernames with a leading -, which could then be passed through as command-line flags — letting an attacker slip past the ACLs into a root session. A classic argument-injection bug, but in a spot where it really hurts.

TS-2026-008: a single malformed HTTP request to Serve/Funnel could permanently pin a CPU core — DoS from a single request. TS-2026-007: Tailscale services accepted incoming traffic on ports they never advertised, exposing listeners bound to localhost. All fixed in 1.98.9.

Heads up If you use Tailscale SSH or Serve/Funnel, 1.98.9 is the version you want to be on. Not the end of the world — but TS-2026-009 is exactly the kind of hole you don't want sitting unpatched in a VPN mesh.

Spring: six new HIGH CVEs — right in the middle of the Java stack

Outside the strict 7-day window (published Jun 9–12), but still unresolved and squarely in Daniel's territory: the Spring Security advisory index currently lists six new HIGH entries across three projects. In Spring Security that's CVE-2026-40988, CVE-2026-40993, and CVE-2026-41003 (HIGH) plus CVE-2026-41694 (LOW).

The most eye-catching one comes out of the AI building block: Spring AI — CVE-2026-47835, a SQL injection via the vector store interfaces. Exactly the kind of software second-order effect this brief cares about — not the model, but a classic injection flaw in the framework wrapped around it. Plus Spring Cloud Gateway — CVE-2026-47825 (HIGH), which per the index affects seven versions.

A note on the numbers: the spring.io/security index publishes severity and the count of affected versions, but not individual CVSS scores — so deliberately just "HIGH" here, not a decimal point. Check your own Spring BOM against the index before the next dependency bump.

Stack maintenance: Django 6.0.7/5.2.16 and a Postgres JDBC downgrade

Django shipped the security releases 6.0.7 and 5.2.16 on Jul 7 — three low-severity CVEs, all routine: CVE-2026-48588 (a cached Set-Cookie could leak to the wrong user), CVE-2026-53877 (a ~32-byte heap over-read in GDALRaster), and CVE-2026-53878 (a newline in DomainNameValidator enabled header injection). Nothing dramatic, but worth grabbing on the next deploy.

More relevant for Daniel (Postgres in the stack) is the PostgreSQL JDBC driver 42.7.12 from Jul 6, against CVE-2026-54291: connections with channelBinding=require could be silently downgraded from SCRAM-SHA-256-PLUS (with channel binding) to plain SCRAM-SHA-256.

Channel binding: a cryptographic seal that ties your login to this exact TLS connection — so an attacker in the middle can't replay it over a different wire. A silent downgrade removes the seal without anyone noticing: you think you're protected, but you're not. That's exactly why "silent" is the dangerous word here.

Container corner: containerd, Docker Desktop, Kubernetes

Three maintenance items from the container world — all DoS-class, outside the 24h window, so bundled together without an alarm. containerd: CVE-2026-47262 (CVSS 6.5) — a maliciously crafted image can drive the containerd process into an out-of-memory kill via unbounded group parsing; fixed in 2.3.2 / 2.2.5 / 2.1.9 / 1.7.33.

Docker Desktop: CVE-2026-8936 (unbounded recursion in the grpcfuse module → VM panic, fixed in 4.76.0) and CVE-2026-31431 (a copy.fail write to the host page cache could lead to root inside the VM).

Kubernetes: a recent 1.33.11 patch release addresses CVE-2026-35469 (memory amplification in SPDY frame parsing) and CVE-2026-33814 (an infinite loop via SETTINGS_MAX_FRAME_SIZE in HTTP/2). The exact release date couldn't be pinned down cleanly — sources place it in "June/July 2026" without a specific day; read it as the "most recent patch release," not a fixed date.

EU Parliament: chat control is back, via a procedural trick

Standing thread, EU regulation: on July 9, the EU Parliament reinstated "voluntary" chat control — a "temporary" exemption that lets services like WhatsApp, Microsoft, or Google voluntarily scan private chats for CSAM material, set to run until April 2028. The trick is in the mechanics: 314 MEPs voted against it, only 276 for — but a formal rejection would have needed 361 votes (an absolute majority). Those never materialized, so the motion counts as adopted.

For developers in the EU this stays relevant because it marks the direction the debate around client-side scanning and end-to-end encryption keeps heading — even though this particular decision only covers the voluntary variant, not the disputed mandatory scanning.
