GhostLock: the container is no longer a wall

The most interesting find of the week isn't an appliance hole — it's a bug that has been sitting in the Linux kernel since 2011, introduced with Linux 2.6.39 and undetected for roughly fifteen years. CVE-2026-43499, dubbed "GhostLock," is a use-after-free in the rtmutex priority-inheritance logic: the cleanup function remove_waiter() tidies up the currently running task instead of the one actually waiting, leaving a dangling pointer into freed kernel stack memory.

Use-after-free: A program frees a chunk of memory but keeps the address around — and later writes to it again. Like checking out of a hotel room, keeping the key, and coming back at night: someone else is living there by then. If an attacker can control who moves in, the kernel suddenly writes data the attacker prepared.
Priority inheritance (futex PI): If a low-priority task holds a lock a high-priority task is waiting on, the kernel briefly lends the low-priority one the higher priority — so it finishes faster and releases the lock. Like letting the intern jump the checkout line because the boss is standing right behind him. GhostLock tangles up exactly this handoff.

It's triggered via three futexes and several threads deliberately driven into a priority deadlock; the attacker then re-occupies the freed stack frame with a forged waiter structure and works their way up to arbitrary kernel read/write and, ultimately, root. The flaw only requires CONFIG_FUTEX_PI — which is enabled in practically every distro kernel. It was found by Nebula Security with their analysis tool VEGA; the exploit is, per their own testing, 97% reliable and earned $92,337 via Google's kernelCTF program.

The part that actually matters to Daniel: GhostLock breaks out of containers. The CVSS score of 7.8 ("high," not "critical") only looks that moderate because the attacker must already be logged into the machine. In a container world, though, that condition is cheap: every process inside a container is a local user on the host kernel.

Heads up Containers share the host's kernel — isolation is a kernel feature, not a fence. A kernel LPE is therefore not a "merely local" bug in Docker/Kubernetes; it's the classic path from "I can run code in a pod" to "I own the node." If you run untrusted code in containers (CI runners, multi-tenant setups), treat this 7.8 more like a 9.

Three months of distro lag — and a fix that needed a fix

The kernel fix has been available for a long time — much longer than the headlines suggest. The finders' timeline: reported on 18.04.2026, fixed in mainline on 20.04.2026 as commit 3bfdc63936dd, backports into the stable branches starting 04.05, made public only on 07.07. AlmaLinux had patched kernels in its testing repo by 09.07 — specifically kernel-4.18.0-553.141.2.el8_10 (AlmaLinux 8), kernel-5.14.0-687.24.1.el9_8 (9), and kernel-6.12.0-211.32.1.el10_2 (10) or later. In other words: two days after disclosure.

And Ubuntu? A look at Canonical's own CVE page today, 17.07 — ten days after disclosure and almost three months after the mainline fix — shows 24.04 LTS (Noble) listed for most kernel packages as "Vulnerable, work in progress," while 22.04 LTS (Jammy) and 20.04 LTS (Focal) are simply marked "Vulnerable." Priority: "High." What's fixed so far is essentially just 26.04 (Resolute).

The road there wasn't a straight line: the original fix itself needed a correction — it introduced a crash bug tracked as CVE-2026-53166, reverted upstream in June. But that doesn't explain Ubuntu's current lag: it was resolved five weeks before disclosure. Why the LTS kernels are still open regardless, Canonical doesn't say — the page just says "work in progress."

For Daniel's stack Concretely: apt list --upgradable | grep linux-image won't help you right now — for Jammy/Focal there's simply nothing to pull yet. What you can do: watch Ubuntu's CVE page and reboot after the kernel update lands (a kernel fix only takes effect afterward; Livepatch only if you're running it). Until then, the old rule applies doubly: no untrusted code in containers on a host you care about.

AsyncAPI: why --ignore-scripts wouldn't have helped this time

Update since Wed The AsyncAPI breach was already a thread here on 15.07. Now Microsoft Threat Intelligence has followed up with an analysis — and it's more interesting than the incident itself, because it dismantles a convenient assumption.

Affected are five poisoned versions across four packages, published on 14.07: @asyncapi/specs (6.11.2-alpha.1 and 6.11.2), @asyncapi/generator (3.3.1), @asyncapi/generator-components (0.7.1), and @asyncapi/generator-helpers (1.1.1). The key line from Microsoft's report: "When any consuming build or application imports a poisoned package, the injected block runs immediately."

That's the point. The standard defense against poisoned npm packages — and the feature npm v12 is currently selling as a big win — is disabling install scripts. Here, though, the malicious code doesn't fire on install; it fires on import/require. That means npm install --ignore-scripts runs into a dead end: the package sits quietly on disk, waiting for your build to touch it.

Install hook vs. import-time: An install hook is a script that runs the moment a package is unpacked — you can hold that door shut. Import-time code, by contrast, lives inside the package itself and runs the moment your program uses it. Like a package that doesn't go off when unpacked but the first time it's used: the security check at the front door sees nothing.

According to Microsoft, it fetches a roughly 8.2 MB encrypted bundle (sync.js) over IPFS, which rolls out a C2 runtime (framework "M-RED-TEAM," campaign ID "miasma-train-p1"). IPFS as the delivery channel is the clever part: no domain name to block, no server to seize. The takeaway, unchanged since Wednesday: lockfiles and pinned versions protect you here more than any script block ever could.

Gitea thread: 13 days from advisory to first attack

Follow-up to a thread from 08. and 11.07 — not a new incident, but a number worth remembering. The Gitea Docker flaw CVE-2026-20896 (CVSS 9.8) arises because the official image hardcodes REVERSE_PROXY_TRUSTED_PROXIES = *: Gitea then trusts the X-WEBAUTH-USER header from any source IP — letting anyone impersonate any user. Fixed in 1.26.3.

Sysdig observed the first real-world exploitation attempt 13 days after disclosure (observation around 07.07, reported mid-July — so not a same-day figure). Roughly 6,200 Gitea instances are publicly reachable by count; how many of those are vulnerable, the number explicitly doesn't say.

Two weeks sounds relaxed — it isn't. It's the realistic upper bound of your patch window for a critical, well-documented flaw in a self-hosted web app. If you run Gitea in a homelab or for clients: check your image version, and if you have a reverse proxy in front, pin the trusted-proxy list to a real IP instead of *.

Google in the EU: what's final — and what isn't

Standing thread on EU regulation, today mostly a clarification. Headlines are circulating claiming the EU just imposed a record DMA fine on Google. As of today that's not accurate — and the risk of mixing things up is high, because there really are two proceedings.

Final: the ECJ definitively upheld the Android antitrust fine on 02.07 — €4.125 billion, originally €4.3 billion from 2018, slightly reduced by the EU General Court in 2022. That closes out an eight-year proceeding. It's done and was covered here on 05.07.

Not decided: the separate, newer DMA case over self-preferencing in Google Search and the anti-steering rules in the Play Store. The Commission's decision is expected before Parliament's summer recess — there's no fixed date, so it hasn't landed yet. The often-cited 27.07 date has nothing to do with the fine: it's the statutory deadline for two separate specification decisions (DMA.100220 on Android AI access, DMA.100209 on sharing search data).

Source honesty A headline from 16.07 claims the DMA fine has already landed; the article itself wasn't accessible (403), and cross-checking still says "decision pending." Until a Commission primary source says otherwise, the standing line here is: expected, not imposed. The Android ruling is a separate matter — don't conflate the two figures.
