wp2shell: a default installation is enough

WordPress shipped a security release round on 17.07 — 6.9.5, 7.0.2, 6.8.6, and 7.1-beta2 — closing two core flaws. The more severe of the two, CVE-2026-63030, has picked up the name "wp2shell" along the way: a pre-auth RCE, meaning code execution on the server without any login.

The chain has two links. First, a route confusion in the REST API's batch endpoint: WordPress can bundle multiple API calls into a single HTTP request, and when resolving which internal route a bundled call actually refers to, the system can be fooled. Second, a SQL injection in WP_Query that becomes reachable behind this incorrectly resolved route. Together, according to Rapid7, this amounts to "complete compromise of the website and its underlying data."

Route confusion: A web app decides, based on the path, which internal function handles a request — and which permission check sits in front of it. Like a doorman who decides who to let in by reading the sign on the door. But in a batch request, the sign sits inside the package instead of on the door: the doorman checks the harmless label, and the forbidden door gets opened anyway.

What makes this dangerous isn't sophistication — it's the total lack of prerequisites: no account, no plugin, no special configuration. A freshly installed WordPress instance with default settings is reachable via a single anonymous HTTP request.

Affected: 6.9.0 through 6.9.4 and 7.0.0 through 7.0.1. Everything below 6.9 is not affected by this CVE — which is no comfort, see below: the second flaw from the same Friday round reaches back to 6.8. WordPress has pushed forced auto-updates for the affected branches; that's well-intentioned, but auto-updates regularly fail silently due to file permissions or disabled cron. Check which version is actually running instead of assuming it.

Two scores for the same flaw — the CNA says 9.8, the ADP says 7.5

Scroll down on the NVD entry for CVE-2026-63030 and you find two CVSS ratings sitting side by side. The CNA — the official assigner, here WPScan — gives it 9.8 CRITICAL with the classic vector AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H: unauthenticated over the network, all three impacts (confidentiality, integrity, availability) set to "high." The CISA ADP record gives it 7.5 HIGH with a reduced vector: C:H/I:N/A:N — the attacker can read data out, but the ADP rating doesn't factor write access or availability damage into the math. Rapid7 quotes both figures in one sentence: "While the official GitHub security advisory classifies the severity as Critical, the vulnerability has currently been assigned a CVSS score of 7.5."

This isn't a "the number will still get adjusted" story — it's a genuine disagreement, and a practically relevant one: anyone who ties prioritization to a score threshold ("immediate above 9.0, next window below that") and sees the 7.5 first will delay. Rule of thumb: when there are multiple ratings, the higher one governs. And more important than the number anyway are the conditions — no login, no plugins, internet-reachable.

The second CVE reaches further back

Alongside wp2shell, Friday's round also carries CVE-2026-60137, a SQL injection reported by a team of TF1T, dtro, and haongo (GHSA-fpp7-x2x2-2mjf). It's getting less attention but affects more installations: it reaches back into the 6.8.x line and is fixed there in 6.8.6.

That's the awkward combination for anyone running "we deliberately stay one minor version behind": you're off the hook on the spectacular RCE, but exposed on the quieter SQLi — and because it doesn't make headlines, nobody checks. Anyone running WordPress should simply go through all four branches today: 6.8.6, 6.9.5, 7.0.2, 7.1-beta2. Per the release post, anything below 6.8 isn't affected.

Seven hours and forty-one minutes

The real lesson of this weekend is a matter of timestamps. Cloudflare rolled out its two new WAF rules — one each for the SQLi and the RCE — by its own account "at 17:03 UTC on July 17 2026," with the default action Block and explicitly "to protect all customers, including those on free and paid plans." Rapid7 wrote the same day that technical exploit details hadn't been published yet, but that a public PoC was "highly likely […] in a short period of time."

The first public PoC repository for the chain was created, per the GitHub API, at 00:44:18 UTC on 18.07.2026 — seven hours and forty-one minutes after Cloudflare's rules, and still on the day of the patch. It describes itself as a detection PoC; securityonline reports the tool confirms the injection and can read the database, including admin password hashes.

Heads up Confirmed in-the-wild exploitation doesn't exist so far — not according to Rapid7 ("not aware of publicly confirmed in-the-wild exploitation," 17.07), nor securityonline ("No in-the-wild exploitation has been confirmed.", 18.07). That's a snapshot, not an all-clear: for a pre-auth flaw in the world's most widely used CMS codebase, mass scanning is the norm within days, not weeks.

The consequence for your own workflow is uncomfortable but simple. The old model — a patch appears, you schedule a maintenance window for the coming week — carries the silent assumption that days separate a fix from a weapon. But because the patch itself is the instructions (anyone can read the diff), that window has shrunk to hours. For anything reachable from the internet without a login, the rule is: on the day of the patch, or with a filter in front of it beforehand.

SharePoint 9.8: the deadline expires tomorrow

CVE-2026-58644 is a deserialization flaw in Microsoft SharePoint with CVSS 9.8 (vector AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H). The NVD description is terse: "Deserialization of untrusted data in Microsoft Office SharePoint allows an unauthorized attacker to execute code over a network." The flaw is being actively exploited and was added to the CISA KEV catalog on 16.07; the deadline for US federal agencies ends on 19.07 — tomorrow.

Deserialization: A program takes in a data stream and rebuilds it into live objects in memory. Like a furniture kit with instructions. The problem: if the attacker writes the instructions, the program doesn't build a chair — it builds whatever the instructions say — and some objects execute code as they're assembled.

Patched versions: SharePoint Server 2016 from 16.0.5556.1005 and 2019 from 16.0.10417.20153 (both taken from NVD's CNA/CPE fields and consistent). For the Subscription Edition, the number is ambiguous: NVD's CNA/CPE field lists 16.0.19725.20384, while NVD's own "Initial Analysis" block lists …20434 instead — both on the same page. What's reliable here is Microsoft's KB5002655, which you should check the installed version against rather than trusting either NVD figure. This isn't a direct exposure for Daniel's own stack — it's included here because SharePoint servers are extremely common in German SME networks, and because a KEV deadline on a Sunday is a good reason to check in with clients.

7-Zip: the fix is three weeks old, the advisory three days

CVE-2026-14266 is a heap buffer overflow that can occur when processing data in xz format — CVSS 7.0, tracked as ZDI-26-444 (ZDI-CAN-30169), reported by Landon Peng of Lunbun LLC. Affected are all versions before 26.02; it's fixed in 7-Zip 26.02 — released already on 25.06.2026.

Exploitation requires user interaction: the victim has to open a crafted archive. In-the-wild exploitation isn't claimed in the write-up; this was coordinated disclosure via the Zero Day Initiative. That's the relaxed version — with one footnote: 7-Zip doesn't update itself. The fix has been ready for three weeks, and that's exactly why it isn't running on most machines yet. 7z i shows the installed version.

npm, two breaches in one week

Update The AsyncAPI incident ran here in detail yesterday. What got a bit lost in that: it wasn't the only one. On 11.07, several versions of the jscrambler package and its build plugins were compromised via stolen publishing credentials — three days before AsyncAPI.

Reported as poisoned: jscrambler 8.14.0, 8.16.0, 8.17.0, 8.18.0, and 8.20.0, along with jscrambler-webpack-plugin 8.6.2, gulp-jscrambler 8.6.2, grunt-jscrambler 8.5.2, and jscrambler-metro-plugin 9.0.2; clean versions are reportedly 8.22.0 and 8.6.3 / 8.6.3 / 8.5.3 / 9.0.3 respectively. The malicious code loaded platform-specific infostealers written in Rust from a roughly 7.8 MB, obfuscated container — targeting cloud credentials, CI tokens, browser sessions, crypto wallets, and the configuration files of AI coding tools.

What's interesting is the delivery. It changed mid-campaign: according to Socket, quoted by The Hacker News, 8.14.0, 8.16.0, and 8.17.0 fire from a preinstall hook — npm install --ignore-scripts helps against those. In 8.18.0 and 8.20.0, the dropper moved into the package's main code and the CLI, "so it fires when the package is imported or run, and npm install --ignore-scripts does not stop it." So the attacker tried both routes within a single week.

That makes the comparison to AsyncAPI messier, but more instructive: the script lock covers one path and not the other — and because the same attacker can run both, it's no substitute for the more mundane defenses. What would have caught both is unspectacular: pinned versions plus a few days' delay before a fresh version is allowed into the build.

Stack maintenance, briefly

Angular 22.0.7

Released 16.07; alongside compiler fixes, it includes a safeguard against ReDoS when processing source-map URLs. ReDoS means: a maliciously crafted string makes a regular expression compute for exponentially long — the process hangs, without anyone having actually broken in. The parallel branch v22.1.0-next.6 landed the same day.

PostgreSQL 19 Beta 2

Also 16.07, a cleanup beta: a regression in vacuumdb --analyze-in-stages on partitioned tables is fixed, plus corrections to temporal tables (FOR PORTION OF), SQL/PGQ property graphs, and a race condition when enabling logical decoding. GA remains targeted for "around September/October 2026."
