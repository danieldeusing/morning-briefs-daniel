Three npm Breaches, Three Different Doors

Sunday, no fresh advisory in the sources we monitor — so today's a tally of a week that had plenty going on. In eleven days, three npm packages were hijacked: @injectivelabs/sdk-ts (08.07), jscrambler (11.07), @asyncapi/* (14.07). Individually, these already ran here. Together they form a pattern that strikes me as more important than any single case: each one used a different execution path. And of all days, on 08.07 — the day of the first breach — npm v12 turned off allowScripts by default, a defense that would have blocked exactly one of the three routes. Not an argument against the change; one against considering it solved. Also in this issue: why the "record Patch Tuesday" is roughly three-quarters Chromium, a SharePoint score where Microsoft says 5.3 and the NVD says 9.8, and the NIS2 registration deadline in twelve days.

Three Breaches, Three Different Doors

Three hijacked npm packages in eleven days is unpleasant but not unusual — in 2026 this happens on a weekly cadence. What's interesting is putting them side by side: none of the three attackers used the same execution path. Anyone relying on a single defense measure would have missed twice this week.

Execution path: For a malicious package to cause harm, its code has to run at some point. There are several moments for that — during installation, during import into your own code, or when a CLI command starts. Like a burglar choosing between the front door, a basement window, and the garage door: an extra lock on the front door only helps if that's the door he picks.

08.07 — Injective SDK: Keys in the HTTP Header

Through a compromised GitHub account, @injectivelabs/sdk-ts@1.20.21 was published (tens of thousands of downloads a week — ~50,000 per The Hacker News, >30,000 per Datadog). The detail that matters for defense: the code ran neither as an install script nor at import time. Datadog documents two hooks placed directly in the key-derivation functions PrivateKey.fromMnemonic() and PrivateKey.fromHex() — they fire at runtime, only once a wallet is loaded. The harvested mnemonics and private keys then went out base64-encoded in the X-Request-Id header of outgoing requests — a field almost nobody looks at, since it normally just carries a correlation ID. Injective found the commit and shipped 1.20.23 after about 49 minutes.[1][2][24]

11.07 — jscrambler: the Attacker Switches Doors Mid-Attack

The malicious jscrambler@8.14.0 went live at 16:12:40 BST; Socket's research team spotted it six minutes later, with around 1,500 downloads. Five versions were affected (8.14.0, 8.16.0, 8.17.0, 8.18.0, 8.20.0); 8.22.0 is clean. The notable part is in the middle: the early versions ran via a preinstall hook on dist/setup.js. The later ones dropped the install script and injected a self-executing function into the package's main code — one that fires on import or CLI invocation and slips past any scanner that only checks install scripts.[3][4]

14.07 — AsyncAPI: Import-Time, Precisely Timed

Five compromised versions across four packages (@asyncapi/specs 6.11.2-alpha.1 and 6.11.2, generator@3.3.1, generator-components@0.7.1, generator-helpers@1.1.1). The way in was not a trusted maintainer but an external attacker using a CI pwn request: a workflow (manual-netlify-preview.yml) triggered on pull_request_target but checked out the pull request's own code — foreign code running in the base repository's context, with access to its secrets. Via PR #2155 (05:08 UTC, 14.07), a token belonging to the asyncapi-bot service account was stolen.[25][26] Microsoft's reconstruction reads like a release plan: first re-publish around 07:10 UTC, the stable version with a byte-identical payload at 08:30:09 UTC, first confirmed download at 08:49:22 UTC. Here too, the code runs on import; npm install --ignore-scripts was useless.[5]

npm v12 Closed the Right Door — There Are Just Several

On 08.07 — the same day the Injective backdoor went live — npm v12 was released. The key change: allowScripts is now off by default. npm install no longer runs preinstall, install, and postinstall from dependencies unless explicitly allowed — including implicit node-gyp rebuilds. --allow-git and --allow-remote are now also set to none.[6]

The right call, arriving late. But the tally for the week is uncomfortable: the new default would have stopped part of a single incident — the early jscrambler versions. Injective and AsyncAPI bypassed install scripts entirely, and the jscrambler attacker switched to that path himself mid-attack. The safeguard wasn't defeated; the attackers moved elsewhere.

Install scripts blocked → Payload moves into the main code → Execution on import

The takeaway is unspectacular but concrete: --ignore-scripts is basic hygiene, not a defense. What actually helps is the time axis — lockfiles with exact versions and a grace period before adopting a fresh release. Six minutes of detection time only benefits someone who doesn't install in the first six minutes.

For Daniel's Stack The Angular/TypeScript branch is the exposed part: an npm ci in a GitHub Action pulls the same dependency tree as locally, just unattended and often with registry tokens sitting in the environment. The jscrambler malware targeted AWS/GCP/Azure credentials, crypto wallets, and configurations for AI coding assistants[7] — exactly what's lying around on a dev machine. Maven and pip have the same problem structurally.

SharePoint: Microsoft Says 5.3, the NVD Says 9.8

Update as of 16.07 Thursday's issue said the on-paper score for CVE-2026-56164 sat "more toward the middle" at 5.3. That was Microsoft's number. The NVD has since published its own assessment — and lands at 9.8, Critical. That's not a rounding difference; those are two different worlds.[8]

Both fully agree on the attack vector: AV:N/AC:L/PR:N/UI:N — over the network, low complexity, no privileges, no user interaction needed. The disagreement is entirely about impact. Microsoft sets C:N/I:L/A:N: no confidentiality breach, low integrity impact. The NVD sets C:H/I:H/A:H — everything high.

CVSS: A scoring system from 0 to 10 that rates a vulnerability by attack path and potential damage. Like a restaurant guide: the stars exist, but who awards them matters too. A vendor often rates the flaw within its own component, while an outside analyst rates what an attacker can ultimately achieve with it.

In practice, the dispute doesn't matter, and that's the point. The flaw has been in the CISA KEV catalog since 14.07 — the same day the patch shipped — and US agencies had to remediate by 17.07. Affected: SharePoint Server 2019 (< 16.0.10417.20175), Subscription Edition (< 16.0.19725.20434), and Enterprise Server 2016 (< 16.0.5561.1001).[9] The Zero Day Initiative puts it more dryly than I could: "patch it now and worry about the score later".[10]

My read: a KEV entry is an empirical finding, a CVSS score is a model assumption. Where the two diverge, the finding wins. The score is good for sorting, not for deciding.

The "Record Patch Tuesday" Is Three-Quarters Chromium

July's Patch Tuesday made the rounds as the biggest ever — and the headlines cheerfully contradict each other: 622 at The Hacker News, 621 at Security Affairs, 570 at BleepingComputer, 569 in Tenable's own count. Anyone citing a single one of these numbers as fact is citing a counting rule.[11][12]

The Zero Day Initiative untangles it: of the 621 CVEs, around 480 are Chromium and Chromium-based Edge — bugs assigned by a different CNA that Microsoft merely carries along. That leaves roughly 141 genuine Microsoft flaws. By severity: 63 Critical, six Moderate, one Low, the rest Important.[10]

CNA: The body allowed to assign a CVE number — usually the vendor itself. Like a residents' registration office: if someone moves from one town to another, they might show up in both towns' statistics. When Microsoft pulls a Chromium flaw into its own Edge, it shows up in Microsoft's update guide even though Google assigned it.

That doesn't defuse the number — 141 bugs with 63 Critical is plenty, and the ZDI rightly notes the year-to-date total already exceeds any full year of the last two decades. But anyone who has to justify a patch window gets further with "141 of our own, 63 of them critical, two actively exploited" than with a number the trade press disagrees on by fifty.

AD FS: the Second Zero-Day That Stayed Quieter

Alongside SharePoint, CVE-2026-56155 was on the 14.07 list — privilege escalation in Active Directory Federation Services, CVSS 7.8 (AV:L/AC:L/PR:L/UI:N), actively exploited and confirmed by Microsoft. The cause is an overly permissive ACL on the DKM keys; exploiting it lands you at administrator rights.[13] The local attack vector makes it less conspicuous than the SharePoint flaw — it's the second step of a chain, not the first. Doesn't run in Daniel's stack, but belongs in every Windows domain conversation this week.

Browser Week: Firefox with a PoC, Chrome with Ozone

Firefox 152.0.6 arrived on 14.07 with the advisory MFSA 2026-67 and closes two flaws rated Critical, CVE-2026-15718 and CVE-2026-15719. Publicly accessible exploit code exists for both; Mozilla says it is aware of no active exploitation.[14][15] Part of the trade press headlined this as a "zero-day" — it isn't, as long as nobody documents in-the-wild use. Published PoC code means "the clock is running," not "it's too late."

Google released a stable update for desktop on 16.07: 150.0.7871.124/.125 for Windows and macOS, 150.0.7871.124 for Linux, with 15 security fixes. Two of them are critical use-after-free bugs in Ozone, Chrome's graphics/windowing abstraction on Linux: CVE-2026-15764 and CVE-2026-15765.[16][17]

Use-after-free: The program frees a chunk of memory but then accesses it again anyway. Like turning in your hotel room key, having the room reassigned — and unlocking it again regardless. Whoever moves in fast enough decides what's there when the program looks back.

Both updates are just-restart-it affairs, no drama — and a good chunk of the 480 Chromium CVEs from the paragraph above comes from exactly this pipeline.

NIS2: Twelve Days, and the "Deadline" Isn't One

The hardest deadline for a German reader falls on 31 Jul 2026. Until then, the BSI is tolerating late registrations under the NIS2 implementation act — the regular deadline was 6 Mar 2026. The state of things is sobering: of around 29,500 expected entities, fewer than half had registered by the original deadline; by the end of May it was around 18,500.[18]

The part usually lost in the coverage: this grace period is not a statutory deadline extension — it's administrative forbearance. The obligation has existed since March; anyone who missed it is already in default and is counting on the BSI not to sanction them. Legally, that's a different thing from "I still have time until the end of July."

NIS2: An EU directive that obligates companies in critical sectors to meet minimum IT security standards and to report incidents — in force in Germany since December 2025. The first step is mundane: register with the BSI and say "we exist." Like registering your address at the residents' registration office — nothing happens as a result, but skip it and you'll have a problem later.

The fine range goes up to €500,000, and up to €10 million for certain violations, plus personal liability for management.[18] Also relevant for those not directly affected: registered entities push their security requirements down the supply chain — the next tender questionnaire will ask about it.

CRA: the Deadline After the Deadline

Deliberately outside the seven-day window, but the next block after NIS2: the Cyber Resilience Act's reporting obligations take effect on 11 Sep 2026 — a 24-hour early warning, 72-hour full report for actively exploited vulnerabilities and severe incidents. Full application on 11 Dec 2027.[19]

The Commission maintains an FAQ page on this that gets updated continuously — the most recent date shown on the page itself is 23 Apr 2026; a July date circulating in search snippets couldn't be confirmed on the document. Worth another look before September, since this exact FAQ defines who counts as a "manufacturer" — the decisive boundary question for a one-person software operation.

Stack Maintenance, Briefly

PostgreSQL 19 Beta 2 (16.07)

The cleanup beta, focused on the cycle's two big new features: several fixes for the SQL/PGQ property graph feature and the temporal syntax FOR PORTION OF. Also included: a regression in vacuumdb --analyze-in-stages on partitioned tables and an autovacuum bug where the multixact age score could go to infinity.[20] GA is targeted for Sept./Oct. — so: test it, don't migrate to it.

Angular 22.0.7 / 22.1.0-next.6 (16.07)

A maintenance release with a security component: ReDoS prevention in the regex check for sourcemap URLs in platform-browser. The prior version, 22.0.6 from 08.07, brought the "safe property lookups" hardening in the router.[21]

ReDoS: A regular expression that, given a cleverly crafted input, tries an exponential number of combinations and pegs a CPU core in the process. Like a bouncer who runs through every combination on the guest list for each visitor: a name with lots of hyphens brings the entrance to a standstill.

Tailscale 1.98.9 (14.07)

Six security items, TS-2026-004 through -009: Unix socket forwarding in Tailscale SSH now respects symlink permissions, the serve socket proxy is restricted to root, Serve and Funnel now catch a path walk that could peg a CPU with non-absolute paths, and SSH rejects purely numeric usernames as well as ones starting with a hyphen.[22] The most relevant of the three lines here for a tailnet with active Serve/Funnel.

It's Sunday, July nineteenth. There was no new advisory over the weekend in the sources we track, so no breaking news today — instead, a tally of a week in which three npm packages got hijacked.

On July eighth, it was the Injective SDK. A compromised GitHub account, a poisoned release, tens of thousands of downloads a week. The malicious code ran neither on install nor on import — it hooked the functions that derive the private key, so it only fired once someone loaded a wallet. Then it stole the keys and smuggled them out through an HTTP header called X-Request-Id — a field nobody normally looks at. The team caught the commit and shipped a clean version in just under fifty minutes.

On July eleventh, it was jscrambler's turn. Socket caught the malicious version six minutes after it went live. What's interesting is what the attacker did next: the first versions ran through an install script. Once that got noticed, the attacker dropped the script and hid the payload directly in the package's main code, where it fires on import instead. The defense wasn't broken — the attacker just moved to a different door.

On July fourteenth came AsyncAPI. The attacker came from outside: he sent a pull request that a badly configured GitHub Action then executed, secrets and all. Microsoft reconstructed the timeline: the stable poisoned version went live at eight-thirty UTC, and nineteen minutes later the first project downloaded it. Again, the code ran on import, not on install.

And on that same July eighth — the day of the first breach — npm version twelve shipped. Its headline change: install scripts from dependencies no longer run automatically. That's the right call. But of the three incidents, it would have stopped only part of one of them. The lesson for your own practice: blocking install scripts is basic hygiene, not a defense. What actually helps is time — pinned versions in your lockfile, and a few days' delay before adopting a fresh release.

Second story: a fight over a number. For the SharePoint flaw tracked as CVE 2026-56164, Microsoft rates the severity at five point three — middle of the road. The NVD, the U.S. vulnerability database, puts it at nine point eight — critical. Both agree on the attack path; they only disagree on the damage. The Zero Day Initiative summed it up best: patch first, argue about the score later. The flaw is being actively exploited, and that finding beats any model.

That fits with the third story. Microsoft's July Patch Tuesday was celebrated as a record, with counts ranging from five hundred sixty-nine to six hundred twenty-two, depending who you read. The Zero Day Initiative worked out that about four hundred eighty of those are Chromium and the Edge browser built on it. That leaves roughly a hundred forty genuine Microsoft flaws, sixty-three of them critical. Still a lot — but a different story than a triple record.

For German readers, the most important deadline: NIS2 registration with the BSI runs out on July thirty-first — twelve days from now. Of roughly twenty-nine thousand five hundred expected organizations, only about eighteen thousand five hundred had registered by the end of May. Worth knowing: this grace period isn't a legal extension, just the regulator looking the other way for now. The obligation has existed since March. Fines run up to five hundred thousand euros, and up to ten million for certain violations, plus personal liability for management.

Quick maintenance notes. PostgreSQL nineteen beta two cleans up the property-graph feature and the new temporal syntax; the final release is targeted for September or October. Angular twenty-two point zero point seven fixes a regex-exhaustion bug. Tailscale one point ninety-eight point nine closes six security issues, mostly around Serve and Funnel. And on browsers: Firefox one fifty-two point zero point six closes two critical flaws with public exploit code; Chrome one fifty ships fifteen fixes, including two critical ones in its Ozone graphics layer.
