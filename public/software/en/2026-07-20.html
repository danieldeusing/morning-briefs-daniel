SharePoint: the fourth flaw, and the worst one

CVE-2026-58644 hits on-prem SharePoint — Subscription Edition, 2019, and 2016 — and carries a CVSS v3.1 score of 9.8 (Critical). The root cause is deserialization of untrusted data; the result is arbitrary code execution by an attacker who doesn't even need to log in. That exact combination — no login, full execution, a server sitting on the internal network — is what turns a flaw like this into a gateway for everything that follows.

Deserialization: A program takes in a chunk of data and rebuilds it into live objects in memory. Picture an IKEA shelf that ships with assembly instructions: normally you end up with a shelf. But if the attacker gets to write the instructions, your server builds whatever he wants instead — say, "run this command." That's why "untrusted deserialization" has been one of the most reliable paths to remote code execution for years.

On 16 Jul 2026, the US agency CISA added the flaw to its KEV catalog. That's no longer a theoretical score — it's a finding: this is being used.

KEV catalog: "Known Exploited Vulnerabilities" — the list of flaws for which CISA has documented evidence of real-world attacks. It's the difference between "this lock could theoretically be picked" and "this street has already had break-ins using exactly that method." For US federal agencies, patching becomes mandatory once a flaw lands on it; for everyone else, it's the best free public prioritization signal there is.

In parallel, Germany's BSI published its own cybersecurity warning on 14 Jul 2026: "Microsoft SharePoint — zero-day and critical vulnerabilities closed," version 1.0. The warning covers the whole SharePoint Patch Tuesday bundle: the actively exploited flaw there was CVE-2026-56164; CVE-2026-58644 was added to the CISA KEV two days later.

Four KEV entries since April — that's a campaign, not a bug

The real point isn't this one flaw — it's the chain. Four SharePoint CVEs have landed in the KEV catalog since April:

  • CVE-2026-32201 — KEV addition 14 Apr 2026
  • CVE-2026-45659 — KEV addition 1 Jul 2026
  • CVE-2026-56164 — KEV addition 14 Jul 2026
  • CVE-2026-58644 — KEV addition 16 Jul 2026

Two of those landed within 48 hours of each other. When a product racks up freshly loaded zero-days at that pace, it says less about any single bug than about the state of the attack surface: someone is working through the same component with persistence and system. My read: by 2026, on-prem SharePoint has arrived in the same category as Exchange did in 2021 — a product whose on-premises operation has to be a deliberate, continuously defended decision, not a default assumption anymore.

Context No "patch tonight" panic, but the pragmatic consequence stands: anyone running on-prem SharePoint should check whether all four entries are covered. A patch level from last week doesn't cover the fourth flaw, because it was added two days later.

LegacyHive: a Windows zero-day, dropped on purpose

A researcher going by "NightmareEclipse" — according to The Register, likely a disgruntled former Microsoft employee — published a local privilege-escalation flaw in the Windows User Profile Service on 14/15.07. The trick: arbitrary loading of registry hives. A regular user can use it to mount another user's hive — including an administrator's — into their own Classes Root.

Privilege escalation (local): The attacker is already inside the house, but only as a guest. The flaw is the master key sitting in the cleaning closet. Worthless on its own — but combined with initial access (phishing, a hijacked dev package, a compromised build runner), it's the second step toward a full takeover.

The timing is the message: published just hours after the July Patch Tuesday, meaning maximum wait time until the next regular fix. The public PoC is deliberately trimmed down (it needs extra credentials), which keeps things at "serious, but not immediately exploitable at scale" for now.

Chrome: second critical round in 48 hours

Google bumped the stable channel to 150.0.7871.128/.129; its own release post is dated Thursday, 16.07.2026. Content-wise, it's use-after-free bugs again — the same error class that already dominated the round two days earlier.

Use-after-free: The program frees up a chunk of memory but keeps the address around and accesses it again later. Like the house key to an apartment you moved out of long ago — someone else lives there now. If an attacker can control who moves in after you, the browser ends up running their code.
Sourcing, unresolved I'm deliberately not listing CVE numbers here. Google's own release post, when fetched, only returned a title and a date — no body text. Two secondary sources cite different sets of CVEs for the very same version number — one lists three critical use-after-free bugs in the camera, GPU, and network components, the other a single high-severity bug in the Aura UI framework. Both can't be the complete list. The fix itself isn't in dispute: restart the browser so the update takes effect.

The month's pattern is more noteworthy than any single number: a bundle round with 382 resolved issues in early July, one with 27 vulnerabilities including two code-execution bugs around 15.07, and a single stable update carrying two build numbers on 16.07. Chrome is now patching at a cadence where "I restart my browser once a week" becomes genuinely dangerous.

Angular switches to an annual cadence

This week's most interesting framework signal isn't sitting in a blog post — it's in a pull request: Angular documents the switch from its previous ~6-month rhythm to an annual major-version cycle — explicitly in response to community feedback on how frequently breaking changes land. At the same time, 22.0.7 and 22.1.0-next.6 shipped on 16.07.

My take: this is the right correction, and it's arriving late. A twice-yearly major is simply faster than a framework living in enterprise-Java territory can get customers to plan migration budgets for. Two majors a year mean two upgrade projects a year, and at some point a team just stops keeping up — that's exactly how the Angular 8 legacy baggage everyone later marvels at gets created. An annual cadence turns the upgrade into a plannable line item instead of a constant state. The price: new features take longer to reach the majors, which makes the minor releases more important.

Caddy: two advisories right inside our own stack

On 10.07, the Caddy project published two security advisories, both rated Moderate. GHSA-6365-7ppr-5r92: when combining forward_auth + reverse_proxy, a connection can end up going to the wrong upstream. GHSA-j8px-rmrx-76h9: three vulnerabilities in v2.11.3.

The first one is the more unpleasant of the two. forward_auth is exactly the construct you use to put an auth service in front of multiple backends — if the connection lands on the wrong upstream, the worst-case logical consequence is that a request arrives in the context of a different service. "Moderate" is correct in CVSS terms and still underestimates how unpleasant that is in a multi-service configuration.

Relevance to our stack Caddy sits in Daniel's own toolchain. Two Moderates aren't cause for alarm, but they are the cheapest patch of the week: check the version, pull the current one, done.

npm 12: what the shutoff actually blocks

npm v12 is set to roll out later this month according to the announcement, and it turns off lifecycle install scripts as well as --allow-git and --allow-remote by default; anyone who needs them has to explicitly allow them via npm approve-scripts. The GitHub changelog note on this is dated 9 Jun 2026; the rollout date itself was announced on 08.07.

Lifecycle script (postinstall): An npm package is allowed to run its own command while it's being installed — meant for things like "quickly compile this native library here." In practice, it's a note tucked inside the package that says "please run this before you unpack me," and your machine just does it, no questions asked. That's exactly why it's been the most popular entry point for supply-chain attacks for years.

The change is the right call — nobody should mistake it for a solution, though. The AsyncAPI incident from 14.07 didn't run through postinstall — it fired at import time: the malicious code triggered the moment any build loaded the package via import/require. Neither the new default nor the classic --ignore-scripts helps against that path. The @injectivelabs/sdk-ts case from 08.07 — a package with roughly 175,000 installs a month — also used an execution path that script-blocking doesn't cover.

What's left is the uncomfortable answer: lockfiles, pinned versions, fast reaction times, and a deliberate eye on the number of direct dependencies. The new default takes one door out of play. The house has several.

NIS2: eleven days left

The registration requirement with the BSI ends on 31 Jul 2026 — pushed back from the original 6 Mar 2026. The state of play is still thin: by the end of May, around 18,500 of an estimated 29,500 obligated entities had registered. Missing the pure registration requirement alone puts up to €500,000 on the table; for the broader NIS2 obligations of "essential" entities, that rises to up to €10 million or 2% of worldwide annual revenue.

For a solo dev in Brazil, this isn't a direct obligation — but it is a supplier issue. German clients who themselves fall under NIS2 push their requirements down the chain: reporting channels, patch deadlines, evidence. Starting in August, that shows up in tenders and supplier questionnaires, not as statutory text.
