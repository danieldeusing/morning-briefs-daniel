Brussels dictates terms to Android: two binding DMA decisions against Google

DMA & gatekeepers: The Digital Markets Act is the EU law that forces the handful of platforms nobody online can avoid to open up to competitors. A "gatekeeper" is exactly that — Google is one, via Android and search. Imagine someone owns the only bridge into a city: the DMA says they also have to let competitors' delivery trucks cross on fair terms.

On July 16, the European Commission issued two binding DMA specification decisions against Google — they don't just state that Google has to open up, but spell out exactly how. The first decision: Android must let rival AI assistants into eleven functional areas, so a third-party assistant can take actions inside apps on the user's behalf — not just answer questions, but for example send a message or create a calendar event. The measures are meant to take effect starting with Android 18; the Commission cites August 1, 2027 as the deadline (simultaneous hotword detection doesn't arrive until Android 19). ec.europa.eu · winbuzzer.com

The second decision, arguably the heavier one for competition, concerns search: Google must share the same data it uses to optimize its own search with search and AI rivals, anonymized — via a "multi-layered anonymization method." The roadmap: the data set is to be finalized by November 2026, with a pricing offer for access due by January 2027. Anyone who misses these deadlines risks fines of up to 10% of Alphabet's global annual revenue, rising to up to 20% for repeat violations. winbuzzer.com

For developers, the data side is the real lever: what Google now has to hand over is exactly the signal that makes its search so hard to catch up to — which results people actually click for which queries. If search and AI providers get that aggregated, anonymized data set, the biggest barrier to entry in the search and assistant market drops noticeably. The interoperability side, meanwhile, turns what used to be an optional platform favor into a legal obligation: anyone building an assistant today could only hope for Android's Assistant hooks — starting with Android 18, they're enforceable. That's the real story for anyone working on agents and assistants, regardless of which model runs underneath.

Context The precedent is the interesting part, not the speed. For the first time, the EU is technically dictating to an operating system which AI-assistant hooks it must expose to third parties — for anyone building assistants or agents, Android 18 effectively becomes an open platform by law. In practice, nothing changes before 2027, and Google will likely implement the requirements as minimally as it can while fighting them in court. Source caveat: one EC secondary page states the rollout month as "starting July 2027" instead of August — we're going with the later, more specifically dated deadline.

Django security release 6.0.7 / 5.2.16

On July 7, Django 6.0.7 and 5.2.16 shipped with three low-severity fixes: a cache-poisoning flaw where a cached Set-Cookie response could pass a cookie along to other users (CVE-2026-48588), a heap over-read in GDALRaster (CVE-2026-53877), and a header-injection issue because DomainNameValidator allowed line breaks through (CVE-2026-53878). No drama, but worth picking up on the next dependency bump. (Versions and CVE IDs via djangoproject.com, cross-checked with a second source.) djangoproject.com

wp2shell: WordPress core RCE, actively exploited

Chained RCE (SQLi → RCE): Two smaller flaws are stitched together into one big one. First the attacker smuggles a statement into the database via SQL injection that has no business being there — like a forged note in an order system. The second flaw then turns that into "remote code execution": someone else's code runs on the server as if the operator had launched it themselves.

"wp2shell" is a chain of two flaws in WordPress core itself — not in a plugin: a SQL injection (CVE-2026-60137) followed by a remote code execution bug built on top of it (CVE-2026-63030). It's been actively exploited since roughly July 19; before that, VulnCheck had verified more than two dozen independent proof-of-concepts. The full chain is patched in 7.0.2 and 6.9.5; the 6.8.x line only gets the SQLi fix in 6.8.6, since the RCE flaw only entered core with WordPress 6.9. The fact that the chain lives in core rather than a swappable plugin makes it broadly relevant: every WordPress instance on the vulnerable versions is affected, regardless of which theme or plugin set runs on top. Anyone who can't update immediately should shield the affected endpoints and the admin area behind a WAF or access controls and check the instance for signs of compromise — with an actively exploited core RCE, that's no longer a theoretical precaution. tenable.com · thehackernews.com

Number caveat The CVSS scores diverge notably between sources. We're following Tenable's structured advisory table: RCE (CVE-2026-63030) = 9.8 (critical), SQLi (CVE-2026-60137) = 5.9. thehackernews cites different scores for the same CVEs — if you're passing a number along, pick one source and stick with it rather than mixing them.

Autodesk Fusion: July update adds on-demand configurations

Autodesk Fusion got its July major update (v.2704.1.15) on July 6. Highlights: "Create Configurations On-Demand" (generate part variants only when needed, instead of keeping them all around), "Find Similar Components," a hub-based thread library, Altium project import, the 3+2 clearing milling strategy, and — for anyone scripting Fusion — TypeScript support (beta) alongside the existing Python API. Two follow-up builds cleaned up stability issues: 2704.1.23 (Jul 9) and 2704.1.36 (Jul 19), the latter fixing, among other things, a crash when opening documents with version history (patch builds per Autodesk blog roundups, cross-checked with a second source). help.autodesk.com

nginx: heap overflow in the map regex engine — there since 2011

Heap buffer overflow: The program reserves a fixed-size drawer, then puts more into it than fits — the overflow spills into the neighboring drawer. Someone who feeds that overflow deliberately can overwrite other values and, in the worst case, crash the service or inject their own code.

CVE-2026-42533 is a heap buffer overflow in the regex processing of nginx's map module. The alarming part: the code has been in there since 2011 — practically every nginx version from 0.9.6 to 1.31.2 is affected. The NVD entry (published Jul 15) lists the score at 8.1 (CVSS v3.1); thehackernews additionally cites 9.2 under v4. Fixed in 1.30.4 (stable), 1.31.3 (mainline), and NGINX Plus 37.0.3.1. Small mercy: as of Jul 20 there was no public exploit code, and the flaw wasn't on CISA's KEV catalog — so patch it on the normal schedule, no fire drill. nvd.nist.gov · thehackernews.com

GhostLock: 15-year-old kernel bug, PoC now public

Use-after-free: The kernel frees up a piece of memory but still remembers the old address and later accesses it again — like a room key that still works after checkout. Anyone fast enough to put their own contents into that freed "room" can trick the kernel into treating foreign data as legitimate — and go from a normal user to root.

CVE-2026-43499 ("GhostLock," CVSS 7.8 per thehackernews — single-sourced, an NVD score wasn't available at time of research) is a use-after-free in the Linux kernel that has been sitting in the code for roughly 15 years. What's new isn't the flaw itself, but the fact that a reliable root and container-escape exploit is now public: "97% reliable," about five seconds to root; Google's kernelCTF bounty paid $92,337 for the find. The kernel fix (commit 3bfdc63936dd) landed back in April, but the distro rollout is lagging: as of early July, Ubuntu still listed several LTS versions as "vulnerable or in progress," and the fix itself introduced a follow-up crash bug (CVE-2026-53166). Anyone running their own Linux hosts or container fleets should check their distro's patch status now. thehackernews.com
