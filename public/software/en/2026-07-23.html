SharePoint on-prem, round two: the KEV deadline expires Saturday

Deserialization & Machine Keys: "Serializing" means packing an object from a running program into a data stream so it can be sent somewhere; "deserializing" is unpacking it back into an object. If the server doesn't check who packed the parcel when it unpacks it, an instruction can hide inside instead of data — like a package the recipient opens blindly even though it has a stranger's return address on it. The "Machine Keys" are the keys SharePoint uses to sign such parcels. Whoever has them once can keep packing valid parcels themselves — even after the patch.

CVE-2026-50522 is a deserialization vulnerability in SharePoint Server's SessionSecurityTokenHandler that allows unauthenticated remote code execution — CVSS 9.8. CISA added it to the Known Exploited Vulnerabilities catalog on July 22; the remediation deadline listed there is July 25, 2026. The timeline before that: a public proof-of-concept on July 20, active exploitation "within hours" according to consistent reports. Only on-prem installations are affected — SharePoint Enterprise Server 2016, 2019, and the Subscription Edition. SharePoint Online in Microsoft 365 is not on this list. cisa.gov (KEV) · thehackernews.com

KEV catalog: A list maintained by the US cybersecurity agency CISA of flaws confirmed to actually be under attack — not just theoretically dangerous. For US federal agencies, the date next to each entry is a binding deadline. For everyone else, the list is the best free prioritization filter there is: whatever's on it is genuinely being used right now.

Important context: this is not the spring SharePoint wave, nor the Patch Tuesday zero-day. CISA already reported active exploitation on July 14 of CVE-2026-32201 (spoofing, disclosed in March), CVE-2026-45659 (RCE, public in June), CVE-2026-56164 (the July 14 Patch Tuesday zero-day); CVE-2026-58644 was only added to KEV on July 16. CVE-2026-50522 piles on a week later. So anyone who patched back in spring isn't off the hook here. theregister.com · thehackernews.com

Order matters For this class of attack, the patch alone isn't enough. If the Machine Keys leaked before the patch was applied, access persists — the attacker keeps signing valid requests. So rotate the keys after deploying the patch, and only then consider the farm clean. On sourcing: the KEV data (CVE ID, addition and deadline dates) comes directly from CISA's own JSON feed. The exploitation timeline and Machine Key details are from trade press (The Hacker News, BleepingComputer) — CISA's own alert page returned a 403 on request.

Microsoft's record Patch Tuesday: 569 or 622 — depending on who you ask

On July 14, Microsoft delivered the largest Patch Tuesday in its history. How large, honestly, is up for debate. Tenable counts 569 CVEs — 56 critical, 510 important, three moderate — while Krebs on Security writes of "at least 570" with "nearly 60" critical. Malwarebytes and others, meanwhile, arrive at 622 with 59 critical. Both camps are right: the lower number is the CVEs first disclosed on that specific day; the higher one is the cumulative state of the Security Update Guide for the month of July, which also counts addenda and re-releases. Two counting methods, one event. tenable.com · krebsonsecurity.com · malwarebytes.com

Where all sources agree — and this is the part that matters: three zero-days, two of them already actively exploited at the time of disclosure. CVE-2026-56155 is a privilege escalation in Active Directory Federation Services, CVE-2026-56164 a remote privilege escalation in SharePoint Server. The third, CVE-2026-50661 (BitLocker), was publicly known but not exploited. That's the prioritization list; the grand total is headline material. tenable.com

Why we won't pick a single number Microsoft's own release-notes page in the Security Update Guide renders entirely client-side — a fetch returns only the page title and nothing else. So there's no machine-readable primary figure to check the aggregators against. We're therefore citing both counts by outlet name instead of stating one as fact. A fitting symptom on the side: the vendor with the largest Patch Tuesday in history doesn't publish its own scope as a plain, retrievable number.

Oracle: 1,449 patches in a single day, ten of them a perfect score

Seven days after Microsoft, Oracle rolled out its quarterly Critical Patch Update on July 21 — also the largest in its history: 1,449 new security patches across 32 product families. Fusion Middleware took the hardest hit, with 355 fixed vulnerabilities, 219 of them exploitable over the network with no login at all — and ten hit a clean CVSS 10.0. The E-Business Suite got 410 patches, 45 of them unauthenticated-remote. csoonline.com

CVSS 10.0 means in practice: no credentials, no user interaction, reachable over the network, full takeover. For Fusion Middleware that's especially nasty, because the stuff typically sits as an integration layer between systems — whoever has it often has everything behind it too. Oracle CPUs traditionally get deployed sluggishly because maintenance windows have to be negotiated with the business side; that's exactly why Oracle CVEs show up in attack campaigns months later. Anyone running a Fusion Middleware or EBS instance should set the window now, not in September.

On sourcing Oracle's own advisory page (oracle.com/security-alerts/cpujul2026.html) and its security blog both returned HTTP 403 on request. The topline figure of 1,449 patches across 32 families is therefore cross-checked against three independent secondary sources (CSOonline loaded directly, corroborated by Tenable and The Cyber Express) — they agree. The detailed Fusion Middleware and EBS numbers accordingly rest on secondary-source footing.

AsyncAPI: the token was never the problem — the pipeline was

pull_request_target: A GitHub Actions trigger that starts a workflow when a pull request comes in — but with the permissions and secrets of the target repository, not those of the submitter. Picture a stranger dropping a note in your mailbox, and your building super carrying out whatever it says, master key in hand. Useful when the workflow only reads metadata. Dangerous the moment it checks out and runs the submitted code.

On July 14, between 07:10 and 08:30 UTC, five package versions across four package names were republished with malicious code at AsyncAPI — @asyncapi/specs (6.11.2-alpha.1 and 6.11.2), @asyncapi/generator 3.3.1, @asyncapi/generator-components 0.7.1, and @asyncapi/generator-helpers 1.1.1. All within roughly 90 minutes. Microsoft's security blog documented the sequence, and the mechanism is the real point: no npm token was stolen. A malicious pull request exploited a misconfigured pull_request_target workflow to run its own code and, with it, grab the asyncapi-bot's personal access token. The legitimate, trusted publishing pipeline then signed and distributed the malicious packages itself. microsoft.com/security

That's the lesson for anyone running GitHub Actions — which is most of us: token hygiene, trusted publishing, and signatures protect against a stolen key. They don't protect against a hijacked CI, because the compromised pipeline signs everything perfectly correctly. The concrete check is specific: any workflow using pull_request_target that subsequently checks out the PR branch or runs its scripts is the bug. For outside contributions, use pull_request instead, restrict permissions to read, and put anything sensitive in a job gated behind an environment approval.

Single-sourced — unverified We deliberately did not treat two other reports from the same space as fact: an alleged compromise of Jscrambler npm packages via stolen publishing credentials (July 11) and a Windows zero-day dubbed "LegacyHive" with partially published exploit code (The Register, July 15). Both currently rest on a single source, without a confirmed CVE ID or independent corroboration, respectively. Once a second source turns up, they'll be back.

MDASH: Microsoft explains, in its own words, why Patch Tuesdays are exploding

The obvious suspicion behind two record Patch Tuesdays — something is suddenly finding a lot more flaws — is no longer speculation. On May 12, Microsoft introduced MDASH on its own security blog (Multi-Model Agentic Scanning Harness), a system for automatically discovering and fixing vulnerabilities that orchestrates more than a hundred specialized agents across an ensemble of large and distilled models. On the CyberGym benchmark (1,507 tasks) it scores 88.45% — the top of the published leaderboard, roughly five percentage points ahead of the runner-up. It was built by Microsoft's "Autonomous Code Security" team. microsoft.com/security

And Microsoft draws the connection to the patch volume itself: on July 9, the Windows Experience Blog published a post titled "Evolving Windows vulnerability management to meet the speed of AI-powered discovery" — five days before the record Patch Tuesday. In plain terms, the company is saying that its vulnerability management has to adapt to AI-driven discovery speed. blogs.windows.com

This shifts something fundamental for operators. If the discovery rate is structurally rising, "patch everything that's red" finally becomes unaffordable, and prioritization shifts away from the CVSS score toward exploitation evidence — that is, toward the KEV catalog and EPSS-style probabilities. A second, less comfortable consequence: the same class of tools is also available to the other side. The window between patch release and a working exploit is more likely to shrink than grow — CVE-2026-50522, with a PoC on Monday and exploitation the same day, is exactly this pattern.

Scope note MDASH is here because it's changing the patch cadence — not as AI-model news. This week's AI model and coding-tool releases (Gemini 3.6 Flash, GPT-5.6 Sol, Managed Agents, PyTorch 2.13, the Copilot/VS Code CVEs) are covered in the ai-dev brief.

And the deadline that's easy to push out of mind in July: on September 11, 2026, the reporting obligations of the EU Cyber Resilience Act take effect. From then on, manufacturers of software and connected products must report actively exploited vulnerabilities and severe security incidents — that's 50 days from today. The Commission followed up with an implementation FAQ on July 2. This doesn't just affect large corporations: a one-person company that puts a product with digital elements on the EU market counts as a manufacturer under the regulation. It's Thursday, July twenty-third. Today's software news revolves entirely around patching — two record-breaking patch days within seven days of each other, and a hard deadline expiring on Saturday.

Let's start with the deadline, since it's the most concrete piece. A flaw in SharePoint Server, affecting only self-hosted installations, is tracked as CVE twenty twenty-six, five-oh-five-two-two. Its severity score is nine point eight out of ten. The U.S. cyber agency CISA added it to its catalog of actively exploited flaws on July twenty-second and set July twenty-fifth as the deadline. A public proof-of-concept had surfaced on July twentieth, and according to consistent reports, attacks began within hours.

The important caveat here: the patch alone isn't enough. Attackers are grabbing what are called Machine Keys — the keys the server uses to sign its own data packages. Whoever gets hold of them can keep issuing valid requests even after the patch is applied. So the order matters: patch first, then rotate the keys. And to be clear, this is not the spring SharePoint wave, and it's not the Patch Tuesday zero-day either — it's a third, separate issue.

The second big story is Microsoft's Patch Tuesday from July fourteenth, the largest in company history. Exactly how large is something the trade press can't agree on. Tenable counts five hundred sixty-nine flaws, fifty-six of them rated critical, and Krebs on Security writes of at least five hundred seventy. Malwarebytes, on the other hand, arrives at six hundred twenty-two. Both figures are defensible: the lower one only counts flaws first disclosed that specific day, while the higher one is the cumulative monthly total from Microsoft's own update catalog. That's why we're citing both numbers by source name — Microsoft's own overview page can't be read by machine, so there's no primary figure to check against. What everyone agrees on: three zero-days, two of them already actively exploited at disclosure.

Seven days later, on July twenty-first, Oracle followed with a record of its own: one thousand four hundred forty-nine new security patches across thirty-two product families. Fusion Middleware took the hardest hit, with three hundred fifty-five vulnerabilities fixed, ten of which hit the maximum score of ten point zero. In practice that means: no login required, no user interaction, full takeover over the network. Oracle's own advisory page was blocked when we tried to load it, so the figures are cross-checked against three independent trade sources.

But the most instructive story of the week is a supply-chain attack, because its mechanism is such a clear lesson. At the open-source project AsyncAPI, five package versions were published with malicious code within ninety minutes on July fourteenth. No publishing token was stolen in the process. Instead, the build pipeline itself was hijacked — through a misconfigured GitHub Actions trigger that runs code from a submitted pull request with the permissions of the target project. That let attackers grab the project bot's access token, and the trusted pipeline then signed and distributed the malicious packages entirely on its own. Microsoft documented the whole sequence on its own security blog.

The lesson applies to almost every project out there: signatures and trusted publishing protect against a stolen key, but not against a hijacked build environment. The concrete thing to check is any workflow that listens for the pull-request-target event and then checks out or runs the submitted code.

And then there's the question hanging over all of it: why are Patch Tuesdays suddenly this big? Microsoft answers that one itself. Back in May, the company introduced a system called MDASH that orchestrates more than a hundred specialized agents to automatically hunt for vulnerabilities, scoring eighty-eight point four five percent on the CyberGym benchmark — the top of the published leaderboard. Five days before the record Patch Tuesday, Microsoft published a post arguing that vulnerability management has to keep pace with AI-driven discovery. For operators, that means prioritizing by exploitation evidence rather than severity score, because patching everything red is becoming unaffordable.

SAP closed sixteen flaws, Citrix fixed six in NetScaler, and Samsung patched five critical and forty-two high-severity issues. Angular reached twenty-two point oh point eight, React reached nineteen point two point eight — routine maintenance, nothing more. And from Brazil: Microsoft licenses got more expensive as of July first, and the new alphanumeric CNPJ company ID is forcing every system with a digit-based validation to adapt.
