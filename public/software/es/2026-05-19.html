Seguridad & cadena de suministro

Cisco SD-WAN CVE-2026-20182 — CVSS 10,0, en libertad

Es bastante raro que CISA suba un bypass de autenticación en edge con score CVSS máximo al catálogo KEV — y este viene con atribución explícita. CVE-2026-20182 en Cisco Catalyst SD-WAN Controller y Manager permite que un atacante remoto no autenticado obtenga privilegios de administrador. La propia Cisco atribuye la explotación "con alta confianza" al cluster UAT-8616 — el mismo grupo que weaponizó el CVE-2026-20127. CISA agregó el CVE al KEV el 14/05; la fecha límite de patch FCEB fue 17/05 [1][2].

El stack de Daniel no está directamente afectado — pero los clientes DACH con edges Cisco SD-WAN definitivamente sí. La lección topológica es más interesante: los controladores SD-WAN suelen vivir en VLAN de gerenciamiento con segmentación débil hacia producción. Si el controlador cae, en el peor caso se lleva el control de rutas del mesh completo del sitio. CVE-2026-20133, -20128, -20122 del batch KEV de abril están en la misma ruta — los atacantes los están encadenando [3].

pgvector 0.8.2 — integer underflow en HNSW paralelo

pgvector pasó de hobby de fin de semana a índice vectorial por default en stacks Postgres — quien hospeda su propio RAG lo tiene en el cluster. CVE-2026-3172 vive en el build paralelo del índice HNSW: un integer underflow produce una lectura fuera de límites en la memoria del backend. Un usuario de DB autenticado puede usar eso para filtrar datos de otras relations, o tirar el server. CVSS 8,1. Afectadas 0.6.0–0.8.1, corregido en 0.8.2.

HNSW (Hierarchical Navigable Small Worlds): imagina buscar una palabra en un libro. En vez de leer cada página, tienes un índice que te lleva groseramente a la sección correcta — y ahí, letreros locales hacia palabras vecinas. HNSW es exactamente eso para búsqueda vectorial: varias capas, la de arriba gruesa (pocos nodos, saltos largos), la de abajo fina. Resultado: vecino más cercano aproximado en milisegundos en vez de segundos. La variante paralela construye el índice en varios workers — y en pgvector 0.6+ la aritmética de tamaño pasaba por un entero sin signo que se desbordaba a negativo.

En la práctica: cualquier cluster Postgres con pgvector activo y usuarios DB multi-tenant (o sea, el default para SaaS) está expuesto. Actualizar a 0.8.2 es tarea de esta semana; cross-check hecho contra NVD [1][2].

Spring AI 1.0.7 / 1.1.6 / 2.0.0-M6 — tres CVEs de memoria de una sola

Si alguien usa Spring AI en serio en ruta productiva, la oleada de releases del 08/05 es lectura obligatoria. Tres CVEs cierran huecos que aparecen estructuralmente en agentes de chat basados en memoria cuando la API se usa con descuido. CVE-2026-41712: el `DEFAULT_CONVERSATION_ID` por default hacía que la memoria de chat sin scope explícito de usuario se filtrara entre sesiones — el clásico "usuario A ve la respuesta de usuario B" en stacks de PoC. CVE-2026-41713: un atacante puede armar prompts que aterrizan en la conversation memory y se interpretan en un turno posterior como "instrucción" — memory poisoning. CVE-2026-41705: filter injection en el conector Milvus vector store.

La corrección no es solo un upgrade — la API PromptChatMemoryAdvisor quedó deprecada; el sucesor obliga un parámetro explícito de conversation ID. Quien tenga código Spring AI parado del Q1 2026 arregla no solo el build sino el threat model. Spring AI 2.0 está anunciado para el 28/05 [1][2][3].

Nota sobre threat model Memory poisoning no es bug de código, es premisa de arquitectura. Quien usa chat memory como almacenamiento de "lo que dijo el usuario" está tratando input de usuario como system prompt confiable. Eso se rompe sistemáticamente apenas un tool-use en el mismo turno lee la memoria. La nueva API impone el scope por conversation ID — eso resuelve la fuga cross-user. El problema de instrucción-que-viene-de-memoria sigue siendo una pregunta de arquitectura.

Firefox 151 — Container Style Queries cross-browser-stable

Mozilla saca Firefox 151 como estable hoy, y el item más importante no está en la línea de marketing de arriba: Container Style Queries ahora están estables en un tercer navegador principal. ¿Qué significa? Un componente puede reaccionar al valor de una custom property en su container — no solo a su tamaño. Hasta ahora, theming se modelaba vía selector de ancestro (".dark .card"). Desde hoy, "el container tiene `--theme: dark` puesto" basta — theme switching sin JS, sin toggle de clases, local por subtree. Además: `CSSContainerRule.conditions` como nueva property, varias conditions por rule.

Tres ítems más que importan en producción: el Document Picture-in-Picture API (contenido DOM en ventana flotante always-on-top — Chrome lo tiene desde ~24, Firefox lo alcanza), Fullscreen Keyboard Lock (juegos en browser capturan Ctrl-W), y respaldos locales de perfil en Linux y macOS con restore cross-platform. Firefox 151 también es el primer build estable que, según Mozilla, lleva preparación para el nuevo "Firefox Home" — el rediseño mayor de UI llega con el 152 [1][2].

Temporal: el objeto Date de JavaScript es una de las piezas más notorias de la biblioteca estándar del lenguaje — meses numerados desde 0, mutaciones en vez de inmutabilidad, manejo sucio de timezone. Temporal lo reemplaza: valores inmutables, separación limpia entre Instant (punto en el tiempo), PlainDate (día de calendario) y ZonedDateTime (punto en el tiempo + timezone). Código JavaScript ahora escribe `Temporal.PlainDate.from('2026-05-19').dayOfWeek` en vez de `new Date('2026-05-19').getDay() + 1`.

M&A & política OSS

Anthropic ↔ Stainless — el taller de SDKs cae

Anthropic confirmó oficialmente el deal Stainless por más de US$ 300 mi el 18/05. Stainless es el pequeño taller de NYC (fundado en 2022 por el ex-Stripe Alex Rattray) que, por tres años, ha generado automáticamente y mantenido los SDKs oficiales de OpenAI, Google, Cloudflare y Runway desde sus specs OpenAPI — Python, TypeScript, Go, Java. Anthropic está cerrando los productos Stainless hospedados y manteniendo la tech internamente. Los clientes existentes conservan sus SDKs generados y derechos plenos de modificación.

Este es el movimiento de plataforma más agudo del mundo IA en mayo de 2026 — y no es uno que produzca titular de "nuevo modelo". Lectura: Anthropic no está comprando a la competencia; está comprando la infraestructura compartida de la competencia. OpenAI y Google tienen, a partir de hoy, un hueco interno de tooling en la pipeline de SDK — tres a seis meses hasta que estén listas soluciones de reemplazo. Para consumidores de SDK (o sea, todos los devs) la capa se vuelve lentamente más fragmentada, no más uniforme [2].

Efecto de SDK lock-in Stainless genera SDKs desde specs OpenAPI — las specs siguen siendo de los respectivos vendors, así que OpenAI/Google siguen siendo completamente consumibles a nivel de API. Lo que falta es la capa de SDK cross-language con actualización automática. Si la construyen ellos mismos, lleva meses; si la subcontratan a Speakeasy / orval / openapi-generator, pierden el acoplamiento estrecho a features propietarias de la API (convenciones de streaming, encoding de tool-call, etc.) que Stainless modelaba en el camino.

Comentario — la capa debajo del modelo

Tres movimientos de mayo muestran el mismo patrón: la competencia ya no se juega solo en la capa de modelo. Primero: Anthropic compra el taller de generación de SDKs de la competencia — la capa que nadie tiene en el slide de conferencia porque no produce benchmark score. Segundo: Spring AI abandona su default de API de memoria en el mismo respiro, porque la capa de flujo de datos debajo de la llamada LLM es la capa real de riesgo, no el modelo en sí. Tercero: Google I/O esta noche va a hablar mucho de Gemini, pero el slide comercialmente decisivo es el del logo MCP — quien controla el tool schema controla lo que un agente puede hacer, no solo lo que dice.

¿Qué significa eso para el stack-DNA de Daniel? Los stacks Java-Spring-Postgres-Caddy tienen, en la ola IA hasta aquí, la virtud de que el modelo sigue siendo un servicio llamado externamente, no un componente arquitectural. Quien esté construyendo Spring AI dentro del stack tiene desde ayer una nueva API que aprender, tres CVEs que revisar, y un threat model que antes no estaba. Camino pragmático: llamadas HTTP al proveedor IA, estado de conversación explícitamente en el propio schema Postgres, sin magia de memory-advisor. Eso no es "estar atrás" — es la arquitectura que sobrevive una auditoría.

Mañana Quarkus 3.36 va final, Google I/O entrega el Developer Day, y TC39 día 2 ocurre en Amsterdam — la semana ruidosa dura tres días.
