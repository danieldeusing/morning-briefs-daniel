Docker — tres caminos del contenedor a root del host vía docker cp

Este es el parche obligatorio de la semana, y cae justo en el stack de Daniel. El Docker Engine 29.5.1 (18/05) cierra tres fallas que comparten un núcleo: un contenedor malicioso puede escapar durante el aparentemente inofensivo docker cp y alcanzar el sistema de archivos del host, porque la lógica de copia corre con privilegios de root del host. CVE-2026-41567: los binarios de descompresión (xz, unpigz) se resuelven por el PATH dentro del contenedor — un binario plantado corre entonces como root del host. CVE-2026-42306: una carrera TOCTOU redirige un bind mount a una ruta arbitraria del host. CVE-2026-41568: un intercambio de symlink TOCTOU crea archivos/directorios arbitrarios en el host. Docker califica las tres de "critical"; los registros GHSA correspondientes asignan High/High/Medium [1].

TOCTOU (time-of-check / time-of-use): Como el de seguridad revisando tu identificación, tú cambiándola por una falsa en un parpadeo, y el sello cayendo en la tarjeta equivocada. El programa verifica una ruta, el atacante la cambia en la fracción de segundo intermedia, y la acción privilegiada golpea el objetivo plantado — aquí, una ruta del host en vez de la ruta esperada del contenedor.

El 29.5.2 (20/05) es el complemento: el 29.5.1 introdujo una regresión en la que docker cp con bind mounts que atraviesan symlinks dentro del contenedor (/var/run → /run) fallaba con "mkdirat: file exists". Así que si ya saltaste al 29.5.1, toma el 29.5.2 de inmediato [1].

Qué hacer hoy Revisa la versión de la engine (docker version) y sube a la 29.5.2. El ataque requiere un contenedor comprometido u hostil que intercepte un docker cp — quien corre imágenes de terceros o contenedores de CI disparados por PR es el objetivo principal. Hasta parchar, no corras docker cp contra contenedores no confiables.

containerd — runAsNonRoot era una petición, no un cerrojo

Un piso abajo, el mismo día: CVE-2026-46680 (GHSA-fqw6-gf59-qr4w, CVSS v4 7,3) en containerd. El manejo de user-ID puede engañarse de modo que un contenedor que debería correr sin ser root vía runAsNonRoot lo haga de todos modos. Eso derriba una de las garantías de hardening estándar de las que dependen los perfiles de pod-security de Kubernetes. Afectados: 1.7.27–1.7.31, 2.0.4–2.0.8, 2.1.0–2.2.3 y 2.3.0-beta–2.3.0; corregido en 2.3.1 / 2.2.4 / 2.0.9 / 1.7.32 (20/05). Corrección al brief del miércoles: la CVE de esta ola es la 2026-46680, no el número de spdystream citado ahí [1].

OpenJDK — JDK 27 se consolida, tres JEPs cerca del target

La línea Early-Access está en el build 22 (desde ~17 el miércoles). Tres JEPs pasaron a Proposed to Target, con la ventana de revisión cerrando el 19/05: JEP 523 (G1 como GC por defecto en todos los entornos), JEP 534 (Compact Object Headers por defecto — la promoción del JEP 519 del JDK 25) y JEP 537 (Vector API, 12ª incubadora). Nuevo en Candidate: JEP 538 (encodings PEM de objetos criptográficos). El RDP1 (feature freeze) se bifurca el 04/06, GA en septiembre. Aviso de honestidad: que los tres JEPs pasaran de Proposed to Target a de hecho Targeted no se pudo confirmar de forma final en la página primaria de openjdk.org (que devuelve 403) — ventana de revisión cerrada, targeting esperado [1][2].

Compact Object Headers: Cada objeto Java carga un pequeño header de control. JDK 25 lo encogió de 12–16 a 8 bytes — como darle a cada caja del almacén una etiqueta más chica: en millones de objetos eso libera heap de verdad y así recorta las pausas de GC (~15% menos GCs en SPECjbb2015). El JEP 534 lo vuelve el comportamiento por defecto en JDK 27 en lugar de una flag opt-in.

TC39 #114 — using ya es ECMAScript

El plenario de Ámsterdam (hospedado por JetBrains) cerró el jueves; los cambios de stage están en el repositorio tc39/proposals, con las notas narrativas todavía por venir. Lo destacado: Explicit Resource Management — las declaraciones using y await using que muchos conocen de TypeScript — alcanzó el Stage 4 y ya es parte de ECMAScript 2026. También en Stage 4: Atomics.pause (una pista de spin-loop para memoria compartida) y Joint Iteration (Iterator.zip). Avanzaron a Stage 3, entre otros, iterator chunking/includes/join y un accessor estandarizado de error.stack. Decorators pasó solo de 2 a 2.7 — la vieja salvedad se mantiene: ninguna de las grandes engines quiere ser la primera en enviar, así que el calendario sigue flojo [1][2].

using / Explicit Resource Management: Como el try-with-resources de Java, pero para JavaScript: escribes using file = openFile() y el recurso se cierra automáticamente cuando termina el bloque — ya sea que salga normal o por excepción. Se acabaron los bloques finally { close() } olvidados.

Comentario — las fronteras en las que confiamos

Tres de las historias principales de hoy son la misma historia en tres capas. El contenedor no debería ver el host — el docker cp y el bypass de runAsNonRoot muestran que la pared de aislamiento es más delgada que el modelo mental. El ORM debería hacer imposible la inyección SQL — la falla de Drupal muestra que "solo en Postgres" basta para tumbar esa premisa. Y la provenance SLSA debería probar la confianza de la cadena de suministro — Shai-Hulud muestra desde hace dos semanas que un build secuestrado hereda esa confianza. El patrón: no es la cripto ni la feature lo que está roto, sino la frontera donde dejamos de pensar.

Para el stack de Daniel eso significa algo pragmático y manejable: sube Docker al 29.5.2 y no corras docker cp contra contenedores no confiables; si Postgres-Drupal corre en algún lado, parcha hoy; sigue vigilando lockfiles de npm y compromisos de mantenedores. La buena noticia viene de la misma industria que está mudando de piel: el using ya es ECMAScript, el JDK 27 vuelve los Compact Object Headers el default, Node tiene su parche LTS. El lunes la UE entrega la siguiente pieza de soberanía con el CADA — y ahí veremos si el párrafo filtrado sobre nube de EE. UU. se vuelve texto legal de verdad [1].

  1. europarl.europa.eu/legislative-train

Es viernes, 22 de mayo. El hilo conductor de hoy es incómodamente concreto: las fronteras en las que confían los desarrolladores están filtrándose todas a la vez — en el aislamiento de contenedores, en la base de datos y en la cadena de suministro.

La historia más grande cae justo en el stack de Docker. El Docker Engine veintinueve punto cinco punto uno cierra tres fallas que permiten a un contenedor malicioso escapar durante el aparentemente inofensivo comando de copia docker cp y escalar a root del host — una vez porque los programas de descompresión se resuelven por la ruta dentro del contenedor, y dos veces por una carrera de tiempo que redirige un punto de montaje al sistema de archivos del host. Docker califica las tres de críticas. Un piso abajo, containerd corrige el mismo día un bypass que deja a un contenedor que no debería correr como root hacerlo de todos modos. El consejo de hoy: sube la engine al veintinueve punto cinco punto dos, porque la primera corrección introdujo una regresión — y no corras docker cp contra contenedores en los que no confías.

La segunda capa es la base de datos. El release de Drupal anunciado el miércoles llegó: una inyección SQL sin autenticación en el core que afecta solo a instalaciones PostgreSQL. El propio Drupal la califica en veintitrés de veinticinco, mientras que la base de vulnerabilidades estadounidense la llama solo media — y ambos valores merecen mención. Y desde hoy, Drupal dice que se observan intentos de explotación en la práctica. Si corres Drupal sobre Postgres, parcha hoy.

La tercera capa es la cadena de suministro. El gusano autorreplicante del brief anterior, el Shai-Hulud, no se enfrió — está escalando. El diecinueve de mayo una quinta ola rodó por el ecosistema AntV: a través de un mantenedor secuestrado, cerca de seiscientas treinta y nueve versiones maliciosas en trescientos veintitrés paquetes se publicaron en aproximadamente media hora. Peor, el código del gusano ya es público, y los imitadores son el nuevo riesgo. Además, la agencia estadounidense CISA agregó nuevas fallas activamente explotadas, incluida una en Langflow con puntuación nueve punto cuatro.

También hay buenas noticias, y vienen de la misma industria que está mudando de piel. Quarkus y Spring siguen tranquilos — el final de Quarkus esperado el miércoles aún no llega.

UU. para datos gubernamentales sensibles — no es prohibición, y no para la iniciativa privada.
