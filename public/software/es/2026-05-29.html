Vencimiento Exchange-OWA hoy — aún sin parche

El vencimiento KEV CISA para la brecha Exchange on-prem CVE-2026-42897 vence hoy, vie 29/05. Microsoft confirmó explotación activa el 14/05, CISA lo agregó al catálogo el 15/05 y dio 14 días para mitigar — la ventana se cierra ahora [1].

XSS en OWA: Outlook Web Access es el frontend web de Microsoft para Exchange. Cross-Site Scripting significa: el atacante contrabandea código JavaScript extranjero en tu pestaña OWA autenticada, que luego se ejecuta bajo tu login. En la práctica: un correo preparado llega, lo abres en OWA, y su código roba cookies de sesión, copia tu buzón o configura reglas de correo. El servidor no se toca — la brecha secuestra el navegador que ya habías autenticado.

El problema: sin parche en el sentido clásico. Microsoft solo está distribuyendo una mitigación de reescritura de URL llamada "M2.1.x" a través del Exchange Emergency Mitigation Service [2]. EMS está activo por defecto en Subscription Edition y 2019; en 2016 y CUs más antiguas tienes que parchear manualmente. Cualquiera que haya desactivado EMS por razones de cumplimiento (más agencias gubernamentales de las que crees) está técnicamente sin mitigar hoy. CVSS 8,1 [3]. Para empresas mexicanas de mediano tamaño que ejecutan Exchange híbrido: verifica el log EMS hoy, no la próxima semana.

Gitea: cuatro años entregando imágenes de contenedor privadas a solicitudes anónimas

Gitea, la alternativa auto-hospedada para GitHub Enterprise/GitLab, confirmó CVE-2026-27771 el 27/05 y lo cerró con 1.26.2. CVSS 8,2. El problema: el registro de contenedores incorporado respetaba la bandera private en la UI, pero entregaba manifiestos y capas de blob a los endpoints del registro (GET /v2/<name>/manifests/<tag>, /blobs/<digest>) incluso a solicitudes anónimas. Conocer el nombre del repo y la etiqueta era suficiente — sin autenticación, sin cuenta, sin token [4].

Registro de contenedores: El gestor de paquetes para imágenes Docker — como npm para Node, excepto por configuraciones de servidor completo pre-construidas. Construyes localmente (docker build), empujas al registro, y el servidor tira a través de docker pull. Registro "privado" significa autenticación solamente — en las capas de imagen residen código fuente, archivos .env, claves de API incrustadas, credenciales AWS. Si fallan los controles de autenticación, es peor que un repo Git abierto: no solo código, sino el bundle construido, funcionando de secretos.

El bug existió durante casi cuatro años. Exposición: >30.000 instancias en 30+ países, top 5 China, USA, Alemania, Francia, Reino Unido [5][6]. Forgejo heredó el mismo código de registro y también se ve afectado. Descubierto en abril por "NoScope", un agente autónomo de pentesting basado en IA — una vista previa de los próximos 12 meses [7]. Mitigación: actualiza a ≥1.26.2; medida temporal [service].REQUIRE_SIGNIN_VIEW = true [8]. Para desarrolladores independientes y empresas de mediano tamaño con reflexo de soberanía de seguridad, Gitea/Forgejo es la stack auto-hospedada obvia de Git+CI+registro — si atiendes clientes: actualiza ahora, audita los logs del registro (qué IPs anónimas extrajeron qué manifiestos?), rota secretos.

C# obtiene un toque Rust — tipos de pointer fuera de "unsafe"

El equipo .NET de Microsoft esbozó el 26/05 en el blog .NET la mayor reforma de memory-safety para C# en años. PM Richard Lander explica en "Improving C# Memory Safety": la palabra clave unsafe se está redefiniendo — según el modelo de Rust, sin abandonar el GC [9].

"unsafe" en un lenguaje con GC: C# normalmente limpia memoria a través de recolección de basura. En el momento en que utilizas pointers crudos (para código C nativo, hardware), desactivas esa red de seguridad — eso es lo que hace la palabra clave unsafe. Rust es más preciso: solo la operación realmente peligrosa (pointer dereferencing, es decir, leer desde una dirección extranjera) va en un bloque unsafe; la aritmética con el pointer se mantiene segura.

Eso es exactamente hacia donde Microsoft quiere llevar C#. Los tipos de pointer y la aritmética de pointer se volverán seguros; solo el dereferencing se mantiene unsafe. Efecto: los wrappers FFI e interop se vuelven más legibles, el paso peligroso se destaca en lugar de esconderse entre cálculos inofensivos en el mismo bloque — la misma lógica que Rust [10].

Roadmap: vista previa en .NET 11, producción en .NET 12 / C# 16, opt-in. El código antiguo se ejecuta sin cambios; las bibliotecas de runtime serán opted-in [11]. Se ajusta a la imagen más grande: Microsoft ha estado traduciendo Windows C/C++ a Rust internamente desde 2024; ahora .NET está apretando el tornillo en su propio lenguaje. Para desarrolladores Java/Spring, un punto a observar; más relevante si tocas .NET/Unity — el modelo llega en vistas previas de .NET 11 este verano.

npm "staged publishing" — la puerta 2FA humana es GA

Respuesta de GitHub a TanStack: GA desde 22/05. En lugar de publicar directamente en npmjs.com, el tarball cae en una cola de staging — un mantenedor debe aprobar explícitamente con 2FA. Requiere npm CLI 11.15.0+ y npm stage publish [17].

Trusted Publishing / OIDC: Un CI (p. ej., GitHub Actions) se autentica con un token OIDC de corta duración para npm y envía un nuevo paquete automáticamente — sin clics. Conveniente, pero: si un atacante compromete el CI (solicitud pwn, envenenamiento de caché), el CI publica su paquete envenenado con una firma válida. Así sucedió TanStack. Staged publishing lo invierte: el CI carga, pero solo entra en línea después de un segundo paso, humano con 2FA.

"Prueba de presencia" incluso para publishes CI/CD. Advertencia: se defiende contra la ruta TanStack (robo de token), no la ruta Nx (cuenta del mantenedor comprometida). Y solo funciona si los mantenedores la habilitan — los valores por defecto se mantienen por ahora.

JDK 27 RDP1 el 04/06 — Feature Freeze con solo un JEP

El próximo jueves (04/06), OpenJDK JDK 27 se bifurca en Rampdown Phase One — feature freeze, solo estabilización después. Resto del cronograma: RDP2 el 16/07, RC inicial 06/08, RC final 20/08, GA el 14/09/2026 [21].

Llamativo: solo un JEP oficialmente objetivo — JEP 527: Intercambio de claves híbrido post-cuántico para TLS 1.3. El resto se convierte en una versión de corrección de errores — editorialmente mucho más pequeña que 21. Cualquiera que apueste por una LTS con un grupo de características espera 28+. JEP 527 es serio de todas formas: harvest-now-decrypt-later se vuelve operacional en los próximos dos años, KEM híbrido en TLS 1.3 es la respuesta [22].

Es viernes, 29 de mayo, y el día concentra dos tipos muy diferentes de "abierto durante años". Primero: el vencimiento CISA de hoy para el zero-day Exchange OWA de Microsoft, CVE veintiséis cuarenta y dos ochenta y siete. El parche permanente no existe — Microsoft solo ha distribuido una mitigación temporal de reescritura de URL llamada M dos punto uno sobre el Emergency Mitigation Service. Si eres una empresa mexicana ejecutando Exchange híbrido, verifica los logs EMS hoy, no la próxima semana.

Segundo: Gitea, la alternativa Git auto-hospedada, cerró el miércoles una brecha que entregó imágenes privadas de contenedor a solicitudes web anónimas durante casi cuatro años. El registro incorporado respetaba la bandera de privacidad en la UI pero no la implementaba en los endpoints de imagen. Si conocías el nombre del repo y la etiqueta, obtenías las imágenes construidas con todos los secretos incrustados — sin cuenta, sin token. Los investigadores de seguridad estiman alrededor de treinta mil instancias expuestas a internet, concentradas en China, Estados Unidos y Alemania. El fork Forgejo también se ve afectado. La corrección es la versión uno punto veintiséis punto dos de Gitea. Un detalle notable: el bug fue encontrado por NoScope, un agente autónomo de pentesting alimentado por IA, en abril de este año.

Tercero, mucho más silencioso pero significativo a largo plazo: el equipo dot-net de Microsoft esbozó la mayor reforma de memory-safety del C-sharp en años en el blog dotnet. La palabra clave unsafe se redefinirá a imagen de Rust. Trabajar con pointers se volverá seguro en adelante; solo dereferencing — el acceso real a la memoria — aún requiere un bloque unsafe. Oracle entregó ayer la primera Critical Security Patch Update mensual: treinta y cinco parches, nuevo ritmo mensual en el tercer martes. El razonamiento: el descubrimiento de vulnerabilidades impulsado por IA ha superado la antigua ventana trimestral. Si ejecutas una base de datos Oracle on-premises, planifica ahora doce ventanas de parche por año en lugar de cuatro.
