Miasma: un gusano npm que se propaga a través de binding.gyp

La historia de la semana no es un lanzamiento de modelo, es un incidente de cadena de suministro — y uno con mecánica novedosa. Un gusano npm autorreplicante llamado Miasma atacó en una onda el 3 de junio y de acuerdo con StepSecurity comprometió 57 paquetes npm en más de 286 versiones maliciosas en menos de dos horas. Primero atacó @vapi-ai/server-sdk (más de 408 mil descargas/mes) a las 23:30 UTC, luego incluyendo ai-sdk-ollama (más de 120 mil/mes).[1]

binding.gyp: Un archivo de configuración que permite que los paquetes npm creen componentes C/C++. Piénsalo como las instrucciones de ensamblaje que un mueble ejecuta automáticamente cuando se desempaca — normalmente inofensivo. Miasma usa un archivo de 157 bytes con sustitución de comandos gyp para ejecutar código durante npm install — completamente eludiendo los scripts de ciclo de vida package.json usuales que cada scanner observa. El punto ciego es el mecanismo en sí.

El payload recopila credenciales de múltiples nubes ampliamente: claves AWS y tokens IMDS, cuentas de servicio de GCP, identidades de Azure, secretos de GitHub Actions (incluyendo raspado de memoria del ejecutor), tokens npm, credenciales de Vault y SSH. Con tokens robados, el gusano luego publica más paquetes infectados — de ahí "gusano" en lugar de solo "robo".[1]

Linaje: Miasma se deriva del malware Mini-Shai-Hulud de acuerdo con análisis de Microsoft, que un actor publicó como código abierto el 12 de mayo — esencialmente un kit de herramientas que múltiples actores ahora pueden reconstruir.[2] Quién podría ser afectado: verifica tu lockfile, bloquea scripts de instalación en CI con --ignore-scripts, rota tokens.

Red Hat confirma: 32 paquetes npm comprometidos

La primera gran onda Miasma golpeó a la propia Red Hat el 1 de junio: bajo el namespace @redhat-cloud-services, 96 versiones en 32 paquetes fueron comprometidas — totalizando aproximadamente 116.991 descargas/semana (RHSB-2026-006).[3] Son bibliotecas frontend de la Hybrid Cloud Console. El punto de entrada fue una cuenta GitHub comprometida utilizada para inyectar código malicioso pasando la revisión de código.

La confirmación de Red Hat es instructiva: ningún lanzamiento de Console fue publicado durante el incidente, y el proceso de publicación elimina scripts de instalación antes del despliegue — la capa protectora se mantuvo. Para clientes finales "ninguna acción necesaria". Pero quien sacó los paquetes directamente de npm no está protegido por esa misma pared.[3]

Contexto En paralelo, Microsoft reportó una campaña separada: 33 paquetes npm usando confusión de dependencias para espiar entornos de desarrolladores.[4] Dos ataques independientes en una semana — la capa de registro es el punto crítico ahora, no el código dentro.

Cuando la firma miente: procedencia SLSA como camuflaje

La parte más desagradable de Miasma no es el robo, es el camuflaje. El gusano intercambia tokens robados por certificados de firma genuinos de Sigstore de Fulcio y crea entradas en el registro de transparencia Rekor — los paquetes reinfectados se ven legitimamente construidos, con procedencia SLSA válida.[1]

Procedencia SLSA: Un recibo firmado que prueba "este paquete fue construido en este pipeline a partir de este commit". Diseñado como un sello de autenticidad — como un sello notarial. Miasma muestra: quien roba al notario puede falsificar cada sello. El sello prueba origen, no benignidad.

Consecuencia práctica para nosotros: "tiene procedencia" no es un pase libre. La cadena es solo tan fuerte como su eslabón de token más débil. Lo que sigue siendo valioso es profundidad — hashes fijados, bloqueo de scripts de instalación, y escepticismo hacia paquetes con archivos que no son entry-point desproporcionadamente grandes.

Atención Miasma también deja archivos de puerta trasera en repositorios — .claude/setup.mjs, .cursor/rules/setup.mdc, .vscode/tasks.json. Envenena la capa de herramientas a través de la cual el código es generado, no solo el código en sí. También audita estos directorios.[1]

Build 2026, la mitad no-IA: Windows se vuelve amigable con Linux para desarrolladores

Los titulares del keynote fueron para los modelos — viven en el brief de ai-dev. Para nosotros más interesante es la mitad de la plataforma, y es sustancial esta vez.

Coreutils para Windows (GA)

Microsoft está haciendo disponibles más de 75 herramientas de línea de comandos estilo Unix nativamente en Windows — ls, cat, cp y más sin WSL, sin trucos Git Bash. Vienen del proyecto de código abierto uutils, una reimplementación en Rust de GNU Coreutils.[5] Para scripts multiplataforma, reducción real de fricción.

Contenedores WSL (Preview)

Los contenedores Linux ahora pueden crearse y ejecutarse directamente desde WSL con su propia CLI/API y gestión de políticas empresariales. Esto abiertamente apunta a reducir la dependencia de Docker Desktop y otros.[6] Desde que WSL fue abierto como código en Build 2025, Microsoft reporta más de 200 pull requests por mes entrando — el proyecto está vivo.[5]

Terminal Inteligente (experimental)

Una terminal que detecta comandos fallidos, muestra contexto y sugiere correcciones — vía un Protocolo de Capacidad de Agente (ACP). Interesante, pero experimental; anotado aquí como contexto de plataforma, la sustancia del agente es territorio de ai-dev.[6]

  2. blogs.windows.com

Azure Linux 4.0 & Cobalt 200: La sala de máquinas Linux de Microsoft

Azure Linux 4.0 está en preview público — y es la primera distro de servidor de Microsoft declarada como de propósito general, no solo un host de contenedores más. Ahora es basada en Fedora, código abierto en GitHub (azurelinux), con huella de paquete más pequeña y política SELinux de negación predeterminada más rigurosa.[7]

En el lado del hardware, la generación de VM Cobalt 200 entró en preview de acceso temprano: segunda generación de servidor Arm de Microsoft (SoC en Arm Neoverse V3), que según Microsoft trae hasta 50% más desempeño de generación versus Cobalt 100.[8] Para quien ejecuta cargas de trabajo ARM en la nube, ese es el número clave de la semana.

  2. azure.microsoft.com

RTX Spark Dev Box & GitHub Enterprise Local

Dos anuncios de hardware/infraestructura que importan para el debate local-versus-nube. La Surface RTX Spark Dev Box es una estación de desarrollo compacta con silicio NVIDIA RTX Spark, 1 petaflop de computación y 128 GB de unified memory — suficiente para ejecutar modelos más allá de 120 mil millones de parámetros localmente, sin facturas de API. Disponible más adelante en 2026, inicialmente solo en los EE.UU. a través de Microsoft.com; precio aún por determinar.[6]

Más relevante infraestructuralmente: GitHub Enterprise Local, una variante de GitHub Enterprise que funciona en Azure Local y puede operarse conectada o aislada — con inferencia on-prem vía Foundry Local. Para ambientes regulados o impulsados por soberanía (agencias de la UE, por ejemplo) una señal; precio y tiempo aún abiertos.[9]

CISA KEV: Vulnerabilidad WebLogic de dos años se activa

CISA agregó CVE-2024-21182 en Oracle WebLogic Server al catálogo Known-Exploited-Vulnerabilities el 1 de junio — la vulnerabilidad es activamente explotada. CVSS 7,5 (HIGH), acceso de red no autenticado vía T3/IIOP, afecta versiones 12.2.1.4.0 y 14.1.1.0.0. Oracle la parchó en CPU de julio 2024; está siendo explotada contra rezagados.[10][11]

Catálogo KEV: Lista de CISA de vulnerabilidades con explotación activa comprobada en la naturaleza. No "teóricamente arriesgada", sino "siendo usada ahora mismo". Las agencias federales estadounidenses deben parcharla dentro del plazo — aquí era ayer, 04.06. Para nosotros es una buena señal de priorización, incluso sin el mandato.

La exposición es real: según Security Affairs, más de 1.592 servidores WebLogic están en línea y vulnerables (961 en 12.2.1.4.0, 631 en 14.1.1.0.0).[12] Ejecutando Fusion Middleware: verifica el estado de tu CPU.

Más adiciones KEV: cgroups Linux & framework Android

CISA agregó dos más el 2 de junio: CVE-2022-0492 (kernel Linux, escalada de privilegios de cgroups, CVSS 7,8 per NVD) — un atacante local puede escapar de un contenedor — y CVE-2025-48595 (framework Android, desbordamiento de enteros, CVSS 8,4, Android 14–16). El 3 de junio vino CVE-2026-45247 (Mirasvit Full Page Cache Warmer, deserialización insegura de datos).[13][14]

La vulnerabilidad de cgroups es la para nosotros: si ejecutas contenedores en kernels antiguos, alinea tu nivel de parche — escape de contenedor no es un riesgo periférico.

Nota CISA.gov está devolviendo 403 a los crawlers ahora; los detalles de adición provienen de los títulos de páginas de alerta de CISA y se verifican contra securityaffairs/thehackernews y NVD (CVSS de WebLogic directamente de NVD).

UE/DE: El aparato CRA se pone en marcha

Hilo de regulación en curso: El 11 de junio las disposiciones para nombrar órganos de evaluación de conformidad bajo la Ley de Resiliencia Cibernética entran en vigor — en Alemania, el BSI como autoridad notificadora publica los procedimientos.[15] Suena burocrático, pero es la pista en la que las obligaciones CRA para fabricantes de productos conectados se hacen concretas a partir de 2027.

El plazo más importante para desarrolladores llega en el otoño: comenzando el 11 de septiembre, el deber de notificación para vulnerabilidades activamente explotadas entra en vigor — dentro de 24 horas.[15] Multas por incumplimiento: hasta EUR 15 millones o 2,5% de los ingresos anuales globales; el BSI está contratando 141 posiciones hasta 2029 para hacerla cumplir.[16]

Releases en el espejo retrovisor: Node 24 & PostgreSQL 18.4

Tranquilo en el frente de lenguaje/DB, pero dos releases de mayo merecen atención. Node.js 24 (21.05) trae la propuesta TC39 Explicit Resource Management nativamente vía V8 13.6 — la palabra clave using para limpieza determinística de recursos, similar a try-with-resources en Java.[17]

PostgreSQL 18.4 (14.05) es una actualización pura de seguridad/mantenimiento de la línea 18 — 11 correcciones de seguridad y más de 60 bugfixes, ningún nuevo recurso.[18] Los destaques reales vinieron con PostgreSQL 18.0 (sep 2025): subsistema I/O asincrónico (hasta 3× en cargas pesadas de I/O), columnas generadas virtuales, y uuidv7() nativa — la última ahorra a muchos de generadores personalizados para IDs ordenables por tiempo. Si estás en 18, actualiza a 18.4 por las correcciones de seguridad.

  2. postgresql.org

Es viernes, cinco de junio. La semana de software gira en torno a la cadena de suministro — y a una conferencia Microsoft Build cuyos elementos más interesantes no tienen nada que ver con inteligencia artificial.

En el corazón está un gusano npm llamado Miasma. En una única onda el tres de junio, StepSecurity reporta que comprometió cincuenta y siete paquetes en menos de dos horas. Lo nuevo: se propaga a través de un archivo de compilación llamado binding.gyp y ejecuta su código durante la instalación — completamente eludiendo los lugares donde los scanners de seguridad normalmente miran. Roba credenciales de la nube de Amazon, Google, Azure y GitHub y las usa para publicar el próximo lote de paquetes infectados.

La primera gran onda golpeó a la propia Red Hat el primero de junio: noventa y seis versiones en treinta y dos paquetes, totalizando aproximadamente ciento diecisiete mil descargas por semana. La capa protectora de Red Hat se mantuvo, pero el incidente muestra cuán vulnerable es la capa de registro ahora.

Lo más desagradable es el camuflaje. El gusano intercambia tokens robados por firmas genuinas a través de Sigstore — y da a sus paquetes procedencia SLSA válida. La lección para nosotros: una firma prueba de dónde viene un paquete, no si es benigno.

En Microsoft Build, la mitad de plataforma fue sustancial. Windows recibe más de setenta y cinco comandos Unix nativamente, escritos en Rust. WSL ahora puede ejecutar contenedores Linux directamente sin Docker Desktop. Y Azure Linux cuatro punto cero es la primera verdadera distribución de servidor de propósito general de Microsoft, ahora basada en Fedora.

En seguridad, CISA agregó una vulnerabilidad Oracle WebLogic de dos años a su lista de vulnerabilidades activamente explotadas, con una gravedad de siete coma cinco. Más de mil quinientos servidores están vulnerables en línea. El plazo para agencias era ayer, cuatro de junio — rezagados deben parchar ahora.

Y en Europa el aparato de la Ley de Resiliencia Cibernética continúa: comenzando el once de junio, el BSI alemán nombra los órganos de evaluación. El plazo más importante para desarrolladores llega en septiembre, cuando el requisito de notificación de veinticuatro horas para vulnerabilidades explotadas entra en vigor.

Mañana será más tranquilo — las sesiones de Build permanecen disponibles bajo demanda, y continuamos observando avisos npm para ver si Miasma envía otra onda. De vuelta a la visión general.
