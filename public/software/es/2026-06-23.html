Splunk: RCE no autenticada, explotada activamente, plazo KEV ya pasó [1][2]

La vulnerabilidad del fin de semana: CVE-2026-20253 en Splunk Enterprise permite ejecución remota de código no autenticada a través del PostgreSQL Sidecar incluido — CVSS 9,8. Afectadas están las versiones < 10.2.4 y < 10.0.7; corregidas en 10.2.4 y 10.0.7 respectivamente. Según helpnetsecurity, está siendo atacada activamente desde aproximadamente el 15 de junio. CISA la agregó al catálogo KEV — el plazo de remediación para agencias federales estadounidenses era el domingo, 21 de junio, por lo que ya pasó.

Splunk frecuentemente se encuentra en el corazón de una pila de seguridad como SIEM; una RCE no autenticada allí es exactamente el tipo de vulnerabilidad que un atacante usa para profundizar desde la capa de monitoreo hacia la infraestructura. Si ejecutas una instancia local: la actualización no es opcional, y "accesible internamente" no es protección si el canal de telemetría llega lejos en la red.

libssh2: RCE crítica — y sin lanzamiento aún [3]

CVE-2026-55200 en libssh2 (CVSS 9,2) se puede activar a través de paquetes SSH malformados: errores de memoria que llevan a ejecución de código, incluyendo DoS. Afectadas están todas las versiones ≤ 1.11.1 — y la parte incómoda: aún no hay lanzamiento oficial. La corrección existe solo como commit de GitHub 7acf3df; Debian ha empujado 1.11.1-3 a Testing. Una segunda vulnerabilidad (CVE-2026-55199, High) viene con ella.

libssh2 está en todos lados donde SSH se incrusta: enrutadores, dispositivos IoT, herramientas de gestión de servidores, clientes Git. Si vincula libssh2 estáticamente, debes aplicar el commit o esperar el paquete de la distribución en lugar de esperar una etiqueta upstream que aún no está allí.

Oracle CPU: RCE de PeopleSoft explotada activamente [4][5]

La segunda actualización mensual de seguridad de Oracle trajo 243 CVEs en 245 parches, con 122 calificadas como críticas. El punto notable: CVE-2026-35273, una RCE no autenticada en PeopleTools 8.61 y 8.62. Según Tenable, ShinyHunters (UNC6240) han comprometido ~300 instalaciones, con 68% en educación superior. Oracle proporcionó una corrección fuera de cronograma el 10.06, y CISA agregó la vulnerabilidad al catálogo KEV el 12.06.

PeopleSoft es ERP de campus — datos de estudiantes, registros de personal, finanzas. Que un grupo de ransomware esté específicamente apuntando a universidades se ajusta al patrón: pilas empresariales antiguas, equipos de TI sobrecargados, datos sensibles.

Cadena de suministro de WordPress golpeada dos veces [6][7]

Dos incidentes de cadena de suministro en el ecosistema WordPress. Primero: en Awesome Motive (OptinMonster), atacantes robaron una clave de API de CDN a través de una vulnerabilidad de UpdraftPlus e inyectaron JavaScript malicioso en OptinMonster, TrustPulse y PushEngage — 1,2 millones de sitios afectados, según Sansec. Segundo: en ShapedPlugin, atacantes comprometieron la cadena de construcción y distribución a través de Easy Digital Downloads; el malware robó credenciales de administrador además de secretos 2FA y configuró acceso remoto — ventana vulnerable de abril a junio de 2026.

En ambos casos el objetivo no era el sitio individual sino el canal de actualización. Esa es la lección incómoda para cualquiera que actualiza plugins automáticamente: la actualización automática también distribuye automáticamente una versión envenenada.

Atención Si ejecutas plugins afectados: verifica scripts inyectados por CDN y cuentas de administrador inesperadas, rota secretos 2FA.

AWS Lambda obtiene MicroVMs — hasta 8 horas [8]

AWS anunció Lambda MicroVMs el 22.06: ambientes de ejecución aislados, stateful con tiempo de ejecución hasta 8 horas (vs. el antiguo límite de 15 minutos), hasta 16 vCPU, 32 GB RAM, 32 GB disco, ARM64, construida sobre Firecracker, con suspend/resume. Inicialmente en cinco regiones de AWS (N. Virginia, Ohio, Oregon, Irlanda, Tokio).

La intención declarada es clara: ejecutar código generado por usuarios o IA en un sandbox. AWS está siguiendo el patrón que otros venden como "sandbox de intérprete de código" — solo directamente en la plataforma Lambda con tiempo de ejecución más largo y estado. Para cargas de trabajo de agentes que necesitan más que unos pocos minutos y ejecutan código no confiable, esto cierra una brecha real.

MicroVM: una máquina virtual minimal que inicia en milisegundos (Firecracker es la tecnología detrás de esto). Aislamiento más pesado que un contenedor, más ligero que una VM clásica — ideal para encerrar de forma segura código de terceros.

Spring Boot 4.1.0 llegó — y 3.5 está terminando [9][10]

Spring Boot 4.1.0 está disponible desde el 10.06. Destacados: configuración automática de Spring gRPC, protección SSRF para el cliente HTTP (nuevo InetAddressFilter), integración OpenTelemetry actualizada, rotación de archivos Log4j, propiedades de configuración Jackson y soporte Kotlin 2.3. Marca en tu calendario: Spring Boot 3.5 alcanza EOL el 30.06 — después no hay correcciones gratuitas en la rama OSS.

Para tiendas Java/Spring, ese es el anuncio de migración del mes. El filtro SSRF es más que cosmético: aborda exactamente la clase de falsificación de solicitud del lado del servidor a través de la cual se rasguñan frecuentemente los puntos finales de metadatos internos.

AMD elimina cifrado de memoria — silenciosamente [11]

Historia extraña e desagradable: AMD eliminó Transparent Secure Memory Encryption (TSME) de los CPUs Ryzen 9000 para consumidor (no-Pro) con firmware AGESA 1.2.7.0 — sin ningún anuncio. Un usuario de Linux lo descubrió en abril a través de auditoría HSI; AMD confirmó que la eliminación fue intencional pero se mantuvo callado en preguntas posteriores. Una actualización de BIOS en julio se supone que restaurará TSME.

TSME cifra la RAM de forma transparente contra ataques físicos (arranque en frío, espionaje DMA). Apagarlo silenciosamente significa que los dispositivos pierden una característica de seguridad sin que sus propietarios lo sepan — el peor tipo de regresión de seguridad.

TSME: cifra toda la memoria automáticamente en hardware. Protege contra atacantes con acceso físico que quieren leer el contenido de la RAM directamente — especialmente relevante para laptops y sistemas de quiosco.

FastAPI 0.138.0 ahora sirve frontends [12]

Pequeño pero práctico para la multitud de Python: FastAPI 0.138.0 (20.06) trae app.frontend() / router.frontend() para servir un directorio dist/ construido directamente desde el framework — además de correcciones de deprecación de Pydantic v2 y un bump a Starlette 1.0.0. Ahorra un servidor estático separado para configuraciones pequeñas de SPA-plus-API; especialmente para un backend FastAPI con frontend Angular o React que quieres enviar como un contenedor único, esto es una simplificación bienvenida. El salto Starlette 1.0 es la parte que merece una mirada más cercana — un major 1.0 generalmente significa algunos cambios de comportamiento bajo el capó.

Patch Tuesday: cifra récord de CVEs, seis zero-days [13][14]

El patch day de junio de Microsoft (09.06) fue inusualmente grande: dependiendo de cómo cuentes, 198 a 209 CVEs, 32–33 de ellos críticos, 28 RCEs críticas. Además de seis zero-days — pero solo uno explotado activamente: CVE-2026-42897 (suplantación de Exchange Server). Los otros cinco eran conocidos públicamente pero aún no atacados, incluyendo CVE-2026-45585 (bypass de BitLocker "YellowKey"), CVE-2020-17103 (Cloud Files EoP), CVE-2026-45586 (CTFMON EoP, 7,8 — privilegios SYSTEM a través de procesos de reconocimiento de voz) y CVE-2026-49160 (DoS HTTP.sys, 7,5 — "bomba" HTTP/2, mitigada a través de nuevo valor de registro MaxHeadersCount). Separadamente, CVE-2026-32193 destaca: una RCE crítica del Servicio Azure Kubernetes con CVSS 8,8.

El DoS HTTP.sys es lo más interesante para desarrolladores web: una inundación de encabezado HTTP/2 que puede abrumar la pila HTTP de Windows — relevante para cualquier cosa detrás de IIS o HTTP.sys. Pero el único ya explotado activamente es la suplantación de Exchange Server (CVE-2026-42897) — exactamente el tipo de fallo que un atacante busca primero en una cadena de phishing o identidad. Seis zero-days en un único patch day es inusualmente alto; en suma, este es uno de los mayores patch days de Microsoft jamás.
