UniFi: tres fallos CVSS-10, explotados — y el plazo es hoy

La única cosa que realmente importa hoy. El 23/06, CISA agregó tres fallos de Ubiquiti UniFi OS al catálogo KEV — todos con severidad máxima 10,0 y todos siendo activamente explotados — y dio a las agencias federales tres días para parchear. Esos tres días terminan hoy.[1][4]

Catálogo KEV: La agencia estadounidense CISA mantiene una lista de vulnerabilidades que no son solo teóricamente peligrosas sino que son activamente usadas por atacantes — como un boletín de búsqueda de herramientas de robo que ya están en circulación. Una vez que un fallo entra en esa lista, la discusión "esperar o parchear" termina.

La cadena: CVE-2026-34908 permite cambios remotos no autorizados, -34909 es un fallo de path traversal (acceso a archivos + manipulación de cuentas), y -34910 es una debilidad de validación de entrada que lleva a ejecución remota de código via inyección de comandos. Encadenados, permiten control completo de la red sin autenticación. El parche se lanzó el 21/05 en UniFi OS Server 5.0.8 — si administras una instancia UDM/Cloud Key/UniFi OS y aún no estás en 5.0.8, esta es tu tarea de viernes por la tarde.[1][2]

Nota de Fuentes CISA.gov retornó 403 en acceso directo; la fecha de adición a KEV y el valor CVSS 10 fueron verificados cruzados via BleepingComputer y GBHackers, los scores individuales de CVE del NVD no están en nuestras fuentes primarias. La dirección — tres fallos de severidad máxima, activamente explotados, plazo hoy — es clara.
  1. bleepingcomputer.com — CVSS 10, adición a KEV el 23/06.
  2. gbhackers.com — Parche en UniFi OS Server 5.0.8 (21/05).
  3. cisa.gov — Alerta KEV, plazo de tres días.

"Miasma": una cuenta Red Hat contamina 32 paquetes npm

Exactamente el tipo de historia para la cual existe este boletín: sin titular de IA, sino cadena de suministro como superficie de ataque. El 01/06, Wiz divulgó una campaña llamada Miasma — una cuenta GitHub comprometida perteneciente a un empleado de Red Hat fue usada para inyectar malware en 32 paquetes npm @redhat-cloud-services. Juntos, sumaron aproximadamente 80 mil descargas por semana.[5]

El truco es lo que asusta. El atacante solicitó tokens OIDC via GitHub Actions y publicó los paquetes contaminados con proveniencia SLSA válida — así que los paquetes llevaban exactamente las credenciales de autenticidad de cadena de suministro que se suponía construían confianza.[5]

SLSA Provenance / OIDC: SLSA Provenance es como una prueba sellada de origen, mostrando "este paquete fue realmente construido por este pipeline". OIDC es la tarjeta de identificación que el pipeline usa al publicar. Aquí, el atacante falsificó la tarjeta de identificación y así colocó un sello genuino en un paquete falso — el sello es correcto, pero el contenido no.

En la instalación, un hook preinstall disparó un dropper ofuscado de 4,29 MB, descargó el tiempo de ejecución Bun y cosechó credenciales de GitHub, npm, AWS, Azure y GCP — cada infección individualmente encriptada para que una firma no las capture todas. Red Hat tiene un advisory (RHSB-2026-006) publicado; la remediación está en progreso.[6]

No solo Es temporada ocupada: el 17/06, un easy-day-js tipificado incorrectamente (similar a dayjs) contaminó 140+ paquetes en el ecosistema Mastra AI via dropper postinstall. El eje aquí también es la cadena npm, no el modelo.[7]
  1. wiz.io — 32 paquetes, 80 mil descargas/semana, abuso OIDC/SLSA.
  2. access.redhat.com — Detalles de payload, advisory.
  3. upwind.io — Mastra AI, easy-day-js.

Apple adquiere el Swift Package Index

Anunciado el 23/06: Apple está adquiriendo el Swift Package Index, construido por Dave Verwer y Sven A. Schmidt — el catálogo de facto para paquetes Swift con más de 10 mil paquetes indexados. Permanece código abierto, y el plan declarado es firma de paquetes más su propio registro para "salirse completamente del modelo GitHub".[8]

Para quienes no escriben Swift, aún relevante: es el mismo movimiento que vemos en todas partes ahora — después de los incidentes npm, los ecosistemas se están alejando de "cuenta GitHub = confianza" hacia registros firmados y autohospedados. Apple puede hacer esto para Swift porque es dueña de la plataforma. Para todos los demás, es un dato sobre hacia dónde se dirige la seguridad de paquetes.

  1. swiftpackageindex.com — Adquisición, 10 mil+ paquetes, firma + registro.

Check Point VPN: bypass de autenticación IKEv1, explotado por Qilin

CVE-2026-50751 (CVSS 9,3) es un fallo de lógica en validación de certificados para el intercambio de claves legado IKEv1: un atacante no autenticado puede eludir la autenticación del usuario y establecer una conexión VPN de acceso remoto sin una contraseña válida. Afectados son Quantum Security Gateway (R80.40–R82.10), Firewalls Spark y varias versiones de Gaia OS.[9]

Este no es un riesgo teórico: la explotación comenzó el 07/05, y actividad post-compromiso por un afiliado del ransomware Qilin ha sido confirmada en al menos un caso. CISA agregó el fallo al catálogo KEV (plazo era 11/06); el parche se lanzó el 08/06. Si aún tienes IKEv1 activo: deshabilítalo o parchealo.[10]

  1. nvd.nist.gov — CVSS 9,3, bypass de autenticación IKEv1, productos afectados.
  2. helpnetsecurity.com — Afiliado Qilin, explotación desde 07/05.

Oracle PeopleSoft: RCE zero-day, ShinyHunters en 100+ organizaciones

CVE-2026-35273 (CVSS 9,8) afecta el componente PSEMHUB de PeopleSoft — fácilmente explotable, sin autenticación, el acceso HTTP puro es suficiente para compromiso completo. El grupo ShinyHunters explotó esto como zero-day desde 27/05 hasta el parche el 10/06 y reclama 300 instancias en 100+ organizaciones, de las cuales 68% están en educación superior.[11]

Para Daniel, sin impacto directo en el stack (PeopleSoft es HR/ERP empresarial), pero un patrón instructivo: una ventana de explotación de semanas antes del parche, dirigida a un sector con equipos de TI limitados. Cualquiera que ejecute PeopleSoft debería haber parcheado hace mucho y revisado registros de mayo.[11]

  1. thehackernews.com — ShinyHunters, 300 instancias, 68% educación superior.

Patch Tuesday de junio: 200 vulnerabilidades, 3–6 zero-days

La ronda de junio de Microsoft el 09/06 estableció récords: 200 vulnerabilidades, de las cuales 33 son críticas (28 RCE) y 3–6 zero-days. El más importante es CVE-2026-42897 — un XSS de Exchange Server: un correo preparado, cuando se abre en OWA, ejecuta JavaScript arbitrario en el navegador del destinatario sin que el atacante necesite privilegios. Activamente explotado.[12][13]

Los zero-days restantes varían desde una escalada de privilegios de Windows CTFMON (exploit GreenPlasma) hasta un DoS HTTP.sys via "bomba HTTP/2". Las fuentes discrepan sobre cuántos fueron activamente explotados versus solo conocidos públicamente (BleepingComputer cuenta 6, Tenable 3–4) — seguimos la lista detallada de BleepingComputer.[12]

  1. bleepingcomputer.com — 200 vulnerabilidades, 33 críticas, 6 zero-days.
  2. techcommunity.microsoft.com — Exchange XSS CVE-2026-42897.

CRA de la UE: Capítulo IV en vigor desde 11/06

Hito de conformidad callado pero relevante: desde 11/06, el Capítulo IV de la Ley de Resiliencia Cibernética está en vigor — notificación de organismos de evaluación de conformidad. Los estados miembros ahora deben designar autoridades de notificación; esta es la infraestructura a través de la cual los productos más tarde obtendrán marcado CE de seguridad. Próximo hito: obligaciones de informe comenzando el 11/09/2026, cumplimiento total entonces el 11/12/2027.[14]

  1. digital-strategy.ec.europa.eu — Capítulo IV desde 11/06/2026, próximos plazos.

Angular 22 llegó — y está consolidando fuerte

Angular 22.0 aterrizó el 03/06, patch 22.0.2 el 17/06. Es un gran lanzamiento de consolidación: Signal Forms, Angular Aria y las APIs de reactividad asincrónica (resource/httpResource) ahora son estables, OnPush es el nuevo predeterminado, hay componentes sin selector y el cliente HTTP ahora usa la API Fetch por predeterminado.[15]

Para el código Angular de Daniel, dos cosas importan: strictTemplates ahora está activado por predeterminado (espera nuevos errores de compilación al migrar), y la Angular CLI trae soporte MCP estable para herramientas de IA — el último es más un tema ai-dev, solo una nota aquí.[16]

  1. angular.dev — 22.0 (03/06), 22.0.2 (17/06), APIs estables.
  2. blog.ninja-squad.com — strictTemplates predeterminado, CLI MCP.

Spring Framework 7.0.8: endurecimiento, y 6.2 termina el martes

El lanzamiento 7.0.8 del 08/06 es más endurecimiento de seguridad que caída de características, y exactamente por eso importa: la deserialización Jackson en convertidores JMS ahora puede ser restringida a paquetes en lista blanca via setTrustedPackages(...), y la evaluación SpEL ahora está limitada a diez mil operaciones por predeterminado.[17]

Lista Blanca de Deserialización / Límite SpEL: Deserialización convierte datos recibidos en objetos — sin una lista blanca, un atacante puede forzar clases que ejecutan código. SpEL es el mini-lenguaje Spring para expresiones dinámicas; un límite previene que una expresión maliciosa trabe la CPU en un bucle infinito. Ambas hacen a los clásicos RCE/DoS comunes mucho más difíciles.

Y la nota de calendario: Spring Framework 6.2.x llega a fin de vida el 30/06 — el próximo martes. Si aún estás en 6.2, deberías estar planeando el salto a 7.0 ahora.[18]

  1. endoflife.date — 7.0.8 (08/06), lista blanca JMS + límite SpEL.
  2. endoflife.date — 6.2.x EOL el 30/06/2026.

PostgreSQL: actualización consolidada, PG19 Beta 1, PG14 terminando

La consolidación del 14/05 (18.4 / 17.10 / 16.14 / 15.18 / 14.23) cierra 11 CVEs y más de 60 bugs — incluyendo tres fallos CVSS 8,8: un desbordamiento de entero (CVE-2026-6473), un problema de symlink en pg_basebackup (-6475) y una sobrescritura de stack libpq (-6477). Si aún no estás en la versión menor más reciente de tu versión, planifica para ello.[19]

Mirando hacia adelante: PG19 Beta 1 está fuera (final esperada sep/oct), y PostgreSQL 14 llega a fin de vida el 12/11/2026. Para el stack PG de Daniel, esto significa mover instancias 14 a una ruta de actualización durante el verano.[19][20]

  1. postgresql.org — 11 CVEs, PG14 EOL el 12/11/2026.
  2. releasebot.io — PG19 Beta 1.

Microsoft construye una verdadera distro de servidor: Azure Linux 4.0

Presentado en el OSS Summit NA (mediados de mayo): Azure Linux 4.0 (Preview) — una distribución derivada de Fedora, basada en RPM, que Microsoft construye y mantiene para VMs Azure. Lo notable es la "primera vez": es la primera distro Linux soportada de Microsoft más allá de alojamiento puro de contenedores.[21]

En paralelo, Azure Container Linux llegó a GA (anunciado a mediados de mayo en el OSS Summit; rollout más amplio el 02/06 en Build) — un host OS inmutable, optimizado para contenedores, basado en el proyecto Flatcar adquirido. Juntos, esto es la confianza Linux de Microsoft como plataforma, no solo como invitado.[22]

  1. infoq.com — Azure Linux 4.0, derivado de Fedora, primera distro de servidor.
  2. opensource.microsoft.com — Azure Container Linux GA, Flatcar.
