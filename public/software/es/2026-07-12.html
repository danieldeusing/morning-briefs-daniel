Dos paquetes de npm secuestrados: la cadena de suministro sigue siendo el frente

Un domingo tranquilo, pero no vacío: el hilo de noticias de esta semana es la cadena de suministro de software, y está bajo ataque activo. En pocos días secuestraron dos paquetes de npm: ayer jscrambler, con un infostealer en Rust, y antes de eso el SDK de Injective, con una puerta trasera que robaba claves de wallet. Esta es exactamente la clase de ataque que los nuevos valores predeterminados de npm v12 buscan detener, motivo para preguntarnos con honestidad qué detiene realmente este endurecimiento y qué solo desplaza. Además: una brecha silenciosa de downgrade en el driver JDBC de Postgres, justo en el stack de Daniel, y CodeQL suma una query contra la inyección de prompts.

  • Sec paquete npm jscrambler secuestrado (11/07): cinco lanzamientos maliciosos en ~3 horas, un infostealer en Rust para perfiles de navegador, la extensión de Bitwarden y Steam; en la 8.18.0 el dropper se movió del install hook a dist/index.js.
  • Sec SDK de Injective con puerta trasera (08/07): @injectivelabs/sdk-ts v1.20.21 robaba seed phrases y llaves privadas: 18 paquetes, ~50,000 descargas/semana, <1 hora activo, sin daños. Corregido en 1.20.23.
  • Infra npm v12: evaluación: habría bloqueado el primer hook de jscrambler, pero no el dropper en dist/ ni la puerta trasera de Injective en el código del SDK. Necesario, no suficiente.
  • Sec PostgreSQL JDBC 42.7.12: cierra el CVE-2026-54291: channelBinding=require se degradaba silenciosamente de SCRAM-PLUS a SCRAM simple. Afecta de la 42.7.4 a la 42.7.11.
  • OSS GitHub CodeQL 2.26.0 (10/07): Kotlin hasta 2.4.0, más una nueva query, js/system-prompt-injection, que detecta entradas no confiables que llegan a los prompts de sistema de una IA.
  • Infra Microsoft Foundry hosted agents (GA): sandbox aislado por hipervisor en cada sesión, agnóstico de framework, scale-to-zero: la historia es el aislamiento, no los agentes.
  • Sec CVE de Langflow, en contexto: el NVD lo califica en 8.4 (alto), el fabricante en 9.9 (crítico), y aun así se explota activamente desde el 25/06. El KEV le gana al puntaje CVSS.

Paquete npm jscrambler secuestrado: cinco lanzamientos maliciosos en tres horas

El golpe más reciente llegó ayer, sábado 11 de julio: el paquete de npm jscrambler fue comprometido. Según la prensa de seguridad (The Hacker News, basándose en datos de Socket), la versión publicada el 11/07, 8.14.0, traía un hook preinstall que deja y ejecuta un binario nativo, un build para Windows, macOS y Linux. En cerca de tres horas, el atacante publicó cinco lanzamientos maliciosos (8.14.0/8.16.0/8.17.0/8.18.0/8.20.0). La carga útil: un infostealer escrito en Rust que roba perfiles de Chrome/Brave/Edge/Chromium, la extensión de navegador de Bitwarden y sesiones de Steam.

Infostealer: Un programa cuyo único trabajo es copiar en silencio contraseñas guardadas, cookies del navegador, wallets de cripto y tokens de sesión, y enviarlos al atacante. Como un ladrón que no rompe nada: solo fotografía cada llave y contraseña que encuentra por ahí, y se va.

El detalle instructivo: En la 8.18.0, el atacante movió el dropper del hook preinstall directamente a los archivos entregados dist/index.js y dist/bin/jscrambler.js: a propósito, para engañar a los escáneres que solo revisan los scripts de instalación. Con eso, el malware ya no se ejecuta al instalar, sino al importar o ejecutar el paquete. Este mismo truco será importante en un momento, cuando evaluemos npm v12. (La lista de lanzamientos y el movimiento del dropper vienen de resúmenes de la prensa de seguridad; las páginas de detalle de Socket y THN bloquearon el acceso directo, así que hay que leerlos como información secundaria, no telemetría primaria.)

SDK de Injective con puerta trasera: llaves de wallet de 18 paquetes

Pocos días antes, el miércoles 8 de julio, la misma clase de ataque con un objetivo más robusto. @injectivelabs/sdk-ts, el SDK en TypeScript para la blockchain de Injective, recibió una puerta trasera a través de una cuenta de desarrollador de GitHub comprometida. La versión 1.20.21 capturaba en silencio frases de recuperación de wallet (seed phrases) y llaves privadas. Alcance según BleepingComputer y StepSecurity: 18 paquetes afectados, alrededor de 50,000 descargas por semana, activo menos de una hora, sin daños. Corregido en la 1.20.23: quien use el SDK de Injective debe actualizar.

Seed phrase: Las 12 a 24 palabras que funcionan como llave maestra de una wallet de cripto: quien las tiene, es dueño de la wallet, sin necesidad de contraseña. Como la llave maestra de una caja fuerte que por casualidad cabe en una postal: una vez robada, todo lo que hay dentro queda expuesto para siempre.

El contraste con jscrambler es el punto clave: dos puntos de entrada distintos, el mismo resultado. jscrambler llegó por una cuenta de publicación secuestrada, con carga en el script de instalación y en el código; Injective, por una cuenta de GitHub secuestrada que subió la puerta trasera directamente al código fuente del SDK. Ambos evaden la máxima simple de "solo no ejecutes scripts de instalación": el código malicioso está en el propio código, que de todas formas se va a ejecutar.

Fuentes: thehackernews.com · bleepingcomputer.com

Lo que npm v12 detiene, y lo que no

npm v12 activa tres valores predeterminados seguros este mes: nada de scripts preinstall/install/postinstall de dependencias, nada de dependencias de git y nada de dependencias con URL remota sin autorización explícita (changelog de GitHub). La mecánica se explicó a fondo en el brief del 10/07; la pregunta fresca y honesta de hoy es: ¿esto habría detenido los dos casos de este fin de semana?

Install hook → npm v12 bloquea · Código en dist/ → sigue corriendo

En parte. Los primeros lanzamientos de jscrambler usaban un hook preinstall, que npm v12 habría bloqueado por defecto. Pero en la 8.18.0 el dropper se movió a dist/index.js, es decir, código que corre al importar el paquete: ahí npm v12 no hace nada. Y la puerta trasera de Injective estuvo desde el inicio en el código fuente del SDK, ejecutándose en tiempo de ejecución, tal como estaba previsto: también intacta ante el bloqueo de scripts de instalación.

Conclusión, con postura: npm v12 sube el piso y elimina el ataque más barato, el drive-by por script de instalación. Pero la carga útil simplemente se muda a código de importación y de tiempo de ejecución. Necesario, no suficiente. Las verdaderas líneas de defensa siguen siendo las mismas: fijar versiones, lockfile más hashes de integridad, verificar procedencia/attestation, y no actualizar a ciegas solo porque hay una versión nueva.

PostgreSQL JDBC 42.7.12: downgrade silencioso de channel binding (CVE-2026-54291)

Justo en el stack de Daniel (Java/Spring sobre Postgres): el release de seguridad 42.7.12 del driver JDBC de PostgreSQL (publicado el 29 de junio, reportado en las noticias de postgresql.org el 6 de julio) cierra el CVE-2026-54291. El núcleo del problema: las conexiones con la configuración channelBinding=require podían degradarse en silencio de SCRAM-SHA-256-PLUS (con channel binding) a SCRAM-SHA-256 simple (sin él), anulando justo la protección que se había pedido de forma explícita. Las versiones afectadas van de la 42.7.4 a la 42.7.11; la severidad se clasifica como media.

Channel binding: Cuando tu app inicia sesión en Postgres por TLS, el channel binding ata el handshake de contraseña exactamente a esa conexión cifrada; un atacante que se coloque en medio y reenvíe tu login a otro servidor no puede reutilizarlo. Como una firma que solo vale para este sobre sellado, no para una copia. Un "downgrade silencioso" significa que el cliente cree haber recibido la protección, pero nunca la tuvo.

Quien haya configurado channelBinding=require contra ataques MITM debería pasar a la 42.7.12; de lo contrario, el endurecimiento pudo haber corrido en vano. Del ecosistema de Postgres, esta semana también salieron CloudNativePG 1.30.0 (nuevo recurso DatabaseRole), pg_ivm 1.15 y Barman 3.19.0/3.19.1: no son correcciones de seguridad, pero valen la pena para quien corre Postgres en Kubernetes.

GitHub CodeQL 2.26.0: Kotlin 2.4.0 y una query contra la inyección de prompts

CodeQL es el motor de análisis estático de código de GitHub: la máquina detrás de Code Scanning. La versión 2.26.0 (10 de julio) ahora soporta Kotlin hasta la 2.4.0 y trae, la parte interesante para una herramienta de seguridad, una nueva query, js/system-prompt-injection. Detecta cuando una entrada de usuario no confiable llega al system prompt de un modelo de IA, y suma nuevos sinks de inyección de prompts para los SDKs de OpenAI, Anthropic y Google GenAI. Esto pertenece claramente al brief de software: es una función de escaneo, no el lanzamiento de un modelo.

Sink de inyección de prompts: Si tu app mete entradas de usuario en las instrucciones que le da a un modelo de IA, un usuario malicioso puede colar comandos que secuestren al modelo ("ignora lo anterior, haz X"). Un "sink" de CodeQL es un punto final conocido como peligroso que el escáner vigila; la nueva query rastrea si datos no confiables llegan hasta el sink del system prompt, la misma idea de rastreo de taint que se usa en la inyección SQL, aplicada a llamadas a LLM.

La lectura: entre más aplicaciones conectan LLMs a su backend, más se convierte la inyección de prompts en una clase de AppSec como XSS o SQLi, y ahora las herramientas de SAST convencionales empiezan a modelarla. Para quien ya usa code scanning, esto llega gratis con la actualización de CodeQL.

Fuente: github.blog

Microsoft Foundry: sandboxes aislados por hipervisor para agentes (GA)

Como nota de infraestructura, bien acotada: los "hosted agents" de Microsoft Foundry llegaron a disponibilidad general en julio (byteiota.com). Cada sesión recibe su propio sandbox aislado por hipervisor: cómputo, memoria y sistema de archivos dedicados, y esto de forma agnóstica al framework: LangGraph, Claude Agent SDK, OpenAI Agents SDK o un Dockerfile propio, todos corren. Precio: $0.0994 por hora de vCPU más $0.0118 por hora de GiB, con scale-to-zero cuando está inactivo.

El ángulo relevante para software es a propósito no "qué framework de agentes" (eso es territorio de ai-dev/ai-usecases, ahí están los detalles), sino el modelo de aislamiento: quien ejecuta código no confiable generado por un LLM necesita una frontera de hipervisor por sesión, el mismo problema que gVisor o Firecracker resuelven para el cómputo multiquilino. Que las grandes nubes ahora entreguen esto como valor predeterminado gestionado es la verdadera historia de infraestructura.

Eso lleva a la pregunta honesta sobre npm versión doce, que este mes busca contener justo este tipo de ataque bloqueando por defecto los scripts de instalación y las dependencias de git y remotas. La mecánica ya se explicó en el brief del diez de julio; hoy toca la evaluación: ¿esto habría detenido los dos casos?

Del propio stack de Daniel viene una brecha silenciosa. Quien haya configurado esta protección contra ataques de intermediario debería actualizar, o el endurecimiento pudo haber corrido en vano. La inyección de prompts se está convirtiendo poco a poco en una clase de seguridad como la inyección SQL, y las herramientas estándar del mercado empiezan a modelarla.
