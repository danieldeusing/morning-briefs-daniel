Segurança & cadeia de suprimentos

Cisco SD-WAN CVE-2026-20182 — CVSS 10,0, em ação

Raro o suficiente para a CISA elevar um bypass de autenticação em edge com score CVSS máximo ao catálogo KEV — e este vem com atribuição explícita. CVE-2026-20182 em Cisco Catalyst SD-WAN Controller e Manager permite que um atacante remoto não autenticado obtenha privilégios de administrador. A própria Cisco atribui a exploração "com alta confiança" ao cluster UAT-8616 — o mesmo grupo que weaponizou o CVE-2026-20127. A CISA adicionou o CVE ao KEV em 14/05; o prazo de patch FCEB foi 17/05 [1][2].

O stack do Daniel não é diretamente afetado — mas clientes DACH com edges Cisco SD-WAN, com certeza. A lição topológica é mais interessante: controladores SD-WAN ficam frequentemente em VLAN de gerenciamento com segmentação fraca em relação à produção. Se o controlador cai, no pior caso leva consigo o controle de rotas do mesh completo do site. CVE-2026-20133, -20128, -20122 do batch KEV de abril estão no mesmo caminho — os atacantes estão encadeando [3].

pgvector 0.8.2 — integer underflow no HNSW paralelo

O pgvector deixou de ser hobby de fim de semana para virar o índice vetorial padrão em stacks Postgres — quem hospeda RAG por conta própria tem ele no cluster. CVE-2026-3172 está no build paralelo do índice HNSW: um integer underflow produz uma leitura fora dos limites na memória do backend. Um usuário de banco autenticado pode usar isso para vazar dados de outras relations, ou derrubar o servidor. CVSS 8,1. Afetadas 0.6.0–0.8.1, corrigido em 0.8.2.

HNSW (Hierarchical Navigable Small Worlds): imagine procurar uma palavra em um livro. Em vez de ler cada página, você tem um índice que leva você grosseiramente à seção certa — e ali, placas locais até palavras vizinhas. HNSW é exatamente isso para busca vetorial: várias camadas, a do topo grossa (poucos nós, saltos longos), a do fundo fina. Resultado: vizinho mais próximo aproximado em milissegundos em vez de segundos. A variante paralela constrói o índice em vários workers — e no pgvector 0.6+ a aritmética de tamanho passava por um inteiro sem sinal que estourava para negativo.

Na prática: qualquer cluster Postgres com pgvector ativo e usuários DB multi-tenant (ou seja, o padrão para SaaS) está exposto. Atualizar para 0.8.2 é tarefa desta semana; cross-check feito contra NVD [1][2].

Spring AI 1.0.7 / 1.1.6 / 2.0.0-M6 — três CVEs de memória de uma vez

Se alguém usa Spring AI a sério em caminho produtivo, a leva de releases de 08/05 é leitura obrigatória. Três CVEs fecham brechas que surgem estruturalmente em agentes de chat baseados em memória quando a API é usada com pouco cuidado. CVE-2026-41712: o `DEFAULT_CONVERSATION_ID` padrão fazia com que a memória de chat sem escopo explícito de usuário vazasse entre sessões — o clássico "usuário A vê a resposta do usuário B" em stacks de PoC. CVE-2026-41713: um atacante consegue elaborar prompts que aterrissam na conversation memory e são interpretados em um turno posterior como "instrução" — memory poisoning. CVE-2026-41705: filter injection no conector Milvus vector store.

A correção não é só um upgrade — a API PromptChatMemoryAdvisor foi deprecada; o sucessor força um parâmetro explícito de conversation ID. Quem tem código Spring AI parado do Q1 2026 conserta não só o build, mas o threat model. Spring AI 2.0 está anunciado para 28/05 [1][2][3].

Nota sobre threat model Memory poisoning não é bug de código, é premissa de arquitetura. Quem usa chat memory como storage para "o que o usuário disse" está tratando input do usuário como system prompt confiável. Isso quebra sistematicamente assim que um tool-use no mesmo turno lê a memória. A nova API impõe o escopo por conversation ID — isso resolve o vazamento cross-user. O problema de instrução-vinda-da-memória continua sendo uma pergunta de arquitetura.

Firefox 151 — Container Style Queries cross-browser-stable

A Mozilla lança o Firefox 151 como estável hoje, e o item mais importante não está na linha de marketing do topo: Container Style Queries agora estão estáveis em um terceiro navegador principal. O que isso significa? Um componente pode reagir ao valor de uma custom property no seu container — não apenas ao seu tamanho. Até agora, theming tinha de ser modelado via seletor de ancestral (".dark .card"). A partir de hoje, "o container tem `--theme: dark` setado" basta — theme switching sem JS, sem toggle de classe, local por subtree. Além disso: `CSSContainerRule.conditions` como nova property, várias conditions por rule.

Mais três itens que importam em produção: o Document Picture-in-Picture API (conteúdo DOM em janela flutuante always-on-top — o Chrome tem desde ~24, Firefox está alcançando), Fullscreen Keyboard Lock (jogos no browser capturam Ctrl-W), e backups locais de perfil em Linux e macOS com restore cross-platform. O Firefox 151 é também o primeiro build estável que, segundo a Mozilla, leva a preparação para o novo "Firefox Home" — o redesign maior de UI chega com o 152 [1][2].

TC39 114 em Amsterdam — Temporal aterrissa de vez

Três dias de plenary TC39 em Amsterdam começando hoje, sediados pela JetBrains. O maior movimento aconteceu antes do meeting: Temporal avançou para Stage 4 no meeting 113 de março — significa que aterrissa no ECMAScript 2026 e implementações podem ir para produção. Avanços de stage concretos do meeting 114 atual serão publicados nas notes nos próximos dias [1].

Temporal: o objeto Date do JavaScript é uma das peças mais notórias da biblioteca padrão da linguagem — meses numerados a partir de 0, mutações em vez de imutabilidade, tratamento sujo de timezone. O Temporal substitui isso: valores imutáveis, separação limpa entre Instant (ponto no tempo), PlainDate (dia de calendário) e ZonedDateTime (ponto no tempo + timezone). Código JavaScript agora escreve `Temporal.PlainDate.from('2026-05-19').dayOfWeek` em vez de `new Date('2026-05-19').getDay() + 1`.

M&A & política OSS

Anthropic ↔ Stainless — a oficina de SDKs cai

A Anthropic confirmou oficialmente o deal Stainless por mais de US$ 300 mi em 18/05. A Stainless é a pequena loja de NYC (fundada em 2022 pelo ex-Stripe Alex Rattray) que, há três anos, gera automaticamente e mantém os SDKs oficiais de OpenAI, Google, Cloudflare e Runway a partir das specs OpenAPI deles — Python, TypeScript, Go, Java. A Anthropic está desligando os produtos Stainless hospedados e mantendo a tecnologia internamente. Clientes existentes mantêm os SDKs já gerados e direitos plenos de modificação.

Este é o movimento de plataforma mais agudo no mundo IA em maio de 2026 — e não é um que vire manchete de "novo modelo". Leitura: a Anthropic não está comprando a concorrência; está comprando a infraestrutura compartilhada da concorrência. OpenAI e Google têm, a partir de hoje, um buraco interno de tooling na pipeline de SDK — três a seis meses até soluções de substituição estarem prontas. Para consumidores de SDK (ou seja, todos os devs) a camada vira lentamente mais fragmentada, não mais uniforme [2].

Efeito de SDK lock-in A Stainless gera SDKs a partir de specs OpenAPI — as specs continuam pertencendo aos respectivos fornecedores, então OpenAI/Google permanecem totalmente consumíveis no nível de API. O que falta é a camada de SDK cross-language com atualização automática. Se construírem por conta própria, leva meses; se terceirizarem para Speakeasy / orval / openapi-generator, perdem o acoplamento estreito a features proprietárias da API (convenções de streaming, encoding de tool-call etc.) que a Stainless modelava no caminho.

Comentário — a camada abaixo do modelo

Três movimentos de maio mostram o mesmo padrão: a concorrência não está mais jogando só na camada de modelo. Primeiro: a Anthropic compra a oficina de geração de SDKs da concorrência — a camada que ninguém tem no slide de conferência porque não produz benchmark score. Segundo: o Spring AI abandona seu default de API de memória no mesmo fôlego, porque a camada de fluxo de dados abaixo da chamada LLM é a camada real de risco, não o modelo em si. Terceiro: o Google I/O hoje à noite vai falar muito sobre Gemini, mas o slide comercialmente decisivo é o do logo MCP — quem controla o tool schema controla o que um agente pode fazer, não só o que ele diz.

O que isso significa para o stack-DNA do Daniel? Stacks Java-Spring-Postgres-Caddy têm, na onda IA até aqui, a virtude de que o modelo continua sendo um serviço chamado externamente, não um componente arquitetural. Quem está construindo Spring AI dentro do stack tem desde ontem uma nova API para aprender, três CVEs para revisar, e um threat model que não estava ali antes. Caminho pragmático: chamadas HTTP para o provedor IA, estado de conversa explicitamente no próprio schema Postgres, sem magia de memory-advisor. Isso não é "atrasado" — é a arquitetura que sobrevive a uma auditoria.
