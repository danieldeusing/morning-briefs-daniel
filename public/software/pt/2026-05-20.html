Mini Shai-Hulud — o worm que forja assinaturas legítimas

Esta é a história mais consequente da semana, e ela não faz keynote nenhuma. Em 11/05 (entre 19h20 e 19h26 UTC), um worm autorreplicante publicou 84 artefatos maliciosos em 42 pacotes npm @tanstack/* — publicados pela própria pipeline de release legítima da TanStack, sob sua identidade OIDC confiável, depois que o código do atacante sequestrou o runner no meio do workflow. Em 48 horas, a campanha cresceu para 172 pacotes em 403 versões no npm e no PyPI — entre eles Mistral, Guardrails AI, UiPath, OpenSearch, Bitwarden CLI, SAP. CVE-2026-45321, CVSS 9,6, confirmado pela NVD [1][2].

Pwn Request
pull_request_target → Cache do Actions
envenenado → Token OIDC
da RAM do runner → Publish com provenance
SLSA-L3 válida → Autopropagação
via repos das vítimas
Provenance SLSA: Como uma nota de entrega lacrada que prova criptograficamente que um pacote foi mesmo construído pela pipeline de CI legítima — a resposta de praxe do setor a ataques à cadeia de suprimentos. O assustador: o worm não quebrou a assinatura, ele sequestrou a própria pipeline, de modo que ela assinou o malware. O Sigstore verificou o processo de build corretamente — o que o SLSA nunca garantiu é que o código construído fosse inofensivo. Primeiro ataque a npm com provenance SLSA Build L3 válida.

O worm rouba PATs do GitHub, tokens de npm, chaves SSH, credenciais de cloud e K8s, além de arquivos .env, persiste via hooks do VS Code e do Claude Code e se propaga sozinho pelos repos das vítimas. Atribuído ao grupo "TeamPCP", que teria comprometido o scanner Trivy em março e a Bitwarden CLI em abril. Consequência: a TanStack tira a lição e estuda passar a aceitar PRs apenas por convite daqui pra frente — um estreitamento drástico da colaboração que mostra o quanto o incidente pesou [3][4].

O que checar hoje O stack do Daniel está exposto (GitHub Actions + npm + Claude Code). Três checagens: (1) dar grep nos workflows atrás de pull_request_target com checkout de código de PR e fixar as Actions por SHA em vez de por tag. (2) revisar os hooks de ~/.claude e do VS Code atrás de entradas desconhecidas. (3) rodar npm audit signatures + cruzar os lockfiles contra as versões comprometidas, e depois rotacionar PATs/tokens de npm caso workflows tenham rodado dentro da janela. A verificação de SLSA sozinha não teria protegido aqui.

Linux „Copy Fail" — isolamento de containers quebra, prazo já foi

Direto no caminho Docker/Linux do Daniel: o CVE-2026-31431 („Copy Fail", CVSS 7,8) fica no template de cripto authencesn do kernel — um usuário sem privilégios pode escrever quatro bytes controlados no page cache de qualquer arquivo legível e escalar isso até root. O inquietante é a trivialidade: um script Python de 732 bytes, rodando sem mudança em toda distribuição relevante desde 2017. O bug tem nove anos. Segundo a Theori, ele também funciona como primitiva de escape de container e atinge nós K8s, porque o page cache é compartilhado por todo o host. No KEV desde 01/05, o prazo FCEB (15/05) já passou. Correção no kernel 6.18.22 / 6.19.12 / 7.0 [1][2].

Exchange CVE-2026-42897 — e a disputa de score

A CISA incluiu em 14/05 um XSS de Exchange on-prem no catálogo KEV (prazo FCEB 29/05): por meio de um e-mail preparado, dá pra executar JavaScript arbitrário no contexto do browser do OWA. O interessante é o conflito de avaliação: a NVD classifica como 6,1 MÉDIO (Scope changed, C:L/I:L), e a própria Microsoft como 8,1 ALTO (C:H/I:H). Citar os dois valores, não afirmar nenhum como „o" score — exatamente o cross-check que a disciplina de pesquisa exige. Afetados o Exchange 2016/2019; Exchange on-prem ainda é numeroso no middle market da região DACH [1].

Trimestre do PostgreSQL — 11 CVEs, uma RCE de verdade

A atualização trimestral do PostgreSQL de 14/05 (18.4 / 17.10 / 16.14 / 15.18 / 14.23) fechou 11 CVEs e mais de 60 bugs. O destaque é de verdade: CVE-2026-6637 (CVSS 8,8) é um stack buffer overflow no módulo refint que permite execução de código. Além de duas coisas de injection/memória: CVE-2026-6476 (SQL injection no pg_createsubscriber via nome da subscription, CVSS 7,2) e CVE-2026-6477 (libpq: um servidor malicioso sobrescreve memória do stack do cliente). Atualização sem dump/reload, basta trocar os binários. O PG 14 chega ao fim de vida em 12/11/2026 [1].

OpenJDK — JDK 27 EA e pós-quântico para TLS

A linha ativa de Early-Access é o JDK 27 (build ~17); o RDP1 (feature freeze) faz fork em 04/06, com GA em setembro. Única JEP oficialmente no alvo até agora: a JEP 527 (troca de chaves híbrida pós-quântica para TLS 1.3) — três esquemas híbridos (X25519MLKEM768, SecP256r1MLKEM768, SecP384r1MLKEM1024), ativos desde o build 6 do EA. Baseia-se na JEP 496 (ML-KEM) do JDK 24 [1][2].

Troca de chaves híbrida pós-quântica: No handshake TLS, browser e servidor combinam uma chave secreta de sessão. "Híbrida" quer dizer: eles usam dois métodos ao mesmo tempo — um clássico e um resistente a quântico. Mesmo que um futuro computador quântico quebre um deles, a conexão segue segura. O mesmo motivo do tema OpenSSF/Sigstore lá embaixo — o setor está endurecendo sua criptografia antes de a ameaça quântica virar real.

CRA — obrigações de reporte a partir de setembro, retroativas

As obrigações de reporte do Cyber Resilience Act entram em vigor a partir de 11/09/2026: falhas ativamente exploradas e incidentes graves precisam ser reportados — alerta inicial em até 24h, notificação completa em até 72h, relatório final no máximo 14 dias após a correção estar disponível. O detalhe para o Daniel e para qualquer um que entregue software: a obrigação vale retroativamente para todo produto colocado no mercado antes de 11/12/2027 (Art. 69(3)). Comentário — confiança é a cadeia de suprimentos

O worm Shai-Hulud é tão desagradável justamente porque não explorou uma brecha — ele forjou o próprio comprovante de confiança: provenance SLSA válida, o recibo criptográfico que cinco anos de supply-chain security venderam como resposta. A lição não é "o SLSA está quebrado", é mais sutil: o SLSA prova que a pipeline construiu, nunca o que ela construiu. Quem sequestra a pipeline herda a confiança dela. É exatamente por isso que o endurecimento quântico do Sigstore (amanhã, no OpenSSF) e a JEP 527 no JDK são só metade da batalha — eles protegem a assinatura, não o contexto de build ao redor.

Para o DNA do stack do Daniel — GitHub Actions, npm, Java/Spring, Postgres, Caddy/Tailscale, Claude Code — isso significa, na prática: aplicar ao CI o mesmo princípio de menor privilégio de todo o resto. Banir pull_request_target dos workflows ou cercá-lo com rigor, fixar Actions por SHA, versionar e revisar os diretórios de hooks, manter tokens de vida curta. Digno de nota: o worm persistiu via os mesmos hooks de tooling de agentes (VS Code, Claude Code) que deveriam trazer produtividade — a camada de tooling é também vetor de persistência. (Os próprios lançamentos de agentes de IA desta semana — Google I/O — pertencem tematicamente ao brief de ai-dev; aqui só conta o lado de superfície de ataque deles.) Três pontos de controle pra hoje: não perder a janela do Drupal hoje à noite, dar grep nos gatilhos dos workflows atrás de pull_request_target, e checar se os clusters Postgres estão no nível de patch de 14/05. Amanhã o OpenSSF Community Day entrega a sessão de Sigstore quântico e o TC39 suas notas de encerramento.
