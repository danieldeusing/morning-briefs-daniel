Docker — três caminhos do contêiner ao root do host via docker cp

Este é o patch obrigatório da semana, e ele fica bem no stack do Daniel. O Docker Engine 29.5.1 (18/05) fecha três falhas que compartilham um núcleo: um contêiner malicioso pode escapar durante o aparentemente inofensivo docker cp e alcançar o sistema de arquivos do host, porque a lógica de cópia roda com privilégios de root do host. CVE-2026-41567: os binários de descompressão (xz, unpigz) são resolvidos pelo PATH dentro do contêiner — um binário plantado então roda como root do host. CVE-2026-42306: uma corrida TOCTOU redireciona um bind mount para um caminho arbitrário no host. CVE-2026-41568: uma troca de symlink TOCTOU cria arquivos/diretórios arbitrários no host. A Docker classifica todas as três como "critical"; os registros GHSA correspondentes atribuem High/High/Medium [1].

TOCTOU (time-of-check / time-of-use): Como o segurança conferindo sua identidade, você trocando-a por uma falsa num piscar de olhos, e o carimbo indo para a carteirinha errada. O programa verifica um caminho, o atacante o altera na fração de segundo entre as etapas, e a ação privilegiada atinge o alvo plantado — aqui, um caminho do host em vez do caminho esperado do contêiner.

O 29.5.2 (20/05) é o complemento: o 29.5.1 introduziu uma regressão na qual o docker cp com bind mounts atravessando symlinks dentro do contêiner (/var/run → /run) falhava com "mkdirat: file exists". Então, se você já pulou para o 29.5.1, pegue o 29.5.2 já [1].

O que fazer hoje Verifique a versão da engine (docker version) e suba para a 29.5.2. O ataque exige um contêiner comprometido ou hostil que intercepte um docker cp — quem roda imagens de terceiros ou contêineres de CI disparados por PR é o alvo principal. Até corrigir, não rode docker cp contra contêineres não confiáveis.

OpenJDK — JDK 27 se consolida, três JEPs perto do target

A linha Early-Access está no build 22 (de ~17 na quarta). Três JEPs foram a Proposed to Target, com a janela de revisão encerrando em 19/05: JEP 523 (G1 como GC padrão em todos os ambientes), JEP 534 (Compact Object Headers por padrão — a promoção do JEP 519 do JDK 25) e JEP 537 (Vector API, 12ª incubadora). Novo em Candidate: JEP 538 (encodings PEM de objetos criptográficos). O RDP1 (feature freeze) bifurca em 04/06, GA em setembro. Aviso de honestidade: que os três JEPs passaram de Proposed to Target para de fato Targeted não foi possível confirmar de forma final na página primária do openjdk.org (que retorna 403) — janela de revisão fechada, targeting esperado [1][2].

Compact Object Headers: Todo objeto Java carrega um pequeno header de controle. O JDK 25 o encolheu de 12–16 para 8 bytes — como dar a cada caixa do armazém uma etiqueta menor: em milhões de objetos isso libera heap de verdade e assim corta pausas de GC (~15% menos GCs no SPECjbb2015). O JEP 534 torna isso o padrão no JDK 27 em vez de uma flag opt-in.

TC39 #114 — using agora é ECMAScript

O plenário de Amsterdã (hospedado pela JetBrains) terminou na quinta; as mudanças de stage estão no repositório tc39/proposals, com as notas narrativas ainda por vir. Destaque: Explicit Resource Management — as declarações using e await using que muitos conhecem do TypeScript — atingiu o Stage 4 e agora faz parte do ECMAScript 2026. Também no Stage 4: Atomics.pause (uma dica de spin-loop para memória compartilhada) e Joint Iteration (Iterator.zip). Avançaram para o Stage 3, entre outros, iterator chunking/includes/join e um accessor padronizado de error.stack. Decorators foi só de 2 para 2.7 — a velha ressalva se mantém: nenhuma das grandes engines quer ser a primeira a enviar, então o cronograma fica frouxo [1][2].

using / Explicit Resource Management: Como o try-with-resources do Java, mas para JavaScript: escreva using file = openFile() e o recurso é fechado automaticamente quando o bloco termina — seja saindo normalmente ou por exceção. Acabaram os blocos finally { close() } esquecidos.

Comentário — as fronteiras em que confiamos

Três das principais histórias de hoje são a mesma história em três camadas. O contêiner não deveria enxergar o host — o docker cp e o bypass de runAsNonRoot mostram que a parede de isolamento é mais fina do que o modelo mental. O ORM deveria tornar a SQL injection impossível — a falha do Drupal mostra que "só no Postgres" basta para derrubar essa premissa. E a provenance SLSA deveria provar a confiança da cadeia de suprimentos — o Shai-Hulud mostra há duas semanas que um build sequestrado herda essa confiança. O padrão: não é a cripto ou a feature que está quebrada, mas a fronteira onde paramos de pensar.

Para o stack do Daniel isso significa algo pragmático e administrável: suba o Docker para o 29.5.2 e não rode docker cp contra contêineres não confiáveis; se Postgres-Drupal roda em algum lugar, corrija hoje; siga de olho em lockfiles npm e comprometimentos de mantenedores. A boa notícia vem da mesma indústria que está trocando de pele: o using agora é ECMAScript, o JDK 27 torna os Compact Object Headers o padrão, o Node tem seu patch LTS. Na segunda a UE entrega a próxima peça de soberania com o CADA — e aí veremos se o parágrafo vazado sobre nuvem dos EUA vira texto legal de verdade [1].

  1. europarl.europa.eu/legislative-train

É sexta-feira, 22 de maio. O fio condutor de hoje é desconfortavelmente concreto: as fronteiras em que os desenvolvedores confiam estão todas vazando ao mesmo tempo — no isolamento de contêineres, no banco de dados e na cadeia de suprimentos.

A maior história fica bem no stack do Docker. O Docker Engine vinte e nove ponto cinco ponto um fecha três falhas que permitem a um contêiner malicioso escapar durante o aparentemente inofensivo comando de cópia docker cp e escalar para root do host — uma vez porque os programas de descompressão são resolvidos pelo caminho dentro do contêiner, e duas vezes por uma corrida de tempo que redireciona um ponto de montagem para o sistema de arquivos do host. A Docker classifica as três como críticas. Um andar abaixo, a containerd corrige no mesmo dia um bypass que deixa um contêiner que não deveria rodar como root fazer isso mesmo assim. O conselho de hoje: suba a engine para vinte e nove ponto cinco ponto dois, porque a primeira correção introduziu uma regressão — e não rode docker cp contra contêineres em que você não confia.

A segunda camada é o banco de dados. O release do Drupal anunciado na quarta chegou: uma SQL injection sem autenticação no core que atinge apenas instalações PostgreSQL. O próprio Drupal a classifica em vinte e três de vinte e cinco, enquanto a base de vulnerabilidades americana a chama só de média — e os dois valores merecem ser citados. E a partir de hoje, o Drupal diz que tentativas de exploração estão sendo observadas na prática. Se você roda Drupal sobre Postgres, corrija hoje.

A terceira camada é a cadeia de suprimentos. O verme autorreplicante do brief anterior, o Shai-Hulud, não esfriou — está escalando. Em dezenove de maio uma quinta onda rolou pelo ecossistema AntV: por meio de um mantenedor sequestrado, cerca de seiscentas e trinta e nove versões maliciosas em trezentos e vinte e três pacotes foram publicadas em aproximadamente meia hora. Pior, o código do verme agora é público, e imitadores são o novo risco.
