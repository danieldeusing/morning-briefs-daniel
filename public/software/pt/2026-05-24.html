O ataque à cadeia de suprimentos migra: do npm ao Composer e para dentro da CI

Domingo, e os feeds oficiais estão parados — a CISA KEV segue em 2026.05.22. Mas o tema da semana mudou de território da noite para o dia: o ataque à cadeia de suprimentos não é mais um problema de npm. Em dois dias, a mesma lógica atingiu o Composer/Packagist (quatro pacotes Laravel-Lang, cada tag Git reescrita) e o GitHub diretamente (a campanha "Megalodon" sequestrou 5.561 repositórios via workflows de CI injetados). Denominador comum: não uma nova versão maliciosa, e sim o encanamento de confiança por baixo — tags Git e identidades de bots de CI nas quais confiamos cegamente. Ao lado, a falha de PostgreSQL no Drupal escala de forma mensurável (Imperva: mais de 15.000 tentativas de ataque), e o lado dos releases está tão quieto que a notícia do dia é uma correção do nosso próprio brief: o Firefox 151 stable saiu lá em 19/05, e o Chrome está em 148, não 149.

  • Cadeia Laravel-Lang (22/05): o invasor reescreveu cada tag Git de quatro pacotes Composer (só o laravel-lang/lang tem 502 tags) → o Packagist ingeriu commits de um fork como releases legítimos. O payload roda a cada vendor/autoload.php e rouba credenciais de nuvem/CI.
  • Cadeia Megalodon (18/05): 5.718 commits maliciosos em 5.561 repositórios do GitHub em ~6 h, via identidades de bots de CI forjadas que injetaram workflows do GitHub Actions. Exfiltra segredos para um C2. Impacto real: o pacote npm @tiledesk/tiledesk-server.
  • Sec Drupal CVE-2026-9082: a Imperva relata >15.000 tentativas de ataque contra ~6.000 sites em 65 países (EUA 61,8%). Só PostgreSQL. Prazo KEV 27/05. Ainda majoritariamente reconhecimento.
  • Sec Defesa da cadeia: o npm Staged Publishing (GA 22/05) separa o publish da CI de uma aprovação humana com 2FA; os cooldowns do pip (desde a 26.1, abril) teriam barrado 8 de 10 ataques. As ferramentas endurecem — a mecânica continua a mesma.
  • Infra Levas do kernel Linux (23/05): sete releases stable/longterm (7.0.10 + seis ramos LTS). Ao fundo, a LPE Fragnesia (CVE-2026-46300, 7,8) da família "Dirty Frag" — patches das distros desde meados de maio.
  • Sec Dois 10.0: plugin LiteSpeed cPanel CVE-2026-48172 (explorado ativamente, root via toggle de Redis) e Cisco Secure Workload CVE-2026-20223 (REST sem autenticação, ainda não explorado). Ambos pontuados pelo CNA, avaliação NVD pendente.
  • Release Calmaria + correção: Node, Angular, Spring, Quarkus, Deno, Postgres inalterados. Firefox 151 stable está fora desde 19/05 (152 em beta), Chrome stable = 148 (149 ~02/06) — o brief de ontem errou nos dois.
  • Cloud-native OpenTelemetry é projeto graduado da CNCF desde 21/05 — o padrão de fato de observabilidade, 12.000+ contribuidores. O OpenTofu 1.12 entregou o recurso que o Terraform nunca entregou.
  • UE CADA: adoção pela Comissão segue prevista para seg 27/05 (pacote de soberania tecnológica: CADA + Chips Act 2 + estratégia de open source). O texto real — incluindo a cláusula da nuvem dos EUA — chega amanhã.
  • BR Decretos de Lula: a oposição protocolou vários PDLs na Câmara para suspender os dois decretos de plataformas (12.975/12.976) — disputa sobre o novo poder de fiscalização da ANPD. O CGI.br apoia a autoridade.
  • Quieto Shai-Hulud: nenhuma nova onda na janela. NGINX "Rift" (CVE-2026-42945) ainda explorado, ainda fora da KEV. As notas do TC39 #114 seguem inéditas. O Quarkus 3.36 Final ainda não saiu.

Laravel-Lang — quem é dono da tag Git é dono do pacote

A principal história do fim de semana não está no npm, e sim no Composer, o gerenciador de pacotes do mundo PHP/Laravel. Em 22/05, um único invasor sequestrou quatro pacotes muito usados da organização Laravel-Lang — laravel-lang/lang, /http-statuses, /attributes e /actions. O truque: ele não publicou uma nova versão maliciosa que um mantenedor pudesse notar. Ele reescreveu cada tag Git já existente (só o carro-chefe laravel-lang/lang tem 502 tags), de modo que as tags apontassem para um commit em um fork fora da organização. As reescritas começaram às 22:32 UTC e terminaram por volta da meia-noite — cerca de 700 versões históricas nos quatro pacotes [1][2].

Reescrita de tag Git + Packagist: uma tag Git é um adesivo com um nome (por exemplo "v3.0") colado em um determinado estado do código. O Packagist (a registry do Composer) verifica a qual código uma tag está colada e publica isso como release. O invasor moveu os adesivos — o nome ficou, o conteúdo foi trocado. Como o rótulo "orgânico" que alguém descola e cola em outro pote: os dizeres estão certos, o conteúdo não.

O dano real está no payload: os commits adicionam um src/helpers.php ao autoload.files do Composer — a lista de arquivos carregados automaticamente a cada vendor/autoload.php, ou seja, a cada inicialização de uma app Laravel/Symfony. O ladrão de credenciais pega metadados EC2 (169.254.169.254), tokens do Kubernetes, HashiCorp Vault, segredos do Jenkins e senhas do navegador. Quem puxou algum desses pacotes em ou após 22/05 22:32 UTC via composer install deve tratar como comprometido todo segredo alcançável e rotacioná-lo [3][4].

Megalodon — 5.561 repositórios em seis horas, direto pela CI

Se o Laravel-Lang mostra que o Composer também é atacável, o Megalodon vai um passo além: a campanha atacou o próprio GitHub, na camada dos pipelines de CI/CD. Em 18/05, numa janela de seis horas (11:36–17:48 UTC), um invasor automatizado enviou 5.718 commits maliciosos para 5.561 repositórios. Tudo disfarçado por identidades de bot forjadas como build-bot, ci-bot ou pipeline-bot — nomes que ninguém olha duas vezes num histórico de commits [1][2].

Injeção de workflow de CI/CD: um pipeline de CI é o robô que, a cada push, compila, testa e faz deploy automaticamente — com acesso a todos os segredos do projeto. Quem contrabandeia um arquivo de workflow malicioso faz esse robô entregar as chaves no próximo push. É como passar sorrateiramente uma segunda tarefa ao zelador que tem a chave-mestra do prédio.

Os workflows contêm bash em base64 que exfiltra segredos de CI, credenciais de nuvem, chaves SSH e tokens OIDC para um servidor C2. Duas variantes: a de massa adiciona um workflow que dispara a cada push; a direcionada substitui workflows existentes por gatilhos workflow_dispatch — backdoors adormecidas, acionadas depois via API. Impacto real: o pacote npm @tiledesk/tiledesk-server (2.18.6–2.18.12) propagou a variante direcionada porque o mantenedor publicou de um ambiente comprometido [3].

SQLi do Drupal — de "exploração confirmada" para mensurável

A injeção de SQL só-PostgreSQL do início da semana (CVE-2026-9082, SA-CORE-2026-004) agora tem números. Órgãos FCEB precisam corrigir até 27/05; boa regra de ouro para todos os demais. Quem roda Postgres-Drupal não tem nada a decidir aqui — só a fazer [3].

A cadeia de suprimentos reage — npm, pip, proveniência

O Staged Publishing do npm segue a maior alavanca

O Staged Publishing do npm (GA desde 22/05) separa o publish da CI de uma aprovação humana com 2FA — um tarball publicado por pipeline cai primeiro numa fila, antes de ficar instalável. Justamente a brecha que Megalodon e Shai-Hulud exploram (pipeline sequestrado → pacote no ar na hora) ganha um ponto de controle humano. Além disso, os quatro interruptores --allow-* (npm 11.15.0) contra fontes fora da registry [1].

Cooldowns do pip — simples, grande efeito

Combina com isso, vinda do campo Python e disponível desde o pip 26.1 (fim de abril): os cooldowns de dependências. O parâmetro --uploaded-prior-to (por exemplo P5D) impede que versões recém-publicadas — e portanto potencialmente recém-sequestradas — entrem direto no build. Segundo a análise citada, um cooldown de 7 dias teria barrado 8 de 10 ataques analisados antes de chegar ao usuário final. Não é bala de prata, mas é barato [2][3].

Shai-Hulud — sem nova onda

O verme npm auto-replicante não emendou nova onda na janela; a última confirmada segue sendo o surto do AntV em 19/05. Vale notar apenas: a família agora forja proveniência Sigstore em tempo de execução — a camada de assinatura deixou de ser passe livre [4].

Kernel Linux — leva stable nova, e a Fragnesia por trás

No lado de infra, a notícia que toca o stack do próprio Daniel: em 23/05 saiu um conjunto inteiro de kernels stable e longterm — 7.0.10 como stable atual, mais os ramos longterm 6.18.33, 6.12.91, 6.6.141, 6.1.174, 5.15.208 e 5.10.257. É manutenção cumulativa de rotina, não release de emergência; quem roda ramos stable atualiza no ritmo normal [1].

Ao fundo, porém, corre a história mais séria: a Fragnesia (CVE-2026-46300, CVSS 7,8), uma escalada local de privilégios da família "Dirty Frag", descoberta por William Bowling (Zellic). Fica no subsistema XFRM / ESP-in-TCP e permite a um usuário local sem privilégios alterar arquivos somente-leitura — inclusive binários setuid — via manipulação do page cache, virando root. As distribuições corrigem desde meados de maio (AlmaLinux, Ubuntu/Canonical, CloudLinux); a mitigação rápida sem reboot é descarregar ou colocar em denylist os módulos esp4, esp6 e rxrpc onde não forem necessários [2][3].

LPE de page cache: o kernel mantém em RAM os arquivos muito lidos (page cache) para que nem todo acesso vá ao disco. A "Dirty Frag" & parentes enganam o kernel para alterar essa cópia em cache de um arquivo de sistema que deveria ser só leitura — por exemplo, um programa que roda com permissões de root. Não se falsifica o original no cofre, mas a cópia de trabalho na mesa, que todos usam.
Nota de pesquisa A leva stable de 23/05 (kernel.org) e a Fragnesia têm fontes separadas; que esses releases específicos citem a CVE-2026-46300 explicitamente no changelog não foi verificado individualmente aqui — as correções da Fragnesia vêm saindo pelos kernels das distros desde meados de maio.

Dois 10.0 e um caso teimoso do NGINX

Plugin LiteSpeed cPanel CVE-2026-48172 — explorado ativamente

Pontuação máxima, e já na prática: um usuário cPanel logado pode executar scripts como root via o recurso de habilitar/desabilitar Redis do LiteSpeed User-End cPanel Plugin — CVE-2026-48172, pontuado em CVSS 4.0 10.0 pelo CNA (MITRE) (avaliação NVD ainda pendente), publicado em 20/05. Versões anteriores à 2.4.5 são afetadas; correção na 2.4.5/2.4.7. Para o Daniel é puro valor de alerta (sem cPanel/LiteSpeed no stack), mas um padrão instrutivo: um suposto recurso de conveniência vira o caminho para root [1][2].

Cisco Secure Workload CVE-2026-20223 — 10.0, mas (ainda) não explorado

Também 10.0, perfil diferente: uma falha de API REST sem autenticação no Cisco Secure Workload permite a um invasor ler e configurar entre fronteiras de tenant como Site Admin. Correção na 3.10.8.3/4.0.3.17, SaaS já corrigido, sem workaround; a Cisco não relata exploração. Aqui o que conta é a classe, não o alarme [3].

NGINX "Rift" — ainda explorado, ainda fora da KEV

Cloud-native — OpenTelemetry se gradua, OpenTofu entrega

Em 21/05, no Observability Summit em Minneapolis, a CNCF anunciou a graduação do OpenTelemetry — o nível máximo de maturidade da Cloud Native Computing Foundation, após auditoria de segurança independente e revisão de governança. O OTel é agora oficialmente o padrão de fato para telemetria (traces, métricas, logs); a fundação cita 12.000+ contribuidores de 2.800+ empresas e a segunda maior velocidade de projeto depois do Kubernetes. Ao lado, no lado de IaC: o OpenTofu 1.12 (14/05), o fork do Terraform, trouxe o prevent_destroy dinâmico — enfim acoplável a uma variável, de modo que prod protege e dev não (um pedido no tracker do Terraform desde 2016), além de checksums de provider mais completos numa única passada de init. E o containerd corrigiu em quatro ramos em 20/05 (2.3.1/2.2.4/2.0.9/1.7.32) [2][3].

CADA — na segunda Bruxelas vai a sério

O Cloud and AI Development Act segue na pauta da reunião da Comissão para segunda-feira, 27/05, como parte do pacote de soberania tecnológica — agrupado com o Chips Act 2, uma estratégia de open source e um roteiro de "IA na energia". O objetivo é o mesmo: triplicar a capacidade de data centers da UE, regras de elegibilidade em toda a UE para provedores de nuvem, uma política unificada de aquisição de nuvem para a administração pública [1].

Soberania digital: a questão de quem pode puxar o plugue numa emergência. Se dados de governo ficam numa nuvem dos EUA, o provedor está sujeito à lei americana — um governo estrangeiro poderia exigir acesso ou cortar o serviço. "Soberania" significa manter dados críticos sob a sua própria jurisdição. Como um cofre no seu próprio banco em vez de um num país cujas leis você não ajuda a definir.

A questão sensível segue no rascunho vazado (CNBC, 07/05): cogita-se restringir o uso de nuvem dos EUA para dados sensíveis de governo (saúde, finanças, justiça) — explicitamente não uma proibição, não para o setor privado. O que é novo é o vento contrário diplomático: um telegrama do Departamento de Estado dos EUA de fevereiro instrui diplomatas a combater iniciativas estrangeiras de soberania de dados — então a adoção de segunda cai num ambiente acirrado. Quem atende contratos do setor público na UE lê o texto real na segunda [2].

Brasil — escala a disputa sobre a fiscalização da ANPD

Desdobramento dos decretos de plataformas de Lula, de 20/05: a oposição protocolou vários PDLs (Projetos de Decreto Legislativo) na Câmara para suspender os dois decretos 12.975 e 12.976. A acusação: eles impõem às plataformas deveres "inéditos" de monitoramento, remoção e mitigação de risco e ampliam os poderes da autoridade de proteção de dados ANPD sem base legal suficiente. Uma voz de peso em sentido contrário: Renata Mielli, coordenadora do CGI.br, chama o papel de fiscalização da ANPD de "coerente". Os decretos entram em vigor 60 dias após a publicação — ou seja, a briga política corre em paralelo com um prazo correndo [1].

PDL (Projeto de Decreto Legislativo): um instrumento do Congresso brasileiro que permite ao Parlamento derrubar um ato do Executivo (aqui: um decreto presidencial). Não é uma lei nova, e sim uma ferramenta de veto contra normas do governo. Como um conselho de administração que retoma, por maioria, uma decisão da diretoria.

Em paralelo, o próprio prazo de pessoal da ANPD segue correndo: até 31/05 é preciso autorizar o concurso para 200 vagas efetivas (Lei 15.352/2026), com edital projetado para ~24/09 e provas para o início de 2027. Não se ganha fiscalização só com decreto — é preciso também ter gente para isso [2].

Comentário — nunca foi um problema de npm

Quem leu as últimas semanas como "o npm tem um problema de segurança" corrige isso esta semana. O Laravel-Lang atingiu o Composer, o Megalodon atingiu o GitHub Actions — nenhum precisou de uma nova versão maliciosa. Eles sequestraram a camada de baixo: a tag Git em que o Packagist confia cegamente, e a identidade de bot de CI que ninguém verifica num histórico de commits. O padrão não é o ecossistema, e sim a premissa implícita de confiança — "esta tag aponta para o código que o mantenedor quis", "este commit veio do pipeline".

É exatamente aí que entram as boas notícias: o Staged Publishing coloca um humano entre o pipeline e o pacote no ar; os cooldowns compram tempo até um sequestro ser notado; e o fato de o Shai-Hulud agora forjar proveniência só significa que a assinatura também é uma medida, não uma prova. Para o stack do Daniel: corrigir o Postgres-Drupal hoje (prazo KEV 27/05). Verificar se os pipelines de CI checam a proveniência de tag/commit — e se um cooldown cabe na política de build. Atualizar o Linux stable, descarregar esp4/esp6 se não usados. Na segunda, Bruxelas entrega o próximo bloco de soberania com o CADA, e vamos ver se a cláusula vazada da nuvem dos EUA vira texto de lei de verdade [1].

É domingo, vinte e quatro de maio. Os feeds oficiais de segurança estão parados — o catálogo de falhas exploradas ativamente não tem entrada nova desde sexta. Mas o grande tema da semana mudou de território da noite para o dia.

Por semanas a fala era: o npm tem um problema de cadeia de suprimentos. Esta semana mostra que isso era estreito demais. Em dois dias, a mesma lógica atingiu dois lugares completamente diferentes.
