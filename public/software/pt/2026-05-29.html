Prazo Exchange-OWA hoje — ainda sem patch

O prazo KEV CISA para a lacuna Exchange on-prem CVE-2026-42897 expira hoje, sex 29/05. Microsoft confirmou exploração ativa em 14/05, CISA adicionou ao catálogo em 15/05 e deu 14 dias para mitigação — a janela fecha agora [1].

XSS em OWA: Outlook Web Access é o frontend web da Microsoft para Exchange. Cross-Site Scripting significa: o atacante contrabandeia código JavaScript estrangeiro na sua aba OWA autenticada, que depois roda sob seu login. Na prática: um email preparado chega, você abre em OWA, e o código deles rouba cookies de sessão, copia sua caixa de correio ou configura regras de email. O servidor não é tocado — a lacuna sequestra o navegador que você já havia autenticado.

O problema: sem patch no sentido clássico. Microsoft está apenas distribuindo uma mitigação de reescrita de URL chamada "M2.1.x" via o Exchange Emergency Mitigation Service [2]. EMS está ativo por padrão em Subscription Edition e 2019; em 2016 e CUs mais antigas você precisa patchear manualmente. Qualquer um que tenha desligado EMS por razões de conformidade (mais agências governamentais do que se pensa) está tecnicamente não mitigado hoje. CVSS 8,1 [3]. Para empresas brasileiras rodando Exchange híbrido: verifique o log EMS hoje, não na próxima semana.

Gitea: quatro anos entregando imagens de container privadas para requisições anônimas

Gitea, a alternativa auto-hospedada para GitHub Enterprise/GitLab, confirmou CVE-2026-27771 em 27/05 e fechou com 1.26.2. CVSS 8,2. O problema: o registro de container embutido respeitava a flag private na UI, mas entregava manifestos e blob layers aos endpoints do registro (GET /v2/<name>/manifests/<tag>, /blobs/<digest>) até para requisições anônimas. Conhecer o nome do repo e a tag era suficiente — sem autenticação, sem conta, sem token [4].

Registro de container: O gerenciador de pacotes para imagens Docker — como npm para Node, exceto por setups de servidor inteiro pré-construído. Você constrói localmente (docker build), envia ao registro, e o servidor puxa via docker pull. Registro "privado" significa autenticação apenas — nos layers da imagem residem código-fonte, arquivos .env, chaves de API baked-in, credenciais AWS. Se verificações de autenticação falham, é pior que um repo Git aberto: não apenas código, mas o bundle construído, rodando de segredos.

O bug existiu por quase quatro anos. Exposição: >30.000 instâncias em 30+ países, top 5 China, EUA, Alemanha, França, Reino Unido [5][6]. Forgejo herdou o mesmo código de registro e também é afetado. Descoberto em abril por "NoScope", um agente autônomo de pentesting baseado em IA — uma prévia dos próximos 12 meses [7]. Mitigação: atualize para ≥1.26.2; medida temporária [service].REQUIRE_SIGNIN_VIEW = true [8]. Para devs solo e empresas de médio porte com reflexo de soberania BSI, Gitea/Forgejo é a stack auto-hospedada óbvia de Git+CI+registro — se você atende clientes: atualize agora, faça auditoria dos logs de registro (quais IPs anônimos puxaram quais manifestos?), rotacione segredos.

C# ganha um toque Rust — tipos de pointer fora de "unsafe"

O time .NET da Microsoft esboçou em 26/05 no blog .NET a maior reforma de memory-safety para C# em anos. PM Richard Lander explica em "Improving C# Memory Safety": a keyword unsafe está sendo redefinida — seguindo o modelo Rust, sem descartar o GC [9].

"unsafe" em uma linguagem com GC: C# normalmente limpa memória via garbage collection. No momento em que você usa pointers crus (para código C nativo, hardware), você desabilita essa rede de segurança — é isso que a keyword unsafe faz. Rust é mais preciso: apenas a operação realmente perigosa (pointer dereferencing, i.e., ler de um endereço estrangeiro) vai num bloco unsafe; aritmética com o pointer fica safe.

É exatamente para onde Microsoft quer levar C#. Tipos de pointer e aritmética de pointer ficarão safe; apenas dereferencing fica unsafe. Efeito: wrappers FFI e interop ficam mais legíveis, o passo perigoso se destaca em vez de se esconder entre cálculos harmoniosos no mesmo bloco — a mesma lógica que Rust [10].

Roadmap: preview em .NET 11, produção em .NET 12 / C# 16, opt-in. Código antigo roda inalterado; bibliotecas de runtime serão opted-in [11]. Encaixa no quadro maior: Microsoft vem traduzindo Windows C/C++ para Rust internamente desde 2024; agora .NET aperta o parafuso em sua própria linguagem. Para devs Java/Spring, um ponto a observar; mais relevante se você toca .NET/Unity — o modelo chega em previews do .NET 11 este verão.

npm "staged publishing" — o portal humano 2FA é GA

A resposta da GitHub ao TanStack: GA desde 22/05. Em vez de publicar direto em npmjs.com, o tarball cai numa fila de staging — um maintainer deve aprovar explicitamente com 2FA. Requer npm CLI 11.15.0+ e npm stage publish [17].

Trusted Publishing / OIDC: Uma CI (p.ex., GitHub Actions) se autentica com um token OIDC de curta duração para npm e envia um novo pacote automaticamente — sem clique. Conveniente, mas: se um atacante compromete a CI (pwn request, cache poisoning), a CI publica seu pacote envenenado com uma assinatura válida. Assim aconteceu TanStack. Staged publishing inverte isso: a CI faz upload, mas ele vai online apenas após um segundo passo, humano com 2FA.

"Prova de presença" até para publishes CI/CD. Ressalva: defende contra o caminho TanStack (roubo de token), não o caminho Nx (conta maintainer comprometida). E funciona apenas se maintainers a habilitarem — defaults continuam por enquanto.
