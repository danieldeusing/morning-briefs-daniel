Build 2026: Windows vira uma caixa Unix apropriada [1][2]

O grande show de IA na Microsoft Build (2–3 de junho) é domínio do brief de ai-dev — aqui apenas o que os devs ganham direto de ferramentas além de agentes de IA. E desta vez é inusitadamente tangível.

Coreutils para Windows agora está disponível em geral: mais de 75 comandos padrão Linux — ls, cp, mv, rm, cat, grep, find — rodam nativamente no Windows, sem WSL e sem VM. Construído a partir do projeto de código aberto uutils, uma reimplementação do GNU Coreutils em Rust.[1] Tecnicamente um único coreutils.exe com hardlinks NTFS (ls.exe, cp.exe …) apontando para ele.[2] Para qualquer um que alternar entre servidores Linux e laptop Windows, isso remove um ponto de atrito diário no shell.

Coreutils: Estes são os pequenos comandos fundamentais que você usa no Linux para navegar pastas, copiar arquivos e filtrar texto — como os utensílios básicos numa cozinha. Até agora o mesmo conjunto de ferramentas tinha nomes diferentes e vivia numa gaveta diferente no Windows; agora o kit Linux fica bem ali na cozinha Windows.

Containers WSL e configs de dev: menos atrito de setup [2]

Segundo ponto sólido da Build: Containers WSL — uma forma integrada de criar, iniciar e gerenciar containers Linux através de CLI e API familiares diretamente do Windows; preview pública, segundo a Microsoft, "em breve".[2] É o mecanismo de container um pouco mais fundo no SO, sem o desvio de uma aplicação desktop separada.

Mais Configurações do Desenvolvedor Windows: de máquina nova para ambiente pronto para o trabalho "em minutos", com ferramentas pré-configuradas — WSL, PowerShell 7, Git, GitHub CLI, VS Code, Python.[2] Para a stack de Daniel (Java/Spring, Angular, Python, Docker), isso significa menos acompanhamento manual ao configurar, mais ponto de partida reproduzível. Não é um milagre, mas economiza horas reais em cada novo dispositivo.

Criptografia pós-quântica chega à pilha TLS do Windows [2]

Terceiro ponto da Build pelo ângulo da confiança: Windows expande criptografia pós-quântica com troca de chave híbrida PQ na pilha TLS e nos serviços de certificado.[2] "Híbrido" significa: um algoritmo clássico e um resistente a quântico rodam juntos, então a conexão permanece segura mesmo se um deles se tornar quebrável depois.

"Capturar agora, descriptografar depois": Atacantes interceptam tráfego criptografado hoje e o armazenam — esperando descriptografá-lo em dez anos com um computador quântico. Como alguém roubar uma carta fechada e guardá-la até finalmente ter a chave certa. Criptografia PQ garante que a carta permaneça fechada.

Praticamente nenhuma pressão de ação hoje, mas a direção está definida — bibliotecas TLS e configurações verão suites híbridas como padrão nos próximos anos. Qualquer um operando seus próprios endpoints TLS (Caddy, reverse proxies) deve manter isso no radar.

Chrome 149 estável: decorações de gap CSS e scroll promises [3][4]

Chrome 149 chegou ao canal estável em 02.06.2026 (Android, ChromeOS, Linux, macOS, Windows).[3] Relevante para trabalho frontend: decorações de gap CSS — gaps em grid/flexbox finalmente podem ser estilizados diretamente (novas propriedades como column-rule-inset, row-rule-visibility-items), análogo a column-rule em layout multi-coluna.[4]

Mais scroll promises: element.scrollTo() agora retorna uma promise — uma lacuna antiga que permite esperar uma animação de scroll terminar sem ginástica de timer. Além disso, uma página com WebSocket aberto agora pode entrar no cache back/forward porque o socket fecha proativamente na entrada. Pequenas coisas individualmente, mas exatamente as lacunas de plataforma que devs Angular/React de outra forma preenchem com polyfills.

npm "Miasma": pacotes Red Hat como vetor de worm [6][7]

Em 01.06.2026, Wiz Research descobriu um comprometimento de cadeia de suprimentos no namespace npm @redhat-cloud-services: pelo menos 32 pacotes, totalizando 96 versões, continham mudanças não autorizadas não correspondendo aos repos de origem.[6] O gatilho foi uma conta GitHub comprometida de um funcionário Red Hat; o atacante enviou workflows GitHub Actions maliciosos para três repos RedHatInsights.[7]

O malware — apelidado "Miasma", uma variante da família Mini-Shai-Hulud — rodava via script preinstall: o payload inicia automaticamente em npm install, antes do código real da aplicação e sem aviso visível. Coleta tokens GitHub, chaves SSH e credenciais em nuvem (novo nesta variante: identidades GCP e Azure).[6] Cada infecção recebe um payload criptografado unicamente — IOCs baseados em hash são bons apenas por versão de pacote.

Script preinstall: npm tem permissão de executar um script antecipadamente durante a instalação de pacote — feito para trabalho de setup. Como um móvel que, quando desempacotado, automaticamente despacha um assistente. Se o pacote é malicioso, código estrangeiro roda em sua máquina antes de você até ter importado uma única linha.
Atenção Se você usa pacotes @redhat-cloud-services: audite versões afetadas, rotacione tokens GitHub/chaves SSH/credenciais em nuvem, inspecione ambientes CI/CD. Geralmente, npm install --ignore-scripts como padrão protege contra exatamente essa classe.

33 pacotes npm falsos via dependency confusion [8]

Quase simultaneamente (28–29.05), Microsoft Threat Intelligence descobriu uma segunda campanha: 33 pacotes npm maliciosos em três ondas, distribuídos por nove escopos organizacionais, personificando namespaces reais de empresas internas — dependency confusion clássica.[8] Por trás de três apelidos de mantenedor (mr.4nd3r50n, ce-rwb, t-in-one) senta, segundo infraestrutura C2 compartilhada, um ator único.

Notável: Os pacotes ainda estavam em modo de reconhecimento — coletavam info de sistema, nomes de hosts, variáveis de ambiente e contexto de desenvolvedor, com um kill switch integrado para exploração total em ataques subsequentes. Então coleta de inteligência antes do ataque real.

Dependency confusion: Empresas frequentemente usam pacotes internos com seus próprios nomes. Se alguém fazer upload de um pacote com mesmo nome publicamente, o gerenciador de pacotes pode acidentalmente buscar o público (malicioso) em vez do interno. Como um entregador pegando um pacote impostor com seu nome nele.
Mitigação Microsoft aconselha: scope-lock registries internos (sem fallback para npm público), desabilite scripts pós-instalação, bloqueie egresso para domínios C2, inspecione logs CI/CD. Para seus próprios projetos: pacotes com escopo e um proxy de registry upstream.

CISA KEV: dois registros novos em 02.06 [9][10]

CISA adicionou duas vulnerabilidades ativamente exploradas ao catálogo KEV em 02.06.2026:[9]

  • CVE-2022-0492 — Linux kernel, cgroups v1: um bug de lógica em release_agent permite escalação de privilégio e potencialmente escape de container. Afeta a arquitetura mais antiga, mas ainda largamente implantada de cgroup-v1.[10] Uma vulnerabilidade antiga (2022) agora aparentemente ativamente abusada novamente — corrija hosts de container ou audite cgroup v2.
  • CVE-2025-48595 — Android framework, integer overflow: escalação de privilégio local sem interação do usuário. Google fala de "exploração limitada e direcionada" — o padrão frequentemente ligado a fornecedores de spyware.
Catálogo CISA KEV: Uma lista de vulnerabilidades comprovadamente exploradas na natureza. Agências federais americanas devem fechá-las até um prazo — para todos os outros é um "corrija estes primeiro" priorizado porque atacantes estão realmente usando.

Regulação UE: dever de reporte CRA começando 11.09 [13]

Thread permanente porque o prazo está se aproximando: As obrigações de reporte do Cyber Resilience Act (CRA) da UE entram em vigor a partir de 11.09.2026. Fabricantes de "produtos com elementos digitais" devem reportar vulnerabilidades ativamente exploradas e incidentes graves — aviso antecipado dentro de 24 horas, relatório completo dentro de 72 horas — via plataforma única de reporte CRA que ENISA está construindo atualmente.[13] Os deveres principais do CRA entram em vigor em 11.12.2027, mas a obrigação de reporte é o primeiro prazo firme.

CRA / SBOM: O CRA exige, entre outras coisas, uma lista de materiais de software legível por máquina para componentes de software (software bill of materials) — como uma lista de ingredientes numa embalagem, apenas para todas as bibliotecas no produto. Sem um processo SBOM maduro, os prazos de reporte são difíceis de cumprir.

Relevante para Daniel/twiceD uma vez que um produto com elementos digitais entre no mercado UE: planeje pipeline SBOM e um processo de reporte cedo, não se apressando em setembro.

No lado de segurança, npm está pegando fogo duas vezes. Em primeiro de junho, a firma Wiz encontrou um comprometimento no espaço de pacotes Red Hat: noventa e seis versões em trinta e dois pacotes foram envenenadas com um worm chamado Miasma que rouba tokens GitHub, chaves SSH e acesso em nuvem. Roda via script pré-instalação, então já antes de você importar uma única linha.

Quase ao mesmo tempo, Microsoft descobriu uma segunda campanha: trinta e três pacotes npm falsos personificando nomes reais de empresas internas. Ainda estavam em modo de reconhecimento, reunindo informações de ambiente, mas tinham um kill switch para o ataque completo posterior já integrado. A lição para ambos os casos é a mesma: desligue scripts de instalação por padrão e tranche fontes de pacotes internas bem fechadas.

A agência americana CISA adicionou duas vulnerabilidades ativamente exploradas ao seu catálogo de patch obrigatório em segundo de junho: um bug antigo do kernel Linux em grupos de controle que permite escape de container, e um buraco Android aparentemente sendo usado deliberadamente para espionagem. A atualização de junho do Google para Android fecha cento e vinte e quatro fraquezas.

No navegador, Chrome cento e quarenta e nove atingiu estável em segundo de junho: você agora pode estilizar as lacunas em layouts grid e flexbox diretamente, e rolar por script finalmente retorna uma promessa que você pode limpar esperar.

Fabricantes devem reportar buracos ativamente explorados dentro de vinte e quatro horas mais cedo e dentro de setenta e duas horas em cheio.
