Microsoft Constrói Hardware Dev Novamente: Surface RTX Spark Dev Box [1][2]

A maior surpresa da Build 2026 não foi um recurso de software, mas um dispositivo: Microsoft anuncia com a Surface RTX Spark Dev Box seu primeiro aparelho construído especificamente para desenvolvedores em anos.[1] No núcleo está o "RTX Spark" superchip da NVIDIA — uma CPU Grace (ARM) mais GPU RTX Blackwell — que segundo Microsoft entrega 1 petaflop de computação IA e 128 GB de Unified Memory. A promessa: rodar ou fazer fine-tune em modelos de 120 bilhões de parâmetros com contexto de 1 milhão de tokens localmente, sem fatura de GPU em nuvem.[1]

Unified Memory: Normalmente CPU e GPU têm memória separada e devem copiar dados constantemente de um lado para o outro — como dois chefs com despensas separadas passando ingredientes um ao outro. Unified Memory é uma despensa compartilhada: CPU e GPU acessam o mesmo pool. Para modelos IA gigantes, esse é o truque crucial, porque um modelo de 120 bilhões de parâmetros não caberia na memória GPU de outra forma.

O gabinete é um bloco monolítico de alumínio com aproximadamente 1.000 furos de ventilação, funcionando simultaneamente como dissipador — feito para trabalhos longos de treinamento sem throttling.[2] É construído com propósito: Windows 11 Pro com PowerShell 7 como shell padrão, WSL 2 com passthrough GPU e CUDA, VS Code, Git, Python, Node — utilizável direto da caixa.[1]

A pegadinha: preço e prazo exato estão faltando. Microsoft apenas diz "este ano, nos EUA, exclusivo via microsoft.com" e uma lista de espera — Nadella brincou que ele mesmo está na fila.[2] Analistas estimam USD 3.000–3.500; para comparação, o DGX Spark da NVIDIA agora custa USD 4.699 (elevado de 3.999 em fevereiro por escassez de memória), e o Ryzen AI Halo PC da AMD, USD 3.999.[2] O que importa: essa é a resposta da Microsoft à tendência Mac Studio/DGX Spark — estações de trabalho IA locais para devs. Menos interessante como objeto de compra do que como sinal — Windows está sendo posicionado como uma plataforma de dev de primeira classe, incluindo hardware ARM com GPU séria.

Coreutils for Windows: Comandos Unix Nativos — e uma Declaração de Licença [3][4]

Desde a Build, Coreutils for Windows está geralmente disponível: 75 comandos Linux familiares (ls, cp, grep, find, xargs, du, uptime …) rodam nativamente no Windows, instaláveis via WinGet, agrupados com findutils e um grep compatível GNU.[3] Para quem pula entre servidor Linux e laptop Windows, isso é alívio real de atrito: scripts de shell ficam mais portáveis sem precisar iniciar WSL primeiro.

A parte interessante está na nota de rodapé: Microsoft não constrói sobre os coreutils GNU clássicos, mas em uutils — uma reimplementação em Rust multiplataforma — e isso é licenciado MIT, enquanto os originais GNU são GPL.[4]

MIT vs. GPL: Ambas são open source, mas com caracteres diferentes. GPL é "compartilhe igualmente": se você coloca código GPL em seu produto e o distribui, deve divulgar seu próprio código nos mesmos termos — uma cola para liberdade. MIT é "faça o que quiser": inclua, modifique, até coloque em produtos fechados, sem obrigação de retorno. Para uma empresa como Microsoft, MIT é muito mais conveniente — sem risco de "infecção" GPL embutido no SO.

Que um gigantesco fornecedor de plataforma troque um kit de ferramentas GPL por uma variante MIT e a coloque no SO é parte de uma tendência OSS maior (rewrites Rust, licenças permissivas). Não impactante para o dia a dia de Daniel, mas um marcador de para onde estão indo as ferramentas de sistema. Tropeço prático: comandos como dir, more, paste, whoami estão propositalmente fora (colisão com built-ins Windows), e coisas POSIX como chmod, chown, chroot, nohup faltam completamente.[3]

WSL Containers + Windows Limpo: a Ponte Linux Fica Mais Larga [5]

Dois anúncios menores mas relevantes para operações: WSL Containers (WSLC) trazem uma forma embutida, tipo Docker, de iniciar containers Linux via CLI e API direto no WSL — incluindo passthrough GPU para cargas de trabalho IA locais. Status: "em breve" em Preview Público.[5] Isso preenche uma lacuna entre "WSL para uma shell" e "Docker Desktop completo" e pode ser uma configuração de containers local mais leve.

Junto a isso, Windows Developer Configurations agora estão GA: um dev-config.winget mais scripts que tornam um Windows fresco pronto para dev em minutos — Explorer limpo, tema escuro, sem widgets, sem "sugestões", sem popups.[5] Pequeno, mas qualquer um que já tenha despojado uma máquina Windows para trabalho sabe que economiza tempo real.

PostgreSQL 19 Beta 1 Chegou [7]

O momento de banco de dados da semana: O PostgreSQL Global Development Group lançou PostgreSQL 19 Beta 1 em 04.06.[7] Isso significa que o conjunto de recursos para a próxima versão major está efetivamente congelado — o que está na beta vai (exceto bugs) para o lançamento final, previsto para setembro de 2026.

Beta / Feature-Freeze: Até a beta, novos recursos ainda podem chegar; a partir da beta, é estabilização, testes, correções de bugs — como um restaurante que fixa seu menu e agora apenas pratica cozinhar os pratos perfeitamente. Beta vai em um ambiente de teste, não seu banco de dados de produção: o formato on-disk ainda pode mudar até GA.

Na prática: qualquer um que mantenha extensões customizadas, ORMs ou scripts de migração deve testar contra beta agora — esse é o ponto inteiro da fase. No stack de Daniel (Spring/FastAPI em PostgreSQL), uma olhada cedo em melhorias de query planner e logical replication faz sentido antes do GA de setembro. Não solte isso em dados de produção.

Magento RCE na Lista KEV: CVE-2026-45247 [8][9]

O alerta de segurança único, mas pesado: CISA adicionou CVE-2026-45247 ao catálogo KEV em 03.06 — uma execução remota de código sem autenticação ativamente explorada com CVSS 9,8 no Mirasvit Full Page Cache Warmer, extensão de cache amplamente usada para Magento 2 / Adobe Commerce.[8]

Lista KEV: O catálogo "Known Exploited Vulnerabilities" da agência EUA CISA é o rol de lacunas para as quais há atividade de ataque real confirmada em campo — não "poderia teoricamente acontecer" mas "está sendo usado agora". Algo na lista não é mais uma situação "patch quando tiver tempo".

O mecanismo é clássico: a extensão coloca parte do cookie CacheWarmer não validado direto no unserialize() do PHP, sem restrições em quais classes podem ser instanciadas — um gadget de injeção de objeto PHP faz, um cookie elaborado em qualquer request normal da loja é o bastante para execução de código, sem login necessário.[9] Versões afetadas antes da 1.11.12; o patch está disponível desde 25.05. O prazo para agências federais EUA foi 06.06. A Sansec (que reportou primeiro) aponta como forte indício um cookie começando com CacheWarmer: seguido por Tz, Qz ou YT; a Imperva relata tentativas ativas de exploit.[9] Se você executa uma loja Magento/Adobe Commerce com essa extensão: atualize.

Camada Infra: GitHub Air-Gapped, Azure Linux 4.0, HorizonDB [11][12]

Do canto de infra de Build, fácil de perder no barulho IA, três coisas que importam para setups regulados e on-prem:

GitHub Enterprise Local (Preview) traz GitHub Enterprise Server para Azure Local — conectado ou air-gapped, com runners auto-hospedados.[11] Para governo, bancos e qualquer coisa com regras estritas de soberania de dados (muito relevante para DE/EU) "GitHub sem internet" é um pitch sério.

Air-gapped: Um sistema com zero conexão internet — fisicamente separado, como se você tivesse cortado os cabos de rede. Usado para ambientes altamente sensíveis (militar, infraestrutura crítica) onde até um caminho teórico de ataque online é demais.

Azure Linux 4.0 (Public Preview) é a distro própria da Microsoft, nova com base Fedora; já alimenta milhões de cores em AKS, Azure SQL e Cosmos DB.[12] E Azure Container Linux (GA), a próxima iteração Flatcar, visa hosts mínimos otimizados para container. Ambas são movidas padrão de plataforma Linux — sem ângulo IA.

No lado de banco de dados, Azure HorizonDB (Preview) flanqueia o dia Postgres: serviço gerenciado compatível com PostgreSQL com resiliência de zona e armazenamento elástico até 128 TB.[12] Significado: Microsoft se posiciona ainda mais forte como casa Postgres — relevante se Daniel alguma vez avaliar Postgres gerenciado em Azure.

Plataforma Web: WebMCP & HTML-in-Canvas em Origin Trial [13]

Do carryover de web standards de Google I/O (maio), agora em implementação: Chrome 149 lança um Origin Trial para WebMCP, um padrão web aberto proposto que expõe funções JavaScript e formulários HTML como "tools" estruturados para que agentes de navegador executem tarefas com mais confiabilidade.[13]

Origin Trial: A forma do Chrome de testar um novo recurso de navegador, ainda não padronizado, em produção real. Um site faz opt-in e recebe um token que desbloqueia o recurso apenas para seus visitantes — como um beta test para uma web API, time-limited e sem rollout global.

Junto com isso, a HTML-in-Canvas API (também Origin Trial): elementos reais do DOM podem embutir em um Canvas com WebGL/WebGPU e permanecer pesquisáveis e acessíveis — previamente uma escolha difícil entre "lindamente renderizado" e "utilizável".[13] Para o trabalho frontend de Daniel (Angular/React), ambos são sinais cedo, não recursos de produção — mas a direção é clara: a plataforma navegador está se preparando para agentes e UIs dirigidas por GPU.

Regulação EU: Relógio CRA Corre para 11.06 [14]

Thread permanente, ainda ativo: O Cyber Resilience Act atinge um marco administrativo em 11.06 — até então a autoridade designadora (na Alemanha, o BSI) deve ter estabelecido e publicado os procedimentos pelos quais avaliadores de conformidade são avaliados, nomeados e notificados.[14]

Cyber Resilience Act (CRA): Lei EU exigindo todos os produtos "com elementos digitais" — de lâmpadas smart-home a software B2B — serem construídos, mantidos e reportados como seguros ao longo de sua vida. Uma espécie de requisito de inspeção de segurança para produtos de software conectados.

Para devs solo e pequenas casas de software, ainda não é um prazo de entrega, mas a corrida para a navalha real: de 11.09.2026 em diante, um dever de notificação de 24 horas entra em vigor para falhas ativamente exploradas e incidentes severos em produtos conectados (primeiro aviso dentro de 24 h, follow-up dentro de 72 h).[14] Se você vende software na EU, prepare seu processo de notificação agora, não em setembro.
