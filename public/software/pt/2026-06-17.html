Patch Tuesday junho: o maior desde o início do programa — com uma brecha OWA em chamas

Esta é a manchete do dia, e é por duas razões. Primeiro, o puro volume: a Microsoft fechou ~198 CVEs em junho (segundo a contagem da Tenable; BleepingComputer fala em 200) — de acordo com a Tenable, "o maior lançamento do Patch Tuesday desde o início do programa Patch Tuesday".[1] Desses, 32 são classificados como críticos. A distribuição é previsível: 31,8% das brechas são elevação de privilégios, 27,3% execução remota de código.[1]

Segundo, e esta é a parte que realmente importa hoje: entre os três zero-days conhecidos publicamente está uma brecha já sendo explorada ativamente no Outlook Web Access do Exchange. CVE-2026-42897 (CVSS 8,1) é um bug de falsificação que surge de uma fraqueza de cross-site scripting na OWA; afeta Exchange Server 2016, 2019 e Subscription Edition.[2] A Microsoft entregou atualizações de segurança em 9 de junho para Exchange SE RTM, Server 2019 (CU14/CU15) e Server 2016 (CU23).[3]

XSS (Cross-Site Scripting): Imagine um quadro de avisos onde qualquer um pode pregar um papel — e alguém prende um papel que não diz "olá", mas sim uma instrução que o zelador executa automaticamente ao passar. Em XSS, um atacante introduz código em uma página da web (aqui: a caixa de entrada de webmail), que o navegador da vítima então executa como legítimo. No contexto da OWA, isso significa: conteúdo preparado na caixa de entrada pode agir em nome da vítima.

Os outros dois zero-days são menos agudos, mas nomeados: CVE-2026-45586 (CTFMON EoP, CVSS 7,8) permite escalação para privilégios SYSTEM[1], e o bug "bit war" CVE-2026-50507 (CVSS 6,8) permite que um atacante com acesso físico contorne a criptografia de dispositivo BitLocker — PoC existe, mas até agora sem exploração em estado selvagem.[1] Além do clássico DoS CVE-2026-49160 ("HTTP/2 Bomb", CVSS 7,5), onde quantidades minúsculas de dados forçam o servidor a alocações de memória desproporcionalmente grandes.[1]

Atenção Sem alarme geral sobre 198 CVEs — mas a prioridade é clara: Exchange OWA primeiro, porque sendo explorada ativamente. Quem executa Exchange on-prem deve implantar a atualização de 09/06 primeiro; o resto do lote segue conforme a exposição.

Incidente npm TanStack: dessa vez a pipeline foi sequestrada, não uma senha roubada

Em 11 de maio, um ataque de cadeia de suprimentos atingiu o ecossistema TanStack: 42 pacotes em 84 versões comprometidas.[4] O incidente tem a identificação CVE-2026-45321 com uma pontuação CVSS de 9,6.[5] Snyk e Heise o classificam sob o padrão "mini-Shai-Hulud" — worms npm auto-replicantes.

O que é tecnicamente notável é o vetor: os pacotes não foram publicados por um atacante que roubou credenciais. Foram publicados da pipeline de lançamento legítima do TanStack usando sua identidade OIDC confiável — depois que código controlado pelo atacante sequestrou o executor CI no meio do fluxo de trabalho.[4] O malware então exfiltrou credenciais IMDS AWS, tokens GitHub e chaves SSH.

OIDC em CI/CD: Em vez de colocar uma senha de longa duração na pipeline de construção, a pipeline prova sua identidade como uma carteira de identidade emitida recentemente (token OIDC) que o registro aceita brevemente. Parece mais seguro — e é contra senhas roubadas. Mas se o atacante está na execução de construção, ele apenas usa a verdadeira carteira de identidade válida enquanto é boa. Exatamente isso é o que aconteceu aqui: nenhuma chave roubada, mas uma máquina sequestrada usando sua própria chave legítima.

Para verificar se foi afetado: @tanstack/query*, table*, form*, virtual*, store e o meta-pacote @tanstack/start (não os subpacotes start-*) foram não comprometidos.[4] Encaixa no quadro maior: ReversingLabs relata um aumento de 73% em pacotes de código aberto maliciosos para 2025 versus o ano anterior.[6]

Atenção Tarefa de casa: verificar arquivos de bloqueio contra as janelas de versão afetadas — e absorver a lição de que a confiança da pipeline OIDC é tão boa quanto a integridade do executor. Tokens de curta duração protegem contra roubo, não contra uma máquina sequestrada no meio da execução.

Chrome V8: quinto zero-day sendo explorado ativamente do ano [7][8]

O Google corrigiu outro zero-day do navegador. CVE-2026-11645 é uma leitura/escrita fora dos limites no mecanismo JavaScript V8; o Google confirma que "existe um exploit para CVE-2026-11645 em estado selvagem".[7] Corrigido em Chrome 149.0.7827.102/.103, implantado a partir de 8/9 de junho. O Google não atribui uma pontuação CVSS numérica; a brecha é classificada como "severidade alta". É o quinto zero-day Chrome explorado em estado selvagem desde o início do ano — e o segundo a atingir V8.[8] O CISA o adicionou à KEV em 9 de junho.

Leitura/Escrita Fora dos Limites: Como um livro com páginas numeradas — o código deve acessar apenas páginas 1 até 100, mas acidentalmente lê ou escreve página 101 ou −1. Em V8 (o mecanismo que executa JavaScript no navegador), isso significa: JavaScript preparado em uma página da web pode acessar memória estrangeira — o primeiro passo para escapar do navegador.

No mesmo lote da KEV de 9 de junho também estavam Arista EOS (CVE-2026-7473) e Cisco SD-WAN (CVE-2026-20245).[9] Dica útil: Chrome apenas precisa ser reiniciado para que a atualização tenha efeito — muitas instalações esperam dias em segundo plano.

Microsoft Work IQ API GA: uma nova camada de API para agentes empresariais

Desde 16 de junho, os endpoints de API do Work IQ estão geralmente disponíveis.[10] A promessa de IA não é o que importa aqui — isso é território de ai-usecases — mas a superfície de plataforma/API que está abrindo para equipes de dev. Work IQ fornece 10 ferramentas genéricas M365 (mail, calendário, arquivos, pessoas, chat, sites) e fala três protocolos: A2A, MCP (servidor remoto recém-reformulado), e REST.[11]

Dois detalhes importam para o lado de engenharia de software. Primeiro, faturamento: baseado em consumo via Copilot Credits e explicitamente independente do licenciamento M365 Copilot.[12] Isso move uma conexão de API da negociação de licença para um modelo de custo baseado em uso — relevante para qualquer um que precisa precificar uma integração M365. Segundo, governança: um mecanismo de política baseado em Rego, cada invocação de ferramenta é registrada e avaliada, além de rate-limiting e cumprimento de conformidade em tempo real.[13]

Rego / Policy-as-Code: Em vez de colocar regras de acesso no código da app, você escreve como uma linguagem de regra separada e legível (Rego, conhecido do Open Policy Agent) — como um conjunto de regras separado na porta que decide "permitido / não permitido" a cada solicitação. A vantagem: regras são versionáveis, auditáveis e alteráveis sem reimplementar a aplicação.

Work IQ foi anunciado na Microsoft Build 2026 (2 de junho), e a prévia agora passou para GA.[14] Contexto de referência: para o mundo de Daniel, isso é menos um tópico de IA do que uma nova peça de engenharia de plataforma empresarial — uma ponte padronizada, faturável e controlada por política para dados M365 que você pode colocar serviços em cima.

Nível de patch junho Android: brecha sendo explorada ativamente — e uma confusão de números

O boletim Android de junho fecha oficialmente 96 CVEs distintos nos níveis de patch 2026-06-01 e 2026-06-05, 11 delas críticas.[15] Importante aqui: CVE-2025-48595 (elevação de privilégio, sem interação do usuário) afeta Android 14, 15, 16 e 16-QPR2 e está sob "exploração limitada e direcionada" de acordo com o Google.[15] O CISA adicionou a brecha à KEV em 2 de junho.[16]

Uma nota sobre a diligência: vários veículos (incluindo SC World, The Hacker News) citam 124 vulnerabilidades — notavelmente mais que os 96 do boletim.[17] A lacuna é inexplicada; plausível é que a contagem mais alta inclua patches específicos de Pixel ou de parceiros/fornecedores não no boletim AOSP principal. Na dúvida, a fonte primária vence — então 96 — com a ressalva de que a mídia conta diferentemente.

Como bônus, mas útil: a atualização de sistema de junho traz um Padrão de Troca de Credenciais que torna senhas e passkeys portáveis entre gerenciadores de terceiros e o Google Password Manager.[18]

Linha de segurança PostgreSQL: 11 CVEs, três com CVSS 8,8

O grupo PostgreSQL lançou as versões 18.4, 17.10, 16.14, 15.18 e 14.23 (14 de maio) — 11 brechas de segurança e mais de 60 correções de bugs.[20] Três dos CVEs estão em CVSS 8,8 e afetam todas as versões 14–18: CVE-2026-6473 (wraparound inteiro na alocação de memória), CVE-2026-6475 (sobrescrita de arquivo via pg_basebackup/pg_rewind), e CVE-2026-6637 (stack buffer overflow mais SQL injection no módulo refint).[21] Uma atualização menor agendável — sem drama, mas três 8,8s em um lote é uma boa razão para não adiar a próxima janela de manutenção.

Python 3.15 Beta 2: JIT mais rápido, Lazy Imports por padrão

Em 2 de junho, Python 3.15.0 Beta 2 foi lançado.[22] O compilador JIT fica mais rápido: média geométrica de 8–9% em x86-64 Linux versus o intérprete padrão, 12–13% em AArch64 macOS versus o intérprete tail-calling.[22] Destaques de conteúdo: PEP 810 (lazy imports explícitos para inicialização mais rápida), PEP 814 (frozendict como built-in), PEP 661 (sentinel como built-in), e PEP 686 (UTF-8 como padrão).[23] A próxima beta (3.15.0b3) está agendada para 23 de junho, o RC para 4 de agosto.[24]

Marque o prazo: Spring Framework 6.2 cai do suporte OSS em 30/06

Para todos com Java/Spring na stack: Spring Framework 6.2.x atinge fim de vida OSS em 30 de junho.[25] Depois disso, não há patches de segurança gratuitos — novas CVEs se acumulam sem caminho de patch a menos que você migre ou compre suporte comercial.[26] O movimento vai para a linha 7.0.x; atualmente estável é 7.0.7 (17 de abril), Spring Boot 4.0.6 de 23 de abril, com suporte JDK para 17, 21 e 25.[27] Este é o único prazo de stack deste mês que merece uma entrada no calendário.

É quarta-feira, dezessete de junho, e o dia está dominado por atualizações de segurança — lideradas por um Patch Tuesday que quebrou recordes.

A Microsoft fechou aproximadamente cento e noventa e oito vulnerabilidades em junho — segundo a Tenable, o maior Patch Tuesday desde o início do programa. Trinta e dois deles são críticos. A parte realmente urgente, porém, é um único buraco: um erro de cross-site scripting no webmail do Exchange, chamado CVE-vinte-vinte-seis-quarenta e dois mil oitocentos noventa e sete, com uma pontuação de severidade de oito vírgula um. Já está sob ataque ativo e afeta Exchange dois mil e dezesseis, dois mil e dezenove e a Subscription Edition. A Microsoft entregou o patch em 9 de junho. Se você executa Exchange você mesmo, essa atualização específica deve entrar primeiro.

A segunda grande história vem da cadeia de suprimentos de código aberto. O projeto TanStack teve quarenta e dois pacotes em oitenta e quatro versões comprometidas, com uma severidade de nove vírgula seis. Aqui está o que é diferente: ninguém roubou uma senha. Os atacantes sequestram a pipeline real e confiável enquanto ela rodava, e publicaram os pacotes envenenados usando a própria identidade válida do TanStack. A lição é desconfortável: tokens de curta duração protegem contra roubo, mas não contra uma máquina tomada no meio da execução.

O Google também teve que corrigir. O mecanismo JavaScript no Chrome continha um bug que permite código acessar memória estrangeira — e já existe um exploit em estado selvagem. É o quinto zero-day Chrome assim neste ano. Reiniciar o navegador é tudo que é preciso para que a atualização tenha efeito.

A Microsoft também ativou uma nova camada de plataforma: Work IQ está geralmente disponível desde 16 de junho. O que é interessante aqui não é a promessa de IA, mas a tecnologia direta — dez ferramentas padrão para dados Microsoft, três protocolos, faturamento de consumo e seu próprio sistema de permissão baseado em regras. Para equipes de desenvolvimento, é principalmente uma nova peça de infraestrutura empresarial que você pode colocar serviços em cima.

No lado mais leve: PostgreSQL lançou uma linha de segurança com onze buracos, três deles severidade oito vírgula oito. Python três ponto quinze está em sua segunda beta, com compilação just-in-time oito a nove por cento mais rápida e lazy imports. E Android entregou seu nível de patch junho — oficialmente noventa e seis vulnerabilidades, embora a mídia relata cento e vinte e quatro; uma delas está sendo explorada ativamente.

Da Europa: o Cyber Resilience Act deu um passo em 11 de junho, mas o estágio realmente duro com obrigações de notificação não vem até setembro. E há uma data para marcar no calendário de stack: 30 de junho é quando Spring Framework seis ponto dois perde suporte gratuito. Se você estiver nele, você precisa mudar para versão sete ponto zero.

Amanhã, o foco provavelmente será nos desdobramentos do Patch Tuesday e se novas entradas aparecem na lista de exploração ativa. Até então.
