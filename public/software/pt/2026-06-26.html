UniFi: três falhas CVSS-10, exploradas — e o prazo é hoje

A única coisa que realmente importa hoje. Em 23/06, a CISA adicionou três falhas do Ubiquiti UniFi OS ao catálogo KEV — todas com severidade máxima 10,0 e todas já sendo ativamente exploradas — e concedeu aos órgãos federais três dias para corrigir. Esses três dias terminam hoje.[1][4]

Catálogo KEV: A agência americana CISA mantém uma lista de vulnerabilidades que não são apenas teoricamente perigosas, mas que são ativamente usadas por atacantes — como um boletim de procura de ferramentas de invasão que já estão em circulação. Uma vez que uma falha entra nessa lista, a discussão "esperar ou corrigir?" acabou.

A cadeia: CVE-2026-34908 permite alterações remotas não autorizadas, -34909 é uma falha de path traversal (acesso a arquivos + manipulação de contas), e -34910 é uma fraqueza de validação de entrada que leva à execução remota de código via injeção de comando. Encadeadas, elas permitem controle completo da rede sem autenticação. A correção foi lançada em 21/05 no UniFi OS Server 5.0.8 — se você administra uma instância UDM/Cloud Key/UniFi OS e ainda não está na 5.0.8, esta é sua tarefa de sexta à tarde.[1][2]

Nota de Fontes CISA.gov retornou 403 no acesso direto; a data de adição ao KEV e o valor CVSS 10 foram verificados via BleepingComputer e GBHackers, os scores individuais de CVE do NVD não estão em nossas fontes primárias. A direção — três falhas de severidade máxima, ativamente exploradas, prazo hoje — é clara.
  1. bleepingcomputer.com — CVSS 10, adição ao KEV em 23/06.
  2. gbhackers.com — Patch no UniFi OS Server 5.0.8 (21/05).
  3. cisa.gov — Alerta KEV, prazo de três dias.

"Miasma": uma conta Red Hat contamina 32 pacotes npm

Exatamente o tipo de história para a qual este boletim existe: sem manchete de IA, mas cadeia de suprimentos como superfície de ataque. Em 01/06, a Wiz divulgou uma campanha chamada Miasma — uma conta GitHub comprometida pertencente a um funcionário da Red Hat foi usada para injetar malware em 32 pacotes npm @redhat-cloud-services. Juntos, somavam aproximadamente 80 mil downloads por semana.[5]

O truque é o que assusta. O atacante solicitou tokens OIDC via GitHub Actions e publicou os pacotes contaminados com proveniência SLSA válida — então os pacotes carregavam exatamente as credenciais de autenticidade da cadeia de suprimentos que deveriam construir confiança.[5]

SLSA Provenance / OIDC: SLSA Provenance é como uma prova selada de origem, mostrando "este pacote foi realmente construído por este pipeline". OIDC é o cartão de ID que o pipeline usa ao publicar. Aqui, o atacante falsificou o cartão de ID e assim afixou um selo genuíno a um pacote falso — o selo está correto, mas o conteúdo não.

Na instalação, um hook preinstall acionou um dropper ofuscado de 4,29 MB, buscou o tempo de execução Bun e coletou credenciais do GitHub, npm, AWS, Azure e GCP — cada infecção individualmente criptografada para que uma assinatura não capture todas. A Red Hat tem um aviso (RHSB-2026-006) publicado; a remediação está em andamento.[6]

Não é sozinho É temporada movimentada: em 17/06, um easy-day-js tipificado incorretamente (similar a dayjs) contaminou 140+ pacotes no ecossistema Mastra AI via dropper postinstall. O eixo aqui também é a cadeia npm, não o modelo.[7]
  1. wiz.io — 32 pacotes, 80 mil downloads/semana, abuso de OIDC/SLSA.
  2. access.redhat.com — Detalhes de payload, aviso.
  3. upwind.io — Mastra AI, easy-day-js.

Apple adquire o Swift Package Index

Anunciado em 23/06: a Apple está adquirindo o Swift Package Index, construído por Dave Verwer e Sven A. Schmidt — o catálogo de fato para pacotes Swift com mais de 10 mil pacotes indexados. Permanece código aberto, e o plano declarado é assinatura de pacotes mais seu próprio registro para "sair completamente do modelo GitHub".[8]

Para quem não usa Swift, ainda relevante: é o mesmo movimento que estamos vendo em todos os lugares agora — após os incidentes npm, ecossistemas estão se afastando de "conta GitHub = confiança" para registros assinados e auto-hospedados. A Apple pode fazer isto para Swift porque é dona da plataforma. Para todos os outros, é um dado sobre onde a segurança de pacotes está indo.

  1. swiftpackageindex.com — Aquisição, 10 mil+ pacotes, assinatura + registro.

Check Point VPN: bypass de autenticação IKEv1, explorado por Qilin

CVE-2026-50751 (CVSS 9,3) é uma falha de lógica na validação de certificado para a troca de chaves legada IKEv1: um atacante não autenticado pode contornar a autenticação do usuário e estabelecer uma conexão VPN de acesso remoto sem uma senha válida. Afetados são Quantum Security Gateway (R80.40–R82.10), Firewalls Spark e várias versões do Gaia OS.[9]

Este não é um risco teórico: a exploração começou em 07/05, e atividade pós-comprometimento por um afiliado do ransomware Qilin foi confirmada em pelo menos um caso. A CISA adicionou a falha ao catálogo KEV (prazo era 11/06); a correção foi lançada em 08/06. Se você ainda tem IKEv1 ativo: desabilite ou corrija.[10]

  1. nvd.nist.gov — CVSS 9,3, bypass de autenticação IKEv1, produtos afetados.
  2. helpnetsecurity.com — Afiliado Qilin, exploração a partir de 07/05.

Oracle PeopleSoft: RCE zero-day, ShinyHunters em 100+ organizações

CVE-2026-35273 (CVSS 9,8) afeta o componente PSEMHUB do PeopleSoft — facilmente explorável, sem autenticação, acesso HTTP puro é suficiente para comprometimento completo. O grupo ShinyHunters explorou isto como zero-day de 27/05 até o patch em 10/06 e reivindica 300 instâncias em 100+ organizações, das quais 68% são no ensino superior.[11]

Para Daniel, nenhum impacto direto no stack (PeopleSoft é HR/ERP empresarial), mas um padrão instrutivo: uma janela de exploração de semanas antes do patch, visando um setor com equipes de TI enxutas. Qualquer pessoa executando PeopleSoft deveria ter corrigido há muito e revisado logs de maio.[11]

  1. thehackernews.com — ShinyHunters, 300 instâncias, 68% ensino superior.

Patch Tuesday de junho: 200 vulnerabilidades, 3–6 zero-days

O round de junho da Microsoft em 09/06 estabeleceu recordes: 200 vulnerabilidades, das quais 33 são críticas (28 RCE) e 3–6 zero-days. O mais importante é CVE-2026-42897 — um XSS do Exchange Server: um email preparado, quando aberto no OWA, executa JavaScript arbitrário no navegador do destinatário sem o atacante precisar de privilégios. Ativamente explorado.[12][13]

Os zero-days restantes variam de uma escalação de privilégio do Windows CTFMON (exploração GreenPlasma) para um DoS HTTP.sys via "bomba HTTP/2". As fontes discordam sobre quantos foram ativamente explorados versus apenas conhecidos publicamente (BleepingComputer conta 6, Tenable 3–4) — seguimos a lista detalhada da BleepingComputer.[12]

  1. bleepingcomputer.com — 200 vulnerabilidades, 33 críticas, 6 zero-days.
  2. techcommunity.microsoft.com — Exchange XSS CVE-2026-42897.

CRA da UE: Capítulo IV em vigor desde 11/06

Marco de conformidade tranquilo mas relevante: desde 11/06, o Capítulo IV da Lei de Resiliência Cibernética está em vigor — notificação de organismos de avaliação de conformidade. Os estados-membros agora devem designar autoridades de notificação; esta é a infraestrutura através da qual os produtos mais tarde receberão marcação CE de segurança. Próximo marco: obrigações de relatório começando 11/09/2026, imposição total então em 11/12/2027.[14]

  1. digital-strategy.ec.europa.eu — Capítulo IV a partir de 11/06/2026, próximos prazos.

Angular 22 chegou — e está consolidando pesadamente

Angular 22.0 aterrissou em 03/06, patch 22.0.2 em 17/06. É um grande lançamento de consolidação: Signal Forms, Angular Aria e as APIs de reatividade assíncrona (resource/httpResource) agora estão estáveis, OnPush é o novo padrão, há componentes sem seletor e o cliente HTTP agora usa a API Fetch por padrão.[15]

Para o código Angular de Daniel, duas coisas importam: strictTemplates agora está ativado por padrão (espere novos erros de build ao migrar), e a Angular CLI traz suporte MCP estável para ferramentas de IA — o último é mais um tópico ai-dev, apenas uma nota aqui.[16]

  1. angular.dev — 22.0 (03/06), 22.0.2 (17/06), APIs estáveis.
  2. blog.ninja-squad.com — strictTemplates padrão, CLI MCP.

Spring Framework 7.0.8: endurecimento, e 6.2 encerra terça

O lançamento 7.0.8 em 08/06 é mais endurecimento de segurança do que drop de recursos, e exatamente por isso importa: a desserialização Jackson em conversores JMS agora pode ser restrita a pacotes na lista branca via setTrustedPackages(...), e a avaliação SpEL agora é limitada a dez mil operações por padrão.[17]

Lista Branca de Desserialização / Limite SpEL: Desserialização converte dados recebidos em objetos — sem uma lista branca, um atacante pode forçar classes que executam código. SpEL é a mini-linguagem Spring para expressões dinâmicas; um limite impede que uma expressão maliciosa prenda a CPU em um loop infinito. Ambas tornam clássicos RCE/DoS comuns muito mais difíceis.

E a nota de calendário: Spring Framework 6.2.x chega ao fim de vida em 30/06 — próxima terça. Se você ainda está na 6.2, deveria estar planejando o salto para 7.0 agora.[18]

  1. endoflife.date — 7.0.8 (08/06), lista branca JMS + limite SpEL.
  2. endoflife.date — 6.2.x EOL em 30/06/2026.

PostgreSQL: atualização consolidada, PG19 Beta 1, PG14 encerrando

A consolidação de 14/05 (18.4 / 17.10 / 16.14 / 15.18 / 14.23) fecha 11 CVEs e mais de 60 bugs — incluindo três falhas CVSS 8,8: um wraparound de inteiro (CVE-2026-6473), um problema de symlink em pg_basebackup (-6475) e uma sobrescrita de stack libpq (-6477). Se você não está ainda na versão menor mais recente de sua versão, planeje para isto.[19]

Olhando para frente: PG19 Beta 1 está fora (final esperada set/out), e PostgreSQL 14 chega ao fim de vida em 12/11/2026. Para o stack PG de Daniel, isto significa mover instâncias 14 para um caminho de upgrade ao longo do verão.[19][20]

  1. postgresql.org — 11 CVEs, PG14 EOL em 12/11/2026.
  2. releasebot.io — PG19 Beta 1.

Microsoft constrói uma verdadeira distro de servidor: Azure Linux 4.0

Revelado no OSS Summit NA (meados de maio): Azure Linux 4.0 (Preview) — uma distribuição derivada de Fedora, baseada em RPM, que a Microsoft constrói e mantém para VMs Azure. A coisa notável é a "primeira vez": é a primeira distro Linux suportada da Microsoft além do hosting puro de container.[21]

Em paralelo, Azure Container Linux saiu para GA (anunciado em meados de maio no OSS Summit; rollout mais amplo em 02/06 na Build) — um host OS imutável, otimizado para container, baseado no projeto Flatcar adquirido. Juntos, isto é a confiança de Linux da Microsoft como plataforma, não apenas como convidado.[22]

  1. infoq.com — Azure Linux 4.0, derivado de Fedora, primeira distro de servidor.
  2. opensource.microsoft.com — Azure Container Linux GA, Flatcar.
