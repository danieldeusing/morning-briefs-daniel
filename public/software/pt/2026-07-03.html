SharePoint RCE: CISA anula avaliação de risco da Microsoft [1]

A história de hoje é menos sobre o buraco que sobre a contradição por trás dele. A CISA adicionou CVE-2026-45659, uma execução de código remoto no SharePoint on-premises (desserialização insegura), ao seu catálogo KEV após confirmar exploração ativa. Afetadas: SharePoint Server Subscription Edition, 2019 e Enterprise 2016. O ponto: quando a Microsoft corrigiu o bug em maio, rebaixou a exploração no mundo real como "Improvável" — CISA inverteu essa decisão agora.[1]

Catálogo KEV: A lista Known Exploited Vulnerabilities (Vulnerabilidades Conhecidas e Exploradas) da autoridade americana CISA é como um cartaz de procurado para buracos de segurança que atacantes já estão usando no mundo real. Se um bug está nela, não é teórico — alguém o está explorando hoje. Para agências federais dos EUA, corrigir se torna obrigatório com prazo.

O que torna o buraco especialmente desagradável é a baixa barreira: segundo The Register, qualquer conta com permissões de "Site Member" mais credenciais válidas é suficiente para executar código arbitrário — sem privilégios de admin.[1] De acordo com a Diretiva Operacional de Vinculação 26-04, agências devem corrigir até 04/07/2026. Para qualquer um executando SharePoint on-premises, esse é o sinal para finalmente aplicar a correção de maio — não porque estou levantando um alarme, mas porque está sendo explorado na prática.

libssh2: dois buracos pré-autenticação, um PoC, nenhum lançamento [2][3]

libssh2 fica como uma biblioteca cliente SSH em inúmeras ferramentas e ligações de linguagem — em qualquer lugar onde um programa se conecta via SSH sem invocar o cliente OpenSSH diretamente. Dois bugs recém-divulgados afetam todas as versões ≤ 1.11.1, e ambos acionam antes da autenticação.

  • CVE-2026-55200 (CVSS 9,2) — Escrita fora dos limites em transport.c. Um servidor SSH malicioso pode injetar código em um cliente. O detalhe crítico: uma prova de conceito pública já está circulando.[2]
  • CVE-2026-55199 (CVSS 8,2) — Leitura fora dos limites mais loop infinito no manipulador SSH_MSG_EXT_INFO (packet.c), tornando isso uma negação de serviço pré-autenticação via esgotamento de CPU.[3]
Atenção As correções estão mescladas no master (commits 7acf3df / 1762685), mas não há nenhum lançamento oficial marcado ainda. Se você tem libssh2 vendido ou o puxa via pacotes de distribuição, não há nada acabado para implantar agora — se você se conecta para servidores SSH não confiáveis (runners de CI, scrapers, ferramentas de deploy), mantenha as correções do master em sua mira.

Uma observação sobre fonte por nossa conta: o 9,2 para CVE-2026-55200 vem do banco de dados oficial GitHub Advisory; o 8,2 para o bug DoS está atualmente apenas com fonte secundária (snippet Hacker News, verificado contra a entrada irmã) — então leia esse número com ressalva até a entrada CVE oficial se atualizar.

npm worm "Miasma": auto-propagação via binding.gyp [4]

O incidente mais interessante da semana em engenharia de software é um worm de cadeia de suprimentos na linha Shai-Hulud. Ele infectou 57 pacotes npm em centenas de versões maliciosas — e diferentemente dos suspeitos usuais, não se espalha via scripts de ciclo de vida mas através do arquivo binding.gyp que node-gyp processa ao compilar módulos nativos.[4]

binding.gyp / node-gyp: Alguns pacotes npm contêm código C/C++ nativo que deve ser compilado durante a instalação. node-gyp é a ferramenta de construção, e binding.gyp é sua receita. Imagine pedir móveis e as instruções de montagem por si mesmas executarem um comando quando você abre a caixa — é onde o worm se conecta.

O mecanismo é classicamente vicioso: a expansão de comando do node-gyp executa código durante `npm install`, o worm lista todos os pacotes pelo mantenedor via registry.npmjs.org, baixa cada alvo, injeta seu próprio binding.gyp mais index.js, e republica. Maiores vítimas por alcance: @vapi-ai/server-sdk (~86.500 downloads/semana) e ai-sdk-ollama (~36.900/semana).[4]

Contexto Que pacotes de SDK de IA se classifiquem entre os maiores alvo não é acaso — a camada de agente e ferramental ao redor de IA é uma superfície de ataque em rápido crescimento, frequentemente recém-mantida. Mas o problema raiz permanece software — execução de código em tempo de instalação — não o modelo por trás dele. Endurecimento prático: `--ignore-scripts` não funciona aqui porque o vetor é expansão de comando node-gyp, não um hook de ciclo de vida. Fixe lockfiles, inspecione novos bumps maiores em pacotes nativos.

Spray de senha M365: 81 milhões de tentativas, MFA contornada [5]

Huntress documentou uma campanha agressiva de spray de senha contra ambientes Microsoft 365: mais de 81 milhões de tentativas de login entre 12 e 26 de junho de 2026 (pico em 22), com 78 contas comprometidas em 64 organizações no final.[5]

Spray de senha: Em vez de bombardear uma conta com muitas senhas (o que dispara bloqueios), o atacante tenta uma senha comum contra muitas contas — como um ladrão testando uma chave mestra em cada porta da rua em vez de tentar todas as chaves em uma porta.

A dor real está no mecanismo: atacantes usaram o fluxo OAuth ROPC, que envia a senha diretamente para o endpoint /token — sem um prompt MFA interativo. Mesmo se Acesso Condicional exige MFA, o segundo fator não ativa neste fluxo. Contramedidas: bloqueie ROPC completamente via política Conditional Access, desabilite autenticação herdada.[5]

BR Tech: nenhum item não-IA hoje

Bandeira de honestidade: a cena de tecnologia brasileira não tinha história de software/engenharia não-IA no escopo deste brief para hoje — os hits (política de trabalho híbrido Nubank, um investimento de R$ 45 bilhões em 2026) são tópicos de consumidor/RH sem substância de dev. Sem motivo para preencher; amanhã abre com olhos frescos.

É sexta-feira, dia três de julho. Não é um dia de grandes lançamentos, mas um de cadeias de suprimentos e prazos de correção — e um em que a Microsoft recebe uma lição em avaliação de risco.

A história principal: autoridades americanas adicionaram um buraco do SharePoint à sua lista de procurados que a Microsoft rebaixou para improvável em maio. Agora está em ataque ativo. A pontuação é oito ponto oito, e segundo The Register, qualquer conta de membro regular é suficiente para executar código. Agências federais devem corrigir até amanhã. Se você executa SharePoint on-premises, agora é a hora de aplicar essa correção de maio.

Quase tão urgente é libssh2, a pequena biblioteca SSH que vive em inúmeras ferramentas. Dois buracos recentes afetam todas as versões até um ponto um ponto um, ambos antes do login. O crítico, pontuando nove ponto dois, permite que um servidor malicioso injete código em um cliente — e uma prova pública de conceito já está circulando. As correções estão no master, mas sem lançamento oficial ainda.

Da cadeia de suprimentos: um worm chamado Miasma. Infectou cinquenta e sete pacotes npm e se replica a si mesmo — não através dos scripts de ciclo de vida usuais, mas através do arquivo binding-ponto-gyp que é executado ao compilar módulos nativos. Como instruções de montagem que executam um comando ao abrir a caixa. Snyk diz que o mais afetado foi o principal SDK de IA, com mais de oitenta e seis mil downloads por semana.

Depois um spray de senha contra Microsoft 365: mais de oitenta e um milhões de tentativas de login em duas semanas, setenta e oito contas assumidas. Segundo Huntress, os atacantes contornaram verificação de dois fatores através de um fluxo OAuth antigo que envia senhas direto para o endpoint de token, sem prompt interativo.

Depois duas peças mais quietas. Chrome um-cinquenta corrigiu trezentos e oitenta e dois problemas de segurança, quinze deles críticos, nenhum explorado até agora. E curl está fechando sua porta de recompensa de bug para o mês todo porque relatórios chegam quatro a cinco vezes mais rápido que vinte e vinte e quatro — release oito ponto vinte e dois ponto zero está deslizando para segundo de setembro.

Amanhã o prazo de SharePoint expira — primeiro olhar é se libssh2 finalmente consegue um lançamento marcado e se ataques a SharePoint se disseminam mais. Até então.
