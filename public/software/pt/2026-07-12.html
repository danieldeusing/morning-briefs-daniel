Dois pacotes npm sequestrados — a cadeia de suprimentos continua na linha de frente

Um domingo tranquilo, mas não vazio: o fio condutor da semana é a cadeia de suprimentos de software, e ela está sob ataque ativo. Em poucos dias, dois pacotes npm foram sequestrados — ontem o jscrambler, com um infostealer em Rust, e antes disso o SDK da Injective, com um backdoor que roubava chaves de carteira. Essa é exatamente a classe de ataque que os novos padrões do npm v12 tentam barrar — motivo para perguntar com honestidade o que esse endurecimento realmente impede e o que apenas desloca. Além disso: uma brecha silenciosa de downgrade no driver JDBC do Postgres, bem dentro da stack do Daniel, e o CodeQL ganha uma query contra prompt injection.

  • Sec pacote npm jscrambler sequestrado (11/07): cinco releases maliciosos em ~3 h, infostealer em Rust para perfis de navegador, extensão do Bitwarden e Steam — na versão 8.18.0 o dropper migrou do install hook para dist/index.js.
  • Sec SDK da Injective com backdoor (08/07): @injectivelabs/sdk-ts v1.20.21 roubava seed phrases e chaves privadas — 18 pacotes, ~50.000 downloads/semana, <1 h no ar, sem danos registrados. Correção na 1.20.23.
  • Infra npm v12 — avaliação: teria bloqueado o primeiro hook do jscrambler, mas não o dropper em dist/ nem o backdoor da Injective no código do SDK. Necessário, mas não suficiente.
  • Sec PostgreSQL JDBC 42.7.12: corrige o CVE-2026-54291 — channelBinding=require era silenciosamente rebaixado de SCRAM-PLUS para SCRAM simples. Afeta as versões 42.7.4–42.7.11.
  • OSS GitHub CodeQL 2.26.0 (10/07): Kotlin até 2.4.0 — mais uma nova query, js/system-prompt-injection, que detecta entradas não confiáveis chegando a prompts de sistema de IA.
  • Infra Microsoft Foundry hosted agents (GA): sandbox isolado por hypervisor a cada sessão, agnóstico de framework, scale-to-zero — a história é a isolação, não os agentes.
  • Sec CVE do Langflow, em contexto: o NVD avalia em 8,4 (alto), o fabricante em 9,9 (crítico) — e mesmo assim explorada ativamente desde 25/06. KEV supera o score CVSS.
  • Release Notas rápidas: Angular 22.0.5, Firefox 152.0.5 (153 em 21/07), SQLite 3.53.3 — e o CPU trimestral do OpenJDK em 21/07.

Pacote npm jscrambler sequestrado: cinco releases maliciosos em três horas

O golpe mais recente aconteceu ontem, sábado, 11 de julho: o pacote npm jscrambler foi comprometido. Segundo a imprensa de segurança (The Hacker News, com base em dados da Socket), a versão publicada em 11/07, 8.14.0, trazia um hook preinstall que baixa e executa um binário nativo — um build para cada sistema, Windows, macOS e Linux. Em cerca de três horas, o atacante publicou cinco releases maliciosos (8.14.0/8.16.0/8.17.0/8.18.0/8.20.0). A carga: um infostealer escrito em Rust que rouba perfis do Chrome/Brave/Edge/Chromium, a extensão de navegador do Bitwarden e sessões do Steam.

Infostealer: Um programa cuja única função é copiar silenciosamente senhas salvas, cookies do navegador, carteiras de cripto e tokens de sessão, e enviá-los ao atacante. Como um ladrão que não arromba nada — apenas fotografa cada chave e senha que encontra pelo caminho e vai embora.

O detalhe instrutivo: Na versão 8.18.0, o atacante moveu o dropper de dentro do hook preinstall diretamente para os arquivos entregues dist/index.js e dist/bin/jscrambler.js — de propósito, para enganar scanners que só verificam scripts de instalação. Com isso, o malware deixa de rodar no momento da instalação e passa a rodar na importação ou execução do pacote. Esse mesmo truque será importante daqui a pouco, quando avaliarmos o npm v12. (A lista de releases e a mudança do dropper vêm de resumos da imprensa de segurança — as páginas de detalhes da Socket e do THN bloquearam o acesso direto; portanto, devem ser lidas como informação secundária, e não telemetria primária.)

Fonte: thehackernews.com

SDK da Injective com backdoor: chaves de carteira de 18 pacotes

Poucos dias antes, na quarta-feira, 8 de julho, a mesma classe de ataque, com um alvo mais robusto. O @injectivelabs/sdk-ts — o SDK em TypeScript para a blockchain Injective — recebeu um backdoor por meio de uma conta de desenvolvedor do GitHub comprometida. A versão 1.20.21 capturava silenciosamente frases de recuperação de carteira (seed phrases) e chaves privadas. O alcance, segundo BleepingComputer e StepSecurity: 18 pacotes afetados, cerca de 50.000 downloads por semana, menos de uma hora no ar, nenhum dano registrado. Corrigido na 1.20.23 — quem usa o SDK da Injective deve atualizar.

Seed phrase: As 12 a 24 palavras que servem como chave-mestra de uma carteira de cripto — quem as tem, é dono da carteira, sem precisar de senha. Como a chave-mestra de um cofre que por acaso cabe em um cartão postal: uma vez roubada, tudo dentro fica exposto para sempre.

O contraste com o jscrambler é o ponto central: dois pontos de entrada diferentes, o mesmo resultado. O jscrambler veio por uma conta de publicação sequestrada, com carga em script de instalação e código; a Injective, por uma conta do GitHub sequestrada que fez commit do backdoor diretamente no código-fonte do SDK. Ambos contornam a máxima simples de "só não rode scripts de instalação" — o código malicioso está no próprio código, que de qualquer forma vai ser executado.

Fontes: thehackernews.com · bleepingcomputer.com

O que o npm v12 barra — e o que não barra

O npm v12 ativa três padrões seguros neste mês — sem scripts preinstall/install/postinstall vindos de dependências, sem dependências de git e sem dependências de URL remota sem liberação explícita (changelog do GitHub). A mecânica já foi detalhada no brief de 10/07; a pergunta honesta e nova de hoje é: isso teria barrado os dois casos deste fim de semana?

Install hook → npm v12 bloqueia · Código em dist/ → continua rodando

Parcialmente. Os primeiros releases do jscrambler usavam um hook preinstall — que o npm v12 bloquearia por padrão. Mas, na 8.18.0, o dropper migrou para dist/index.js, ou seja, código executado na importação do pacote — ali o npm v12 não faz nada. E o backdoor da Injective já estava no código-fonte do SDK desde o início, executado em tempo de execução, como previsto — também intocado pela restrição de scripts de instalação.

Conclusão, com posição: o npm v12 eleva o piso e elimina o ataque mais barato, o drive-by via script de instalação. Mas a carga simplesmente se muda para código de importação e de execução. Necessário, mas não suficiente. As verdadeiras linhas de defesa continuam as mesmas: fixar versões, lockfile mais hashes de integridade, verificar procedência/attestation — e não atualizar cegamente só porque existe uma versão nova.

Fonte: github.blog

PostgreSQL JDBC 42.7.12: downgrade silencioso de channel binding (CVE-2026-54291)

Bem dentro da stack do Daniel (Java/Spring com Postgres): o release de segurança 42.7.12 do driver JDBC do PostgreSQL (publicado em 29 de junho, noticiado nas notícias do postgresql.org em 6 de julho) corrige o CVE-2026-54291. O cerne: conexões com a configuração channelBinding=require podiam ser silenciosamente rebaixadas de SCRAM-SHA-256-PLUS (com channel binding) para o SCRAM-SHA-256 simples (sem) — anulando exatamente a proteção que havia sido explicitamente solicitada. As versões afetadas vão de 42.7.4 a 42.7.11; a severidade é classificada como média.

Channel binding: Quando seu app se autentica no Postgres via TLS, o channel binding amarra o handshake de senha exatamente a essa conexão criptografada — um atacante que se posiciona no meio e repassa seu login para outro servidor não consegue reutilizá-lo. Como uma assinatura que só vale para este envelope lacrado específico, não para uma cópia. Um "downgrade silencioso" significa que o cliente acredita ter recebido a proteção — mas nunca a teve.

Quem configurou channelBinding=require contra MITM deve migrar para a 42.7.12; do contrário, o endurecimento pode ter funcionado à toa. Do ecossistema Postgres, esta semana também saíram o CloudNativePG 1.30.0 (novo recurso DatabaseRole), o pg_ivm 1.15 e o Barman 3.19.0/3.19.1 — não é correção de segurança, mas vale a pena olhar para quem roda Postgres em Kubernetes.

Fontes: jdbc.postgresql.org · postgresql.org

GitHub CodeQL 2.26.0: Kotlin 2.4.0 — e uma query contra prompt injection

O CodeQL é o motor de análise estática de código do GitHub — a máquina por trás do Code Scanning. A versão 2.26.0 (10 de julho) agora suporta Kotlin até 2.4.0 e traz — a parte interessante para uma ferramenta de segurança — uma nova query, js/system-prompt-injection. Ela detecta quando entradas de usuário não confiáveis entram no system prompt de um modelo de IA, e adiciona novos sinks de prompt injection para os SDKs da OpenAI, Anthropic e Google GenAI. Isso pertence claramente ao brief de software: é um recurso de scanner, não o lançamento de um modelo.

Sink de prompt injection: Se seu app coloca entradas de usuário nas instruções que dá a um modelo de IA, um usuário mal-intencionado pode contrabandear comandos que sequestram o modelo ("ignore o que veio antes, faça X"). Um "sink" do CodeQL é um ponto final conhecidamente perigoso que o scanner observa; a nova query rastreia se dados não confiáveis chegam até o sink de system prompt — a mesma ideia de rastreamento de taint usada em SQL injection, aplicada a chamadas de LLM.

A leitura: quanto mais aplicações conectam LLMs ao backend, mais o prompt injection se torna uma classe de AppSec como XSS ou SQLi — e agora as ferramentas de SAST mainstream começam a modelá-la. Para quem já usa code scanning, isso chega de graça junto com a atualização do CodeQL.

Fonte: github.blog

Microsoft Foundry: sandboxes isolados por hypervisor para agentes (GA)

Como nota de infraestrutura, bem delimitada: os "hosted agents" do Microsoft Foundry chegaram à disponibilidade geral em julho (byteiota.com). Cada sessão recebe seu próprio sandbox isolado por hypervisor — computação, memória e sistema de arquivos dedicados — e isso independentemente do framework: LangGraph, Claude Agent SDK, OpenAI Agents SDK ou um Dockerfile próprio, todos rodam. Preço: US$ 0,0994 por hora de vCPU mais US$ 0,0118 por hora de GiB, com scale-to-zero em momentos ociosos.

O ângulo relevante para software é propositalmente não "qual framework de agentes" (isso é território de ai-dev/ai-usecases — os detalhes estão lá), mas o modelo de isolamento: quem executa código não confiável, gerado por LLM, precisa de uma fronteira de hypervisor por sessão — o mesmo problema que gVisor ou Firecracker resolvem para computação multi-tenant. O fato de as grandes nuvens agora entregarem isso como padrão gerenciado é a verdadeira história de infra.

Hoje é domingo, dia doze de julho. Um dia de notícias tranquilo — mas não vazio. O fio condutor desta semana é a cadeia de suprimentos de software, e ela está sob ataque ativo. Em poucos dias, dois pacotes npm foram sequestrados.

O golpe mais recente aconteceu ontem. O pacote npm jscrambler foi comprometido. A versão maliciosa, oito ponto quatorze zero, trazia um hook de instalação que baixava e executava um arquivo nativo malicioso, um build para Windows, macOS e Linux. Em cerca de três horas, o atacante publicou cinco versões maliciosas. A carga era um infostealer escrito em Rust — um programa cujo único objetivo é copiar silenciosamente senhas salvas, cookies de navegador e tokens de sessão, e enviá-los para fora. Ele capturava perfis de navegador, a extensão do Bitwarden e sessões do Steam.

O detalhe instrutivo: na versão oito ponto dezoito zero, o atacante moveu o código malicioso do hook de instalação diretamente para os arquivos entregues do pacote. Com isso, o malware deixa de rodar na instalação e passa a rodar na importação — de propósito, para enganar scanners que só olham para scripts de instalação. Isso vai importar em seguida. Essas informações vêm da imprensa de segurança, e não de telemetria primária, porque as páginas de detalhes bloquearam o acesso direto.

Poucos dias antes, em oito de julho, a mesma classe de ataque, com um alvo mais robusto. O SDK em TypeScript para a blockchain Injective recebeu um backdoor por meio de uma conta de desenvolvedor do GitHub sequestrada. A versão um vinte vinte e um capturava silenciosamente as seed phrases e chaves privadas de carteiras de cripto. A seed phrase é a chave-mestra de uma carteira — quem a tem, é dono de tudo o que está dentro dela, sem precisar de senha. Foram afetados dezoito pacotes, com cerca de cinquenta mil downloads por semana; ficou no ar por menos de uma hora, e não houve dano. A correção é a versão um vinte vinte e três.

Os dois casos contornam a regra simples de simplesmente não executar scripts de instalação, porque o código malicioso está no próprio código, que de qualquer forma vai rodar. Isso leva à pergunta honesta sobre o npm versão doze, que neste mês pretende conter exatamente esse tipo de ataque, bloqueando por padrão scripts de instalação e dependências de git e de fontes remotas. A mecânica já rodou no brief de dez de julho; hoje é a hora da avaliação: será que isso teria barrado os dois casos?

Só parcialmente. O primeiro hook do jscrambler, o npm doze teria bloqueado. Mas o dropper posterior, nos arquivos entregues, e o backdoor da Injective, no código-fonte, ambos rodam em tempo de execução — e ali a restrição de scripts de instalação não muda nada. Minha conclusão: o npm doze eleva o piso e elimina o ataque mais barato, mas a carga simplesmente se muda para código de importação e de execução. Necessário, mas não suficiente. As linhas de defesa de verdade continuam sendo: fixar versões, lockfile e hashes de integridade, verificar procedência — e não atualizar cegamente.

Da própria stack do Daniel vem uma brecha silenciosa. O release de segurança quarenta e dois ponto sete ponto doze do driver JDBC do PostgreSQL corrige uma vulnerabilidade em que uma autenticação vinculada à conexão, explicitamente solicitada, era silenciosamente rebaixada para a variante mais fraca. Quem configurou essa proteção contra ataques man-in-the-middle deve atualizar, ou o endurecimento pode ter funcionado à toa. E o scanner de código do GitHub, o CodeQL, na versão dois ponto vinte e seis, ganha suporte a Kotlin até a versão dois ponto quatro — além de uma nova regra que detecta quando entradas de usuário chegam ao system prompt de um modelo de IA. O prompt injection está aos poucos se tornando uma classe de segurança como o SQL injection, e as ferramentas padrão do mercado começam a modelá-la.

Rapidinho: o Microsoft Foundry agora está liberando seus agentes hospedados para disponibilidade geral, cada sessão em seu próprio sandbox isolado por hypervisor — o lado interessante não é o framework, mas o isolamento, quando se está rodando código gerado por IA de terceiros. E, na hora de avaliar vulnerabilidades, vale olhar para o Langflow: o NVD dá oito vírgula quatro, o fabricante dá nove vírgula nove — e mesmo assim vem sendo explorada desde o fim de junho. Entrar no catálogo de vulnerabilidades ativamente exploradas diz mais do que o score puro.

Amanhã é segunda-feira, e com ela vence o prazo de patch para as duas falhas do Joomla de sexta-feira. Na próxima semana, vou verificar se aparecem mais pacotes afetados pelos dois incidentes do npm, e se o npm versão doze de fato está sendo adotado de forma ampla. Até lá.
